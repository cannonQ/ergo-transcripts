# Ergo Developer Knowledge Base — Monthly Summary
**Chat:** Ergo Developer | **Month:** 2024-07

---

## Executive Summary

July 2024 was dominated by three converging themes: finalising Sigma 6.0 (BigInt, bulletproofs, numeric operations) and preparing its DevNet testnet, advancing scalability research (sub-blocks, SigmaChains, merge-mined sidechains), and a broad maturation of ecosystem tooling (Fleet SDK headless wallet, ErgoScript debugging, Paideia DAO launch readiness, Rosen Bridge decimal handling). Multiple protocol-level decisions were made or narrowed, setting a clear post-6.0 R&D roadmap through the end of the year.

---

## Key Topics This Month

### Sigma 6.0 Development & Roadmap

Sigma 6.0 reached approximately 95% completion by end of month. Key PRs merged or under review included unsigned BigInt support (#997), bulletproofs range proof verification, bitwise/shift numeric operations (#1017), and remaining issues #847, #905, and #1020. kushti outlined the post-6.0 sequence: polish and merge all PRs → assemble 6.0 candidate → DevNet activation testing → refactor reference client for sidechains → implement faster confirmations. A direct/alternative ErgoScript compiler (PR #1022) was implemented by ergomorphic, passing all LanguageSpecification tests, with a debug function and `val` logging planned in a follow-up PR. The `executeFromVar` feature (provide ErgoTree in context variable, evaluate to SigmaProp) has existed in ErgoTree since early testnets but lacks ErgoScript compiler support; kushti flagged it as polishable for inclusion in 5.0.15.

### Scalability: Sub-blocks, SigmaChains & Merge-Mined Sidechains

kushti reworked the sub-blocks scalability proposal as a soft fork requiring only miner node updates, providing better transaction inclusion guarantees while allowing non-updated protocol clients to continue operating. Separately, a multi-phase SigmaChain development plan was published: (1) Sigma 6.0 release, (2) flexible blockchain context, (3) Ergo node modifications for merge-mined sidechains (similar to BIP 301, with emission contracts rewarding mainchain miners for delivering sidechain data), (4) first SigmaChain implementation. The key open decision is merge-mined sidechain versus standalone blockchain, pending partner identification. Bitcoin-NG and PRISM proposals were evaluated as design references for the sub-blocks design space.

### ErgoScript Developer Experience & Debugging

A significant multi-week discussion surfaced a cluster of developer tooling gaps: no LSP, no debugger, no EBNF specification, and no way to inspect intermediate variable values during contract execution. ergomorphic proposed and implemented an alternative "direct" compilation path without optimisations to enable `debug()` operations in ErgoTree, as optimisations (CSE) destroy the `ValDef` node structure needed for variable name mapping. The consensus on alternative-language compilers (e.g., Rust) was that they are problematic because producing identical ErgoTree output for the same ErgoScript program across compilers is extremely difficult and creates on-chain tracking risks; LSP support for the existing compiler was identified as the higher-priority tooling investment.

### Fleet SDK & Headless JavaScript Wallet

C8 raised a critical ecosystem gap: existing JS wallets (Nautilus, Fleet) target browsers only, leaving mobile apps and games without a common headless wallet library. arobsn confirmed that `@fleet-sdk/wallet` already provides wallet generation, key derivation, and blind `proveDLog` signing, and announced plans to add Sigma.JS and sigma-rust prover wrappers for multi-sig and Diffie-Hellman tuple support. Reduced TX signing in Fleet remains unresolved due to bundle-size concerns. A UTXO invalidation bug in Fleet's mock-chain (party balance miscalculation) was fixed in v0.5.1; separately, c8 identified that the mock-chain does not enforce storage rent (360 nanoerg/byte) or miner fees by default, making transaction-chaining tests impractical — arobsn added configurable box selection via `ensureInclusion()`.

### Node API & Transaction Ordering

A persistent issue with `/blockchain/transactions/byAddress` returning transactions in ascending rather than descending (newest-first) order was traced to a database slicing change. The fix was confirmed merged and deployed to the 213 server by end of month, with 5.0.22 (including PR #2167 and issue #2093 support) and 5.1.0 targeted for release the following week. HTTP service thread pool contention (indexer and API sharing the same pool, causing service failures on indexed nodes) was also fixed and merged into the 5.0.22 candidate.

### Rosen Bridge

Rosen Bridge saw substantial refactoring: EVM asset decimal handling (dropping insignificant decimal places for assets with up to 18 decimal places), eRSN-to-RSN emission contract implementation, eRSN support in the watcher, CORS enablement, xVerse wallet integration for Bitcoin hardware wallet PSBT signing (after xDefi showed limitations), health-check notification support under review, and raffle v2 integration testing. The guard service was updated to support configurable emission tokens (not just RSN).

### Paideia DAO

luivatra completed near-real-time backend updates using ZMQ pub/sub, deployed RC1 SDK, modified endpoints to interact directly with singletons (reducing UTXO contention), added mempool support with rollback capability for forks, and implemented a 10-block safety window treating recent blocks as unconfirmed. Contract size was reduced from 4.5kb to <2kb by storing only a hash of concatenated config key hashes rather than individual `blake2b256` constants. A bug bounty for 1.0.0 contracts launched mid-month; mainnet launch projected within weeks pending testing.

### Ecosystem Projects (Crystal Pool, duckpools, ChainCash, Sigmanauts Mining Pool)

**Crystal Pool**: Deposit/withdraw UI completed; mobile wallet deposit proxy contract ~30% done; revertible transaction tree execution on mempool deposits in progress; limit order contract tests created with Fleet SDK mock-chain.  
**duckpools**: rsBTC and SPF lending pools added; v2 preliminary testing complete with continuous yield delivery; duckDAO vote launched for RSN and rsADA as collateral options.  
**ChainCash**: Spend-note transaction builder and initial storage implementation merged by dusek; blockchain scanner in progress; CI/CD issue attached with bounty (#46).  
**Sigmanauts Mining Pool DAO**: Officially created with ~12–13 token distribution; tx token swap code ready for MiningCore integration; miner ID minting vote pending (~3 ERG consensus); beta testing imminent; "fan clubs" and "degen mode" features in development.

## Technical Q&A Worth Preserving

- **Q** (@Flying Pig, 2024-07-03): Why does `/blockchain/transactions/byAddress` return transactions in ascending order instead of descending (newest first)?
  **A** (@kushti): Transactions stored in ascending order by insertion/height; a recent database slicing change requires fetching from the end for descending order. Fix confirmed merged and deployed to 213.

- **Q** (@deatgripson, 2024-07-03): Does compiled ErgoTree differ from on-chain version?
  **A** (@kushti): Could be ErgoTree version difference (first byte). If only the first byte differs it is a version issue; if multiple bytes differ, it may be an AppKit/ErgoPy encoding problem.

- **Q** (@mgpai, 2024-07-15): Why does Ergo use AVL trees instead of Merkle Patricia Trees?
  **A** (@kushti): MPT design goals are unclear/undefined; unknown whether MPT qualifies as a 2-party Authenticated Data Structure. Ethereum's MPT implementation loses ~40–50% space in proofs due to encoding; even corrected, MPT is ~40% worse than AVL tree structure. See https://eprint.iacr.org/2016/994.pdf.

- **Q** (@C8, 2024-07-10): What are the chances of a headless JavaScript wallet?
  **A** (@arobsn): `@fleet-sdk/wallet` does this now — generates wallets, derives keys, signs transactions with blind `proveDLog`. Sigma.JS and sigma-rust wrappers for multi-sig and Diffie-Hellman tuples are planned.

- **Q** (@c8, 2024-07-18): Can intermediate ErgoScript variable values be exposed during contract execution for unit testing/debugging?
  **A** (@ergomorphic): Simplest approach is to make the interpreter log values as they are calculated. ErgoTree has `ValDef(id, expr)` nodes; names are lost during compilation but can be saved. The constraint is that optimisations (CSE) may alter `ValDef` structure — a direct compilation path without optimisations is required for stable name-to-value mapping. This is now implemented in PR #1022.

- **Q** (@riceandbeans, 2024-07-26): Where are docs for developing Ergo smart contracts in Rust?
  **A** (@mgpai, @luivatra, @kushti): Develop contracts in ErgoScript (see LangSpec), then write off-chain code in Rust using sigma-rust. You cannot compile ErgoScript to ErgoTree in Rust directly; use JS (Fleet/sigmastate-js) or Scala for compilation. Oracle pools are the canonical example of ErgoScript contracts with Rust off-chain code. Understanding the eUTXO transactional model is the critical prerequisite.

- **Q** (@mgpai, 2024-07-26): Would alternative-language ErgoScript compilers (e.g., Rust) be a good tooling improvement?
  **A** (@kushti): Problematic — producing identical ErgoTree output for the same ErgoScript program across compilers is extremely difficult and creates on-chain tracking risks. LSP support for the existing compiler is the higher-priority investment.

- **Q** (@kvarckvarc, 2024-07-22): How does balance work if input balance ≠ output balance? Should the difference equal the fee?
  **A** (@pulsarz, @kushti): Fees are explicit in Ergo — not implicit in a balance difference. The protocol does not know about fees; they are a sender–miner agreement. You must construct a separate fee output box. Sum of all inputs must always equal sum of all outputs (ERG-wise).

- **Q** (@c8, 2024-07-24): Is there a way to set additional registers for the change box in Fleet SDK's `sendChangeTo`?
  **A** (@arobsn): Currently not possible because a single transaction can produce multiple change boxes depending on token count in inputs, and there is no token order guarantee (unsafe for contracts). The recommended pattern is a Fleet plugin per contract to handle change boxes individually (see Babel Fees plugin as example).

- **Q** (@c8, 2024-07-17): How many input boxes can safely be used in a limit order fill transaction?
  **A** (@kushti, @mgpai): As long as the transaction is under the ~96kb propagation limit (protecting the p2p network), it should be fine. Block size is slightly over 1MB (initially 512kb, raised by miners several times).

- **Q** (@c8, 2024-07-18): Does a transaction require a miner fee to be accepted into the mempool?
  **A** (@kushti): A transaction without a fee will not be accepted by other peers. The mempool will accept it via the node API (default minimum 0.001 ERG). Node operators can raise the minimum fee propagation rate to prevent low-fee transactions from even entering their mempool.

- **Q** (@danshkr, 2024-07-26): How to configure an Ergo node to connect only to specified peer IPs?
  **A** (@aberg): Use `scorex { network { knownPeers = ["host:port"] } }` in config — note this only applies on first start; delete the peer database if the node has already run. (@kushti): Can also use the `/peers/connect` API with a POST request and API key.

- **Q** (@c8, 2024-07-18): Is `SString` usable to input text into registers?
  **A** (@kushti): `SString` is a virtual compiler-only type; no value of that type can be stored in registers. Use `SColl(SByte, stringToBytes("utf8", text))`. Fleet SDK should provide an `SString()` convenience wrapper for this pattern.

- **Q** (@Unknown, 2024-07-17): Will `executeFromVar` in 6.0 optimise "companion contracts" / nested DAOs?
  **A** (@kushti): `executeFromVar` has been supported in ErgoTree since early testnets but is missing ErgoScript compiler support; it can be polished for inclusion in 5.0.15 if needed. (@luivatra): The treasury can send to a p2s address with filled registers, enabling a proxy contract pattern as a current workaround.

1. **6.0 DevNet timeline**: Sigma 6.0 PRs polished but DevNet activation testing not yet started; exact launch window TBD pending final issue completions (#847, #905).
2. **Sub-blocks vs. faster confirmation protocols**: Design space still open — Bitcoin-NG, PRISM, and Flux proposals evaluated but no final selection made; testnet parameter-tweaking group needs a leader.
3. **SigmaChain merge-mined vs. standalone**: Key architectural decision deferred pending identification of a launch partner.
4. **Reduced TX signing in Fleet SDK**: Bundle-size concerns unresolved; arobsn has not finalised idiomatic approach.
5. **`AlicePK && BobPK` multi-sig signing in Fleet**: C8's PR #1012 submitted but flagged as potentially containing errors; awaiting review from arobsn or ergomorphic.
6. **ErgoScript LSP**: Identified as the highest-priority developer tooling gap; no concrete implementation plan or assignee yet.
7. **AVL trees in Go**: Ergo's AVL+ trees differ from Cosmos `iavl`; feasibility of Rust-via-C-bindings exposure to Go unconfirmed; mgpai still evaluating for ErgoNames off-chain.
8. **SigmaUSD oracle migration to v2 with buyback fee**: Feasibility raised but no response or analysis recorded.
9. **Trade House contract open-sourcing**: Timeline not discussed; luivatra indicated contracts are not yet public.
10. **Contract parameter limits (5+)**: luivatra reported script reduces to `false` with 5 parameters but works with 4 + hardcoded value; root cause not confirmed.
11. **Fee estimation API adoption**: Node has a fee estimation API but it is untested and unused by any wallet or dApp; flagged as needing community pickup.
12. **ErgoNames permanent registration sustainability**: Design trade-offs debated but no resolution; long-term name market utility concern unresolved.
