# Ergo Developer Chat — Monthly Summary
**Month:** 2023-06 | **Chat:** Ergo Developer

---

## Executive Summary

June 2023 was a productive month for Ergo developer tooling, with significant progress across the Explorer, wallet infrastructure, and DeFi contracts — culminating in the discovery and community-led recovery of the HodlERG contract exploit in early July. Core protocol development continued steadily with two reference client releases (v5.0.11 and v5.0.12), while broader ecosystem discussions covered Oracle Pool infrastructure, order book DEX design, block speed improvements, and spam mitigation strategies.

---

## Key Topics This Month

### Ergo Protocol Reference Client Releases
Two maintenance releases shipped during the month. v5.0.11 was released on 2023-06-01 and v5.0.12 on 2023-06-14, representing steady cadence on core node development. A technical debt issue was also identified: `ergo_tree_template_hash` indexing prevents clean filtering of p2pk scripts without forcing user migration to the `ergo_tree_template` field — a transition path not yet decided.

### Ergo Explorer Development
Aco Šmrkas shipped continuous improvements throughout the month: token list support, block/token detail views, search by token ID and block ID, NFT previews, case-insensitive search with type filtering, smart contract transaction summaries, token burn status tracking, order filtering, a light CSS theme, testnet support, and an address book. The community-funded raffle for Explorer development was fully funded. A server upgrade to a 1GB machine accompanied these releases. IPFS as an alternative to Cloudflare-IPFS was evaluated but ruled out due to VPS constraints (512MB insufficient for pinning without third-party hosting).

### Wallet Infrastructure: Satergo & Yoroi Recovery
Aberg released a Satergo ErgoPay beta (branch `ergopay`, requiring Java 20), enabling dApp/DeFi interaction with Spectrum, Ergopad, and others — though ergopay link handling currently requires manual copy-paste. Separately, Aberg published the `yoroi-ergo-wallet-recover` tool to help users whose Yoroi wallets had completely broken for Ergo, allowing seed decryption via spending password and fund transfer to a new wallet.

### HodlERG Contract Exploit & Recovery
The dominant event of the period was a bug in the HodlERG contract that allowed an attacker to mint unlimited tokens and drain approximately 950 ERG from a Spectrum DEX liquidity pool. kushti committed to refund original holders via snapshot-based distribution, with manual compensation for LP providers. Community discussion focused on preventing double-spend in refunds, handling tokens received by the attacker, and ensuring fair allocation despite manipulation. An additional 75 ERG from the attacker remained locked in an unidentified smart contract whose ErgoTree could not be reverse-engineered. kushti also committed to writing comprehensive test coverage for HodlERG and MewFinance contracts.

### Address Data Infrastructure & SQL Tooling
A recurring technical thread concerned how to efficiently query the full set of p2pk addresses (~850k+) without hitting the ergo.watch 10k API cap. Solutions explored included paginating the indexed node API, direct database queries, and SQL filtering. qx() generated a dataset of 850k+ unique addresses and published it as CSV. Flying Pig shared a reference SQL query (`contracts_TVL.sql`) for explorer database exploration. The database schema lacks formal documentation; the team is discovering it through exploration. A persistent auto-updating API endpoint for all p2pk addresses was requested but not yet built.

### DuckPools Contract Testing
DuckPools resolved Ergo v5 compatibility issues and a contract redesign, completing security audit testing and advancing to the private testing phase — beginning with the ERG pool, followed approximately 1.5 weeks later by SigmaUSD and SigRSV pools.

### Oracle Pool Infrastructure & Palmyra Ecosystem
Dan Friedman outlined plans for Chainlink integration within the Winter Protocol for proof-of-reserve and on-chain warehouse receipts, supporting decentralized trade financing. The architecture pairs Chainlink oracles with IoT devices (camera, weight/motion sensors) for continuous proof-of-existence of bonded warehouse commodities. Dan noted this is not zero-sum: the plan includes using capital from the Palmyra token sale to build out native Ergo Oracle Pools, which are more secure but require additional tooling — currently only two Oracle Pool v1 instances exist (one run by the Ergo Foundation). Oracle Pool v2 is running for Dexy Gold with GORT rewards.

### Order Book DEX Design: Trade House & Auction House
qx() gave an extended explanation of order book DEX advantages over AMMs: full transparency of liquidity depth, user control over slippage, no impermanent loss, partial order fills, and peer-to-peer matching. Each order lives on-chain as its own box with user-controlled execution and refund conditions. No single upgrade key controls the system. qx() highlighted the ability to set up layered buy orders at different price levels as particularly useful for accumulation strategies. Discussion touched on potential future sharing of Spectrum DEX liquidity pools with Trade House orders.

### Block Speed & Protocol Parameter Discussions
Community discussion explored faster block times and confirmation speeds. Key concerns raised included contracts relying on block height rather than timestamps being affected by parameter changes, and security implications of worst-case scenarios in faster-block regimes. kushti noted a testnet already exists for parameter tweaking (block time, difficulty) but requires community members to lead participation, run nodes, observe scientifically, and compare results. No formal workstream was established; kushti offered Telegram invites to interested parties.

### Fee Market & Spam Mitigation
Discussion covered ERG fee estimation APIs (exist in the node but not widely used by applications), minimum propagation fee rates as a filter against spam, and mempool management. kushti noted that during past spam events, simply raising the fee slider in Nautilus was sufficient to get transactions confirmed. The fee estimation API is present but untested and not used by dApps — flagged as a gap to address.

### Edge Framework & AppKit Abstraction
kii presented Edge, a Scala library abstracting ErgoAppKit for faster backend development (4x efficiency gain reported on a crowdfund project), along with `ergo-play-boilerplate` as a Scala Play starter template. Both lack documentation and active maintenance; the developer sought contributors.

## Technical Q&A Worth Preserving

- **Q** (@koukarin, W22): How to construct a mobile wallet URI for token transfers?
  **A** (@Aberg): Use `ergo:recipientaddress?amount=X&token-TOKENID=AMOUNT` — the first amount is always ERG; `&description=` is optional.

- **Q** (@HQΣr, W22): Can the API limit for address transactions be increased beyond 500?
  **A** (@Aberg): Use pagination via `?page=X&limit=500` per the API docs.

- **Q** (@HQΣr, W23): How to get all p2pk addresses to query DEX order history without hitting the 10k API cap?
  **A** (@Aberg): Paginate `/blockchain/transaction/range?offset=?&limit=?` to get all TX IDs, then fetch `/blockchain/transaction/byId/{txId}` and extract addresses from each transaction.

- **Q** (@qx(), W23): Is there a ranged UTXO API call available?
  **A** (@Aberg): Confirmed gap — no UTXO range endpoint exists; direct database query recommended.

- **Q** (@Luivatra, W23): What is the maximum contract UTXO size and how should complex contracts handle it?
  **A** (@kushti, W23): Store events in separate UTXOs labelled with NFTs; the main contract enumerates NFT labels. The Dexy bank contract implements this pattern to stay within UTXO size limits.

- **Q** (@HQΣr, W24): Why hasn't EIP-76 been implemented?
  **A** (@MrStahlfelge): Lack of community review and perceived low interest prevented implementation despite available developer time.

- **Q** (@Aco, W24): What do `maxBlockGix` and `maxTxGix` represent in `/api/v1/networkState`?
  **A** (@Luivatra): They are global indices — `maxTxGix` equals the total transaction count; `maxBlockGix` equals the total box count.

- **Q** (@Flying Pig, W26): Can you reverse-engineer a smart contract address/ErgoTree?
  **A** (community, implied): No direct method; opaque contracts are difficult to analyse even when on-chain. The HodlERG attacker's 75 ERG locked in an unknown contract remains unanalysed.

- **Q** (@Luivatra, W26): What is the recommended proxy vs. main contract architecture?
  **A** (@Luivatra, self): "I usually write proxy from user PoV and main contract from protocol PoV. Proxy protects user, main protects protocol."

- **Q** (community, W26): How does the Ergo fee market protect against spam attacks?
  **A** (@kushti): Node operators can raise minimum propagation fee rates to prevent spam from even entering the mempool. Unconfirmed spam transactions eventually time out and are flushed. A fee estimation API exists in the node but is not yet used by applications — flagged as a gap.
