# Ergo Developer Chat — Monthly Summary: 2023-02

## Executive Summary

February 2023 was a technically dense month in the Ergo developer chat, dominated by low-level protocol implementation work (ErgoTree serialization, P2P parsing, AppKit transaction signing) alongside ecosystem infrastructure discussions around node discovery, Spectrum DEX API integration, and wallet standards. The month also saw the launch of ErgoHack VI and the release of node version 5.0.7, alongside early signals of cross-chain interest from external projects engaging with the Ergo ecosystem.

---

## Key Topics This Month

### ErgoTree Serialization and P2P Protocol Parsing
The most sustained technical thread of the month involved Aberg's work on a custom ErgoTransaction modifier parser for the P2P protocol. Key challenges included handling SigmaBoolean structures, ContextExtension parsing, and constant deserialization without traversing the full recursive sigma value tree. Alex Slesarenko directed work toward the ErgoTree PDF specification and the `SigmaBoolean.serializer` implementation in the Sigma codebase. Aberg ultimately identified both a bug in a custom `DataInputStream` extension (causing each byte to be read twice) and minor inaccuracies in the ErgoTree PDF itself—specifically that the header field should use `getByte()` not `UInt`, and an "end for" label appears where "end if" is correct. These PDF inaccuracies were flagged for upstream correction.

### Spectrum DEX API Integration and Token Pricing
Early in the month, discussion clarified best practices for fetching token prices from the Spectrum DEX API. The distinction between verified and unverified pool endpoints was established: `https://api.spectrum.fi/v1/amm/pools/summary` returns only verified tokens (~13 at the time), while the markets endpoint with a default epoch timestamp returns all tokens including stale ones with backfilled last-known prices. Related to this, concerns were raised about artificially inflated token valuations in wallets caused by tokens with minimal liquidity pools—a "love token" example illustrated how a 50 ERG seed pool could make a token appear worth $23,000 in wallet displays.

### Node Peer Discovery and Decentralization
A substantive debate arose around maintaining trustworthy node lists for peer discovery without a single point of failure. The Bitcoin-style approach of hardcoding a set of known nodes (updated via new releases) was discussed alongside DNS seeder services. MrStahlfelge noted weaknesses in both approaches: hardcoded lists go stale between releases, and centralized aggregators introduce a point of failure. No definitive solution was reached, and the tension between decentralization and practical maintainability remained open.

### Block Time, Parameter Changes, and dApp Impact
Discussion of potential Ergo block speed increases raised the question of downstream effects on contracts and dApps that rely on block height rather than timestamps. The consensus was that testnet experimentation with scientific observation is the right path forward—kushti noted an existing testnet channel for parameter-tweaking work and invited interested community members to lead the effort. The emission contract implications (tail emission timing) and security considerations around worst-case scenarios were also flagged as needing careful analysis before any mainnet changes.

### HD Wallet Derivation and Signing Standards
Aberg queried whether BIP-32 derivation paths such as `44'/429'/0'/0/0` require all components to be hardened or only intermediate ones. A Medium article on hardened derivation was referenced but no authoritative answer was produced within the chat. Separately, AppKit transaction signing with pre-computed signature byte arrays was resolved by Aberg via internal AppKit APIs.

### Affiliate and Referral Tracking On-Chain
A creative discussion explored whether influencer/content creator referral tracking could be implemented on-chain for Ergo DEXes. Flying Pig outlined a faucet-based approach: new users receive a unique token tied to a referrer's link, enabling DEX interaction tracking. The limitation of derived/private addresses (as used in Nautilus wallet) was noted as a practical obstacle.

### Security Assumptions in Ergo Design
kushti published a forum post comprehensively addressing the security assumptions behind key Ergo design choices, covering conservative vs. non-conservative trade-offs—referenced as essential reading for developers building on or integrating with the protocol.

## Technical Q&A Worth Preserving

- **Q** (@Aberg, 2023-02-01): What is the best way to fetch token prices from Spectrum DEX?
  **A** (@qx()): Use the Spectrum DEX API directly. `https://api.spectrum.fi/v1/amm/markets?from=1653075262280` returns all traded tokens; `https://api.spectrum.fi/v1/amm/pools/summary` returns verified tokens only (~13 tokens). For tokens not recently traded, leave the timestamp at default epoch time and the API backfills the last known price.

- **Q** (@Aberg, 2023-02-13): When parsing ErgoTransaction modifiers in the P2P protocol, how do I skip SigmaValue parsing in ContextExtension without traversing the full recursive SigmaBoolean structure?
  **A** (@Alex Slesarenko): Consult the ErgoTree PDF specification and study `SigmaBoolean.serializer` to understand how `ProveDlog` (a SigmaBoolean subclass) is serialized. For P2PK transactions, you are only dealing with ProveDlog opcodes with EC point arguments—but you cannot easily skip the recursive SigmaBoolean structure without implementing the full parser. Use AppKit or serializers from Sigma directly if possible; otherwise, study the interpreter module in the develop branch.

- **Q** (@Aberg, 2023-02-14): ErgoTree header with value 0 or 16 — is that normal? After parsing header and nConsts, the next byte (data type) reads as 0, which appears invalid.
  **A** (@Alex Slesarenko): Consult the ErgoTree PDF specification. (Subsequently, Aberg identified two inaccuracies in the PDF: header field should be `getByte()` not `UInt`, and "end for" should read "end if".)

- **Q** (@Flying Pig, 2023-02-01): In the Ergo Explorer database, does the `box_id` field in the tokens table reflect the current UTXO location?
  **A** (@Flying Pig, self-resolved): No — `box_id` reflects the historical box in which the token was created. Resolved by repurposing one of the Explorer's own API queries to retrieve the current UTXO location.

- **Q** (@MrStahlfelge, 2023-02-14): How should a wallet or application maintain a decentralized, trustworthy list of Ergo nodes for peer discovery without a single point of failure?
  **A** (@Aberg): Hardcode a set of known nodes in the application config (Bitcoin's approach) and allow peer propagation for further discovery. If hardcoded nodes go down, update the array in a new release. Alternatively, implement a DNS seeder similar to Bitcoin's. (@MrStahlfelge noted both are weak: manual maintenance is slow and centralized aggregators are single points of failure. No fully satisfactory solution was identified.)

- **Q** (@HQΣr, 2023-02-15): Can we create on-chain affiliate/referral tracking for content creators bringing users to Ergo DEXes?
  **A** (@Flying Pig): Addresses cannot store data directly. A faucet-based approach works: new users receive a unique token linked to the referrer via a referral URL, which can then be tracked for DEX interactions. Caveat: this breaks if users employ derived addresses (e.g., private mode in Nautilus wallet).

- **GPU mining and Autolykos DAG size**: Whether GPUs with less than 3 GB DAG capacity can mine Ergo, and whether the DAG size parameter is modifiable in the Autolykos protocol. Raised by @HQΣr but not resolved.

- **Node peer discovery without a single point of failure**: The tension between hardcoded node lists (stale between releases) and centralized aggregators (single point of failure) remains unresolved. A DNS seeder implementation was suggested but not actioned.

- **Block time changes and downstream dApp impact**: Safe approaches to increasing block speed without breaking contracts relying on block height vs. timestamp need testnet validation. A testnet parameter-tweaking group exists but requires someone to take ownership and lead it.

- **BIP-32 hardened derivation path standards for Ergo**: Whether all path components in `44'/429'/0'/0/0` must be hardened, or only intermediate ones — no authoritative answer produced.

- **Spectrum API 24h volume querying**: Exact usage of `from`/`to` filter parameters for volume metrics was not resolved.

- **ErgoTree PDF inaccuracies**: Identified errors (header field type, "end for" vs. "end if") flagged but upstream correction not confirmed within the month.

- **SigmaBoolean parsing complexity in real transactions**: Whether sigma constant size ≥ 1 is common in practice, or whether real-world P2PK transactions always use simple ProveDlog structures — Aberg observed it was null in every transaction checked.
