# Ergo Developer Chat — Monthly Summary: June 2022

---

## Executive Summary

June 2022 was a foundational infrastructure month for the Ergo developer community, dominated by the testnet relaunch targeting 5.0 activation testing, low-level P2P protocol implementation work in Satergo, and the release of Satergo v0.0.3. Coordination challenges around the testnet reset exposed gaps in release communication processes, while parallel work on wallet development tooling (including a B4X-based POS terminal exploration) highlighted the breadth of the developer ecosystem.

---

## Key Topics This Month

### Satergo v0.0.3 Release
Aberg shipped a significant update to the Satergo desktop wallet, reducing the binary size by 50% and adding transaction history, password-protected sending, multi-address selection, price charts (24h–max range), token icons, and keyboard navigation on seed phrase entry. Background operations were optimised for UI responsiveness. The release represents a meaningful maturation of Satergo as a full-featured desktop wallet independent of the reference node UI.

### Testnet 5.0 Relaunch and EIP-27 Activation
kushti announced a testnet relaunch mid-week to test 5.0 activation, with Luivatra preparing tx cost comparison tests between 4.0.31 and 5.0 for staking scenarios. The relaunch was poorly coordinated — the Explorer was pointed at the wrong chain, the faucet was broken, no advance blog post was published, and a seed node suffered an unexpected disk failure. kushti acknowledged the coordination failure while noting the server outage was unpredictable. EIP-27 activation on testnet was flagged as conditional but the rationale was not fully explained. The 720-confirmation requirement (previously inconsistently applied between mainnet and testnet) was also noted.

### P2P Layer Implementation in Satergo
Aberg undertook a full reimplementation of the peer-to-peer layer in Satergo, covering message types and modifier serialization. A key technical clarification emerged: `ModifierTypeId` is a single `byte`, while `ModifierId` is a `byte[32]` — a distinction Aberg resolved through source inspection. This work highlights the depth of protocol-level engineering being done independently of the reference client.

### Java Version Compatibility
Discussion clarified the Java version landscape for building Ergo: published jars are built with Java 8, Java 11 is the latest LTS that builds cleanly, and Java 17 introduces compatibility issues. The language-level target remains Java 7 to preserve Android/iOS cross-compiler compatibility. This is a recurring friction point for new developers attempting to build from source.

### Seed Node Resilience and Faucet Access
The testnet reset exposed the fragility of the limited set of mainnet seed peers (notably `176.9.15.237` and `213.239.x.x`, the latter of which suffered a disk failure). kushti acknowledged the need to expand the seed peer list in follow-up releases but gave no timeline. Separately, Aberg requested removal of the Discord verification requirement from the testnet faucet, arguing Captcha alone is sufficient — Moein agreed and actioned the change.

### B4X / Visual Basic POS Terminal Wallet
A developer (Lee) explored building an ERG-receiving POS terminal using the B4X (Visual Basic-like) framework for Android/iOS. MrStahlfelge clarified that receiving ERG is passive and requires only an address, but that address generation requires cryptographic key math comparable to Bitcoin. The suggestion was to either find a Bitcoin key-derivation library for BASIC or generate addresses offline and import them, significantly simplifying the on-device requirement.

## Technical Q&A Worth Preserving

- **Q** (@Lee, 2022-W25): Can I build an ERG-receiving POS terminal using B4X (Visual Basic-like framework) for Android/iOS?
  **A** (@MrStahlfelge): Receiving ERG is passive — you only need an address. Address *generation* requires cryptographic key math similar to Bitcoin; look for Bitcoin key-derivation libraries for BASIC, or generate addresses offline and import them to simplify the on-device implementation significantly.

- **Q** (@Aberg, 2022-W25): Which Java version is required to build Ergo from source? Java 17 produces sbt errors.
  **A** (@MrStahlfelge): Published jars are built with Java 8. Java 11 is the latest LTS that builds cleanly. Java 17 has compatibility issues. The language-level target is Java 7 to preserve Android/iOS cross-compiler compatibility.

- **Q** (@Aberg, 2022-W25): What is the distinction between `ModifierTypeId` and `ModifierId` in the Ergo P2P protocol?
  **A** (@Aberg, self-resolved via source inspection): `ModifierTypeId` is a single `byte`; `ModifierId` is a `byte[32]`.

- **Q** (@kushti, 2022-W25): Are consecutive leaf proofs more compact in UTXO trees?
  **A** (@kushti): Yes — the proof is the subtree needed to compute the new root, so consecutive leaves share subtree structure and are more compact.

## Open Questions Entering Next Month

- **Testnet reset communication process**: No formal process established for coordinating testnet relaunches — no advance blog post protocol, no Explorer switchover checklist. Risk of recurrence is high.
- **Seed node resilience**: The mainnet seed peer list remains small and vulnerable to DDoS or hardware failure. kushti noted expansion is desirable but no timeline or ownership was established.
- **EIP-27 testnet activation**: Conditional activation of EIP-27 on testnet was mentioned but the triggering condition and rationale were not documented.
- **Java 17 compatibility**: No resolution or tracking issue noted for Java 17 build failures; developers building from source must manually know to use Java 11.
- **P2P layer completeness in Satergo**: Aberg's full P2P reimplementation was ongoing at week's end; no completion milestone announced.
