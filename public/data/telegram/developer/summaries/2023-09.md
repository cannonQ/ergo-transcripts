# Ergo Developer Chat — Monthly Summary
**Month:** September 2023

---

## Executive Summary

September 2023 was a month of steady infrastructure development and technical tooling work across the Ergo ecosystem. Key themes included UTXO set scanning implementation, mining pool development (particularly the Sigmanauts pool with storage rent integration), and ongoing discussion of performance optimisation — from database query tuning on the Explorer to block cost governance and long-term TPS metrics. Community projects including GreasyCEX Mission #2 and Nautilus/Fleet SDK updates also saw meaningful progress.

---

## Key Topics This Month

### UTXO Set Scanning & Node Performance
@kushti identified implementing UTXO set scanning as a significant pending task requiring separate threading to avoid blocking other threads during set modifications, with results needing injection into the block scan process. A bounty was attached to incentivise contributors. Separately, discussions on transaction processing clarified that raw TPS is a misleading metric for smart contract chains due to variable transaction complexity; @Luivatra and @Flying Pig proposed BPM (Blocks Per Minute) as a more meaningful measure. The current mainnet max block cost slightly exceeds 1MB, with 10–12M recommended for approximately a 50% throughput increase. A voting proposal on the Ergo Forum was referenced for increasing maximum computational cost per block.

### ErgoExplorer Performance & Database Optimisation
The public ErgoExplorer lagged significantly during W37, becoming stuck while other nodes progressed, traced to heavy transaction loads (6,000 chained transactions in a single block) causing database processing bottlenecks. @Luivatra explained that the Explorer's data model — separate tables for outputs and inputs — causes excessive joins when querying unspent outputs. Adding appropriate indexes dramatically improved query times (e.g., one index reduced a portfolio query from 3 seconds to 0.5 seconds). @Luivatra recommended using query plans (`EXPLAIN` in PostgreSQL with timing enabled) to diagnose slow queries. @Aberg proposed renaming the public explorer to "legacy explorer" to distinguish it from newer UI/UX iterations.

### Sigmanauts Mining Pool & Storage Rent Integration
@qx() announced the Sigmanauts mining pool entering a burn test phase in W38, with storage rent capture integration targeted approximately one to two weeks out prior to public launch with UI. The burn test was designed to verify block hits and payouts. @Cheese Enthusiast was identified as assisting with storage rent integration. Discussion also touched on miner behaviour around storage rent more generally — miners can choose to include only their own transactions taking rent, though this power is not unique to storage rent contexts.

### Flash Loans & Oracle Manipulation Risk
@kushti raised concern in W38 that DuckPools using Spectrum DEX as a price source opens the protocol to flash loan attacks capable of manipulating on-chain oracles. The feasibility of flash loans in Ergo's eUTXO model was debated — Ethereum's contract state mutation within a single block enables flash loans structurally, whereas Ergo's transaction-based model has different mechanics. @Armeanio characterised flash loans used as a base for mark-to-market financial products as fundamentally unsound, and flagged the topic as suitable material for a Hash It Out discussion. No consensus was reached on Ergo's relative safety.

### Fleet SDK & Nautilus Development
@Capt. Nemo announced Nautilus v0.9.0 in W39, featuring an improved Ledger wallet signing flow. Fleet SDK's `ergographql-client` package was described as nearing completion, with the `@urql/core` dependency replaced by a custom fetch-based GraphQL client and unit tests pending. This represented steady incremental progress on developer tooling infrastructure.

### JVM vs. Rust & Language Choice for Blockchain Nodes
@kushti and @Aberg debated language selection for blockchain node implementations. @kushti argued that JVM garbage collection may be well-suited for blockchain processing with periodic transaction classes, and noted that modern GC (ZGC, Shenandoah) outperforms manual memory management for small object churn. @kushti reviewed Kaspa's Rust implementation and found it "raw." @Aberg recommended always using the latest JDK for performance gains. Both agreed that scaling TPS requires bandwidth optimisation or layer 2/sidechain solutions rather than super-nodes, which sacrifice decentralisation.

### GreasyCEX Mission #2 (Greasy Royale)
GreasyCEX Mission #2 ("Greasy Royale") progressed steadily across the month. By W38, seven sponsors were confirmed (Terahertz, Lilium, Sigmanauts, Comet, ErgCube, Spectrum DEX, Grand Gambit), the Greasy Royale Paper was publicly released, 100-character NFT production was 50% complete, and a website update was 50% done. A partnership with Cyberverse for character NFT integration and unique audio files was announced in W39. An AMA co-hosted by @qx() was scheduled for 2023-09-23. Full mission launch was targeted for the week after the Vegas NFT conference.

### Mempool Configuration & Transaction Fee Management
@Pulsarz provided detailed guidance on mempool settings: `rebroadcastCount` (default 3; recommended ~500) controls how many transactions from the mempool are rebroadcast per block, and `mempoolCleanupDuration` (recommended 2 minutes) governs cleanup frequency. Default mempool limit is 1,000 transactions; blocks typically include 400–500. Configuration is managed via `ergo.conf` overriding `application.conf` defaults. A fee estimation API exists in the node but is reportedly not used by applications and may not be thoroughly tested.

## Technical Q&A Worth Preserving

- **Q** (@Flying Pig, 2023-09-15): Can mempool and rebroadcast settings be configured directly in `ergo.conf`, or must they be amended in `application.conf` and rebuilt?
  **A** (@Luivatra): `application.conf` contains defaults; `ergo.conf` overrides them. `mainnet.conf` sits between them in precedence order.

- **Q** (@Flying Pig, W38): What is the minimum ERG value a box containing a token must hold?
  **A** (@Flying Pig, @Luivatra): Enforced by `minValuePerByte × outputSize` nanoErgs. Testing showed 0.00001 ERG fails; 0.00004 ERG succeeds. Formula: box size in bytes × 360 nanoErgs (from Paideia state code).

- **Q** (@Flying Pig, W38): Is a seed phrase stored in the Ergo node?
  **A** (@Aberg): No seed phrase is stored — only the derived private key. Seed phrase → private key is a one-way transformation. The node implements HD (hierarchical deterministic) key derivation where one master key generates multiple child addresses.

- **Q** (@qx(), W38): How would flash loans work on Ergo given the eUTXO model?
  **A** (implicit, @Armeanio): Ethereum enables flash loans via contract state mutation within a single block. Ergo's transaction-based model requires multiple transactions with intermediate outputs, making the mechanism structurally different. The financial risk is acute when flash loans underpin a mark-to-market asset class.

- **Q** (@HQΣr, 2023-09-30): How can token prices be retrieved when Spectrum DEX and its APIs are down?
  **A** (@Luivatra): Query the Explorer API directly: `boxes/unspent/byTokenId/{pool_id}`, then divide the token amount by the ERG value to derive price from on-chain pool state.

- **Q** (@kushti, W39): When a high-cost transaction reaches the mining mempool but isn't included in blocks, what happens?
  **A** (@kushti): Block assembly removes it due to a security buffer (~50k cost margin). Miners running solo can vote to increase the block cost limit. Max block cost on testnet is approximately 1M.

- **Q** (@Aberg, W37): What resources help with optimising database queries on the Explorer?
  **A** (@Luivatra): Learn to read query plans using `EXPLAIN` in pgAdmin with timing enabled. Identify slow operations, then add indexes. A single index reduced one portfolio query from 3 seconds to 0.5 seconds. Principles translate across database systems.

## Open Questions Entering Next Month

- **Flash loan / oracle manipulation risk**: No consensus reached on whether Ergo's eUTXO model provides sufficient structural protection against flash loan-based oracle manipulation when using Spectrum DEX as a price source. Flagged for Hash It Out discussion but not resolved.
- **ETH/USD Oracle deprecation**: @Luivatra proposed turning off the ETH/USD Oracle Pool in W39 with no community response or decision recorded.
- **Block cost limit governance**: 10–12M is informally recommended for a ~50% throughput increase, but no formal consensus on target value or vote timeline was established.
- **Database snapshot recovery**: @HQΣr proposed incremental database snapshots to allow recovery from specific block heights without full resync from genesis; not directly addressed by @kushti.
- **UTXO Alliance protocol research**: @HQΣr questioned whether other UTXO-based chains actively research protocol improvements; no response documented.
- **Node binary distribution strategy**: Unresolved tension between ease-of-use (installable CLI binaries) and node freshness (automatic updates via Satergo-style wrapper); macOS DMG signing also unresolved.
- **Fee estimation API adoption**: Node contains a fee estimation API but it is reportedly unused by applications and potentially untested; no owner or timeline for improvement identified.
- **Accumulator research**: Community discussed accumulators (referencing Boneh-Bunz-Fisch) as a mechanism to reduce full UTXO set requirements; no concrete next steps recorded.
