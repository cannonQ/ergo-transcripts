# Ergo Developer Chat — Monthly Summary: 2024-02

---

## Executive Summary

February 2024 was a highly active month for Ergo protocol development, with significant work across core infrastructure, wallet tooling, and DeFi protocol readiness. The dominant themes were: preparation for the Dexy Protocol launch (nearing public testnet), Ergo Node 5.0.20 release alongside Scala 3 migration work in scrypto, and the formation of the ErgoDevs DAO as a formal R&D structure for protocol and off-chain infrastructure. Secondary themes included wallet ecosystem improvements (Satergo v1.7.0, Minotaur v2.0.0-rc1, Nautilus refactoring), sidechain prototyping, and meaningful community discussions around spam protection, storage rent mechanics, and cross-chain ecosystem collaboration.

---

## Key Topics This Month

### Dexy Protocol Launch Readiness
Dexy moved through closed UI testing across the month, with transaction chaining bugs identified and addressed by @dmitryusov. Redemption proxy contracts were revisited by kushti, and off-chain redemption support was developed by @code_for_uss. By end of February, the protocol was ready for public testnet, with the remaining blockers being a final audit round (preferably by Four Eyes Consulting, with GORT rewards), public testnet testing, and mainnet deployment. Non-constant product LP proposals were acknowledged as valuable but deferred due to analysis complexity.

### Ergo Node 5.0.20 & Scala 3 Migration
Node 5.0.20 shipped in W07, incorporating scrypto PR #109 (Scala 3 migration). A notable engineering challenge was the discovery of type test incompatibilities between Scala 2 and Scala 3 for opaque types — Scala 2 silently passed broken type assertions that Scala 3 correctly fails. A working but imperfect solution was merged to scrypto (PR #108), with the non-functioning test disabled and a tracking issue opened. The `SigmaMap` context rework (replacing sparse arrays in context extension variable representation) progressed across the month, with efficient implementations planned for 1–4 key variants.

### ErgoDevs DAO
kushti formally proposed the ErgoDevs DAO in W08 — a cyberspace-based R&D structure to conduct protocol research and maintain basic off-chain infrastructure. The proposal was posted to the Ergo Forum and attracted broad interest from contributors. A recurring theme was the need for more Rust developers in the ecosystem. This proposal represents the first formal attempt to institutionalise Ergo core R&D outside the Ergo Foundation.

### Wallet Ecosystem: Satergo, Minotaur, Nautilus
Three wallets had significant releases or development milestones in February. Satergo v1.7.0 shipped in W05 with light node support, chained transactions, stealth address sending, and design improvements. Minotaur v2.0.0-rc1 released in W07 as a major redesign with bug fixes and performance optimisations. Nautilus underwent refactoring with arbitrary message signing added; EIP-44 SigmaProp signature support was in progress, with open questions around how to display signed data safely (particularly for Ledger/screen-constrained devices).

### Rosen Bridge Progress
Rosen Bridge saw steady progress: Bitcoin chain implementation went under review, EVM RPC extractor was under review, Bitcoin UTXO selection package was merged, and WID hash calculation fixes landed in guard-service and scanner. The bridge is converging on multi-chain production readiness.

### Sidechains R&D
kushti progressed a merge-mined sidechain prototype throughout the month — header verification code, transaction verification, and a database stub were implemented. The trustless pegging mechanism via NIPoPoWs-based relay was under active development. Sidechains were discussed in the context of Ergo's layer 2 strategy: kushti noted that having a concrete set of on-chain applications (Oracle Pools, Rosen Bridge, Spectrum DEX, etc.) makes it much easier to reason about and optimise layer 2 designs, unlike blockchains that pursue sharding or parallel chains from day one with empty chains.

### Storage Rent Mechanics
Multiple developer questions in W05–W06 clarified storage rent details: the collection period is 1,051,200 blocks (~4 years); boxes are fully spent on collection (cannot be partially spent); `creationHeight` is the correct field for age calculation; and paying storage rent creates a new box rather than modifying the existing one. qx()'s `epBoxAge.pl` endpoint was highlighted as a useful tool for warning users.

### Stealth Addresses
Aberg successfully sent the first stealth address transaction to kushti's test address in W05, validating the implementation before Satergo v1.7.0 release. Point encoding was clarified as compressed 33-byte format (identical to Bitcoin, with infinity point as 33 zero bytes), using BouncyCastle's `getEncoded(boolean compressed)`.

### EIP-45 Ephemeral Tokens
qx() introduced a draft proposal for incentivised token burning via ephemeral smart contracts in W08. Register allocation (R4–R6 per EIP-4 conventions) and cross-protocol interoperability via data inputs for ownership proofs were discussed. Luivatra raised a concern about compatibility with protocols requiring direct token transfers rather than data-input proofs — unresolved at month end.

### Mempool Spam Protection
A detailed discussion in W09 clarified Ergo's spam protection model: per-peer cumulative transaction cost limits (since version 4.0.4x), global per-block limits, and isolation of spam to small node clusters even with multi-peer attacks. The node's minimum propagation fee rate is configurable, allowing operators to prevent low-fee transactions from even entering the mempool. A fee estimation API exists in the node but was noted as untested and unused by applications.

### Pre-Genesis UTXO Set
In W06, @dusek_ investigated why the treasury allocation transaction showed no inputs in explorers. kushti clarified that three boxes (emission, treasury, proof-of-no-premine) are hardcoded in the protocol's pre-genesis state and are not indexed by explorers — this is by design and not an explorer bug.

### ErgoHack Planning
In W08, @mgpai proposed a March 16–26 window for the next ErgoHack, preferring a two-week format (similar to Alephium's hackathon) to allow ecosystem projects to reach MVP. @AngieHar noted a May format was planned and a formal announcement was forthcoming.

## Technical Q&A Worth Preserving

- **Q** (@Aberg, 2024-01-29): How should point encoding work for stealth addresses?
  **A** (@kushti): Points use compressed 33-byte encoding, identical to Bitcoin except the infinity point is 33 zero bytes. Recommended using BouncyCastle's `getEncoded(boolean compressed)` method.

- **Q** (@Aberg, 2024-01-30): How to warn users about storage rent deadlines for old boxes?
  **A** (@qx()): Use the `epBoxAge.pl` endpoint. Use `creationHeight` to calculate age; boxes cannot persist beyond 1,051,200 blocks (~4 years) as storage rent collection fully spends them.

- **Q** (@Aberg, 2024-01-30): Does paying storage rent create a new box or modify the existing one?
  **A** (@qx()): Storage rent collection spends the box regardless of remaining balance. The UTXO model requires breaking the box and creating a new one — boxes cannot be partially spent.

- **Q** (@Paul, 2024-02-11): What JDK version should be used for AppKit development?
  **A** (@Aberg): Use JDK 21 (latest), preferably from adoptium.net unless GraalVM-specific features are needed. Earlier versions receive no updates outside of paid business support.

- **Q** (@dusek_, 2024-02-10): Why does the treasury allocation transaction show no inputs in explorers despite having an input in the node?
  **A** (@kushti): This is the first transaction spending the treasury box created in pre-genesis state. Explorers only index boxes created in blocks; the protocol hardcodes three pre-genesis boxes (emission, treasury, proof-of-no-premine) directly in the UTXO set.

- **Q** (@peterpan0708, 2024-02-19): How to optimise transaction assembly for an exchange wallet with 5,000+ unspent boxes?
  **A** (@kushti): Set `ergo.wallet.optimalInputs = 80` or `100` and use repeated self-transfers to consolidate. For many small-value inputs, set `ergo.wallet.dustLimit = 1000000000` (1 ERG), rescan and collect, then lower the limit and rescan again. An exchange-specific node configuration guide is available in the Ergo docs.

- **Q** (@Aco, 2024-02-21): Why can tokens be burned without a dedicated burn function?
  **A** (@Pulsarz): FleetSDK enforces token burning checks and requires explicit allowance. ergo-lib-wasm and AppKit lack such safeguards — output validation is entirely the developer's responsibility.

- **Q** (@cannon_q, 2024-02-21): Can a standard PHP payment script be created for Ergo?
  **A** (@liquid_phase): FleetSDK (TypeScript/wasm) is the primary standard; PHP lacks native support. For PHP sites, a JavaScript frontend with Nautilus wallet connector is the recommended pattern. Backend validation logic (CMS signals, download authorisation) is inherently app-specific and cannot be fully templated. An open-source PHP dApp connector for Laravel exists but is not a drop-in solution.

- **Q** (@Aco, 2024-02-21): Why don't mempool transactions appear in the official Explorer API when they're visible in Sigmaspace?
  **A** (@kushti): One node behind the explorer ran out of disk space. The explorer randomly selects nodes rather than aggregating data, causing inconsistent mempool visibility.

- **Q** (@dusek_, 2024-02-28): Why does a transaction pass node validation but fail sigma-rust validation on `box2.R4[Int].get == CONTEXT.selfBoxIndex` when R4=-1 and selfBoxIndex=2?
  **A** (@dusek_, self-resolved): `selfBoxIndex` semantics changed in v5.0 (fixed via sigma-rust PR #772). sigma-rust currently has no version checks, so it cannot correctly validate pre-5.0 transactions using `selfBoxIndex`.

- **Q** (@omniawaha, 2024-02-28): How does Ergo defend the mempool against spam attacks given no punishment for failed transactions?
  **A** (@kushti): Per-peer cumulative transaction cost limits have been in place since version 4.0.4x; a global per-block limit exists; failed transaction costs are recorded per peer; spam is isolated to small clusters even with multi-peer attacks. Operators can also raise minimum propagation fee rates to prevent low-fee transactions from entering the mempool at all.

- **Q** (@Michael, 2024-02-28): Can one node support multiple wallets simultaneously?
  **A** (@krasaviceblasen): No — only one wallet is active at a time (wallets can be swapped or renamed). Multiple dApps (DuckPools, Oracle Pools, Spectrum DEX) can share one node and wallet provided the dApps do not require node wallet inputs. A seed-in-config pattern would enable multi-dApp setups without node wallet dependency but has not yet been standardised.

- **Q** (@mgpai, 2024-02-28): What is kushti's view on Vitalik's stance on L1 consensus complexity and centralisation risk?
  **A** (@kushti): His position on L1 consensus failures is alarming; blockchains where core developers have an appetite for endless centralisation are not to be recommended.

- **Q** (@kii, 2024-03-01): Does Ergo have quantum computing security measures?
  **A** (@kushti): Proof of work is quantum-secure; dlog-based Sigma protocols and signatures will be compromised by sufficiently powerful quantum computers. A hard-fork recovery option exists if the threat materialises. No concrete changes are planned yet — the threat timeline is unclear and post-quantum alternatives remain impractical at scale.

## Open Questions Entering Next Month

- **Dexy non-constant product LP**: Proposals exist to improve capital efficiency but require complex analysis; no decision on inclusion at launch.
- **v5.0.14 release timeline**: Pay2SHAddress backward compatibility fix in progress; no firm release date set.
- **fromIds PR**: Ergomorphic noted "lots of distractions"; to be prioritised in March but no ETA confirmed.
- **RocksDB macOS "Too many open files" error**: Reproducible on fresh clones; root cause unclear (upstream RocksDB issue #10367 suspected); blocker for macOS CI test runner.
- **EIP-45 cross-protocol compatibility**: How ephemeral tokens interact with protocols requiring direct token transfers (not data-input proofs) — Luivatra's concern unresolved.
- **Scala 3 TypeTag alternatives**: Full type-safety for `Digest32`/`Digest64` in Scala 3 still uncertain; macro-based solutions under investigation; test coverage incomplete.
- **Explorer API node aggregation**: Whether the explorer will implement multi-node aggregation (vs. random selection) for consistent mempool visibility — not confirmed.
- **Wallet activity definition (Monthly Active Wallets)**: No consensus reached on whether receive-only wallets should be counted; filtering criteria for airdrop noise unresolved.
- **sigma-rust versioning/activation**: Full versioning support deferred; currently only skipping pre-5.0 blocks in Rust clients; architectural design not yet decided.
- **Seed-in-config standard for multi-dApp nodes**: Desirable pattern identified but no decision on adoption or standardisation.
- **ErgoHack format and timing**: May format confirmed as planned; exact duration and announcement details TBD.
- **SigmaUSD test suite**: Contributors called for; kushti to create initial suite; scope and ownership not fully defined.
- **PHP payment library**: No standardised template or library planned; remains custom per-integration.
- **Ledger/hardware wallet signed data display**: Architectural concern raised for screen-constrained devices in EIP-44 context; no concrete solution proposed.
- **Batch token minting info endpoint**: No existing API identified for fetching minting info for multiple token IDs in one request.
- **Official mining pool address registry**: No canonical list exists; community continues to rely on unofficial cross-referencing.
- **Scalahub memorial**: Community gathering stories/paragraphs; format (story paragraphs vs. repository list) not yet decided.
