# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer
**Month:** 2023-03

---

## Executive Summary

March 2023 developer discussions spanned three broad areas: practical indexing and API tooling (PostgreSQL performance, box retrieval pagination, TVL aggregation), protocol design questions (storage rent mechanics, multi-reserve stablecoin architecture, oracle pool infrastructure), and ecosystem infrastructure updates (Ledger integration, Satergo testnet fixes, Oracle Pool v2 on testnet). Activity was moderate, with the most technically substantive exchanges focused on database indexing workarounds and the compositional stablecoin proposal from kushti.

---

## Key Topics This Month

### API Box Retrieval & Pagination
Developers noted that explorer APIs impose artificial limits on box retrieval, making pagination slow and unwieldy for large queries. While full uncapped responses are theoretically possible, responses in the 500 MB+ range are unlikely given the ~30 GB total blockchain size. The practical workaround is to pass a custom `RestApiErgoClient` endpoint when using AppKit rather than relying on defaults — particularly important on testnet where the default ergoplatform node had been frozen since approximately February 8.

### PostgreSQL Index Performance
A recurring problem emerged across multiple developers: B-tree index size limitations when indexing large ErgoTree values in the `node_outputs` table caused row size errors. The confirmed fix is to use hash indexes instead of B-tree indexes for the `ergo_tree` and `node_outputs__address` columns. Luivatra's Paideia contracts were flagged as contributing to downstream UTXO tracker freezes due to contract size, highlighting a broader tension between expressive ErgoScript contracts and indexer performance.

### TVL Aggregation & Smart Contract Identification
Discussion emerged around whether a unified Ergo TVL dashboard was feasible. DefiLlama captures some Ergo projects but not all. Flying Pig published a custom SQL query covering ERG, SigmaUSD, SPF, NETA, Paideia, ErgoPad, and EGIO across 3,200+ smart contract addresses. A key unresolved question is whether smart contracts carry any built-in project identifier — current consensus is they do not, requiring external metadata mapping.

### Multi-Reserve Stablecoin Architecture (Djed/SigmaUSD)
kushti raised Gresham's law as a structural vulnerability in any multi-reserve-asset stablecoin design: "better" reserve assets tend to be withdrawn, leaving only "worse" ones. The proposed mitigation is to deploy separate single-asset SigmaUSD/Djed instances per reserve asset, then compose them into a metastablecoin backed by a basket, using multi-asset liquidity pool mechanics to manage ratio drift between constituent stablecoins.

### Storage Rent — Purpose and Parameterisation
Flying Pig's analysis of ~307 boxes from the first six months of mainnet (July 2019–January 2020), totalling over 13,000 ERG with one box holding 1,400 ERG, prompted debate on whether a storage rent multiplier was needed for ultra-dormant UTXOs. The clarifying distinction established was that storage rent targets **UTXO set size** (preventing bloat and dust), not historical blockchain data size. A 1,000 ERG box is neither dust nor bloat; the "Satoshi-wallet" concern is secondary to the primary UTXO set hygiene goal.

### Oracle Pool v2 & Ecosystem Infrastructure
Oracle Pool v2 went live on testnet with an XAU reserve asset pool, visible on the testnet explorer. The Ledger Ergo app reached full release — installable from the Ledger store without developer mode — though a "Pending review" prompt remains in the UI pending a new Ledger review cycle. Satergo shipped several unreleased fixes including CoinGecko fiat currency expansion, VM argument configuration, and a testnet host change.

### On-Chain Data Storage and Polling Patterns
Developers explored storing file fragments across multiple boxes using registers for metadata (collection ID, part count, sequence number). The simpler recommended approach is to store data directly in box registers and pass the first box ID in a chain to the recipient, rather than using token-linking mechanisms. A proposed on-chain polling system used token burn (one token per option, burn after vote) to prevent double-voting, with CAPTCHA-gated token distribution as a sybil-resistance measure in the absence of identity infrastructure.

## Technical Q&A Worth Preserving

- **Q** (@Flying Pig, 2023-W09): Does AppKit get boxes from the explorer, and can we specify a custom explorer URL on testnet?
  **A** (@Aberg): Yes — pass a custom `RestApiErgoClient` endpoint to AppKit rather than relying on the default public explorer URL.

- **Q** (@Flying Pig, 2023-W09): How do we write custom data to box registers using AppKit (Python)? GitHub docs are three years outdated.
  **A** (@Luivatra): Token ID is automatically written to R2. For custom data, use `setAdditionalRegisters()` in the transaction builder — see FleetSDK transaction building docs for a current example.

- **Q** (@sh₳d0w, 2023-W09): Same transaction signing code works in React but fails in NextJS — the extension window doesn't open.
  **A** (@sh₳d0w): Use `.toEIP12Object()` instead of `.toPlainObject()` when building unsigned transactions in a NextJS environment.

- **Q** (@Flying Pig, 2023-W11): ~307 boxes from early mainnet total over 13,000 ERG; one holds 1,400 ERG. Is a storage rent multiplier needed for ultra-dormant boxes?
  **A** (@Luivatra): The goal of storage rent is to avoid dust and bloat. A 1,000 ERG box is neither dust nor bloat, so the current mechanism is sufficient for that case.

- **Q** (@Flying Pig, 2023-W11): Isn't storage rent also meant to prevent "Satoshi-wallet" scenarios where large amounts go permanently out of circulation?
  **A** (community): The primary purpose is UTXO set hygiene — preventing the node's UTXO set from being bogged down by dust — not recirculating large dormant holdings.

- **Q** (@Luivatra, 2023-W13): Should minimum fee be read from a config box data input to avoid issues if the minimum fee changes?
  **A** (@kushti): Could be read from a config box provided as data input — that pattern works and is flexible.

- **Q** (@Flying Pig, 2023-W13): Is there a way to extract all smart contract addresses from the explorer database?
  **A** (@kushti): The explorer DB schema has a table for addresses that can be queried directly.

## Open Questions Entering Next Month

- **Smart contract project identification**: No built-in identifier exists on Ergo smart contracts — external metadata mapping is required. A standardised registry or tagging convention has not been proposed.
- **DefiLlama Ergo coverage**: DefiLlama does not yet index all Ergo dApps; whether to extend it or build a dedicated Ergo TVL dashboard remains open.
- **Ledger "Pending review" prompt**: Removal requires a new Ledger review cycle; timeline is unknown and estimated optimistically at six months.
- **Storage rent multiplier for ultra-dormant UTXOs**: Flying Pig's proposal for exponential/multiplier-based rent on boxes untouched across multiple collection periods was neither formally accepted nor rejected.
- **SkyHarbor and Auction House royalty enforcement**: Whether SkyHarbor smart contracts respect Auction House royalties was raised by HQΣr but received no response.
- **Fee estimation API adoption**: kushti noted the node has a fee estimation API that is not tested or used by any known application — a gap flagged but not actioned.
- **Sub-blocks / faster confirmation research**: kushti indicated drafts exist and discussion should be revived post-Ergoversary, but no concrete owner or timeline was assigned.
