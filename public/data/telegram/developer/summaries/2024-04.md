# Ergo Developer Chat — Monthly Summary
**Month:** April 2024

---

## Executive Summary

April 2024 was dominated by two interlocking themes: advancing Ergo's core protocol toward Sigma 6.0 (EIP-50) with new cryptographic primitives and performance improvements, and the maturation of ecosystem infrastructure including the Rosen Bridge, ErgoDevs DAO formation, and a wave of tooling releases across multiple languages. Secondary themes included sustained work on privacy wallet tooling, UTXO selection edge cases in the node API, and sidechain/merge-mining research reaching a point where formal EIPs and testnet experiments became actionable.

---

## Key Topics This Month

### Sigma 6.0 (EIP-50) Development

The month saw substantial progress on formalising Sigma 6.0. kushti published the EIP-50 draft with NBits encoding/decoding as the first reference implementation material, submitted PRs for nbits encoding (`sigmastate-interpreter` PR #962), unused code removal (PR #961), and a `BigInt` constructor for ErgoScript (PR #967). A key addition was `Global.powHit()` — a custom Autolykos 2 PoW validation function enabling trustless relay and sidechain construction with variant "k" parameters. kushti confirmed its inclusion in Sigma 6.0 following confirmation from the Lithos Protocol team that it is essential for implementing Lithos on merge-mined sidechains. The first bounty issues for external contributors were opened against the sigmastate-interpreter repository (issues #963, #964, #675). Sigma v5.0.14 shipped mid-month with progress on EIP-50 implementation, and the AppKit snapshot `1381ea75-SNAPSHOT` was made available based on this release.

### Node API, UTXO Selection, and Mempool Filtering

A recurring pain point emerged around `/blockchain/box/unspent/byAddress` with `includeUnconfirmed=true` not excluding boxes already being spent in mempool, breaking UTXO selection for transaction chaining. @Pulsarz filed issue #2144 and identified that PR #2131 (implementing an `excludeMempoolSpent` flag) was stalled due to `limit`/`offset` parameter complications. The fix did not make it into v5.0.21. The RocksDB branch was merged into the 5.1.0 release cycle; 5.1.x pre-releases are planned while 5.0.x remains the Docker default for several months. A separate issue with unknown token warnings and missing box errors after the 5.0.21 upgrade was resolved by deleting `/history/extra` and rescanning.

### Rosen Bridge Infrastructure and Bitcoin Integration

Rosen Bridge had a high-output month. The rosenet repository went public, scaling tests for peer discovery and direct messaging were completed, and Bitcoin support was added to both guard-service and Watcher. TSS improvements included separate ECDSA/EdDSA signer classes, signature recovery for ECDSA, and salt-based API key authentication to prevent precomputed hash attacks. An EVM address transaction extractor was placed under review. ARM v7 Watcher support and a total locked asset calculator were also integrated. Rosen SDK design discussions were ongoing throughout the month.

### ErgoDevs DAO Formation

The ErgoDevs DAO was bootstrapped in the second half of April. A 3-of-5 multisig was finalised with ccellado, CannonQ, lgd, mgPai, and kushti as keyholders — the same EF-tested standard. A Paideia instance is being set up for roadmap management, and Telegram/Discord bridge coordination was underway. kushti described the DAO's initial scope as covering tutorials, EIP management, self-governance, core development, and research, with an open invitation approach rather than strict entry requirements.

### Speed Improvements and Sub-Block Research

Discussion of potential Ergo throughput improvements surfaced in the weekly developer chat. kushti noted that the current concrete set of on-chain contracts (Rosen Bridge, Oracle Pools, GARD protocols, Spectrum AMM pool contracts, Trade House, lending contracts, SigmaUSD) provides a well-defined workload against which sub-block and sidechain proposals can now be meaningfully benchmarked — an advantage over chains that introduced layer 2 complexity before having real transaction profiles. A Telegram group was noted for those interested in running parameter-tweaking testnets; community members willing to lead empirical block-time studies were invited to reach out to kushti directly. The dependency of some dApp contracts on block height rather than timestamps was flagged as a constraint on block time reduction.

### Address Derivation, Fleet SDK, and Cross-SDK Consistency

A concrete developer friction point was resolved: Fleet SDK and ergo-lib were producing different addresses from the same seed phrase. The fix is aligning on derivation path `m/44'/429'/0'/0` (the Nautilus standard) using `ExtSecretKey.derive()` with DerivationPath in ergo-lib. The Nautilus team also shipped a blind prover for `ProveDlog` in `@fleet-sdk/wallet`. A separate Ledger hardware wallet issue was identified where combined `proveDlog(currentUserPk) && CONTEXT.dataInputs(0).value > 0` scripts fail to sign on Ledger despite each component working individually and the combined script working on Nautilus — root cause unresolved.

### Ecosystem Tooling Releases

Several significant tooling releases shipped in April:
- **ergo-lib-go v0.26.0** — Go wrapper for sigma-rust C bindings, supporting the full sigma-rust v0.26 feature set including transaction building, signing, and token minting.
- **Plasma-Toolkit v1.0.4** — updated with AppKit/EIP-5 support.
- **Sigmanauts Mining Pool dashboard v2** — live with planned backend database integration for multi-period hashrate metrics; pool reached top-ten ranking on miningpoolstats with 35+ GH/s.
- **Blitz TCG** — officially launched on Ergo with open-source repository and pack sale.
- **"The Field" DAO** — peer-to-pool pledging MVP contracts completed and tested on-chain.

### Privacy Wallet Tooling and UTXO Control

Detailed discussion of post-mixing UTXO management surfaced when a developer sought fine-grained box selection. SAFEW was identified as the most capable wallet for this use case via its expert mode and TX builder, but its dApp connector support is absent and address-deselection has known bugs. The ErgoPay cold wallet pattern (offline seed on a second device, read-only address on the hot device, QR-based signing) was recommended as the practical privacy path for dApp interaction. The stealth address contract on Ergo Forum was referenced for more advanced privacy patterns.

## Technical Q&A Worth Preserving

- **Q** (@Pulsarz, 2024-04-10): Does `/blockchain/box/unspent/byAddress` with `includeUnconfirmed=true` intentionally not exclude mempool-spent boxes? This breaks UTXO selection for transaction chaining.
  **A** (@kushti): "Sometimes you want to double-spend boxes spent by others." PR #2131 implements `excludeMempoolSpent` but is stalled on `limit`/`offset` parameter complications; not in v5.0.21.

- **Q** (@Pulsarz, 2024-04-10): Could `minConfirmations` be added to `/blockchain/box/unspent`?
  **A** (@kushti): Already implemented in PR #2131; not merged due to `limit`/`offset` issues.

- **Q** (@Luivatra, 2024-04-08): Are you waiting for EIP-5 support in AppKit to finalise contracts?
  **A** (@ergomorphic): Use AppKit snapshot `1381ea75-SNAPSHOT` based on Sigma v5.0.14, which has EIP-50 progress.

- **Q** (@jossemii, 2024-04-22): Fleet SDK returns different addresses than ergo-lib for the same seed phrase — which derivation path is correct?
  **A** (@Pgr456): Fleet SDK uses `m/44'/429'/0'/0` (Nautilus standard). Use `ExtSecretKey.derive()` with DerivationPath in ergo-lib to align.

- **Q** (@c8, 2024-04-11): Is it possible to sign a spending transaction from a `bobPK && alicePK` address using only two Ergo node API calls?
  **A** (@kushti): Yes. Published a 2-of-2 signing algorithm gist; two rounds of interaction required. Reference implementation in the Zero-Knowledge Treasury (DistributedSigsServer).

- **Q** (@jossemii, 2024-04-22): What is ErgoStateContext's purpose in sigma-rust and how is it obtained?
  **A** (@c8, @ergomorphic): Contains PreHeader plus 10 recent block headers. Fetch via GraphQL API, map into BlockHeaders, then construct ErgoStateContext. Required for contract execution and transaction signing.

- **Q** (@Luivatra, 2024-04-08): Can arbitrage across three AMM pools (ERG/A, A/B, ERG/B) be executed atomically in one transaction?
  **A** (@Paul): Technically possible by building a chained transaction sequence, but vulnerable to fee-bidding interference from other mempool transactions breaking the chain; higher mining fee helps but does not eliminate the risk.

- **Q** (@jossemii, 2024-04-03): Recommended approach to connect Rust code to a testnet node — local vs. provider?
  **A** (@cafebedouin.org): Use the TokenJay peer list (swap ports for testnet), or reference testnet docs at ergoplatform.com.

- **Q** (@jossemii, 2024-04-03): Scans via node API not returning historical boxes — how to retrieve them?
  **A** (@kushti): Scans only work forward from registration time. Run `/wallet/rescan` to find pre-existing boxes, or use explorer APIs with additional indices for historical data.

- **Q** (@kushti, 2024-04-24): Is `Global.powHit()` custom PoW validation worth including in Sigma 6.0?
  **A** (@cheeseenthusiast / Lithos): "Quite useful for implementing Lithos on sidechains with different variations of Autolykos." (kushti subsequently confirmed inclusion.)

- **Q** (@tsm121, 2024-04-25): Can dApps be developed without ErgoScript, e.g. using Python?
  **A** (@Paul, @mgPai): ErgoScript is required only for on-chain contract logic. Off-chain code, tooling, and simple transactions can use Python via ergpy or direct REST calls.

- **Q** (@cafebedouin.org, 2024-04-16): Is anyone collecting storage rent? How much ERG is available?
  **A** (@mgPai): Very little currently — approximately 400 ERG/year based on prior calculations. Collection requires custom node middleware at pool level.

- **Q** (@__daddychill__, 2024-04-16): Is there a wallet providing fine-grained UTXO control after mixing?
  **A** (@Flying Pig): SAFEW supports UTXO selection via expert mode and TX builder, but has known bugs with address deselection and no dApp connector. For dApp interaction with privacy, use ErgoPay cold wallet pattern (offline seed on second device, QR-based signing).

## Open Questions Entering Next Month

1. **`excludeMempoolSpent` release timeline** — PR #2131 is implemented but stalled on `limit`/`offset` parameter complications. When will this land in a stable release, and will it make 5.0.22?

2. **Ledger combined-script signing bug** — Why does Ledger fail to sign `proveDlog(currentUserPk) && CONTEXT.dataInputs(0).value > 0` when each component works independently and the combination works on Nautilus? Root cause unidentified.

3. **Sub-blocks EIP and block time parameter testnet** — kushti noted a testnet group exists for block-time parameter experiments. Does sufficient community leadership exist to run structured studies, and what is the realistic path to a formal EIP vote?

4. **headless-dapp-framework maintenance** — Should the framework be updated to modern ergo-lib? What specific features justify the maintenance overhead over using ergo-lib directly?

5. **Token metadata registry architecture** — Should metadata files and server implementation live in separate repositories? @Pgr456's recommendation was noted but not acted upon.

6. **Bitcoin and Ethereum oracle infrastructure** — @kii asked whether a native ETH oracle exists on testnet and whether a Bitcoin oracle is planned post-Chainlink partnership. No substantive answer was recorded.

7. **SAFEW maintenance status** — Active maintenance status unclear; known address-deselection bugs and absent dApp connector remain unresolved, affecting privacy users post-mixing.

8. **Paideia ConstantPlaceholder handling** — Workaround applied for parameterised EIP-5 contracts, but root cause of ConstantPlaceholder conversion in parameterised templates remains unclear.

9. **Gluon mainnet launch timing** — Testnet testing was noted as underway in W14; UI updates and multisig setup still pending. No launch date confirmed.
