# Ergo Developer Chat — Monthly Summary
## July 2025

---

## Executive Summary

July 2025 was a month of concentrated protocol R&D for the Ergo developer community, dominated by three interconnected workstreams: the progression of the 6.0 soft-fork (sub-blocks, UnsignedBigInt, PoEM), the specification and early implementation of a merged-mined sidechain framework (Braid), and a wave of ecosystem tooling releases across wallets, SDKs, indexers, and DeFi infrastructure. The month closed with the 6.0.1 protocol candidate ready for review, the Braid whitepaper published, and multiple projects (Lithos Protocol, ChainCash, Machina Finance, Minotaur) reaching significant milestones simultaneously.

---

## Key Topics This Month

### Protocol 6.0 Soft-Fork Progression

The 6.0.0 on-chain vote began on 2025-07-06. The release had two non-consensus-critical issues — sonatype publishing failures and an upcast AST node strip serializer bug — addressed in a planned 6.0.1 candidate. The 6.0.1 protocol candidate PR (#2222) was ready for community review by end of July, incorporating sub-block API methods, merged PRs #2231 and #2234, and the sonatype fix. Activation is targeted for September. A notable performance result: a Raspberry Pi achieved 1.5-day full sync on 6.0.0 versus 3+ days on 5.0.25, demonstrating the node performance improvements in this release cycle.

### Sub-blocks Implementation

Sub-blocks were the most technically active thread throughout the month. kushti made stability fixes to the devnet seed node (PR #2055 branch), added API methods for sub-block and best sub-block chain transactions, eliminated "todo:" markers in the codebase, and began composing a dedicated whitepaper. A stable devnet seed node was made available for community testing at `213.239.193.208:9058`. Sub-block injection into mainnet was targeted for late July, with the feature forming the core of the 6.0.1 protocol candidate. PoEM (Proof of Equivocation Misbehavior) for resolving input-block forks was identified as part of the feature scope.

### Merged-Mined Sidechains (Braid Framework)

kushti announced the start of Rust implementation for a merged-mined sidechain framework, with the concrete initial example being a double merge-mined Bitcoin + Ergo sidechain (named "Braid"). The Braid whitepaper first iteration was completed and published by end of July. The technical design involves NFT-identified boxes, transaction class 2 (ordering-block-only spendable), and R4/R5 register usage for refresh logic. Empty Rust repos were made available for contributors. AutoJanitor (RustChain project) expressed collaboration interest, noting their hybrid PoW + hardware-attestation design would not impact Ergo mining performance.

### Oracle Pool v2 & Stablecoin Infrastructure

kushti signalled it is time to migrate the SigmaUSD oracle pool to v2, with GluonUSD and DexyUSD launches planned to follow. A Telegram group for v2 oracle operators was formed, and 2–3 testnet operators were sought to test the v2 voting mechanism. Migration instructions from v1 to v2 were planned. Oracle Pool v2 is already running for Dexy Gold with GORT rewards. A new Gluon beta UI was deployed at `gluon.gold` and `gluon-ergo-ui.vercel.app` during W27.

### DarkERG Monetary System Proposal

kushti proposed implementing Bulletproof-based range proof verification inside ErgoScript to enable DarkERG — a hidden-amount stablecoin featuring virtual UTXOs, stealth addresses, and 1:1 contract-based swaps with ERG. A bounty issue was created at `sigmastate-interpreter/issues/1032`. The proposal generated discussion about the scope of zero-knowledge capabilities in ErgoScript, with the conclusion that Bulletproofs (as a trustless, setup-free Sigma protocol variant) could unlock a broader family of zero-knowledge statements verifiable on-chain without trusted setups.

### Lithos Protocol

CannonQ progressed steadily through the Lithos contract stack: storage rent contract testing completed, NIPoPoWs-based fraud proof contracts nearly done, difficulty contract underway to prevent on-chain 'd' value fraud, and LIT token emission contract started. A "holding contract" (initial rollup component) was deployed on devnet and moved to the 6.0 public testnet. Testnet launch is pending off-chain code completion. Lithos also initiated research on merge-mined Ergo sidechains in Rust to complement kushti's Braid framework work.

### ChainCash

Two pull requests (#60 and #62) were merged into the ChainCash Rust implementation. Transaction collateralization and reserve configurations are functional on mainnet, with first end-to-end applications and a UI underway. kushti shared an optimization technique using `blake2b256` script hash checking to avoid full ErgoTree comparison, which was captured in a dedicated GitHub issue (`sigmastate-interpreter/issues/1075`).

### Machina Finance

arobsn completed final AVL tree serialization support (Fleet SDK v0.10.0) and added optimizations to grid contracts, with limit order contracts next in the pipeline (bytecode minimization focus). A real-time order aggregation backend in Rust is in development. Community contract review is underway.

### Wallet & SDK Ecosystem

July saw an unusually high volume of tooling releases:
- **Fleet SDK v0.10.0** added AvlTree serialization/deserialization and support for all major constant types; **v0.12.0** added ErgoTree construction from ergoc JSON output.
- **ergoc v0.4.0** added constant placeholder tracking; **v0.5.0** adds JSON output.
- **Nautilus Wallet** fixed a non-Ledger dApp Connector rendering bug, added a Banxa fiat on-ramp button, JSON transaction copying in dev mode, and a token blacklist fix.
- **keystone-ergo-js v0.1.0** released with custom serialization schema; reduced transaction JSON support added to sigma-rust (PR #834) for Keystone/Nautilus integration.
- **Minotaur Wallet v3.0.0** released with a breaking change to multi-sig communication data format (not backward-compatible); immediately patched to **v3.0.1** to fix a white screen bug.
- **SigmaLok indexer** (Go, actor framework) completed by MGpai, handling mempool transactions, replacements, and rollbacks, with API docs at `api-sigmalok.phoenixfi.app`.

### Node Infrastructure & Performance

An `AskTimeoutException` in the `/mining/candidate` endpoint was traced by kushti to candidate block regeneration introduced in 5.0.26. A fix was prepared as PR #2231. A testnet parser failure at height 801365 was reproduced and fixed via PR #2229. RocksDB performance on 6.1 was reported as significantly slower than leveldb on 6.0 by arobsn on LUKS-encrypted SSD (24 hours at 39% sync); Pulsarz reported the opposite on ZFS RAID1. Investigation ongoing. Ergonodes.net was restored with 1-year hosting sponsorship.

## Technical Q&A Worth Preserving

- **Q** (@Anastasia Polozova, 2025-07-01): What image format does ergoauctions.org expect for NFT minting, and why aren't images displaying?
  **A** (@Aco Šmrkas): Most Ergo services use Lighthouse IPFS. Pinata uploads are found by ErgoExplorer but take longer to propagate. Use `tools.mewfinance.com` for minting — it uploads images to Lighthouse automatically for faster discovery and is EIP-34 compliant.

- **Q** (@arobsn, 2025-07-02): Is there an optimized way to compare different ErgoTree proposition versions without manual byte slicing?
  **A** (@kushti): Strip the first byte and compare. A shortcut ErgoScript function can be added in 6.0.2 without touching the protocol. Example from ChainCash: `blake2b256(receiptOutputErgoTree.slice(1, receiptOutputErgoTree.size))`. Issue tracked at `sigmastate-interpreter/issues/1075`.

- **Q** (@arobsn, 2025-07-06): Why is the 6.1 (RocksDB) node syncing so slowly — 24 hours at 39% with 93 peers, CPU at 100% — on an Intel i7 SSD with LUKS encryption?
  **A** (@kushti): RocksDB generally performs better on SSDs; LUKS encryption may be a bottleneck despite AES hardware offloading. Pulsarz reported the opposite experience (faster with RocksDB on ZFS RAID1), suggesting environment-specific factors. arobsn switched back to 6.0 (leveldb) for comparison.

- **Q** (@MGpai, 2025-07-08): How frequent are rollbacks on Ergo, and what block depth guarantees finality?
  **A** (@kushti): Ergo follows Bitcoin's probabilistic finality model. Refer to Satoshi's original paper and Pass-Seeman-Shelat (2016) for rollback probability analysis. (@c8): Finality is cost-based, not time-based — e.g., a $1B transaction on Bitcoin requires approximately one week of accumulated cost to make double-spending uneconomical.

- **Q** (@MGpai, 2025-07-10): Does Nautilus support submitting multiple chained transactions with a single signature?
  **A** (@arobsn): Not yet, but it is planned. See GitHub issue #213. Fleet SDK already supports chaining unsigned transactions — see the spec test at `fleet-sdk/fleet/.../ergoUnsignedTransaction.spec.ts#L120`.

- **Q** (@MGpai, 2025-07-16): Why is `UnsignedBigInt` needed as a new type in 6.0?
  **A** (@kushti): Critically needed for cryptographic applications such as signature verification. The new type includes support for modular arithmetic operations — essential for things like checking cryptographic sigs inside ErgoScript contracts.

- **Q** (@Michael, 2025-07-18): Testnet node stuck at height 801365 with `Failed to parse modifier` error from `ConnectedPeer`.
  **A** (@kushti): Reproduced locally, root cause identified. Fixed via PR #2229.

- **Q** (@andrei, 2025-07-23): POST `/mining/candidate` returns `AskTimeoutException` after 5 seconds on 6.0.0; some blocks rejected.
  **A** (@kushti): Likely caused by candidate block regeneration behaviour introduced in 5.0.26. Requested RAM allocation and node config to confirm. Fix prepared as PR #2231 — review requested from Pgr456 regarding `previousCachedCandidate` expiration.

- **Q** (@kushti, 2025-07-23): How can hardware fingerprinting in RustChain's Proof-of-Antiquity consensus be proven to validators without being trivially spoofable?
  **A** (@AutoJanitor): Three-pronged approach: (1) RAM clock drift differences between real and emulated hardware; (2) CPU entropy aging and thermal degradation curves; (3) BIOS/firmware specificity — emulators use generic ROMs that are detectable and punishable. ARM CPUs are cheap and plentiful, so they receive penalties. Removing hashrate races eliminates the incentive to emulate.

- **Q** (@Luca D'Angelo, 2025-07-25): Is Cardano's GuardScripts proposal (contracts controlling which UTXOs can bind to them) possible on Ergo?
  **A** (@kushti): This appears related to Global Transfer Policies (GTP), best explored via sidechains. Key semantic difference: GTP applies to monetary circuits originating from initial UTXOs/tokens, whereas GuardScripts treats contracts themselves as first-class citizens. mgPai confirmed with the CIP author that the Cardano proposal treats contracts as first-class.

- **Q** (@MGpai, 2025-07-29): Is the new `/inputs` API response format a breaking change for existing API consumers?
  **A** (@kushti): Suggested creating new endpoints rather than modifying existing ones. (@MGpai): Clarified that `boxId` and `spendingProof` fields are preserved; no existing keys are removed. `deepMerge` approach confirmed to align with `IndexedErgoTransaction` schema.

## Open Questions Entering Next Month

1. **RocksDB vs. leveldb performance regression**: arobsn's LUKS-encrypted SSD shows dramatically slower sync on 6.1 (RocksDB) vs 6.0 (leveldb). Root cause — LUKS, driver, or RocksDB tuning — remains unconfirmed. Investigation ongoing.

2. **`CONTEXT.HEIGHT` semantics in mempool**: When ErgoScript executes during mempool validation, does `CONTEXT.HEIGHT` evaluate to (confirmed tip height + 1) or something else? Raised by MGpai but unanswered at month end.

3. **Sub-blocks whitepaper scope and dependencies**: The sub-blocks whitepaper is in composition; unclear which protocol milestones (6.0.1 scope, PoEM specification, sidechain framework) depend on its finalization.

4. **Sidechain data box semantics**: Clarity needed on whether current input/ordering block documentation sufficiently specifies NFT-identified box updates, class 2 transaction constraints, and R4/R5 register usage for the Braid refresh logic.

5. **Oracle Pool v2 operator recruitment**: Telegram group forming; 2–3 testnet operators still needed. Migration instructions from v1 to v2 not yet written.

6. **Autolykos `/mining/candidate` timeout root cause**: kushti awaiting RAM and config details from the affected pool operator (andrei) to confirm the 5.0.26 regression hypothesis.

7. **API response format compatibility** (PR #2235): Whether the new `/inputs` endpoint with full box details constitutes a breaking change for existing consumers — review deferred to post-6.0.1 release.

8. **Metadata repository governance**: No consensus reached on centralised vs. fragmented approach for token logos and names. Deferred pending community feedback.

9. **RustChain / Proof-of-Antiquity verifiability**: AutoJanitor's fingerprinting claims (entropy aging, clock drift, BIOS specificity) presented but no whitepaper or audited code provided; kushti requested formal documentation. Meeting with DJ (formalising the code) pending.

10. **EIP compliance for multi-UTXO token minting**: Which EIP standards are affected when distributing tokens to multiple UTXOs in a single mint transaction — requires clarification from wallet and tooling maintainers.
