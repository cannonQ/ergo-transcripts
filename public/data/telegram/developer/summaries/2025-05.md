# Ergo Developer Chat — Monthly Summary: 2025-05

---

## Executive Summary

May 2025 was dominated by two major protocol development threads: the Ergo 6.0 soft-fork preparation (EIP-50 finalization, Sigma 6.0.0 approval, sub-blocks devnet deployment) and a wave of ecosystem tooling releases spanning Fleet SDK, ergoc, and Satergo. Infrastructure reliability concerns—particularly RocksDB crashes in v5.1.3, persistent explorer indexing inconsistencies, and sub-blocks devnet sync instability—ran as a secondary thread throughout the month, generating sustained developer attention.

---

## Key Topics This Month

### Ergo 6.0 Soft-Fork & Sub-Blocks

The 6.0 development track advanced significantly across the month. Sigma 6.0.0 was approved by kushti in W20, with Ergo 6.0.0 polishing underway and a voting start planned at block 1,548,000 (configurable via node flag). EIP-50 was finalized and polished by W22. A tree serialization issue (GitHub #2214) was identified and fixed during W21. A CPU-mining sub-blocks devnet was deployed at `213.239.193.208:9058` on the `weak-blocks` branch in W20, with P2P layer improvements and sync stability fixes pursued through W21. Multiple nodes reported getting stuck despite varying configurations; kushti identified possible fork issues in the sync protocol. Sub-blocks include ordering block announcements designed to minimise retransmission of previously broadcast transactions. SigmaChains support refactoring was flagged as the next step after 6.0 lands.

### Node Stability: RocksDB & Indexing Issues

v5.1.3 introduced RocksDB upgrades from 8.x to 9.x, causing SIGSEGV crashes and database corruption in some Docker environments (reported by Pulsarz and Pgr456 in W19). kushti could reproduce clean deployments on kernel 6.11.0-25-generic but not the failures, pointing to JRE/RocksDB native library mismatches. A separate persistent indexing bug saw two token IDs present in the ergoplatform API but absent from all tested indexed nodes (sigmaspace, ergopool, custom nodes). Log analysis revealed "Unknown box used as input" and "Skipping block" patterns. An orphan block indexing bug was identified and fixed by ⭐Fanta⭐ in W21 (token creation now verified against first input box ID). PRs #2215 and #2216 targeting these issues were included in the planned 5.0.27 release.

### Developer Tooling: Fleet SDK, ergoc & Sigma Rust

Fleet SDK shipped two notable releases: v0.8.5 (SBox constant serialization/parsing, `estimateMinBoxValue()`) and v0.9.0 (additional registers merging on `OutputBuilder`). The ergoc CLI compiler reached v0.1.2 with watch mode, full compiler flags, Base16/Base58 output, and ~55% cold compilation speedup via fullOpt sigmastate-interpreter builds. EIP-5 JSON output support was flagged as the next ergoc priority, with `SigmaTemplateCompiler` JS availability requiring further investigation. Multiple sigma-rust PRs (#820, #823, #824, #825, #826, #827, #829, #830) were reviewed and approved through the month. ErgoTree constant extraction/replacement and template extraction were added to Fleet SDK in W21. Transaction deserialization was clarified as limited to ErgoTrees with size flag enabled (plus P2PK and miner fee trees).

### Rosen Bridge

Development was active across all bridge components: Doge network integration progressed from design through testnet milestone completion by W22; rate-limiting strategy for Doge RPCs and combined/partial network approaches were tested. Bitcoin Runes UI integration reached 2/6 MRs by W22. WalletConnect (Ethereum/Binance Smart Chain) and Xverse wallet (Bitcoin/Ledger) integrations were in progress. Bitcoin multisig design was finalized in W19 with implementation pending. A Bridge Expansion Kit document was published. A technical discussion in W21 clarified that P2S addresses cannot be used as Rosen Bridge target addresses due to cross-chain data size constraints—particularly Bitcoin's 80-byte OP_RETURN limit—leaving approximately 60 bytes for address encoding after overhead.

### Dexy Protocol

LP swap/mint/redeem contracts were deployed to mainnet in W20. A payout contract was under review before remaining non-bank/non-LP contracts would be deployed. Private UI testing began in W21 with a small group (code_for_USS, c8, savonarola). All Dexy contracts were released under CC0 (public domain) license. Public release was pending a confidence threshold in the testing group, with no confirmed date by month end.

### Wallet & dApp Connectivity

Two complementary approaches to wallet↔dApp communication were explored. anon_br investigated Nostr as a transport layer, proposing a forked relay handling Ergo-specific events (`connect`, `get_utxos`, `sign_tx`) with PoW spam protection—inspired by NIP-47 (Lightning) but without Lightning-specific constraints. The Keystone hardware wallet integration (secp256k1 dependency switch, custom ErgoTree serialization keeping trees unparsed during device transfer) was completed by Pgr456 in W19 and submitted for review; end-to-end Nautilus testing was deferred until after a two-week vacation. anon_br also flagged EIP-12 as becoming "a growing Frankenstein" and proposed a new JSON-RPC protocol for wallet communication as a longer-term replacement.

### Celaut P2P Architecture & MGpai Offchain Executor

Josemi detailed Celaut's gRPC-based transport with a custom protocol for efficient service transfer (avoiding retransmission of already-present components), manual peer introduction via `connect ip:port`, and planned on-chain reputation proofs for autonomous peer discovery. MGpai announced a Go-based resilient offchain transaction executor using an actor model for planned Ergo projects.

### ChainCash / Basis & Git Circles

kushti presented the Basis design (foundational on-chain reserve contracts supporting off-chain protocols) to an interested party in W20, with an implementation plan to follow. A forum post on offchain cash options was planned. In W22, kushti published the Git Circles framework for creating community currencies around open-source projects using PoW-based programmable assets.

## Technical Q&A Worth Preserving

- **Q** (@Pulsarz, 2025-05-07): Why does v5.1.3 crash with RocksDB SIGSEGV on fresh data?
  **A** (@kushti): JAR v5.1.3 works on clean Docker states on kernel 6.11.0-25-generic. Issue is likely environment-specific — JRE version or RocksDB native library mismatch. Newer RocksDB (10.x) or alternate JRE may help; Docker build tweaks needed.

- **Q** (@Michael, 2025-05-06): Is it possible to integrate Ergo with WalletConnect?
  **A** (@Michael): No equivalent standard currently exists; most dApps use custom ErgoPayserver. A unified WalletConnect-style standard would reduce developer friction.

- **Q** (@Flying Pig, 2025-05-08): Why can't dApps use indexed nodes instead of relying on Explorer APIs?
  **A** (@kushti): They can in most cases. Explorer API was the first solution for extra data not in basic indexes; many apps remain dependent. Indexed nodes have their own issues (currently debugging missing extra indices). The real blocker is reporting infrastructure — end users don't file GitHub issues, and most dApp failures occur silently in Telegram/Discord.

- **Q** (@kushti, 2025-05-07): How is the P2P network handled in Celaut?
  **A** (@Josemi): Uses gRPC with a custom protocol for efficient service transfer, avoiding retransmission of components already present. Peer introduction is manual via `connect ip:port`. Planned: on-chain reputation proofs will enable autonomous discovery — nodes connect to peers with reputation above a threshold on startup.

- **Q** (@kushti, 2025-05-07): How could Nostr be used for wallet↔dApp communication?
  **A** (@anon_br): Fork a Nostr relay for Ergo-specific events (`connect`, `get_utxos`, `sign_tx`, etc.). Wallets act as Nostr clients. Spam prevented via simple PoW schema. Inspired by NIP-47 (Lightning) but adapted for Ergo without Lightning-specific constraints.

- **Q** (@iospore, 2025-05-08): How do AVL trees solve orderbook design? If the tree is off-chain, who updates it?
  **A** (@c8): The AVL tree serves as an off-chain list of all orders; the tree hash is stored on-chain, with updates submitted via proofs. One-UTXO-one-order enables parallel execution without centralisation. Crystal Pools example: all UTXOs are multisigned, preventing either party from unilaterally transacting; time limits prevent fund freezes but create UX friction from liquidity fragmentation.

- **Q** (@Josemi, 2025-05-21): Is it possible with Rosen Bridge to allow a Cardano box to be spent only if a transaction is sent through the bridge, depositing value and data into an Ergo box with a specific script?
  **A** (@kushti): Theoretically possible, but uncertain how far the current implementation supports it.
  **A** (@zargarzadehmoein): Theoretically possible for Cardano specifically, but P2S addresses cannot be used as Rosen Bridge target addresses for Ergo due to data size constraints across chains — particularly Bitcoin's 80-byte OP_RETURN limit, leaving only ~60 bytes for the address.

- **Q** (@arobsn, 2025-05-26): Can transaction deserialization work for arbitrary ErgoScript boxes?
  **A** (@arobsn): Only boxes with ErgoTrees having the size flag enabled (plus P2PK and miner fee trees) can be deserialized via `deserializeTransaction` due to current limitations.

- **Q** (@Luivatra, 2025-05-28): Can ergoc support EIP-5 JSON output?
  **A** (@arobsn): Yes, will implement. However, `SigmaTemplateCompiler` is unavailable in the JS output — requires closer investigation. Requested EIP-5 JSON examples to better understand constant reconstruction.

- **Q** (@HQΣr, 2025-05-18): What is the minimum box value and can it be lowered?
  **A** (@Luivatra / @Aco): Box value is a function of box size at 360 nanoERG per byte. Use Fleet SDK's `estimateMinBoxValue()` to calculate; a custom value-per-byte can be passed if needed.

- **Q** (@HQΣr, 2025-06-01): How to resolve ~1 minute lag in Explorer API mempool after transaction submission?
  **A** (@Aco / @Pulsarz): Use the node directly. Relying on the Explorer API for mempool state is bad practice; a decentralised node-direct approach is preferred.

- **Q** (@c8, 2025-05-16): How is multisig done with sigmastate-js?
  **A** (@kushti): Interesting gap — will check. If multisig aggregation support is not yet implemented, it should not be hard to add.

## Open Questions Entering Next Month

1. **Sub-blocks devnet sync stability**: Nodes getting stuck at lower heights; root cause (fork handling in sync protocol) not yet resolved. Needs further rework before production readiness.

2. **RocksDB SIGSEGV in v5.1.3**: Affects some Docker/JRE environments but not others. Testing RocksDB 10.x or alternate JRE images proposed but not yet confirmed as fix.

3. **Missing token index bug**: Two token IDs present in ergoplatform API but absent from all tested indexed nodes. "Unknown box used as input" and "Skipping block" log patterns under investigation; root cause unconfirmed.

4. **EIP-12 wallet protocol refactor**: anon_br flagged EIP-12 as a "growing Frankenstein"; JSON-RPC replacement proposed but no decision or timeline set.

5. **`SigmaTemplateCompiler` JS availability**: Required for ergoc EIP-5 output support; status unconfirmed pending investigation.

6. **Multisig aggregation in sigmastate-js**: Gap identified by c8; kushti said it should not be hard to add but no implementation committed.

7. **Nostr wallet↔dApp transport**: anon_br's Nostr relay fork for Ergo events is an early-stage idea; PoW spam threshold, event schema, and relay topology remain open.

8. **Dexy Protocol public release**: Private testing group confidence threshold undefined; public release date TBD.

9. **Keystone Nautilus end-to-end testing**: Deferred until after Pgr456's two-week vacation; no confirmed date.

10. **Explorer API mempool lag and dApp reporting infrastructure**: Recurring concern (noted by Flying Pig); no systematic solution proposed for silent failures in end-user channels.
