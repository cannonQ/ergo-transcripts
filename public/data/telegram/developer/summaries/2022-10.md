# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer | **Month:** 2022-10

---

## Executive Summary

October 2022 was a high-velocity month for Ergo protocol development, marked by a rapid release cycle culminating in v5.0.0 and multiple node updates addressing serialization soundness and protocol-level security improvements. The dominant technical theme was storage rent mechanics — specifically the discovery of creation height vulnerabilities in smart contracts and the design, implementation, and EIP process for a monotonic creation height rule to address them. Alongside protocol work, ecosystem tooling advanced on multiple fronts including Satergo, Dexy Protocol, and exploratory designs for local economy primitives.

---

## Key Topics This Month

### Storage Rent & Creation Height Vulnerability

The month's most sustained technical thread concerned how storage rent interacts with creation height. Storage rent uses the self-reported creation height stored in a box rather than the settlement height, which is by design — settlement height would make pre-inclusion box byte computation impossible and complicate contract design significantly. However, this exposed a gap: off-chain code or modified tooling could set a box's creation height to 0, making boxes eligible for early storage rent collection. The Spectrum DEX was directly affected when a community developer using a modified Sigma Rust bot inadvertently created pool boxes with creation height 0. MrStahlfelge's Mosaik-based box consolidation utility (`mosaikboxconsolidation`) surfaced this issue more broadly.

The proposed mitigation — a monotonic creation height rule requiring that output creation heights be no less than the maximum input creation height — was designed, coded, and submitted as a PR within the same week. kushti confirmed the rule covers observed cases without introducing new problems and that it would be addressed at the protocol level in the 5.0 soft fork. A related open question on what should happen to tokens in boxes with insufficient ERG during storage rent collection (miner takes tokens, tokens are burned, or tokens route to a reemission contract) remained unresolved entering November.

### Protocol Release Cycle: v4.0.x → v5.0.0

October saw an unusually active release cadence. v4.0.104 was released mid-month, followed by v4.0.105, then v5.0.0 and a rapid follow-up v5.0.2 addressing serialization soundness issues in ErgoTransaction handling. Block version 3 activates interpreter 5.0, and AppKit was confirmed to need recognition of block version 3 in the header to function correctly. PaiNet (the protocol testnet) was updated with RC3 settings. Multiple EIPs progressed in parallel, including the monotonic creation height EIP and EIP-37, though EIP-37's specific scope was not fully elaborated in chat.

### Dexy Protocol

Scalahub returned from leave and resumed work on the payout contract implementation within the Dexy Protocol. The decentralized grid trading contract was modified with node tracking rules, and off-chain trading bots were identified as the next development phase. Oracle Pool v2 running for Dexy Gold with GORT rewards was noted as an active deployment.

### Satergo & stratum4ergo

Aberg released Satergo design beta 2 and announced the creation of `stratum4ergo` — a new library and tool for solo mining pool server infrastructure, intended for integration into Satergo. This represented a significant expansion of Satergo's scope beyond wallet functionality into mining infrastructure.

### Local Economy Wallet Concept

Community member koukarin proposed a Local Economy Wallet design that applies local fees when ERG or SigmaUSD transfers occur between cities, incentivizing local commerce. kushti endorsed the concept as viable and noted it could be combined with LETS, ChainCash, and potentially L2 implementations. The proposal drew a connection between several of Ergo's financial primitive designs and real-world local economy applications.

### Explorer Indexing Failure

Late in the month, block height 869219 caused explorer backends to stall with a `ClassCastException: [J cannot be cast to [B` data type mismatch. Nodes on all active versions (4.0.103, 4.0.105, 5.0.2) continued producing blocks normally, confirming the issue was isolated to the explorer's deserialization layer. No resolution was reached within the month.

## Technical Q&A Worth Preserving

- **Q** (@Luivatra, 2022-10-04): Is storage rent calculated using creation height or settlement height?
  **A** (@kushti): Storage rent uses self-reported creation height stored in the box. Settlement height would break useful box properties — you couldn't know box bytes prior to inclusion — and would complicate contract design significantly.

- **Q** (@Luivatra, 2022-10-04): If decentralized off-chain code allows anyone to spend, how do we prevent malicious creation height being set to 0?
  **A** (@kushti): Creation height is stored in the box and protected by transaction signatures.

- **Q** (@Luivatra, 2022-10-04): Can contracts protect against malicious creation height manipulation using `outbox.creationHeight > self.creationHeight`?
  **A** (@kushti): Yes, this is possible. (@MrStahlfelge): This should be standard practice — like checking `SELF.id` — but most contract developers are currently unaware of the requirement.

- **Q** (@Luivatra, 2022-10-05): Can a miner taking storage rent set the resulting UTXO's creation height to 0 and immediately collect rent again?
  **A** (@kushti): No. The protocol requires the miner to set the inclusion block height as the creation height on the updated output; precise timestamping is enforced in this case.

- **Q** (@白白 🍑, 2022-10-15): How many TPS does Ergo achieve, and can it compete with Ethereum's claimed throughput?
  **A** (@Luivatra): High TPS is the domain of layer 2 and does not make sense to pursue at layer 1 where it causes bloat. Ergo's layer 1 is not designed for throughput maximization.

- **Q** (@HQΣr, 2022-10-26): How can real-time token prices from the Spectrum API be fetched for ergcube integration?
  **A** (@MrStahlfelge): A reference implementation is available in the ergo-wallet-app repository under the ergodex API package.

- **Q** (2022-10-26): What is the maximum transaction size configuration for nodes?
  **A** (@kushti): You can set `maxTransactionSize = 200000` in the node config, but other nodes enforce standard limits. This setting is recommended for mining nodes only.

- **Q** (2022-10-26): Why was the token limit per box set to 100?
  **A** (@kushti): The protocol allows up to approximately 4KB per box. Wallet libraries (ergo-wallet lib) conservatively enforce 100 tokens per box; the protocol itself permits higher counts per box and approximately 500 tokens across multiple boxes in a wallet.

## Open Questions Entering Next Month

- **Token handling during storage rent**: What happens to tokens in boxes with insufficient ERG when storage rent is collected? Options (miner takes tokens, tokens burned, tokens routed to reemission contract) were discussed but not resolved.
- **EIP-37 scope and status**: Mentioned by kushti as under active review but not elaborated in detail; outcome unclear.
- **Explorer data type mismatch at block 869219**: The `ClassCastException` in the explorer backend remained unresolved at month end; root cause analysis and fix pending.
- **AppKit serialization breaking changes**: Multiple breaking changes in ErgoTransaction serialization were flagged by kushti; full scope of rollback and fixes not detailed.
- **Monotonic creation height rule edge cases**: Whether the `outbox.creationHeight >= max(input.creationHeight)` rule covers all edge cases — particularly for contracts accepting additional inputs without explicit creation height checks — was acknowledged as potentially incomplete.
- **L2 implementation timeline for Local Economy Wallet**: kushti suggested L2 as the preferred path for reducing mining fees in the design, but no concrete roadmap or timeline was discussed.
- **Payment portal standardisation**: Multiple implementations exist (thierrym1212's version, others on Discord); no canonical recommended implementation was established.
