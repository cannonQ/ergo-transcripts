# Ergo Developer Chat — Monthly Summary
**Month:** 2024-05 | **Chat:** Ergo Developer

---

## Executive Summary

May 2024 was dominated by Sigma 6.0 protocol development, with the sigmastate-interpreter receiving significant new capabilities including `UnsignedBigInt`, `executeFromVar`/`executeFromReg`, unified `DataSerializer`-based serialization, and `Option[T]` support. Alongside core protocol work, the Ergo node v5.0.22 release was prepared with mempool spam mitigations and orphan transaction cleanup, while multiple ecosystem projects (Rosen Bridge, Paideia, Crystal Pool, Analog Ergo atomic swaps) reached significant milestones. ErgoHack VIII was announced and concluded, with trustless Bitcoin relay contracts as a key theme.

---

## Key Topics This Month

### Sigma 6.0 Protocol Development
The month's heaviest technical focus. New capabilities merged or in final review include: `UnsignedBigInt` (256-bit unsigned integers for secp256k1 field arithmetic, Schnorr signature validation, and Bulletproofs verification); `executeFromVar`/`executeFromReg` (P2SH-style execution enabling Merkleized Finite State Machines and optimistic verification schemes like FairSwap); unified `DataSerializer`-based (de)serialization replacing fragmented type-specific serializers; `Option[T]` (de)serialization; `Numeric.toBigEndianBytes`; Header serialization for Lithos Protocol integration; AVL tree serialization; and `SubstConstants` size-bit fix for ErgoTree versions > 0. EIP-50 was updated throughout the month with outstanding issues and bounties. Activation is targeted for approximately November 2024, per Lithos team request. Post-6.0 work identified includes generic Sigma protocol compression and in-ErgoScript Bulletproofs verification.

### Ergo Node v5.0.22 & Mempool Improvements
PR #2156 introduced automatic removal of mempool transactions spending already-confirmed UTXOs (orphan cleanup), resolved after identifying that the default `mempoolCleanupDuration` was too high. OVH anti-DDoS false positives from inter-node requests were investigated; blacklisting OVH subnets was recommended. PRs #2146, #2156, #2158, and #2162 were merged or approved through the month. A fee estimation API exists on the node but was noted as untested and unused by applications — identified as a gap worth addressing.

### Rosen Bridge Multi-Chain Expansion
Bitcoin RPC scanner integration was completed, including a Bitcoin Observation extractor and Rosen Extractor (with `fromAddress` changed to box ID for RPC compatibility). Bitcoin Runes support was actively designed across four candidate solutions, narrowed to two under cost optimization. Guard-service TSS key configuration was revamped, scanner extractor initialization status now persisted to database, and abstract Ergo extractors developed to unify format across chains. A keygen-service was prepared for ECDSA key generation ceremony; BTC bridge launch testing began at month end. RoseNet pub-sub implementation was completed.

### Mempool Spam Resilience
A thorough discussion early in the month covered Ergo's fee market mechanics, mempool timeout behaviour, spam attack precedents (Bitcoin block size war, 2020 mixer activity), and node-level defences. Key points: transactions not picked up within many blocks eventually time out and are dropped; node operators can raise minimum propagation fee rates to prevent spam from entering the mempool; fee estimation API exists but is not yet used by wallets. The v5.0.22 orphan cleanup and existing reference client limits (no more than N transactions from a single peer) were highlighted as existing protections.

### Sub-blocks and GHOST Algorithm
Debate on whether GHOST (Greedy Heaviest-Observed Subtree) fork-choice rule should be adopted for sub-blocks. kushti confirmed GHOST is not in the current draft proposal (which uses longest-chain rule as a velvet fork). The discussion was formally deferred to post-Sigma 6.0 community discussion around Ergoversary. Sub-blocks remain backward-compatible (velvet fork) and do not require a full protocol upgrade. Target implementation window: late June, contingent on contributor availability.

### ErgoHack VIII
ErgoHack VIII (May 25 – June 2, 2024) was co-hosted with CKB and targeted the BTC/Nostr communities. The selected technical theme was trustless relay contracts for Bitcoin and SigmaChains sidechaining, along with Bitcoin hashrate derivatives. kushti worked on Bitcoin transaction inclusion proof validation contracts during the hackathon; ross-weir published relay contracts to GitHub. An ErgoHack updates thread was posted on Reddit.

### Crystal Pool (L1 Order-Based DEX)
A new L1 real-time order-based DEX project reached a significant milestone. Buy and swap contracts completed with vitests; server-side unsigned transaction validation workflow implemented; in-memory contract UTXO database and pool service APIs (`/boxes`, `/swap-transactions/create`, `/swap-transactions/sign`) deployed. A forum post was published explaining the concept, security model, and risks. Positioned as an order book alternative to AMM-based DEXes, building on the same design patterns as Trade House.

### Paideia DAO Platform
EIP-5 contract template standard integration was completed with a 90% test runtime reduction via automatic ErgoScript recompilation on file change. The Create DAO frontend was finished and beta DAO creation reopened despite an unresolved intermittent backend halt bug. Error logging improved with repeat counters. Soft launch target set for end of June / early July. An EIP-5 guide with SigmaTemplateCompiler was published; JS and Scala.js support confirmed; AppKit issue opened for parity.

### Analog Ergo Atomic Swaps
Swap state reloading with client-side minimization became functional. UI-REST API interactions were reduced for backend state management. An Ergo↔BTC-Lightning POC was initiated using regtest/testnet, with preimage processing and contract integration in progress. Custom price feed system for optional market making/taking was planned.

### Sigma-rust Performance & Release
sigma-rust 0.27 was prepared for release with 300–400x performance improvements in extreme cases, transaction validation support, and completed C/Swift/Go bindings for ExtSecretKey, ExtPubKey, and DerivationPath. The signing-with-900-inputs performance problem (30+ minutes) was acknowledged as a known limitation of per-input context recreation; a long-term fix was not yet ready. Pre-built shared libraries on GitHub releases were proposed to simplify adoption.

### ErgoDevs DAO
A Telegram group was established and a multisig was set up and tested. Over 7,000 ERG in donations was received. Leadership structure identified: kushti (R&D + Scala), dusek (Rust). Open positions: Tooling Lead, Infrastructure Lead, Treasury/Fundraising, Business Development, Marketing, Education.

## Technical Q&A Worth Preserving

- **Q** (@ergomorphic, 2024-05-13): Is the `6.0-deserialize` branch buildable?
  **A** (@kushti): Yes, all new tests passing. `sbt clean` resolves stale build artifacts.

- **Q** (@ergomorphic, 2024-05-13): Why do we need both `deserializeRaw` and `deserializeTyped`?
  **A** (@kushti): `deserializeRaw` is needed for round-trip property (e.g., `Box.bytes` → `deserializeRaw` → `Box`). `deserializeTyped` use case is unclear; recommendation is to use a single unified format via `DataSerializer` instead.

- **Q** (@luivatra, 2024-05-14): Would `getVar(inputIndex, varId)` help avoid redundant proof resubmission in multi-input contracts?
  **A** (@kushti): Yes — created bounty issue #983 for v6.0.

- **Q** (@luivatra, 2024-05-14): Can we construct `Option[Long]` from computed values, e.g., `Some(R5[Long].get + R6[Long].get)` as a function parameter?
  **A** (@kushti): Will explore `Some()`/`None()` constructors at ErgoTree or compiler level alongside the bytes/deserialize pattern.

- **Q** (@dusek_, 2024-05-11): Is recursion possible with `executeFromVar`?
  **A** (@kushti): No recursion during substitution — `executeFromVar` works like macros (frontend synonym for `DeserializeContext` ErgoTree node). Macro expansion is non-recursive and happens before AST reduction.

- **Q** (@Pulsarz, 2024-05-09): A short example of `executeFromVar` would be useful.
  **A** (@luivatra): `Blake2b256(SELF.propositionBytes) == getVar[Coll[Byte]](0) && executeFromVar(getVar[Coll[Byte]](0))` — hash should wrap `getVar` and compare to a hardcoded value for P2SH-style validation. Add a condition on current state to obtain a Merkleized FSM contract.

- **Q** (@luivatra, 2024-05-20): Is it intentional that `substConstants` does not include the size/content-length uint in output even if input bytes define it?
  **A** (@ergomorphic): No, this was overlooked — good catch. Issue created for v6.0; should be straightforward to fix. (Issue #994)

- **Q** (@luivatra, 2024-05-20): What is the difference between using ErgoTree version 0 and version 2?
  **A** (@ergomorphic): For version > 0, size bits are required and must be serialized. Currently no functional difference, but this may change with v6.0, so dApps should be explicit about which version they create.

- **Q** (@dΣathgripson, 2024-05-22): Does `UnsignedBigInt` mean ErgoScript is no longer limited to signed 256-bit values?
  **A** (@kushti): The existing `BigInt` type enforces signed 256-bit values with ≤255-bit operation results. For Schnorr validation, off-chain code must discard values requiring 256 bits (as done in ChainCash and Rosen). `UnsignedBigInt` will require non-trivial tests covering Sigma protocols and Bulletproofs verification.

- **Q** (@kushti, 2024-05-22): How does 512-bit support help with secp256k1?
  **A** (@ergomorphic): Any 256-bit integer is also a 512-bit integer, so you can represent q and p, then compute `x mod q` within 512 bits. Cost parameters must be profiling-based to avoid mispriced exponentiation attacks (per the Ethereum 2016 precedent).

- **Q** (@Kartik, 2024-05-22): Where is the Autolykos2 top-level function implemented, and are unit tests available?
  **A** (@kushti): Specification: https://www.docdroid.net/mcoitvK/ergopow-pdf. Implementation: `AutolykosPowScheme.scala` in ergo-core. Tests: `AutolykosPowSchemeSpec`.

- **Q** (@Pgr456, W18): Signing a transaction with 900 inputs takes 30+ minutes in sigma-rust. Does performance scale with input count?
  **A** (@dusek_): Confirmed — sigma-rust recreates context per input and does not scale well. Long-term fix in progress but not ready. "Not enough witness for prover" error occurs above approximately 900 inputs.

- **Q** (@Flying Pig, W18): Can the mempool be cleared of transactions spending already-confirmed UTXOs via API?
  **A** (@kushti): The node should do this automatically but doesn't due to missing indices. Implemented in PR #2156 — will be in v5.0.22.

- **Q** (@ergomorphic, W18): Why was the GHOST fork-choice rule not selected for Ergo? Is it relevant for sub-blocks?
  **A** (@kushti): GHOST was not considered in the current draft, which uses longest-chain rule as a velvet fork. GHOST can be proposed as an alternative on top of the sub-blocks branch; discussion deferred to post-Sigma 6.0 Ergoversary community chat.

- **Q** (@andrei_krx, 2024-05-14): Ergo node v5.0.21 triggers OVH anti-DDoS 10–20 times daily — what causes it?
  **A** (@kushti): Nodes request other OVH nodes, which triggers false "port scanning" detection. Recommended fix: blacklist OVH subnets in firewall.

- **Q** (@jossemii, 2024-05-16): Does Ergo adjust difficulty per block or in epochs? Why didn't it adjust during the testnet halt?
  **A** (@Pulsarz): Difficulty adjusts every epoch (128 blocks). Long gaps between miner activity prevent timely adjustment.

- **Q** (@justblake3911, 2024-05-07): How to sign a transaction with a wallet holding 0 ERG but 10,000 custom tokens using Babel Fees?
  **A** (@anon_BR, 2024-05-08): Babel boxes must be fetched by filtering the blockchain using the Babel ErgoTree contract with the target token ID as parameter — query the contract's hex representation (with token ID substituted) against `/boxes/unspent/byErgoTree`. Reference EIP-0031 for the exact contract hex.

- **Q** (@HQΣr, 2024-05-29): How to convert a public key to an address and vice versa?
  **A** (@Aco, FleetSDK example):
  ```javascript
  // Public key → Address (network type 0)
  let address = fleetSDK.ErgoAddress.fromPublicKey('029...81', 0);
  
  // Address → Public key
  function addressToPubkey(address) {
    let ergoAddress = fleetSDK.ErgoAddress.fromBase58(address);
    return fleetSDK.SSigmaProp(
      fleetSDK.SGroupElement(ergoAddress.getPublicKeys()[0])
    ).toHex();
  }
  ```

- **Q** (@deathgripson, 2024-05-29): Why does `fromBase64` work but `fromBase16` throws "not found in env"?
  **A** (@kushti): Using an older compiler version — `fromBase16` was introduced recently. Update to the latest compiler version.

- **Q** (community, W18): What mempool spam protections does Ergo have?
  **A** (@kushti): (1) Transactions time out and are dropped if not picked up after many blocks. (2) Node operators can raise minimum propagation fee rates to prevent spam from entering the mempool. (3) Reference client limits transactions accepted from a single peer. (4) PR #2156 adds automatic orphan cleanup. A fee estimation API exists on the node but is not yet used by wallets — worth fixing.
