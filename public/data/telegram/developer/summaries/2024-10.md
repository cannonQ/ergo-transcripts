# Ergo Developer Chat — Monthly Summary
**Month:** October 2024

---

## Executive Summary

October 2024 was dominated by the steady march toward Ergo 6.0 protocol activation, with multiple PRs merged across sigmastate-interpreter and the reference client, culminating in the 6.0.0-alpha2 devnet release. Wallet infrastructure saw significant parallel progress — Nautilus gained 0-conf support and Ledger Stax/Flex integration, Satergo shipped v1.8.0 with new tooling, and Keystone hardware wallet integration began. Ecosystem projects including Paideia, Rosen Bridge's RoseNet P2P layer, Sigmanauts Mining Pool, and MewFinance all reached meaningful milestones heading into ErgoHack 9.

---

## Key Topics This Month

### Ergo 6.0 Protocol Development

The dominant technical thread of the month. kushti landed or reviewed a large batch of PRs including `Option` type constructors (`Some`/`None`), unsigned BigInt support, `toBytes`/`fromBigEndianBytes` numeric operations, and AVL+ tree semantics. The sigmastate-interpreter repository was migrated to the ergoplatform GitHub organisation, consolidating development infrastructure. The 6.0.0-alpha2 reference client was released for devnets only, with activation details — including a voteable sub-blocks-per-block parameter (range 2–2048, default 30, implying ~58ms theoretical minimum block time) — advancing through the final polishing phase. Sigma Rust received its first 6.0-targeted PR (#773). Activation proposal finalisation remained pending completion of miner voting mechanics and full test coverage at month end.

### Sub-blocks Architecture

Sub-blocks were a recurring technical discussion throughout the month. The design distinguishes *input blocks* (carrying transactions, earning fees and storage rent) from *ordering blocks* (providing canonical transaction ordering, earning a constant reward). The voteable parameter allows miners to adjust sub-blocks-per-block post-activation without a hard fork. Nodes below version 6.0 will skip unknown parameters and remain functional on-chain. The sustainability of ordering-block rewards after tail emission was raised but not conclusively resolved. A dedicated testnet channel was opened for parameter-tweaking experiments; kushti invited interested community members to lead empirical studies.

### Nautilus Wallet: 0-Conf Support & Ledger Integration

anon_BR shipped v0.15.0-beta.0 with zero-confirmation transaction support, initially released to the Abyss channel pending Google Chrome Web Store Manifest v3 review. Separately, the first implementation of Ledger Stax and Flex device support was completed in ledger-app-ergo PR #19, with memory optimisations still outstanding. Ledger Nano S support was formally removed at Ledger HQ's request; the token limit per transaction was raised from 20 to 100 (applicable to newer app versions). Ergo Foundation and ErgoDevs DAO jointly raised the Keystone integration bounty to 6K ERG total, with pgr456 receiving a Keystone 3 PRO device and making initial progress on address generation and transaction signing.

### AppKit API Improvements

Aberg identified and addressed several long-standing AppKit issues: arrays vs. lists inconsistency, non-empty token list restrictions breaking the builder pattern, obsolete explorer-api and node-api client dependencies causing 5-minute lifecycle failures, and the unnecessary java.sql module inclusion. Multiple PRs were opened and reviewed throughout October. A key design decision — to keep old varargs methods alongside new `List<?>` alternatives without deprecation, preserving backward compatibility — was confirmed by kushti. A Scala compiler infinite-loop bug triggered by varargs array conversion code was also identified. Ergonnection v0.0.1 (Aberg's p2p node communication library) was held pending a sigmastate-interpreter release expected in November.

### Paideia DAO Deployment

After resolving a context variable ordering bug (JSON serialisation corrupting variable order from the Scala side), Luivatra confirmed a max-4-context-variable workaround with passing test cases. Updated contracts were deployed on mainnet with proposal creation and voting successfully tested. The Sigmanauts DAO was created as a free test instance. Web2 backend went live; frontend deployment was pending parallel beta operation for treasury migration support at month end.

### Rosen Bridge — RoseNet P2P Layer

Armeanio issued a community call for globally distributed test nodes (8 GB RAM, 8+ CPU cores preferred) for a ~24-hour decentralised fault-tolerance validation run of the new RoseNet P2P communication layer. Separately, Guard and Watcher apps continued being updated to align with the latest API changes. Watcher v2.2.1 was the recommended upgrade path (`docker compose pull` → `docker compose down` → `docker compose up -d`). A manual TX was executed to refund slashed permits from an accidental high-fee transaction that failed due to a missed signature.

### Sigmanauts Mining Pool

The Miner ID system vote passed at 3 ERG cost, enabling miners to customise payout tokens (ERG, rsBTC, rsADA, CYPX, Gluon, and others). The Mining Swap backend became operational, with the 11.11 release targeting live token swaps. Token Flight vesting contracts (linear, quadratic, exponential, logarithmic) were tested on testnet by Marc the Shark and integrated as the bonus reward distribution mechanism. The pool addresses approximately 20% of 2miners pool participants who prefer BTC over ERG returns.

### SigmaUSD Bank Contract Update

kushti discovered that the SigmaUSD update contract box NFT is vulnerable to storage rent collection in approximately 7 months. A forum voting thread was opened on contract update options including potential key burning. A majority of votes favoured introducing limits; the implementation path for updating the bank (and possibly pool) contracts on testnet was being assessed at month end.

### Ecosystem Projects: Celaut, MewFinance, Crux Finance

jossemii's Celaut project (repository renamed "paradigm") advanced through multi-node testing, with a P2P service execution model using containerised binaries dispatched via Nodo nodes. kushti confirmed the architecture is suitable for Layer 2 digital bearer certificate passing. MewFinance's HQΣr launched the Community Liquidity Bootstrap (CLB), bridged Cardano native tokens to Ergo via MewDex, and began implementing verified collections (EIP standards) on MewMart. Crux Finance fixed a PostgreSQL null character (`0x00`) handling bug in token description fields and identified a Paideia off-chain vote replay bug.

## Technical Q&A Worth Preserving

- **Q** (@Flying Pig, 2024-10-03): How do you force a node reindex?
  **A** (@kushti): Stop the node, remove the `/history/extra` folder from the data directory, restart. An API method for this would be a useful addition.

- **Q** (@Moein Zargarzadeh, 2024-10-01): Nautilus clients getting errors on new versions.
  **A** (@anon_BR): Issue is likely an outdated GraphQL server instance behind the load balancer. Point clients to `https://explore.sigmaspace.io/api/graphql` as a temporary workaround.

- **Q** (@jossemii, 2024-10-06): How to determine if a node is on testnet or mainnet?
  **A** (@Dmitry Usov): Check `http://<node-ip>:9052/info` and inspect the **Network** field in the response.

- **Q** (@jossemii, 2024-10-06): Where does Nautilus specify which node it uses?
  **A** (@Aberg): Nautilus does not use a node directly — it uses only a GraphQL endpoint and the Explorer API. Transactions are submitted via the Explorer API, which routes to the Explorer's own node.

- **Q** (@glasshouse, 2024-10-19): How do you confirm a node is fully synced? What do `fullHeight`, `height`, and `headersHeight` mean?
  **A** (@Aberg): You must compare `headersHeight` **and** `fullHeight` against `maxPeerHeight` from the network. `fullHeight == headersHeight` alone is insufficient — both must equal `maxPeerHeight`. (@Michael): An indexed node is approximately 45 GB.

- **Q** (@wxip, 2024-10-16): Is `window.ergoConnector.nautilus.connect()` sufficient for user verification?
  **A** (@anon_BR): No — a signed message must be verified against the user's public key. Connection alone provides no proof of key ownership.

- **Q** (@jossemii, 2024-10-19): How to encode a tuple of two strings in FleetSDK?
  **A** (@Aco): Use `SPair`. (@c8e4d2a): Use the `SColl(SColl, [1, 2])` pattern. (@anon_BR): Test vectors covering `SPair` usage are in the FleetSDK serializer test vectors file. Note: `STuple` with more than 2 items is not supported.

- **Q** (@jossemii, 2024-10-20): `SPair` is blocking transaction build with `R5: SConstant(SLong(1n))` error.
  **A** (@anon_BR): Do not wrap `SPair` with `SConstant`. (@Aco): Use `SPair(SLong(BigInt(...)), SLong(BigInt(...))).toHex()` directly.

- **Q** (@Paul, 2024-10-17): Do sub-blocks have mining rewards without a hard fork or on-chain contracts?
  **A** (@cafebedouin): Miners earn transaction fees and storage rent from input (sub-)blocks. Ordering blocks earn a constant reward, per the Ergo scalability forum post.

- **Q** (@Marc the Shark, 2024-10-16): AppKit-generated address differs from node wallet address for the same seed phrase.
  **A** (@Aberg): The node wallet may be from a legacy bugged implementation. Newly created wallets should not exhibit this discrepancy.

- **Q** (@Jeanmay, 2024-10-28): How to implement Ergo wallet login in an Android app?
  **A** (@mgpai): Use ErgoPay (EIP-20). Documentation and tutorial links available at the Ergo docs site.

- **Q** (@Luivatra, 2024-10-30): Sending `ModifierRequest` with `ErgoTransaction` type crashes socket connection; `Header` type works fine.
  **A** (@Aberg): Identified a deserialization error in byte-skipping logic (strict vs. lenient validation). Fixed by reverting to lenient skip; also uncovered misinterpreted socket exceptions.

- **Q** (@kushti, 2024-10-28): How does Nodo relate to Celaut? What is a service interface?
  **A** (@jossemii): Nodo is a Celaut node implementation. Services are containerised binaries specifying a file system, entry point, port interfaces, and network access rules. Nodes execute services locally or delegate via RPC `StartService()`. Load balancing follows supply/demand; no consensus is required. Services run per-user and default to blocking external connections unless explicitly specified.

- **Q** (@ioannisleontiadis, 2024-10-02): Master's thesis project on decentralised online tutoring with Ergo — seeking guidance.
  **A** (@kushti): Willing to help. Research paper ideas posted on Ergoforum; manifesto and technical resources (eUTXO, ErgoScript, proof-of-work blockchain) available for study.

## Open Questions Entering Next Month

1. **6.0 Activation Proposal Finalisation**: Miner voting mechanics, activation block/timestamp, and full test coverage still pending. Sub-blocks input vs. ordering block processing ("working slowly") has no confirmed completion date.

2. **Ordering Block Reward Sustainability**: The mechanism sustaining ordering block rewards after tail emission ends was raised but not explicitly resolved in forum or chat discussions.

3. **Ledger Memory Optimisations**: Stax/Flex implementation complete but memory optimisations in ledger-app-ergo PR #19 still outstanding before production release.

4. **Keystone Integration**: pgr456 has address generation working but wallet connection and transaction signing remain incomplete; awaiting further technical detail from the Keystone team.

5. **Sigma Rust Swift README (Issue #759)**: Blocked iOS development for 3+ months; pgr456 acknowledged but gave no ETA.

6. **AppKit Ergonnection Maven Release**: Held pending sigmastate-interpreter release "hopefully in November" — no confirmed date.

7. **SigmaUSD Bank Contract Update**: Majority vote favours limits; implementation path for updating bank and pool contracts on testnet not yet mapped out.

8. **Dexy Mainnet Deployment**: LP audit under review; whitepaper update needed before redeployment; no timeline set.

9. **RocksDB Performance (5.1.0)**: Slow performance identified in testing; DB settings investigation ongoing with no resolution reported.

10. **Scala/JDK Compatibility (Satergo/ProGuard)**: ProGuard generates invalid bytecode under JDK 23+; workaround is building with JDK ≤22; no upstream fix filed or confirmed.

11. **Satergo Wallet Format Security Hardening**: Password iteration strategy for brute-force prevention still under research; completion date unclear.

12. **Nautilus 0-Conf UX**: Zero-confirmation marked experimental; full backend implementation and design/flow finalisation deferred.

13. **Node/Wallet Infrastructure Decentralisation**: Single server outages cascading across dependent services flagged by multiple contributors; no auto-failover strategy or formal wallet endpoint-rotation spec yet defined.
