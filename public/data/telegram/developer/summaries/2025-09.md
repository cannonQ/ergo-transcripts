# Ergo Developer Chat — Monthly Summary
**Month:** September 2025 | **Chat:** Ergo Developer

---

## Executive Summary

September 2025 was dominated by active protocol R&D on the Ergo 6.0 soft-fork and sub-blocks (input/ordering blocks), with kushti driving simultaneous work on p2p layer fixes, test infrastructure expansion, and new financial primitives. The month saw significant ecosystem tooling progress across Hermes, Nautilus, Rosen Bridge, and Gluon Gold, while new economic primitives — the USE stablecoin, ChainCash/Basis contracts, and a proposed perps/LST protocol — entered early deployment or design phases. AI-assisted development (LLM test generation, MCP servers, coding agent governance) emerged as a recurring theme across both core protocol and dApp development.

---

## Key Topics This Month

### Sub-blocks (Input/Ordering Blocks) and 6.0 Protocol Soft-Fork

Sub-blocks were the central technical focus throughout September. kushti reworked the p2p layer to use INV-based announcements and differential mempool syncing, but identified early that propagation aggressiveness was insufficient for input blocks. Multiple bugs in the 6.0 alpha were identified and fixed across the month: a serialization error in the input block transaction announce message caused peer banning (fixed in commits 389c9f5 and 75fcc0c), and duplicate block header delivery was resolved. By late September, PR #1069 (sub-blocks design in SigmaSDK) and PR #2239 (minor 6.0 improvements) were under active review. Sub-blocks will be implemented as a voteable parameter (number per block), requiring a whitepaper, community discussion, and miner adoption before activation — with no sub-blocks possible if clear community disagreement emerges.

### P2P Layer and DevNet Testing

Patato served as the primary devnet tester throughout September, encountering a recurring series of issues: peer blacklisting at ~300ms latency, nodes stuck at sub-seed-height despite full DB sync, and sync decoupling after extended mining runs. kushti iteratively diagnosed and fixed root causes, deploying snapshot builds (e.g., `6.0.1-297-af443163-SNAPSHOT`, `6.0.1-320-4c6fd015-SNAPSHOT`) and periodically resetting the seed node's peer database. By end of September, mining-related desyncs remained under investigation and Patato had paused testing pending return from travel (~2025-09-22).

### ChainCash / Basis Contract

The Basis contract design was substantially reworked to focus on micropayments and p2p network incentivization using minimally trusted trackers that record mutual debt off-chain. On-chain contract-based redemption prevents double-redemptions, and trackers commit state digests to chain periodically. A raw demo of the basis tracker server was completed by late September. The ChainCash Telegram group (https://t.me/chaincashtalks) was created for roadmap sharing and contract testing. A Rust developer is actively being sought to build the full tracker backend; no assignment was confirmed by month end.

### Hermes — Real-Time Box Streamer

anon_br's Hermes project (https://github.com/arobsn/hergmes) was introduced as a real-time, mempool-aware box streamer built on top of the indexed node. Initially designed for Machina Finance, it is intended to be generic enough for any dApp and as a potential GraphQL replacement on Nautilus. Development continued through the month with the addition of an Ergo node client and mempool tracking.

### Nautilus Wallet and Fleet SDK

anon_br migrated Fleet SDK's build system from tsup to tsdown, adding IIFE export support for direct browser access. Nautilus Wallet gained Javanese and Malaysian (Melayu) language support (PRs #316, #317) via community contributors. A Ledger connection issue was under investigation but unresolved by month end. Work on sub-blocks API support in wallet endpoints was noted as a future requirement.

### Rosen Bridge UI

The Rosen Bridge UI saw steady incremental improvements: the Event List page was updated to use the latest UI-Kit, the guard app dashboard was refactored into networks and health sections, and ESLint configuration was standardized across guard-app, rosen-app, and watcher-app.

### Gluon Gold Frontend

Multiple desktop/mobile UI improvements were shipped, and a 14-day transmute volume chart was added. A swap checking restriction was reduced to 75% pending fusion. Two wallet connection issues (GitHub #50, #51) required deeper investigation and blocked the frontend release at month end.

### AI-Assisted Development: LLM Test Generation and MCP Servers

kushti generated ~360 lines of Sigma code using LLM assistance within hours for ErgoScript semantics test coverage. PR #2242 established formal coding agent guidelines: AI agents (e.g., Crush) are permitted for test generation (with ~50% of meaningless output requiring removal) but prohibited for non-testing production code. Separately, Marc the Shark demonstrated an MCP server supporting agent-driven tasks such as crossing Rosen Bridge and purchasing from MewMart, with a proposal to extend this pattern to other ecosystem APIs.

### USE Stablecoin and Oracle Pools

The USE stablecoin (part of the Dexy ecosystem, based on DexyUSD) was announced for deployment during the week of September 23, with initial token issuance planned. A new USE Telegram group was created (https://t.me/USE_stablecoin). Oracle Pool v1 is currently used in SigmaUSD and Raffle v1; Oracle Pool v2 runs for Dexy Gold with GORT rewards. kushti expressed intent to fund native Ergo oracle expansion using capital raised through the Palmyra token sale, noting that only ~2 oracle pools are currently running and more tooling is needed.

### GitCircles

GitCircles migrated from secp256k1 to Ergo addresses (PR #3 merged), implemented token issuance and payment distribution (PRs #5, #6), and made the first contributor payment to bro_etiket for merged PRs. Implementation plan improvements continued through the month.

### RocksDB Migration (Node v6.1)

Pulsarz confirmed that Node v6.1 migrates from LevelDB (v6.0) to RocksDB and has been stable for months, with Docker images available. When booting v6.1.0 from scratch, LevelDB folders (`ldb_main`, `ldb_undo`) still appear in `.ergo/state` as expected behavior per the migration path.

### Data Inputs Cost and Transaction Limits

kushti clarified that data inputs cost 100 nanoERG per input (`dataInputCost` network parameter), making 1,000 inputs feasible but cumulatively costly. Maximum single-transaction inputs under Schnorr signature (p2pk) is estimated at 500–600 due to transaction size limits. Raising `dataInputCost` after 6.0 activation via miner voting was recommended for future consideration.

## Technical Q&A Worth Preserving

- **Q** (@kushti, 2025-W36): What is Hermes?
  **A** (@anon_br): Real-time and mempool-aware box streamer on top of the indexed node; initially designed for Machina Finance but generic for any dApp; can replace GraphQL on Nautilus. Repo: https://github.com/arobsn/hergmes

- **Q** (@kushti, 2025-W36): What is the MCP server and how can it be used in the Ergo ecosystem?
  **A** (@Marc the Shark): MCP server is live on git; can be used for agent trading, crossing Rosen Bridge, MewMart purchases, and various dApp tasks if APIs/code tools are provided. Rosen, Mew, and other ecosystem MCPs can be built to enable LLM tool execution.

- **Q** (@Kubele, 2025-W38): When can sub-blocks be fully live?
  **A** (@kushti): After whitepaper publication and community discussion/approval. In case of clear objections, there will be no sub-blocks. Miners then adopt them or not.

- **Q** (@Josemi, 2025-W38): Is there a limit on data inputs per transaction, and do 1,000 data inputs incur significant cost?
  **A** (@MGpai / @Pgr456): `dataInputCost` is 100 nanoERG per input per network parameters — cumulative but not prohibitive. Transaction size limits also apply.

- **Q** (@Josemi, 2025-W39): What is the maximum number of inputs that can be spent in a single transaction?
  **A** (@kushti): Depends on the protecting condition. For Schnorr signature (p2pk address), approximately 500–600 inputs are feasible.

- **Q** (@yeahokai, 2025-W39): How can development costs be bootstrapped without creating a new token?
  **A** (@HQΣr): Multiple options exist: (1) use MewFinance fund with proportional token distribution, then lock those tokens on MewLock with APR; (2) alternatives available via Stability Nexus' Beene or other mechanisms.

- **Q** (@Luca D'Angelo, 2025-W38): How do you convert block height to timestamp?
  **A** (@HQΣr / @Aco): Ergo Watch Swagger API provides endpoints for this. Example: `https://api.ergoplatform.com/api/v1/blocks?limit=1&offset=1434699&sortBy=height&sortDirection=asc`

- **Q** (@Luca D'Angelo, 2025-W39): Is RocksDB already available in a node release, and are Docker images available?
  **A** (@Pulsarz / @Pgr456): Yes — use v6.1 builds. v6.0 uses LevelDB; v6.1 uses RocksDB and has been stable for months. Docker image available at Docker Hub (`ergoplatform/ergo:v6.1.0`).

- **Q** (@Luivatra, 2025-W37): Has the `extraIndex` config changed in v6?
  **A** (@kushti): Config appears correct; extra index starts only after full sync completes — likely was awaiting initial sync.

- **Q** (@Michael, 2025-W37): Will wallets be able to update (balance, transactions) as fast as sub-blocks are deployed?
  **A** (@HQΣr / @kushti): Yes, if wallet endpoints support sub-blocks; the rest depends on individual wallet implementation.
