# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer | **Month:** 2025-03

---

## Executive Summary

March 2025 was a high-velocity development month for the Ergo ecosystem, dominated by three parallel tracks: the Ergo 6.0 protocol release cycle advancing through RC2→RC4 with sub-blocks P2P implementation and sigmastate-interpreter finalization; a coordinated Scala 3 migration across core libraries (scrypto, scorex-util, debox, sigmastate-interpreter); and ecosystem tooling maturation with major releases from Nautilus, Satergo, and sigma-rust Python bindings. Secondary themes included ErgoScript debugging infrastructure design, mobile node viability, and Dexy Protocol mainnet preparation.

---

## Key Topics This Month

### Ergo 6.0 Protocol Release Cycle

The month saw Ergo 6.0 advance from RC2 through RC4. RC3 was released mid-month for testnet only, with Docker image published on DockerHub and local build issues resolved. RC4 was planned for the final week, incorporating tree version-based versioning that had been missing. The central technical work was finalisation of sigmastate-interpreter PR #1043 (tests and versioning fixes), with mainnet blockchain processing being tested against 6.0 candidates running alongside 5.0 testnet nodes. Testnet peer connectivity required manual configuration throughout the month: users needed to remove `.ergo/peers`, set `peerDiscovery = false`, and specify known peers explicitly. The 6.0 mainnet activation voting date remained unannounced but was described as approaching.

### Sub-blocks Protocol Implementation

Sub-blocks received sustained development effort throughout the month. Initial fork-switching logic was completed in W11, with both common and non-common fork resolution tested. P2P layer propagation for sub-blocks was in progress by W12–W13. The work is part of a broader throughput optimisation research track. Contributors were invited to assist with testing, efficiency improvements, and stylistic refinements. A broader community discussion on block time parameters and faster confirmations noted that contracts tied to block height (rather than timestamps) would be affected by any block time changes, with security implications requiring careful analysis.

### Scala 3 Migration Across Core Libraries

A coordinated migration effort produced multiple library releases: scrypto 3.1.0 (Scala 3 support, updated JS dependencies), debox 0.11.0, and scorex-util 0.2.2. Multiple PRs were reviewed and merged across scrypto, scorex-util, and sigmastate-interpreter. The sigma-rust Python bindings reached alpha and were published to PyPI (v0.28.0) with documentation on ReadTheDocs. A decision was made to move remaining ScorexFoundation repositories under the ergoplatform GitHub organisation for cleaner release management. BouncyCastle version inconsistencies across libraries (bcprov-jdk15on v1.66 vs. bcprov-jdk15to18 v1.80 across different repos) were flagged as requiring standardisation. Scala 3 cross-compilation of sigmastate-interpreter hit scala-js compatibility issues that remained unresolved.

### ExecuteFromSelfReg & DeserializeRegister Semantics

A significant multi-week technical discussion centred on the design of `executeFromSelfReg[T]` in the context of Ergo 6.0. The core question was whether default values should be exposed at the ErgoScript level or remain at the ErgoTree level, and how Option wrapping should be handled. The resolution reached in W12 was to implement two separate methods: `executeFromSelfReg` (without default) and `executeFromSelfRegWithDefault`, both to be documented in the LangSpec. A related finding was that DeserializeRegister is processed in a dedicated substitution pass distinct from AST reduction, meaning it should not have an `eval()` call. JS test failures for `executeFromSelfReg` were traced to a missing AST node translation to JavaScript, separately fixed.

### ErgoScript Debugging Infrastructure

A substantial design discussion in W13 addressed how to implement debug tracing in ErgoScript. The key architectural constraint is that ErgoTree execution is AST reduction (bottom-up), not VM-based execution, making Aiken-style VM trace integration non-trivially portable. Three approaches were discussed: (1) a `debug(value, label)` passthrough function that returns the value with a label and leaves no footprint in the resulting AST — assessed as viable; (2) an implicit trace-on-false `?` operator (similar to Aiken) that traces which boolean subexpression caused a failure — implementation approach unresolved given AST reduction semantics; (3) IDE-level Debug Adapter Protocol integration. The `/script/executeWithContext` API endpoint (available since 2020) was identified as the existing injection point for trace output.

### Mobile Ergo Node via NIPoPoWs + UTXO Snapshot

A significant proof-of-concept was demonstrated: a full Ergo node bootstrapped on Android using NIPoPoWs and UTXO set snapshots, using RocksDB (leveldb has aarch64 incompatibility in Termux). The synced node occupies approximately 2.85 GB and works with Ergo Mobile Wallet for transaction submission. Power consumption while synced is minimal (~160 mAh, ~5.8 MB data/hour); syncing itself generates heat but allows multitasking. The ErgoNodeAndroid repository was published for one-click installation.

### Ecosystem Releases & Tooling

Multiple ecosystem tools shipped in March: **Nautilus v1.0.0** fixed the `get_used_addresses()` error on brand-new wallets; **Nautilus v1.1.0** added SigmaUSD dApp integration, Ledger connection fixes, and Firefox compatibility improvements. **Satergo v1.9.0** added Ledger hardware wallet support, Satergo Offline Vault (Android), in-app updates, and a more secure wallet format. **SkyHarbor v2 UI** launched with speed improvements. The **Ergo Ecosystem Bounty Board** was reworked to support claiming bounties and payment requests with daily automated updates; 108 open issues with 50,920 ERG allocated were tracked.

### SigmaUSD Bank Update

The SigmaUSD bank update box was refreshed with 10 ERG to ensure storage rent sustainability for decades. The update was executed in transaction `fb7947eb2627d85661f07cdf9489c0e82a6027d0ad4aaee679e2775347b3e441`. The bank contract itself was unchanged. Protocol and key management decisions were explicitly deferred with no timeline given.

### Dexy Protocol & Lithos Protocol Status

Dexy Protocol's testnet UI (https://test.ergfi.xyz/) was deployed and performing well. The primary blocker for mainnet launch is update key holder assignment; the core team sought community volunteers willing to hold keys (to be burned after months of successful operation). Lithos Protocol reported testing delays due to stratum issues on rollup contracts, with mainnet voting proposed for May 1st, 2025.

## Technical Q&A Worth Preserving

- **Q** (@CannonQ, 2025-03-04): Would targeting Scala 2.13+ render older Android phones (used for cold wallets) unusable?
  **A** (@novamon_25735): Apps built before the upgrade retain compatibility; only new builds targeting newer JDK are affected. GraalVM may provide a compatibility path.

- **Q** (community, 2025-03-05): What is the best node configuration for light clients? UTXO+NIPoPoWs bootstrap fails to sync; digest-only mode syncs but doesn't work with Ergo Mobile Wallet.
  **A** (@kushti, @MrStahlfelge): Stateless/light node modes have unresolved wallet compatibility issues. UTXO+NIPoPoWs tested internally; UTXO snapshot download progress reporting identified as needed. Full node recommended for Android.

- **Q** (@Anastasia Polozova, 2025-03-06): New wallet fails with error code `-2, "No addresses found"` after successful connection on Safari/Firefox.
  **A** (@arobsn): `get_used_addresses()` throws on brand-new wallets with no prior transactions. Fix queued for next release to return empty array instead of throwing.

- **Q** (@c8e4d2a, 2025-03-05): How to compile ergo-6.0.0-RC2 when `sigmaStateVersion = "5.0.14-518-ca2df921-SNAPSHOT"` is unavailable?
  **A** (@kushti): Publish the sigma version locally via `sbt publishLocal` from the v6.0.0-RC2 sigma branch.

- **Q** (@mgpai, 2025-03-05): Testnet node cannot connect to main testnet node; connection timeout.
  **A** (@kushti): Remove `.ergo/peers`, set `peerDiscovery = false`, and manually define `knownPeers = ["5.161.73.235:9022", "74.69.128.24:9022", "176.9.15.237:9021", "213.239.193.208:9020"]`.

- **Q** (community, 2025-03-12): Can NIPoPoWs + UTXO snapshot bootstrap work on Android? How much storage and power?
  **A** (@rustinmyeye): Yes, working with RocksDB (leveldb has aarch64 issues in Termux). ~2.85 GB total. Syncing uses 100% CPU and generates heat; synced state uses ~160 mAh and ~5.8 MB data/hour. Works with Ergo Mobile Wallet for transaction submission.

- **Q** (@HQΣr, 2025-03-12): LevelDB corruption — missing files in history/extra. How to fix?
  **A** (@kushti): Remove the `extra` folder; the node will rebuild extra indices automatically.

- **Q** (@novamon_25735, 2025-03-13): Should `DeserializeRegister` include `eval()` to minimise cost and avoid duplicate work?
  **A** (@kushti): DeserializeRegister is processed in a dedicated substitution pass, not during AST reduction. It must not have `eval()` — two separate passes occur: substitution first, then AST reduction.

- **Q** (@novamon_25735, 2025-03-19): What's the best approach for integrating secp256k1 in Java — build a C library with JNI ourselves, or use an existing library?
  **A** (@kushti): BitcoinS already implements JNI for libsecp256k1; leverage existing solutions to avoid maintenance burden.

- **Q** (@novamon_25735, 2025-03-21): How should the `executeFromSelfReg` variant without a default be implemented?
  **A** (@kushti): Implement two methods — `executeFromSelfReg` (no default) and `executeFromSelfRegWithDefault`; document both in LangSpec.

- **Q** (@ddobreff, 2025-03-26): Does anyone have a working MiningCore configuration with no duplicate share rejections?
  **A** (@Aberg): MiningCore has been discontinued.

- **Q** (@kushti, 2025-03-28): What is the right feedback channel and injection method for debug info in ErgoScript tooling?
  **A** (@novamon_25735): Most compilers walk the AST from the input node to generate debug output; output can go to a register/var or console via command-line flag. A `debug(value, label)` passthrough function that returns the value and leaves no footprint in the resulting AST will work.
  **A** (@mgpai): Aiken is a good reference: CLI compiles to JSON with optional debug traces; traces surface on contract evaluation failure via `/script/executeWithContext`. An implicit `?` operator (trace-on-false for booleans) is useful for pinpointing which subexpression caused failure in nested logic chains.
  **A** (@kushti): The `/script/executeWithContext` endpoint has been available since 2020. Note: ErgoTree execution is AST reduction (bottom-up), not VM-based, so Aiken-style VM traces cannot be directly replicated.

- **Q** (@ccellado, 2025-03-12): Are there examples of Merkle tree + ErgoScript tests?
  **A** (@Glasgow): Documentation at https://docs.ergoplatform.com/dev/protocol/tx/tx-merkle/; example in ergohack-sidechain repo (`BitcoinMerkleProofSpec.scala`); ChainCash contracts (note.es, reserve.es) in ChainCashLabs repo; AVLTreeScriptsSpecification in sigma repo.

## Open Questions Entering Next Month

- **ErgoScript debug trace implementation**: The `debug(value, label)` passthrough approach is assessed as viable, but the implicit `?` trace-on-false operator (Aiken-style) remains unresolved given ErgoTree's AST reduction model rather than VM execution. Output format for PR #1064 pending kushti review.

- **Scala 3 cross-compilation of sigmastate-interpreter**: scala-js compatibility issues encountered; unclear if resolvable in short term. Downstream impact on ecosystem tooling TBD.

- **BouncyCastle standardisation**: All Ergo libraries should standardise on `bcprov-jdk15to18` (Java 8+), but current state varies across scrypto, sigmastate-interpreter, ergo-appkit, and wallet gradle files. Audit and alignment needed.

- **SigmaUSD protocol & key management**: Explicitly deferred following the update box refresh. No timeline or plan stated.

- **Dexy Protocol update key holders**: Community volunteers sought to hold update keys post-mainnet launch; no confirmed takers recorded in March.

- **Testnet decentralisation**: Concern raised that reliable testnet requires centralized maintainer involvement; no concrete solution proposed for non-interactive testnet participation.

- **6.0 mainnet activation voting date**: Described as "close" but no date announced as of end of March.

- **`executeFromVar` generalisation for reference scripts**: @luivatra's proposal to generalise `executeFromVar` to accept arbitrary ErgoTree bytes (Cardano-like reference scripts) was raised but not addressed.

- **Lithos Protocol stratum issues**: Testing delays due to stratum problems on rollup contracts; status not resolved in March.

- **Node API rate limiting**: Reports of timeouts after ~10 API calls on certain nodes; root cause not identified.

- **Token index inconsistency**: Mew token missing from one node's index while present on others; issue status unknown.
