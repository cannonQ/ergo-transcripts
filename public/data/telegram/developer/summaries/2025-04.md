# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer
**Month:** 2025-04

---

## Executive Summary

April 2025 was dominated by the finalization and release of the Ergo protocol 6.0 reference client, with RC4 and RC5 shipped and the protocol locked down including a critical fix for a big integer injection vulnerability via Autolykos v1 deserialization. Alongside this, significant progress was made on sub-blocks P2P protocol development and the Keystone hardware wallet integration, with both efforts advancing from early prototyping toward testnet-ready and firmware-ready states respectively. Supporting ecosystem work — Nautilus i18n, Oracle Core 2.0.3, Dexy Protocol mainnet token reminting, and CruxFinance LP tracking — filled out a productive and technically dense month.

---

## Key Topics This Month

### Ergo Protocol 6.0 Finalization
The month's headline achievement was shipping two release candidates and finalizing the Ergo 6.0 protocol. RC4 was released in early April and confirmed able to sync with mainnet. RC5 followed mid-month and is considered effectively identical to the final release, pending the mainnet voting height and completion of an ongoing audit. A critical security fix was merged that prevents injection of big integers exceeding 256 bits via Autolykos v1 solution deserialization. All planned PRs for 6.0 were merged, including sigmastate-interpreter #1047. The 5.0.26 reference client also progressed in parallel, with Sigma dependency updated to 5.0.15 and a final review underway.

### Sub-blocks P2P Protocol Development
kushti made sustained progress on sub-blocks throughout April, completing P2P message construction and logic for sub-block propagation and testing the 6.0.0 candidate on mainnet. A key design challenge addressed was transaction classification: "class 2" transactions — those reading block timestamp, miner pubkey, or votes — cannot be included in sub-blocks. Support was implemented in Sigma to disallow these field reads during sub-block validation, with exceptions thrown to distinguish class 1/2 transactions without needing full ErgoTree scanning. Block formation and a private devnet for mining and API testing were in progress by month's end, with a public testnet deployment planned after validation work completes. Multiple community members (mgPai, Pgr456, Luivatra) volunteered to host public-IP devnet nodes.

### Keystone Hardware Wallet Integration
Pgr456 worked throughout the month on integrating Ergo into Keystone's cypherpunk firmware alongside Bitcoin, Zcash, and Monero. The core technical challenge was building a `no_std` version of sigma-rust-mini that replaces k256 with secp256k1 for both security and firmware size reasons. Key substitutions included k256 `Scalar` → secp256k1 `SecretKey`, k256 `ProjectivePoint` → secp256k1 `PublicKey`, and point-scalar multiplication via `PublicKey::mul_tweak`. A further complication emerged: parsing reduced transactions in sigma-rust-mini fails because the miner fee box contains opcodes (HEIGHT, if, substituteconstants) that require full opcode support. The proposed resolution is to extend the serialization schema with a VLQ field before ErgoTree to allow parsing to be skipped, and to derive base58 addresses from unparsed ErgoTree bytes directly via `base58(headByte + ergoTreeBytes + checksum)`. By month's end Pgr456 was adjusting the custom serialization schema and testing address display ahead of signature testing, with the Keystone firmware PR (#1676) open.

### MAST, ErgoScript, and Sigma Protocol Improvements
Several sigmastate-interpreter PRs were merged for the 5.0.16 release cycle, and a notable new capability was demonstrated: Merklized Abstract Syntax Tree (MAST) support in ErgoScript, enabling efficient multi-policy wallets by storing Merkle hashes on-chain with compact one-out-of-many proofs. ccellado resolved a `SBoolean` vs `Boolean` type confusion in ErgoScript during MAST implementation. A LinkedHashMap serialization bug introduced in Sigma 5.0.12 — which caused disagreement between clients on context extensions with more than four values — was fixed in 5.0.13. Scala 3 cross-compilation work for sigmastate-interpreter continued, with JS support adding complexity. Work also began on a debug/inspection callback framework for ErgoScript, with novamon_25735 designing callback injection into the Verifier rather than the Prover.

### Nautilus Wallet: i18n and Multi-wallet Features
Nautilus completed internationalization infrastructure this month: the en-US dictionary was finished, a pt-BR translation was contributed, and custom pluralization rules were added to handle languages like Russian that require singular/paucal/plural forms. A language fallback mechanism was also implemented. Ledger integration remained pending final approval. A community request for an aggregated balance view across multiple imported wallets was raised but left unresolved, directed to anon_br for feasibility assessment.

### Rosen Bridge, Oracle Core, and Dexy Protocol
Rosen Bridge Dogecoin integration neared completion despite API rate-limiting issues. Bitcoin Runes integration surfaced a harder problem: Taproot addresses require Schnorr signature support, and two potential solutions — Bitcoin-Core and FROST — were under investigation with no decision made by month's end. QA testing was expanded to a 3–4 week milestone cycle. Oracle Core 2.0.3 was released, removing dependency on node scans, with Oracle Pool v2 operational and distributing GORT rewards for Dexy Gold. Dexy Protocol mainnet tokens were reminted and contracts prepared for private testing ahead of a public announcement.

### Documentation and Ecosystem Tooling
A significant refresh of docs.ergoplatform.com was completed, covering grammar and navigation improvements, FleetSDK recipes, blockchain indexing tutorials, MiningCore documentation, debugging guides, and FSM examples. The Sigmanauts Mining Pool launched a live dashboard at ergominers.com, with iOS app, MRP UI, and Mining Wave API enhancements in development. CruxFinance added LP token data loading for portfolio value display and LP position tracking. SkyHarbor v2 open-sourced its frontend. AOSSIE announced participation in Google Summer of Code for the 10th consecutive year with up to three Ergo-related projects proposed, seeking additional mentors.

## Technical Q&A Worth Preserving

- **Q** (@Pgr456, 2025-W14): How to replace k256 with secp256k1 in sigma-rust-mini for Keystone firmware, specifically in the Prover trait?
  **A** (@dusek_): Replace EcPoint types throughout `dlog_protocol.rs`: k256 `Scalar` → secp256k1 `SecretKey` (note: secp256k1 `Scalar` is not constant-time and unsuitable for signing); k256 `ProjectivePoint` → secp256k1 `PublicKey`; point-scalar multiplication via `PublicKey::mul_tweak`; ECPoint addition via `PublicKey::combine`. The `compute_commitment` function may be deletable if only needed for signature verification.

- **Q** (@Pgr456, 2025-W14): How to implement secp256k1 equivalents of k256 `generator()` and `identity()` in the EcPoint wrapper?
  **A** (@dusek_): Generator: construct via `PublicKey::from_bytes(concat(4, GENERATOR_X, GENERATOR_Y))` using secp256k1 constants. Identity: secp256k1 lacks a point-at-infinity in affine coordinates — instead catch the `InvalidPublicKeySum` error on `PublicKey::combine` rather than calling `is_identity()`.

- **Q** (@ccellado, 2025-W14): How to convert context-stored `SigmaProp` bytes to a proposition using `getVar`/`executeFromVar` for MAST?
  **A** (@kushti): Use `executeFromVar[SBoolean](idx)` — not `getVar` — to retrieve and deserialize bytes from context extension. `executeFromVar` is macro-like (pre-AST execution) and deserializes the context extension; `getVar` reads evaluated typed values. Resolution: the actual issue was `Boolean` vs `SBoolean` type confusion in ErgoScript.

- **Q** (@Dmitriy B., 2025-W15): What is the best way to verify a transaction is on-chain? Is tracking a singleton token ID reliable?
  **A** (@anon_br): Use the transaction ID to query the blockchain directly to confirm inclusion. Note that a confirmed transaction can disappear due to chain reorgs, so verification logic must account for this. Nautilus polls at 5-second intervals as a practical balance between responsiveness and server load.

- **Q** (@iospore, 2025-W15): How should large BigInt constants be declared in ErgoScript?
  **A** (@iospore, self-resolved): Use `bigInt("7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF")` syntax (Base16 encoding evaluated at compile time). Standard Scala methods such as `.pow()` are not available in ErgoScript — consult the [ErgoScript Language Specification](https://github.com/ergoplatform/sigmastate-interpreter/blob/develop/docs/LangSpec.md). Calculate the hex value externally in Java: `BigInteger.TWO.pow(127).subtract(BigInteger.ONE).toString(16)`.

- **Q** (@ccellado, 2025-W15): Why can't `collection.mutable.LinkedHashMap` be used in Sigma?
  **A** (@kushti): `LinkedHashMap` causes serialization disagreement between clients on transactions with more than four context extension values, because traversal order is not guaranteed to be consistent. This bug was introduced in Sigma 5.0.12 and fixed in 5.0.13.

- **Q** (@Pgr456, 2025-W15): Why do opcodes fail when parsing reduced transactions in sigma-rust-mini?
  **A** (@dusek_): The miner fee box contains opcodes (HEIGHT, if, substituteconstants). Without full opcode parsing, reduced transactions with the size bit set cannot be parsed in a minified implementation.

- **Q** (@Pgr456, 2025-W15 / 2025-W16): What options exist for working with minified sigma-rust that lacks full opcode support?
  **A** (@arobsn): Extend the serialization schema to include the ErgoTree size as a VLQ field before the ErgoTree bytes, allowing parsing to be skipped regardless of the size flag. This would also enable a new schema that reduces transaction size for single-key signing scenarios. Address encoding from an unparsed ErgoTree can be computed directly: `base58(headByte + ergoTreeBytes + checksum)` where checksum = first 4 bytes of `blake2b256(ergoTreeBytes)`.

- **Q** (@Marc the Shark, 2025-W16): API call for Comet token by token ID returns nothing — node issue or API problem?
  **A** (@kushti): Verify the node is running 5.0.25/5.1.2 and that the index was built with that version. Previous versions lost information about boxes on rollback processing.

- **Q** (@c8, 2025-W16): How to handle Russian pluralization (three forms: singular, paucal, plural) in Nautilus i18n?
  **A** (@arobsn): Custom pluralization rules were added. Use the format `{n} неделя | {n} недели | {n} недель` with an optional zero-case prefix. Rules are language-configurable in the pluralization engine.

- **Q** (@kushti, 2025-W16): How are computations metered and pricing determined in Celaut?
  **A** (@Josemi): Detailed answer posted at https://www.ergoforum.org/t/how-are-computations-metered-and-pricing-determined-on-celaut/5140

## Open Questions Entering Next Month

- **Ergo 6.0 Mainnet Voting**: RC5 is ready, but the mainnet voting height has not been set and the audit is ongoing. The timeline for final release and activation remains TBD.
- **Sub-blocks Public Testnet**: Block formation and private devnet are in progress but no confirmed public deployment date. Transaction classification validation and block candidate generation approach still being refined.
- **Keystone Serialization Schema**: The custom schema for unparsed ErgoTree in reduced transactions is still being adjusted and tested. Client compatibility requirements and upstream SDK integration (Keystone SDK USB) are TBD.
- **Rosen Bridge Schnorr/Taproot for Bitcoin Runes**: Two solutions (Bitcoin-Core and FROST) under investigation for Schnorr signature support required for Taproot addresses; no decision made.
- **iOS Mobile App Architecture**: Whether to maintain Scala cross-platform wallet app (with MoboVM compatibility burden) or move to native Android/iOS bindings for sigma-rust remains unresolved.
- **Nautilus Multi-wallet Aggregated Dashboard**: Community request for aggregated balance view across multiple imported wallets flagged but not assessed for feasibility.
- **Celaut Service Specification Packaging**: Josemi identified this as the main blocker for Celaut progress; no ETA provided.
- **Sigma 5.0.16 / Node 5.0.26 Final Release**: Both in final review; `SOption[SUnit]` test coverage and additional PR reviews pending before ship.
- **Debug Callback Architecture for ErgoScript**: kushti review of novamon_25735's Verifier injection approach pending; optimal design pattern not yet confirmed.
