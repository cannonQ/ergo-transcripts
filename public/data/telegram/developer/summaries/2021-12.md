# Ergo Developer Chat — Monthly Summary
**Month:** December 2021
**Chat:** Ergo Developer

---

## Executive Summary

December 2021 was a relatively quiet month for the Ergo developer chat, dominated by infrastructure and tooling discussions rather than protocol-level development. Key themes included wallet implementation challenges (token icons, address ownership verification, mnemonic terminology), node version management and sync testing, and early-stage design discussion around ErgoMixer privacy mechanisms and cross-dApp liquidity. The period closed with kushti releasing node version 4.0.20 as a candidate build and ErgoHack III applications opening.

---

## Key Topics This Month

### Blockchain Data Access Methods
Developers discussed the best approaches for extracting full chain data. Scraping the public Ergo Explorer API was ruled out due to stress and latency concerns. The recommended alternatives were running the explorer backend locally or accessing node data in JSON format directly via the node API. For lighter use cases (block height, timestamp, transaction count, miner, reward, difficulty, size), the node API alone was deemed sufficient without requiring full chain data extraction.

### Wallet Address Ownership Verification
A recurring thread concerned how a backend can verify that a user controls a given wallet address — an OAuth2-style authentication flow. The consensus solution was signature-based verification using the dApp Connector's `ergo.sign_data` method as specified in EIP-0012. Simply checking confirmed balance was identified as insufficient for ownership confirmation. Current wallets outside the dApp Connector cannot support this pattern, leaving a gap for non-browser wallet integrations.

### Token Icon Storage and Standards
Wallet developers needed a reliable source for token icon assets. On-chain storage was deemed inefficient; the preferred approaches were IPFS or centralised servers with on-chain hashes. Existing sources identified included the Nautilus wallet asset mapper and the Spectrum DEX token icon directory. Aberg created a stopgap public resource mapping token IDs to 36×36 SVG icons in the Satergo repository. An EIP proposal for preview picture metadata in token minting was mentioned but not formally tracked.

### ErgoMixer Fee Mechanism and Privacy
kushti explained the ErgoMixer fee design: users purchase fee tokens rather than paying ERG directly. These fee tokens are submitted to a fee emission contract that releases ERG for mining fees. This design prevents privacy leaks that would arise from straightforward ERG fee payments being traceable to mixing participants. Fee tokens are sold above the mining fee cost, generating revenue for mixer developers.

### Cross-dApp Liquidity — Waterspout Proposal
The "Waterspout" concept was discussed: a cross-dApp liquidity pool that could dynamically rebalance LP between Spectrum DEX and ErgoMixer based on mixing demand. The consensus was that dynamic rebalancing is a secondary concern — permanent liquidity presence in the mixer and improved UX are the more pressing needs. Infrastructure was said to need to precede demand, and the Nautilus team was identified as a relevant contact for mixer integration feedback.

### Node Version Management
A recurring pain point was the inconsistent updating of version fields across configuration files (`mainnet.conf`, `application.conf`) across releases. kushti acknowledged this requires a more robust automated solution rather than manual preprocessing. The 3.01 release contained a bug, and users were advised to wait for 3.1.1 or 4.0.20. By early January, kushti released 4.0.20 with improved UTXO and digest mode sync testing.

### Mnemonic Password Terminology
Aberg raised that the term "mnemonic password" is confusing to users and proposed alternatives: "mnemonic salt," "shield," "lock," or "mask." kushti acknowledged the confusion but no final naming decision was reached during the month.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2021-12-22): What is the best way to get a CSV of the full chain?
  **A** (@Aberg): Scraping the public Ergo Explorer API is too slow and stressful on the service. Alternatives are running the explorer backend locally or pulling data in JSON format from the node API directly. For common fields (height, timestamp, tx count, miner, reward, difficulty, size), the node API is sufficient.

- **Q** (@Ilya, 2021-12-24): How do you verify wallet ownership for backend authentication — e.g., an OAuth2-style flow with Yoroi Nightly for user profile updates?
  **A** (@Ilya Oskin): Ask the user to sign something to verify address ownership; optionally verify the address has existing blockchain transactions.
  **A** (@anon_br): Use the dApp Connector's `ergo.sign_data` method per EIP-0012 for signature-based verification.

- **Q** (@Ilya, 2021-12-24): What does "confirmed balance" mean, and can it be used to confirm address ownership on the backend?
  **A** (@Aberg): Confirmed balance means coins verified to belong to an address — it is not sufficient for ownership verification. Address ownership confirmation requires a signature-based solution; current wallets outside the dApp Connector cannot support this.

- **Q** (@Unknown, 2022-01-03): How does ErgoMixer's fee structure preserve privacy?
  **A** (@kushti): Mixer users buy fee tokens (not ERG directly) to participate in mixing. These tokens are paid to a fee emission contract that releases ERG for mining fees. This avoids the privacy leak that would occur if ERG fees were paid directly and traced back to mixing participants. Fee tokens are priced above mining cost, generating developer revenue. See: https://www.ergoforum.org/t/paying-fee-in-ergomix-in-primary-tokens/73

- **Q** (@Unknown, 2022-01-03): Is Radix's Scrypto the same as Ergo's Scrypto?
  **A** (@Glasgow): No. Scrypto is a cryptographic toolkit extracted from Scorex that Ergo uses. Radix independently reused the name.

- **Q** (@Aberg, 2021-12-22): Is there a public mapping of token icons to token addresses? Are icons stored on-chain?
  **A** (@kushti): Most tokens have no icons even at minting time; icons are not stored on-chain.
  **A** (@MrStahlfelge): An EIP proposal exists for preview picture metadata in token minting; images could be stored in the payload or referenced via IPFS or a centralised server with an on-chain hash.

- **Q** (@Aberg, 2021-12-22): Where can token images be obtained right now for wallet embedding?
  **A** (@Capt. Nemo): Nautilus wallet asset logo mapper and icon directory; also the Spectrum DEX frontend token icon assets with a verified token mapper for better sizing.

## Open Questions Entering Next Month

- **Node config version consistency**: No automated solution finalised for keeping version fields synchronised across `mainnet.conf` and `application.conf` on release. Manual preprocessing remains a stopgap.
- **Decentralised node discovery**: Aberg flagged concern about wallets defaulting to centralised nodes (MetaMask/Yoroi pattern). No architectural solution or design proposal was put forward.
- **Address ownership verification for non-browser wallets**: The dApp Connector / EIP-0012 solution works for browser-based wallets. No solution identified for backend auth flows with non-browser wallets.
- **Token icon metadata standard**: Multiple ad-hoc sources identified (Nautilus, Spectrum DEX, Satergo); no unified protocol-level EIP formalised or actively tracked.
- **Mnemonic password terminology**: Alternatives proposed ("salt," "shield," "lock," "mask") but no decision reached; affects wallet UX consistency across implementations.
- **ErgoMixer UX and Waterspout**: Concept discussed but no implementation owner, timeline, or formal proposal established.
