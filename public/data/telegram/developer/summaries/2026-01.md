# Ergo Developer Chat — Monthly Summary
**Channel:** Ergo Developer
**Month:** January 2026

---

## Executive Summary

January 2026 was dominated by sub-blocks (input/ordering blocks) research and devnet testing, with kushti actively iterating on parameters and bug fixes to stabilise the network for broader community participation. Alongside this, significant progress was made on multiple node releases (6.0.2/6.1.2), the Rust node implementation, ChainCash/Basis development, and a growing ecosystem of vibe-coded tooling. Infrastructure reliability — explorer backends, oracle pools, and monitoring — received sustained attention throughout the month.

---

## Key Topics This Month

### Sub-blocks (Input/Ordering Blocks) Devnet Testing
The Matrix whitepaper was published in W02 with implementation described as mostly complete, but requiring broad testing before public testnet activation. By W04–W05, kushti had deployed a devnet with two mining peers and was iterating on parameters: initial settings (30s block interval, 60 input blocks per ordering) caused severe fork proliferation in RAM. After fixing bugs and reducing to a 60s block interval, the network stabilised. Community members (Patato, Netim, Sigmanaut) joined testing but encountered sync issues requiring database resets, and nodes stalled at similar heights (~79731, ~80365). The target for an initial public sub-blocks testnet node is node 6.0.3, approximately 2–3 weeks out from late January. OpenAPI descriptions for input block API methods were added to devnet seed nodes.

### Node Releases: 6.0.2, 6.1.2, and Roadmap to 6.0.3/6.0.4
Multiple PRs from the Unstoppable Hackathon were reviewed in W04; most AI-generated submissions were rejected as low quality, though CI/docs improvements were accepted. Node 6.0.2 and 6.1.2 were released on 2026-01-23, fixing a critical bug in the extra-indices indexer (PR #2236). Users with extra indices enabled were urged to update immediately. The 6.0.3 assembly was underway by end of January with a plan to include sigmastate version 3 by default; 6.0.4 is planned to include the Schnorr message-signing API method (PR #2290). AppKit 6.0 was progressing with sigma 6.0 compatibility work, an IndexedNodeDataSource implementation, and a transaction ID quotation-mark fix.

### Rust Node Implementation
Luivatra built a vibe-coded Rust Ergo node using Claude, achieving header sync of ~700k headers in 40 minutes and 200k+ blocks in 15 minutes with RocksDB tuning. Multiple compatibility issues were identified: sigma-rust has hardfork-ish divergences from the reference Scala interpreter, and a u64 storage rent calculation lacks overflow handling present in the Scala implementation. The AVL tree was incorrectly constructed because Claude omitted genesis boxes, causing a wrong state from block height 1. Luivatra is copying Scala node tests for compatibility verification. Sync stability is not yet 100%; a slowdown-after-time issue requires manual debugging. Community preference for Rust over Scala for sidechain implementation was noted.

### ChainCash, Basis, and Schnorr Signatures
ChainCash non-custodial dApp development progressed significantly. Fitz built an inspector UI to verify contract boxes on testnet and confirmed reserve contract box creation works. The key blocker is that ChainCash transfer/endorsement requires a raw Schnorr signature format not exposed via Nautilus `sign_data`. kushti's PR #2290 adds a node API method to Schnorr-sign a message for a wallet address; the Basis tracker was updated to use the node for tracker signatures during redemption. AVL+ proof generation, note signatures, and redemption data preparation were tested and refined. Unified signature verification algorithms were moved into a new core crate. Help was flagged as needed on the offchain Basis server component.

### Oracle Pools, USE Stablecoin, and Dexy Protocol
Oracle Core 2.0.5 and 2.0.6 were released to fix CoinCap API outages and CoinGecko user-agent requirements, with additional data feeds added for robustness. USE stablecoin analytics went live at cruxfinance.io/use-analytics, cross-chain liquidity on Binance Smart Chain was activated, and USE was added to DefiLlama as a pegged asset. USE Improvement Proposal #1 (UIP-001, balancing interventions) was proposed as a vibe-coding bounty (1 gram Dexy Gold). CannonQ built and automated a GORT/DORT buyback bot running 2× daily at 1% of smart contract balance. A blocker remains: sending from the buyback smart contract to the Oracle Rewards Pool requires Oracle Operator NFT permission and likely needs injection into the Oracle Pool v2 refresh action.

### Explorer Backend and Infrastructure
Both the EF explorer and sigmaspace.io became stuck at block height 1705794 due to node 6.0 introducing outputs with unparseable ErgoTree (allowed by consensus for soft-forkability but more strictly rejected by explorer backends). Pgr456 added a sigmastate-js fallback via an embedded JS engine; the EF fix was nearly complete pending a resolution on whether ErgoTree pretty-printing belongs in testing scope only. The Sigmanauts infrastructure dashboard was enhanced to check both indexed height and full height on nodes with >5-block mismatch alerting. A single-URL redirect was created at `infra-p2p.ergoplatform.com`. Cornell switch failure caused a brief outage requiring node resync. Docker panics on the testnet explorer machine (2 in 2 days, logs lost) remained unresolved.

### Vibe-Coding, Tooling, and Ecosystem Projects
The community produced a notable volume of LLM-assisted tooling. CannonQ released the Ergo Wallet Monthly Statement tool (bank-statement-style portfolio view with T2T/N2T LP pricing, 15,000+ NFT/FT metadata items catalogued) and a Cyberverse NFT/FT viewer covering 12 collections with floor pricing across Ergo/Ronin/Cardano. arkadia built a desktop Ergo DeFi app (Tauri + Rust/React) integrating SigmaUSD, Dexy, ErgoPay, Nautilus, and DuckPools Lending with no third-party APIs. CannonQ also published the Ergo Knowledge Base — 50+ ecosystem meetings (31+ hours) transcribed with executive summaries, Q&A, and searchable interface. Fitz Doud contributed a thesis that vibe-coding shifts competitive moat to legitimacy, credible neutrality, and launch-time path-dependent properties — with Ergo's no-ICO/no-premine/open PoW cited as relevant.

### Machina Finance SDK
The GridOrder module was completed with 100% test coverage. The LimitOrder module was in progress with unit tests. anon_BR fixed a critical input aggregation bug in the limit order contract when fulfilling orders and refunding to owner; the E2T limit contract was finished and ready for review. Separately, Keystone hardware wallet integration raised concerns over 160+ transient dependencies from the Keystone SDK — anon_BR spent 2025 reducing Nautilus dependencies from 120 to 90, and adding 160 is a significant regression. Work to extract minimal Keystone functionality was underway.

## Technical Q&A Worth Preserving

- **Q** (@Josemi, 2026-01-10): When minting a token, how can it end up in two output boxes simultaneously?
  **A** (@kushti): Non-ERG token preservation rule checks (tokens in outputs) ≤ (tokens in inputs). If the token ID equals the ID of the first input (minting transaction), preservation is not checked. Only these two consensus-level rules exist; EIP minting standards assume a single output but are application-level only. The Explorer should skip non-standard multi-output minting tokens rather than display incorrect aggregated totals.

- **Q** (@Pgr456, 2026-01-12): The Rust node uses u64 for storage rent calculation without integer overflow handling present in Scala, risking invalid transactions and hardfork-like divergence.
  **A** (@Luivatra): Acknowledged; captured as a tracking issue. (@kushti): sigma-rust has known and unknown hardfork-ish divergences from the reference interpreter; ensuring complete parity will take time.

- **Q** (@qx(), 2026-01-12): Why not do parallel block grabbing like torrents instead of sequential sync?
  **A** (@Luivatra): Parallel retrieval is a choice; sequential was chosen to avoid disk thrashing on slow drives. RocksDB tuning mitigates the bottleneck. Indexing could theoretically be parallelised but may be excessive.

- **Q** (@kushti, 2026-01-21): Will `sign_data` work for Schnorr signatures, or is a separate `schnorr_sign` for messages needed?
  **A** (@kushti, referencing PR #2290): PR #2290 adds a node API method to Schnorr-sign a message for a wallet address. Basis tracker now uses this for tracker signatures during redemption. Useful for ChainCash, Basis, and gold insurance contracts.

- **Q** (@kushti, 2026-01-21): Any updates on AppKit 6.0 PR?
  **A** (@Cheese Enthusiast): Should be ready the following week after IndexedNodeDataSource implementation — takes advantage of new sigma 6.0 calls and provides an explorer-compatible alternative.

- **Q** (@kushti, 2026-01-21): Are you using the ergo_avltree_rust library?
  **A** (@Luivatra): Yes, the library itself is fine. The issue was that Claude-generated code omitted genesis boxes and did not perform tree actions in the same order as the Scala reference. (@kushti): Genesis state is a commitment to three genesis boxes. Copying Scala node tests over for verification — 99% correctness causes complete AVL tree failure.

- **Q** (@kushti, W05): Can the GQL server switch to another node when one is unavailable or out of sync, as the Explorer does?
  **A** (@qx()): Unclear if the code supports it; currently manual failover. Proposed writing a health-check script to test transactions and rewrite the `.yml` and rebuild GQL if failure is detected.

- **Q** (@Luivatra, W05): What would implementing a stable pool type on Ergo require?
  **A** (@kushti): Spectrum may have a prototype. Stable math is more complex than constant product but is feasible with an offline implementation + ErgoScript + UI.

- **Q** (@Netim / @Patato, W05): Node stuck syncing on sub-blocks devnet — what should we do?
  **A** (@kushti): Delete the `.ergo` folder and resync. The devnet was subsequently relaunched with a 60s block interval (down from 30s) to reduce fork proliferation.

## Open Questions Entering Next Month

- **Sub-blocks sync stalling**: Multiple testers stalled at similar block heights (~79731, ~80365) despite identical configuration. Root cause unknown; possibly related to old JAR mixed with fresh database. kushti reviewing logs.
- **ChainCash Note transfer blocker**: Transfer/endorsement requires raw Schnorr signature format not exposed via Nautilus `sign_data`. Needs contract-side adjustment, node-assisted signing path, or hybrid key approach; under investigation.
- **Oracle Pool v2 buyback→reward transfer**: Whether sending from the DORT buyback smart contract to the Oracle Rewards Pool requires a separate transaction flow or can be injected into the Oracle Pool v2 refresh action; CannonQ to follow up after reviewing notes.
- **Scala vs. Rust for sidechain implementation**: Community preference leaned Rust, but no formal decision recorded.
- **Rust node hardfork-ish parity**: sigma-rust divergences (u64 storage rent overflow, interpreter differences) documented but no resolution timeline established.
- **ErgoTree pretty-printing scope in Sigma 6.0**: kushti needs to determine why pretty-printing was restricted to testing scope before the EF explorer backend fix can be completed.
- **GQL server failover**: Health-check script and automatic failover proposed but not yet implemented.
- **Docker panics on testnet explorer**: Two panics in two days with lost logs; root cause unknown.
- **Keystone SDK integration**: Whether a minimal-dependency Keystone extraction is feasible without a major regression in Nautilus dependency count.
- **ergo-lib release timeline**: Requested by kushti in W04 with no response recorded.
- **AppKit 6.0 / IndexedNodeDataSource**: Targeted for late January/early February; status not confirmed at month end.
- **ZK rollup via MidenVM**: greenhat's proposal to compile ErgoTree to MidenVM for zkSTARKs is conceptually interesting but hash verification strategy and on-chain proof costs are not yet detailed.
- **Stable pool contract on Ergo**: Whether Spectrum's prototype is shareable and what the integration timeline would be.
- **Spectrum DEX backend**: Offline as of 2026-01-24; Spectrum team pinged but status unresolved at month end.
- **EIP-004 multi-output minting allowance**: Whether EIP should permit multiple outputs in minting transactions but restrict registers to one output only; no resolution reached.
