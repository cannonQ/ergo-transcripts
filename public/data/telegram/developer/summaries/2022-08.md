# Ergo Developer Chat — Monthly Summary
## August 2022

---

## Executive Summary

August 2022 was a technically dense month for Ergo development, dominated by node infrastructure improvements, a landmark protocol upgrade, and ongoing work on developer tooling. The most significant event was the commencement of voting for protocol version 5.0 on mainnet, introducing JIT costing — while parallel work on NIPoPoWs integration, explorer redesign, and Android/wallet SDK compatibility kept multiple development threads active simultaneously.

---

## Key Topics This Month

### Protocol Version 5.0 Activation

The month's headline development was the start of voting for Ergo protocol version 5.0 on mainnet (2022-08-26). The voting window spans 32 epochs (1,024 blocks each), with activation requiring ≥90% miner support plus a further 32-epoch delay — meaning full activation will occur approximately 64×1,024 blocks after voting began if the threshold is met. The 5.0 upgrade introduces JIT costing in the interpreter, which kushti had been reviewing throughout the month. Developers were reminded to test contracts on testnet where JIT costing is already activated, as contract execution costs will change at activation.

### Node Reference Client Releases

Four point releases shipped in August: v4.0.37, v4.0.39, v4.0.40, and v4.0.41. The releases addressed a range of issues including unconfirmed transaction optimisations, UTXO set snapshot downloads, and minor byte allocation improvements. kushti noted his primary focus during the month was unconfirmed transaction optimisation, UTXO set snapshot work, and PR reviews alongside the 5.0 interpreter final review.

### Explorer Backend Redesign

Ilya Oskin proposed a significant architectural change to the explorer backend: splitting nodes into "light" (minimal indexes for essential API methods) and "full" (extensive indexes) variants, with intelligent routing to distribute load. The proposal triggered a broader discussion about address index vs. ErgoTree index trade-offs — whether to store addresses directly or decode them on-demand from an ErgoTree index. The address index is compact and offers APIv1 compatibility, but the ErgoTree index reduces footprint. The APIv1 compatibility concern remained unresolved. dApp developers were asked to identify which API methods are essential for their use cases to inform the light/full split.

### NIPoPoWs Integration in AppKit

kushti opened GitHub issue #1813 explicitly requesting NIPoPoWs support be treated as a core priority in AppKit and wallet infrastructure. The reasoning: NIPoPoWs are a key selling point of Ergo, the ecosystem is shifting away from centralised explorer dependency, and lightweight clients need a proper path that doesn't rely on centralised APIs. Discussion touched on alternative SPV architectures — HTTP API, peer-to-peer protocols, and the `ergo-spv` library — with AppKit integration via HTTP identified as the most practical forward path. Privacy implications of SPV queries (analogous to Bitcoin Bloom filter privacy weaknesses) were noted; oblivious transfer-based solutions exist but are complex, and a proxy layer on nodes was suggested as an interim measure.

### Android/Wallet SDK Compatibility

Early-month discussions addressed Scala 2.11/2.12 compatibility for Android wallet development, JDK versioning on Android devices (API level 26+ supports `java.time`), and whether sigma-rust's JNI bindings are necessary given the existence of pure Java/Scala implementations. Aberg was implementing Satergo for Android and encountered WebAssembly being disabled in the JavaFX webkit-based embedded browser — ultimately deciding embedded browser support was overkill for a wallet application. Satergo v1.4.1 shipped during the month with node update fixes, synced sending restrictions, Portuguese translation, and address renaming fixes.

### Mempool Behaviour & Spam Protection

A recurring thread clarified how Ergo's mempool handles unconfirmed transactions. True transaction cancellation does not exist; however, spending the same input boxes in a new transaction effectively supersedes an unconfirmed one. Low-fee transactions time out and flush from the mempool after approximately 10–40 blocks if not picked up by miners. Node operators can raise minimum propagation fee rates to prevent spam from even entering the mempool. kushti noted that Ergo has a fee estimation API that is underutilised by applications and not yet properly tested — hardening this, similar to how Bitcoin wallet fee estimation matured after years of events, was identified as worthwhile work.

### Contract Design Patterns

kushti discussed grid-based trading contract patterns, proposing that price grids be implemented as many simple orders sharing the same contract template but with different prices, combined with AMM LP trades. A preference was expressed for simple contracts with multiple UTXOs over complex single-contract designs. ErgoTree serialisation constraints were clarified: stored ErgoTree cannot contain `executeFromVar`, but `executeFromRegister` works similarly and is supported. Scalable NFT minting contract patterns were also discussed, with a forum thread linked.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2022-08-10): If I send a transaction to the mempool, is there any way to cancel it before it confirms?
  **A** (@MrStahlfelge): You can expect it to be cancelled in most cases if you use the same input boxes — a new transaction spending the same inputs will supersede the unconfirmed one. There is no true cancel on Ergo.

- **Q** (@kii, 2022-08-10): In cases where smart contracts force sending to one address, once the box is spent it can't be unspent in the mempool unless the contract fails when the block updates — correct?
  **A** (@MrStahlfelge): Exactly — we don't have a true cancel.

- **Q** (@Aberg, 2022-08-09): Any reason sigma-rust has JNI bindings when pure Java/Scala implementations exist?
  **A** (@greenhat, implicit): JNI bindings enable native performance on Android, though pure Java/Scala implementations are available as alternatives depending on use case.

- **Q** (@Aberg, 2022-08-09): Did Scala 2.12 update to a new JDK? I think Android is on JDK 11 now?
  **A** (@MrStahlfelge): JDK 11 is the current LTS that Android development tools settle on. No upgrade of device JREs occurred; API level 26+ supports later Java APIs such as `java.time`.

- **Q** (@Aberg, 2022-08-16): Does Ergo have a transaction index by default? Does searching through the blockchain degrade node performance?
  **A** (@kushti, implicit): Following Bitcoin's pattern (where `txindex` is off by default), transaction indexing is optional. Aberg noted this is a good precedent unless overhead proves negligible.

- **Q** (@Luivatra, 2022-08-17): Could a gap in the Merkle tree data occur if it isn't updated within 10 blocks due to congestion?
  **A** (@kushti): Yes, it could. More designs addressing this will be published on the forum.

- **Q** (@kushti, 2022-08-18): Can stored ErgoTree contain `executeFromVar`?
  **A** (@kushti): No — stored ErgoTree cannot contain `executeFromVar`. However, `executeFromRegister` works similarly and is supported.

- **Q** (@Capt. Nemo, 2022-08-26): Could the address index be dropped in favour of an ErgoTree index, with addresses decoded on-demand by the API?
  **A** (@Ilya Oskin, @Capt. Nemo): Feasible, but requires APIv1 compatibility changes. The address index is compact and the decoding cost roughly equals ErgoTree encoding cost. The trade-off between breaking APIv1 compatibility versus reducing index footprint was not resolved.

## Open Questions Entering Next Month

1. **NIPoPoWs in AppKit** — Issue #1813 opened but no assignee or timeline established. Who picks this up, and when?

2. **Explorer indexing strategy** — Address index vs. ErgoTree index trade-off unresolved. APIv1 compatibility concerns need a concrete decision before explorer redesign can proceed.

3. **Privacy for SPV/lightweight clients** — Oblivious transfer solutions are too complex for immediate implementation; proxy layer on nodes proposed as interim but not designed or scoped.

4. **Fee estimation API** — Ergo has a fee estimation API that is untested and unused by applications. Hardening and adoption plan remains undefined.

5. **Testnet configuration** — 5-second block time under current settings is too aggressive, causing propagation issues. Testnet reset was under discussion; outcome not confirmed.

6. **Scala 2.11 Android support** — Whether to drop Scala 2.11 support for Android builds (affecting pre-API 26 devices) remains undecided.

7. **EIP-32 review** — kushti committed to reviewing a proposed EIP-32; status not confirmed by month end.

8. **Contract design pattern documentation** — Additional grid-based and state machine contract design patterns promised for the Ergo forum; not yet published.
