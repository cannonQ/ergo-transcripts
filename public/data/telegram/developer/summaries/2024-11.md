# Ergo Developer Chat — Monthly Summary: November 2024

---

## Executive Summary

November 2024 was a pivotal month for Ergo's core protocol development, with the release of Ergo v6.0.0-RC1 and the publication of the 6.0 soft-fork proposal (EIP-50), advancing unsigned BigInt support, Schnorr signature verification, and new ErgoScript language features. Alongside protocol work, multiple ecosystem projects reached significant milestones: Paideia DAO launched on mainnet, Nautilus's 0-conf workflow merged, Fleet SDK v0.8.0 shipped with deterministic transaction chaining, and the Sigmanauts formally migrated to mainnet Paideia governance. The month also saw sustained groundwork on SigmaUSD oracle pool migration, Lithos Protocol fraud proofs, and sub-blocks/L2 design research.

---

## Key Topics This Month

### Ergo Protocol 6.0 Development

The dominant technical theme of November was the 6.0 soft-fork cycle. kushti finalised the UnsignedBigInt PR (#997) — adding bitwise operations, shifts, and big-endian byte conversions — and merged several other improvements including `Global.some()`/`Global.none()` for `Option[]` construction and Schnorr signature verification of arbitrary byte arrays in ErgoScript. EIP-50 (the 6.0 soft-fork specification) was published to the Ergo Forum on 2024-11-25, and v6.0.0-RC1 was tagged for testnet on 2024-11-18. A separate Sigma 5.0.15 release was planned to ship `executeFromVar` and dependency updates. Outstanding PRs (#979, #1026, #1029, #1021) were awaiting review by ergomorphic as the month closed.

### SigmaUSD Oracle Pool Migration

kushti identified the current SigmaUSD oracle pool setup as problematic and proposed a two-phase migration: first to Oracle Pool v1.0, then v2.0 (which requires bank modifications for buyback contract payment). Testnet bootstrap work progressed through the month — pool token issuance, bank setup, and minimum oracle count configuration — with testnet deployment confirmed active by 2024-11-28. A bounty was posted for testnet deployment assistance and co-voting. Oracle Pool v1.0 currently has EF members plus external participants; v2.0 is being used for Dexy Gold with GORT rewards.

### Sub-blocks, L2, and Sidechain Constructions

kushti signalled a post-Ergoversary return to sub-blocks research, committing to publish an EIP on the topic. Multiple existing PRs for sub-block implementation were flagged for revival and consolidation. Parallel to this, a broader design question was discussed: whether to pursue microblocks, sidechains, or optimistic rollup-style L2 solutions. Luivatra proposed an AVL tree-based hybrid L2 DEX concept for more efficient order matching than transaction chaining, which kushti noted could be efficient but implies some L2 centralisation. A forum post on merge-mined sidechains and BIP 301 comparison was also flagged as forthcoming. The key insight noted: Ergo's known dApp contract profile (GARD protocols, Rosen Bridge, Oracle Pools, Spectrum AMM, Trade House) makes targeted optimisation more tractable than for chains with unknown workloads.

### Nautilus Wallet & Fleet SDK

Nautilus saw sustained progress across the month. anon_BR published comprehensive dApp Connector technical documentation at docs.nautiluswallet.com. The 0-confirmation workflow was designed, built, and merged as an experimental opt-in feature by end of month. The Ledger embedded app completed functional review and entered security audit. Fleet SDK v0.8.0 shipped with deterministic transaction chaining support; EIP-5 compiler output was raised as a GitHub issue. dusek_ progressed sigma-rust `no_std` compatibility for the Keystone hardware wallet target.

### Paideia DAO & Sigmanauts Governance

Paideia DAO launched on mainnet at app.paideia.im during W46, with legacy testnet DAOs remaining at beta.paideia.im. A signing failure in Minotaur wallet was noted (works on Android). The Sigmanauts community formally migrated from beta-Paideia to mainnet Paideia in W48, establishing a new treasury multisig (Luivatra, CannonQ, Marc the Shark, Jake the Snake, Grayman, CW, qx()) and launching a Bluesky social presence. A governance proposal for supERGeometry as a new Sigmanauts member was active at month end.

### Lithos Protocol Fraud Proofs

qx() completed fraud proof contracts for NISPs covering invalid PoW, header validation, and NISP rule violations, with work continuing on transaction collateral and NISP-specific proofs post-vacation. This represents a significant milestone toward Lithos Protocol's trustless sidechain security model.

### ErgoScript Tooling & Developer Experience

Several developer-experience improvements were discussed and initiated. A `debug()` intrinsic for ErgoScript (outputting variable values during execution, similar to Lisp's `inspect`) was proposed, with a design sketch involving fake compiler nodes, prover callbacks, and JSON output. A compiler-level sugar function `equalBoxExcept` was proposed and assigned as GitHub issue #1034 to simplify box equality checks excluding specific registers — identified as a common and security-relevant pattern. The gap between Discord Q&A and formal Fleet SDK documentation was raised, with a migration to GitHub Discussions recommended.

### Ecosystem Projects

- **Celaut**: First tutorial published; remote Git repository compilation, service gas/firewall controls, and dev fee KyA support implemented.
- **Bene**: All platform actions fully implemented and deployed, entering testing phase.
- **Netnotes**: Refactored toward single-window architecture with decoupled UI via messaging system.
- **"The Field"**: CannonQ reported MVP smart contracts complete, dApp UI integration ongoing, alpha testing beginning late November.
- **Sigmanauts Mining Pool**: Bonus token airdrop reactivated as standalone downloadable tool with KYA documentation; EF proposals resubmitted with concrete deliverables.

## Technical Q&A Worth Preserving

**ErgoScript Contract Fundamentals**

- **Q** (@a787978, 2024-11-05): Is a SigmaUSD smart contract essentially an NFT?
  **A** (@Flying Pig / @kushti): A contract is an address; Ergo uses boxes (UTXOs with up to 10 registers), not standalone contract instances. Singleton NFT tokens mark specific contract instances (e.g., the SigmaUSD bank box). Ergo is oriented toward programmable money, not replicated calculators.

**Non-Standard Token Minting (Fleet SDK)**

- **Q** (@b_e_e_e_e_e, 2024-11-05): How to mint the same token into multiple boxes in a single transaction — Fleet SDK throws `NonStandardizedMinting`?
  **A** (@arobsn): Non-standard minting requires manual transaction construction outside EIP-4 safeguards. The `tokenId` must be taken from the first input's `boxId`. Use `OutputBuilder#addTokens` for regular token distribution; use `OutputBuilder#setAdditionalRegisters` for optional metadata.

**Token ID Determination**

- **Q** (@Aberg, 2024-11-15): Is a token ID determined from the input it was minted from?
  **A** (@kushti): Yes — the first input's ID becomes the minted token's ID.

**ErgoScript Register Encoding for Complex Types**

- **Q** (@b_e_e_e_e_e, 2024-11-13): How to encode `string[]` and `[number, string][]` as `Coll[Coll[Byte]]` and `Coll[Pair[Int,Coll[Byte]]]` in Fleet SDK registers?
  **A** (@arobsn): Use `SColl(SColl(SByte), [[...utf8.decode("foo")], [...utf8.decode("bar")]])` for `Coll[Coll[Byte]]`; pairs as arrays: `SColl(SPair(SInt, SColl(SByte)), [[1, utf8.decode("foo")], [2, utf8.decode("bar")]])`. Note: a TypeScript static-type bug exists requiring the deconstructor operator; fix incoming.

**ErgoScript 6.0 Language Changes**

- **Q** (@Luivatra, 2024-11-19): Is there a list of what is added to ErgoScript in 6.0?
  **A** (@kushti): Two approved PRs to merge; changes reflected in `LangSpec.md` diff accessible via PR #930 files.

- **Q** (@Luivatra, 2024-11-19): Is verifying a Schnorr signature over an arbitrary byte array possible in ErgoScript?
  **A** (@dusek_ / @kushti): Already possible. ChainCash uses this pattern — see `note.es` in the ChainCash repo. kushti also posted a detailed forum thread on Schnorr signature verification in ErgoScript.

**Box Register Comparison in ErgoScript**

- **Q** (@b_e_e_e_e_e, 2024-11-19): Is there a generic way to check whether two boxes have the same additional registers?
  **A** (@Luivatra): Compare bytes directly; see `bytesWithoutCreationInfo` in the paideia-sdk for a reference pattern. Registers returning `Any` type fail at runtime — cast to known types individually.

**L2 AVL Tree DEX Efficiency**

- **Q** (@Luivatra, 2024-11-25): Is an L2 DEX with AVL tree balance tracking more efficient than transaction chaining?
  **A** (@kushti): In the optimistic case — where the tree root is updated on-chain only periodically — yes, potentially more efficient. However, it implies some L2 centralisation or an optimistic rollup design.

**NFT Randomness**

- **Q** (@c8e4d2a, 2024-11-25): Are offset header IDs unpredictable enough for NFT randomness?
  **A** (@kushti): A large miner could theoretically redo PoW rather than publish a block to influence randomness (Wesolowski paper), but for low-value NFTs this is economically infeasible. The 2-step pattern — mint with block height, then use `HashballTokenId + NextBlockHeaderId % 159` as index — is sound in practice.

**`equalBoxExcept` Sugar Function**

- **Q** (@kushti, 2024-11-27): Should a compiler-level `equalBoxExcept` sugar function simplify box equality checks excluding specific registers?
  **A** (@luivatra / @arobsn): Yes — it's easier to forget a field to include than to forget one to exclude, making the exclude-list pattern inherently safer. Very common pattern; worth adding for both code clarity and security.

**Fleet SDK Transaction Chaining**

- **Q** (@b_e_e_e_e_e, 2024-11-27): Can Fleet SDK v0.8.0 chained transactions fit in the same block?
  **A** (@c8e4d2a): Yes — as long as the whole chain fits in the next block, which succeeds ~99.9% of the time.

**ErgoScript Debug Intrinsic**

- **Q** (@arobsn, 2024-11-27): Could the interpreter output variable values during execution for debugging (e.g., `debug(SELF.R5[Long].get, "ergPricePerToken")`)?
  **A** (@kushti): The compiler could create fake nodes for debug calls, similar to Lisp's `inspect *`. The prover could register a callback and output debug values as JSON; IDEs could log to terminal and Fleet's mock-chain could test internal values.

**Node Spam and Fee Markets**

- **Q** (@Michael, 2024-11-13): What do "spam attempt" and `spampenalty` mean in node logs?
  **A** (@kushti): Usually delivery of a block section for an unregistered request — can occur if a peer is rogue or if connection latency causes a timeout, triggering a duplicate request after the first completes.

**Block Height vs. Timestamps**

- **Q** (@b_e_e_e_e_e, 2024-11-27): Can block height be converted to an approximate timestamp — is it every 2 minutes from genesis?
  **A** (@Luivatra): It's an average, and should not be treated as exact over shorter periods due to drift. Contracts that must use time should rely on the block timestamp register, not height arithmetic.

## Open Questions Entering Next Month

1. **Sub-blocks EIP path**: kushti committed to publishing the sub-blocks EIP post-Ergoversary, but the implementation approach (microblocks vs. sidechains vs. L2 rollup) remains undecided. Multiple existing PRs need consolidation.

2. **6.0 PR review bottleneck**: PRs #979, #997, #1026, #1029, and #1021 awaiting review by ergomorphic. RC1 is on testnet but mainnet soft-fork activation depends on review completion.

3. **SigmaUSD v2.0 migration timeline**: Phase 1 (v1.0) testnet deployment is underway, but the exact bank contract modifications required for v2.0 and its deployment timeline remain undefined.

4. **Bulletproofs range check bounty**: Assigned (issue #1032) but no one has committed to completing it. Needed to demonstrate 6.0 zero-knowledge capabilities.

5. **`equalBoxExcept` implementation scope**: Whether to include token-difference variants (e.g., `equalBoxExceptTokensDiff(b1, b2, Coll((token1, 1)))`) is unresolved; compiler feasibility also TBD.

6. **ErgoScript debug intrinsic**: Design sketched but no implementation timeline or integration plan with escript.online confirmed.

7. **Minotaur wallet signing failure**: Paideia transaction signing fails in Minotaur but works on Android; root cause not identified.

8. **Fleet SDK / GitHub Discussions migration**: Recommended but not yet implemented; documentation gap between Discord Q&A and formal developer reference remains.

9. **GraalVM native image for Ergo nodes**: mgPai exploring feasibility but viability for production node deployment not yet established.

10. **L2 merge-mined sidechain constructions post**: kushti flagged a forthcoming Ergo Forum post on sidechain constructions and BIP 301 comparison; not yet published.

11. **Block time reduction / throughput improvements**: Flagged as important but deferred to post-Ergoversary community discussion. Security implications (timestamp-dependent contracts, emission schedules) and design approach unresolved.
