# Ergo Developer Chat — Monthly Summary
**Platform:** Ergo Developer (Telegram)
**Month:** 2024-03

---

## Executive Summary

March 2024 was dominated by protocol stabilisation work — finalising the 5.0.21/5.1.0 release cycle, advancing Sigma 6.0 toward feature-completeness, and progressing sub-blocks and sidechain research toward publishable EIPs. Alongside core protocol work, several ecosystem projects (Dexy/GORT mainnet deployment, Paideia DAO MVP, Minotaur v2, Sigmanauts pool UI v2) hit significant milestones, while recurring developer-experience gaps — tooling, ErgoScript IDE support, git workflow standards — were identified and partially addressed.

---

## Key Topics This Month

### Protocol Releases: 5.0.21 & 5.1.0

Preparation for the 5.0.21 release consumed much of the month. Key merges included PR #2100 (mainnet), PR #2139 (integration tests), and PR #2135 (ergo-core CI publishing). P2SH was temporarily disabled again via PR #2140 pending the Sigma 5.0.14 release decision. The develop branch was brought to parity with master by end of month, with new PRs targeting the v5.0.14-RC branch directly. kushti noted that 5.0.x and 5.1.x releases will coexist through 2024. Compilation fixes in PR #2131 addressed the final blockers for both 5.0.21 and 5.1.0 finalization.

### Sigma 6.0 Finalization

kushti identified the likely final outstanding issues (#945 and #958 in sigmastate-interpreter) and began composing the corresponding EIP. `Some(...)` type handling was confirmed for Sigma v6.0 (sigmastate-interpreter issue #462). Code cleaning PR #957 was also raised. ErgoTree versioning — specifically whether P2SH addresses should encode the ErgoTree version byte — generated sustained discussion across weeks 10 and 11, with two paths identified: update P2SH documentation ecosystem-wide, or embed the version byte in the P2SH address encoding itself. No final decision was reached.

### Sub-blocks & Sidechain Research

Sub-blocks development is active in the weak blocks branch (PR #2107, PR #2055), with the sidechains branch building on top. kushti confirmed that sub-blocks will provide ADProofs for stateless/light clients, enabling state-transition verification without storing the full UTXO set. A comprehensive sidechain publication covering merge-mined, dedicated mining, and PoS variants was planned. Ergo-specific constructions — including BIP 301-style trustless pegging and emission contracts rewarding mainchain miners for delivering sidechain data — were described but not yet published as EIPs.

### Dexy / GORT / SigmaUSD Contract Work

GORT dev-emission and buyback contracts were deployed on testnet; Dexy was deliberately depegged on testnet to stress-test bank intervention logic and develop offchain code for extract-to-the-future actions. Pay-to-dev-emission script for GORT reached mainnet. Offchain integration with modified Spectrum bots was being verified. A new SigmaUSD v2 bank contract variant was sketched introducing a 0.5% minting limit per oracle epoch; a new repository with SigmaUSD contracts and tests was created with GORT-denominated bounties for contributors. The cooling-off height was removed from the v2 contract during cleanup.

### Paideia DAO MVP

Luivatra drove significant progress: an ErgoScript `#import` preprocessor for code reuse was implemented, staking rewards were refactored to draw from the DAO treasury on emission rather than pre-locking, and generic refund and config deserialization logic was added. Long-lived contracts were given an automatic refresh mechanism targeting a ~3-year lifecycle. The team is targeting EIP-5 migration after MVP completion and cutting unfinished features to harden the configuration against DAO self-locking. Crux accounting section went live mid-month.

### RocksDB & ARM Platform Support

RocksDB JNI binary compatibility across ARM32, ARM64, macOS, and Windows ARM was investigated. ARM32 is not supported by RocksDB JNI; ARM64 and x86 are functional. An ARM test snapshot binary was provided for Raspberry Pi testing. The Kiosk library was published under the ergo org with sonatype publishing infrastructure set up. ARM64 on some platforms showed a missing `libstdc++.so.6` dependency blocking full deployment.

### Developer Experience & Tooling Gaps

A recurring theme was the gap between Ergo's developer tooling and comparable chains. Compared to Cardano/Aiken, ErgoScript lacks: an import system, automated JSON compilation output, VS Code extensions, auto-formatting, build checks, and serialization/deserialization for composite types. kushti proposed a 1K SigmaUSD bounty for a Blockly-based ErgoScript visual code generator (signature and timelock combinations). Fleet SDK gained library-agnostic Prover/Verifier modules with adapters for sigma-rust and SigmaJS. Nautilus v0.93 was released; the extension↔dApp bridge module requires a full rewrite for Manifest v3 compatibility.

### Git Workflow Standardisation

A multi-day debate across weeks 10 and 11 addressed whether Ergo's core repositories should adopt enforced git standards (squash/rebase on merge, PR templates, issue templates). stenolog opened issue #2137 proposing rebase/squash as a binding rule; kushti supported documented process but resisted centralised enforcement. The discussion was heated enough that stenolog's account was muted. The proposal was framed as a core-dev manual item rather than DAO governance. No formal enforcement mechanism was adopted.

### Sigmanauts Mining Pool UI v2

Marc the Shark led a full dashboard redesign through the month — mobile-friendly layout, URL navigation by mining address, front page overhaul. A multi-user dashboard bug was investigated and fixed by switching from per-container API queries to database queries for scalability to 1K+ users. Locust was integrated for stress testing. The pool itself migrated to a hosted/cloud server with improved ping times; TLS/SSL and storage rent harvesting capabilities were in development. Beta launch was targeted for 29 March, full release the following week.

### Minotaur Wallet v2

Minotaur v2.0.0-rc2 and rc3 were released during weeks 10–11, with v2.0.1 reaching production in week 13. Cold signing and personal multisig wallets became available. A signing server for multisig and a dApp connector are under development. Bitcoin logic was finalised for Rosen Bridge integration. kushti required 3-of-5 multisig cosigners for an incoming ErgoDevs DAO donation; @CannonQ, @ccellado, @filippounits, and @ldgaetano volunteered.

## Technical Q&A Worth Preserving

- **Q** (@omniawaha, 2024-03-04): Can the interpreter process different ErgoTree versions? Can newer versions add new operations?
  **A** (@ergomorphic): The interpreter is the same across versions, but each version may add new operations so behavior is versioned. Current P2SH address implementation does not depend on ErgoTree version — a script can contain code not yet implemented in v5.0 but slated for v6.0, making those boxes unspendable until v6.0 activation.

- **Q** (@krasaviceblasen, 2024-03-12): Is there a protocol lower bound for `creationHeight` on output boxes? Could an attacker send UTXOs with `creationHeight=0` that get immediately claimed by storage rent?
  **A** (@Luivatra, @dusek_): Since the v5.0 soft-fork (Autumn 2022), consensus enforces that output box creation height ≥ max(input box heights). Before this rule, contracts had to guard against height-0 boxes manually. Mainnet has historical examples of boxes wrongly created with height 0 that were subsequently claimed as storage rent.

- **Q** (@kushti, 2024-03-13): If P2SH addresses don't encode ErgoTree version, how will multi-year trackers know which version's tree to monitor as ErgoTree versions evolve?
  **A** (@ergomorphic): Trackers can monitor ErgoTree bytes excluding the first byte (which contains the version), since the version byte is the only structural difference between trees across versions. Two resolution paths exist: (1) update P2SH documentation ecosystem-wide with tracking guidance, or (2) add the tree version byte to the P2SH address encoding (requires documentation updates either way).

- **Q** (@dusek_, 2024-03-12): Is `langspec.md` outdated? Can the compiler compile arbitrary lambdas in `flatMap` in v5.0?
  **A** (@ergomorphic): Yes, no restriction. For sigma-rust development, check the `_eval` methods in `methods.scala` in the Sigma sources.

- **Q** (@Luivatra, 2024-03-14): What is the biggest difference developing on Ergo vs. Cardano?
  **A** (@Luivatra): Both are eUTXO chains. The main practical difference is resources — Ergo has one person solving generic problems where Cardano has 20+, reducing incentive to build generic tooling. The key missing feature in ErgoScript is serialization/deserialization for composite types (structs), which Aiken handles elegantly.

- **Q** (@zkastn, 2024-03-26): Will sub-blocks provide ADProofs for light clients the way full blocks do?
  **A** (@kushti): Yes.

- **Q** (@omniawaha, 2024-03-26): What is an ADProof?
  **A** (@zkastn, @kushti): Authenticated Dictionary Proof. The UTXO set is stored in a special data structure allowing miners to provide small proofs for each transaction's block-state change. This enables stateless and light clients to verify transaction validity without storing the full UTXO set. Reference: https://docs.ergoplatform.com/node/modes/light-full-node/

- **Q** (@aquaritek, 2024-03-27): Can a smart contract on Ergo communicate with an outside API to get data?
  **A** (@mgpai): You would need to run an oracle. The smart contract then interacts with the oracle's on-chain output boxes.

- **Q** (@jossemii, 2024-03-18): Which Rust crate is current for connecting to Ergo — `ergo-headless-dapp-framework` (last commit 2 years ago) or `ergo-node-interface-rust`?
  **A** (@kushti): Use `ergo-node-interface-rust`; the Headless Dapp Framework has been largely unused since the SigmaUSD.io backend.

- **Q** (@__daddychill__, 2024-03-24): Is there a CLI utility to generate Ergo wallets offline?
  **A** (community): The Ergo node itself can initialise offline and generate a seed via CLI using the `/wallet/init` API endpoint with `curl`, providing an `api_key` and password string.

- **Q** (@kushti, 2024-03-13): What is the encryption format for secrets exported from SafeW wallet?
  **A** (@haileypdll): Mnemonic phrase encrypted with AES-256; key derived from `<wallet_password> + "SAFEW_PASSWORD_SALT"` where the salt is hardcoded as `vtvfAKubpNuc6Vn648TTUjh3KmuC8u`; mode CBC with PKCS7 padding (CryptoJS defaults).

## Open Questions Entering Next Month

1. **P2SH ErgoTree version encoding** — No decision reached on whether to embed the version byte in P2SH addresses or rely purely on updated documentation. Requires a formal resolution before Sigma 5.0.14 ships.

2. **Git workflow enforcement** — Rebase/squash proposal (issue #2137) has no enforcement mechanism; needs formalisation as a binding core-dev rule or archiving.

3. **Sub-blocks EIP publication** — kushti plans to publish comprehensive sidechain/sub-block documentation and EIPs; trustless relay and BIP 301-style pegging constructions remain in-progress.

4. **Extra indexes bug (issue #2141 / #2133)** — v5.0.20 users unable to enable extra indexes; test suite gap identified but root cause not isolated.

5. **ARM32 / Windows ARM64 support** — ARM32 not supported by RocksDB JNI; Windows ARM64 failing with AMD 64-bit DLL error; long-term build-from-source path not defined.

6. **SigmaUSD v2 adoption** — New bank contract with 0.5% minting limit per oracle epoch sketched but update key holders and community have not yet decided on adoption.

7. **ErgoScript tooling gaps** — Blockly bounty (1K SigmaUSD) for visual code generation posted but no volunteer developer confirmed; VS Code extension, import system, and composite-type serialization remain unaddressed.

8. **Stealth address receiving** — Satergo supports sending to stealth addresses; full receiving/scanning support across wallet implementations remains unclear.

9. **Nautilus Manifest v3 migration** — Extension↔dApp bridge module requires full rewrite; impact on existing dApp integrations not yet detailed.

10. **HTTP/2 for ErgoNode API** — Akka HTTP configuration flags not enabling HTTP/2 protocol negotiation; root cause and solution remain open.

11. **Stablecoin self-custody model viability** — New team's proposed fiat-backed stablecoin with non-expiring self-custody acknowledged as technically contradictory by qx(); deferred to testnet iteration.
