# Ergo Developer Chat — Monthly Summary
**Month:** June 2025
**Chat:** Ergo Developer

---

## Executive Summary

June 2025 was a landmark month for core Ergo protocol development, culminating in the release of Ergo node v6.0.0 followed rapidly by v6.1.0 with a RocksDB 10.2 storage backend upgrade. Alongside the major protocol milestones, significant progress was made on sub-blocks implementation, Sigmachains architecture research, and a healthy wave of dApp and tooling releases spanning AI agent integration, wallet hardware support, and DEX infrastructure.

---

## Key Topics This Month

### Ergo 6.0.0 & 6.1.0 Releases
The month's dominant technical story was the staged release of the Ergo protocol reference client. RC6 was published in early June for testnet validation, incorporating the EIP soft-fork with input/output block ordering, miner voting mechanisms, and compatibility fixes for ErgoTree v7 blocks. v6.0.0 shipped on 2025-06-23 with voting start height set to 1,561,601 — approximately 11.5 days before majority miner adoption was expected, contingent on pools updating. Within days, v6.1.0 followed with an upgrade from RocksDB 9.x to 10.2, resolving Docker base image compatibility issues that had affected 9.x deployments. Nodes upgrading from 5.0.27 or 6.0.0 require an extra indices resync, which operates locally against the existing database without requiring network re-sync. A 3-month grace period was announced for non-mining nodes, with no urgency for full-node operators to update immediately.

### Sub-blocks Implementation
Sub-block development progressed steadily across both weeks. Chained transaction handling and double-spending fixes were completed early in the month, with p2p layer issues identified and remediated. By late June, multiple-peer mining support and transaction propagation were under active testing, with a public testnet injection planned for July. Logging and event tracing improvements were also completed, strengthening observability for the upcoming broader rollout.

### Sigmachains & Merged-Mined Sidechain Research
kushti published and continued refining a Sigmachains refactoring plan on the Ergo Forum, describing an architecture that enables reuse of Ergo's contractual layer (Sigma), consensus (Autolykos), and p2p layer across off-chain environments, merge-mined sidechains, and dedicated Sigmachains with trustless pegging. A more speculative design was also floated: double Bitcoin + Ergo merge-mined sidechains in which ordering blocks and input blocks are split across the two chains, with Bitcoin UTXO set commitments via AVL+ trees enabling smart contract access. These proposals remained open for community discussion throughout the month.

### DexyGold Protocol & Stablecoin Ecosystem
The Dexy Protocol test UI was made live at `https://test.crystalpool.cc/` for private community testing. Oracle Pool v1 operates with Ergo Foundation members and external participants; Oracle Pool v2 runs Dexy Gold with GORT rewards. Bank reserves were reported to be growing faster than LP size; next milestones include a USD pool v2 upgrade and the launch of GluonUSD and DexyUSD stablecoins. kushti also proposed a stable-to-stable LP implementation as a future ErgoScript experiment, potentially crowdfunded via Ergo Raffle.

### Fleet SDK, ergoc & Tooling Releases
Fleet SDK v0.10.0 shipped in early June with ErgoBox cloning, context extension merging, EIP-12 transaction format support, and SigmaJS direct integration for faster builds and memoized compiler instances. ergoc v0.3.0 added template and contract hash reporting with lazy compiler initialization. A key serialization clarification emerged: the SDK enforces use of `SPair` (not `STuple`) for 2-element tuples in registers; tuples with more than 2 elements are unsupported per EIP.

### Machina Finance Grid Contracts
Machina Finance completed unit test coverage for ERG-to-Token (E2T) grid contracts with auto-compounding and partial fill support. Token-to-Token (T2T) grid contract unit tests were completed in the following week, with E2T undergoing size optimizations before final review. This represented full coverage of both major grid contract variants.

### Nautilus Wallet & Keystone Hardware Integration
Nautilus v1.2.1 was released with rendering fixes. Development focus shifted to Keystone hardware wallet integration, requiring custom ErgoTree serialization for reduced and signed transactions within Rust. A reference implementation was shared via `sigma-rust-mini`, and test vectors were published for community verification. The key insight from serialization discussions: tree headers use VLQ encoding and remain unparsed, with the tree staying in its raw byte representation during custom serialization flows.

### AI Agent Integration & MCP Servers
Multiple projects began leveraging LLM tooling against Ergo infrastructure. CodeUTXO published `ergo-mcp` (natural language node queries) and `ergo-price-mcp` (price data via LLM). Fintelligence AI (Marc the Shark) created RAG agents for smart contract research, generation, and validation. Top Byte was building on-chain token holder analytics. CodeUTXO also released a mempool visualizer with CEX/oracle/P2P colorization and a dead-simple explorer interface, with PostgreSQL direct lookups and server migration planned.

### executeFrom Semantics & Contract Patterns
A substantive discussion around `executeFrom*` function semantics clarified that these are macro-like constructs performing AST substitution from context variables or registers — not computed function arguments. kushti explained that an interesting pattern explored by Luivatra (a dataInput with an ErgoTree template in a register, enabling reusable functions via `executeFrom(substConstants(...))`) would require datainput binding in the contract, making conditions lengthier; this was flagged as a possible Ergo 7.0 feature.

## Technical Q&A Worth Preserving

- **Q** (@Luivatra, 2025-06-09): Why `executeFrom*` functions instead of a generic `executeFrom(byte: Coll[Byte])`?
  **A** (@kushti): `executeFrom*` functions are macros that take abstract syntax subtrees from context variables or registers, substitute `self`, then reduce the resulting AST. They are not computeable arguments — the distinction is AST substitution vs. runtime computation.

- **Q** (@Luivatra, 2025-06-09): Pattern idea — a dataInput with an ErgoTree template in a register, allowing reusable functions without transaction space or context overhead via `executeFrom(substConstants(...))`. Feasible?
  **A** (@kushti): Sounds interesting but requires datainput binding in the contract, making the condition lengthier. Flagged as a possible Ergo 7.0 feature with register-based `substConstants`.

- **Q** (@Pgr456, 2025-06-18): How to handle custom serialization of ErgoTree when working with reduced transactions in Nautilus?
  **A** (@anon_br): Use Fleet's box serializer and `SigmaByteWriter` for custom serialization. Use `SPair` (not `STuple`) for 2-element tuples; tuples with more than 2 elements are unsupported per EIP. Reference: Fleet SDK v0.10.0.

- **Q** (@Pgr456, 2025-06-20): Can you provide test vectors for Keystone custom serialization?
  **A** (@Pgr456): Provided reduced transaction and signed transaction hex test vectors with custom ErgoTree serialization in Rust. Reference implementation: `https://github.com/Alesfatalis/sigma-rust-mini/pull/3/files`

- **Q** (@arobsn, 2025-06-24): Are boxes with size-flag-enabled ErgoTrees affected by custom serialization in the Keystone flow?
  **A** (@Pgr456): Tree stays unparsed (VLQ + unparsed tree bytes); tree header is untouched (VLQ prefix preserved); only the wrapper changes.

- **Q** (@HQΣr, 2025-06-26): Example of how to include a tuple `(Coll[Byte], Long)` in a register using Fleet SDK?
  **A** (@anon_br): Use `SPair` for 2-element tuples. Tuples with more than 2 elements are unsupported per EIP.

- **Q** (@Aco, 2025-06-26): Can I update from 5.1.2 to 6.1.0 without a full resync?
  **A** (@kushti): An extra indices resync is required when upgrading from 5.0.27, 6.0.0, or 6.1.0. This indexing step does not require network access — it runs locally against the existing database.

- **Q** (@Pgr456 / @Pulsarz, 2025-06-23): For v6.1.0, should RocksDB roll back to 8.x or push forward to 10.x?
  **A** (@Pgr456 / @Pulsarz): Push forward to 10.x. Previous issues were Docker base image specific (not RocksDB 9.x itself); no known blockers for 10.x confirmed after testing.

- **Q** (@TMR.ΣRG, 2025-06-10): What is the recommended validation approach for SigmaQuest payment confirmation?
  **A** (@TMR.ΣRG): Server polls Ergo Explorer API every 30 seconds for transaction confirmation; on confirmation, creates a user session on the backend. Frontend polls backend for session existence. Migration to backend validation completed for security.
