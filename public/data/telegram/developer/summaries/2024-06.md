# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer | **Month:** 2024-06

---

## Executive Summary

June 2024 was dominated by three converging themes: finalizing the Sigma 6.0 / Ergo 5.x release pipeline (with significant cryptographic upgrades including BigInt support and Bulletproofs), advancing the Rosen Bridge Bitcoin integration, and maturing the developer tooling ecosystem (Nautilus, Fleet SDK, Trade House, Paideia, and Crystal Pool). Sustained technical debate around block time, throughput, and mempool design produced meaningful architectural consensus — favouring wallet/dApp-layer solutions over protocol-layer changes — while ErgoHack VIII and the approaching Ergoversary catalysed a wave of new project submissions and bounty activity.

---

## Key Topics This Month

### Sigma 6.0 & Ergo Node Releases

Development accelerated throughout the month. Early weeks saw multiple PRs merged or in review: SOption[] serialization (#990), Header serialization (#972), Header.checkPoW (#968), unsigned BigInt support (#997), and modular arithmetic operations (mod, modInverse, plusMod, multiplyMod, subtractMod). kushti also implemented toSigned/toUnsigned conversions and worked toward Bulletproofs range proof and circuit verification in ErgoScript. The EIP-50 draft consolidated all 6.0 scope including new collection methods (distinct, reverse), Bulletproofs, and rollup constructions. Node releases followed a cautious trajectory: 5.0.22 proved unstable and the plan shifted to releasing a slicing-only version first, deferring the larger changeset to 5.0.23. The roadmap through 5.0.24–5.0.25 was laid out to address candidate regeneration and PR #2072.

### Rosen Bridge Bitcoin Launch

The keygen ceremony for Bitcoin was completed in W23, with guards configured for BTC network operation. The team added Bitcoin observation/event trigger extractors, health check parameters, and native Bitcoin token support fixes in APIs. By W26, scanner improvements included Bitcoin RPC authentication, random IDs, and duplicate prevention. UI work covered fee fetching/caching, rate limiting, Xverse wallet core logic, and asset page optimisation. The bridge positions rsBTC as the primary wrapped Bitcoin asset on Ergo, with Analog Ergo's atomic swap approach offering a complementary trustless path.

### Nautilus Wallet & Fleet SDK

Nautilus underwent its most significant refactor in memory during June. Manifest v3 migration was completed, the dApp bridge layer was rewritten, and Ledger integration changes were prepared. Nautilus Abyss (canary channel) progressed from initial release through v0.10.2 and v0.10.4, with the stable v0.11.0 shipping in W26 with full EIP-44 arbitrary data signing support. Fleet SDK shipped v0.4.2 (replacing sigma-rust with SigmaJS as mock-chain backend) and v0.5.0. The community identified persistent gaps in practical documentation; a Fleet docs repo was opened for community tutorial contributions.

### Multisig & Signing Infrastructure

Multisig support matured substantially. The sigma-rust PR #751 was approved and an alpha WASM package (`ergo-lib-wasm-nodejs@0.28.0-alpha-03551ca`) published with `sign_tx_multi_input` support. dusek_ achieved a major performance improvement for `sign_data` — reducing 1000-output signing from 8 seconds to ~900ms in Rust — though C bindings and the ergo-lib-go implementation still showed degraded performance (50s+ for comparable input counts). The sigmastate-js PR #1012 added JavaScript multisig setup. Separately, c8 published a complete multisig-input test suite as a reference implementation.

### Block Time, Throughput & Mempool Architecture

An extended debate across W23–W26 examined the feasibility of shorter block times (e.g. 15 seconds). kushti provided the definitive technical argument against: the 8M execution cost limit per block requires ~8 seconds validation on average hardware, plus multi-hop propagation delays, making 2-minute blocks already conservative. The consensus that emerged was to solve UX problems (perceived slowness) at the wallet/dApp layer — chained transactions, unconfirmed balance visibility — rather than protocol changes. ergomorphic proposed an Atomic Transaction Dependency (ATD) mempool design for chaining transactions atomically; discussion surfaced important questions about propagation, eviction policies, and spam vectors, with the conclusion that dApps must still handle non-atomic inclusion as a failsafe. Sub-blocks, sidechains, and layer-2 solutions were acknowledged as future paths once congestion justifies deployment.

### DeFi Protocol Development

**Trade House** (order book DEX) launched with partial-fill support, full slippage control, and peer-to-peer matching without a centralised engine. **Paideia** contracts underwent a major refactor toward parameter-driven register handling, composable design, generic long-lived update mechanisms, and treasury dust consolidation — targeting community review by end of W25. **Crystal Pool** (Plasma-like offchain bank) demonstrated deposit-collateralised trading with timelock multisig preventing fund hostage-taking; c8 published a deposit proxy for mobile. **ChainCash** had a Fiat-Shamir vulnerability identified and fixed, and its paper was accepted to RAMICS'24. **Dexy Gold** mainnet LP launch was under discussion by month's end.

### ErgoScript Tooling & No-Code Development

c8 built a Blockly-based visual ErgoScript editor at escript.online/blockly, supporting multisig construction (&&, ||, AT_LEAST). The `ergoblocks` npm package was published, enabling fully client-side no-code smart contract creation hosted on GitHub Pages. kushti announced bounties of 1K SigmaUSD for Blockly/ErgoScript integration supporting multisig schemes and 500 SigmaUSD for Bulletproofs verification, plus 200 ERG for EIP-50 instructions implementation.

### Palmyra & Chainlink Integration

Dan (Ergo Foundation Advisor) detailed Palmyra's use of Chainlink's proof-of-reserve oracles with IoT sensors (cameras, weight/motion sensors) for on-chain warehouse receipts underpinning decentralised trade financing. The architecture is not zero-sum with Ergo oracles: the plan includes investing token sale capital into building out native Ergo Oracle Pools (both v1 and v2, the latter already running for Dexy Gold with GORT rewards). The broader point — that companies adopting subsets of the Ergo ecosystem is positive, not a sign of disloyalty — generated substantial community discussion.

### Sigmanauts Mining Pool

Production PPLNS implementation went live with RSN payouts to opted-in miners via custom scripts. Mining Core database was mirrored without modifications to preserve integrity. Dashboard APIs were constructed and the UI moved to Flux infrastructure to reduce server load. NFT register exploration (r9 register) and an rsBTC Bonus Raffle were added; swap APIs for beta finalised by W26.

## Technical Q&A Worth Preserving

- **Q** (@justblake3911, 2024-06-03): Node wallet password error ("Tag Mismatch") after adding full indexing config (`extraIndex = true`, `extraCacheSize = 500`)
  **A** (@Aberg): Delete the `.ergo/wallet` directory and restore with a different password using the seed phrase.

- **Q** (@justblake3911, 2024-06-03): How to obtain total balance of all assets in a multi-address wallet via API?
  **A** (@kushti): Use `/wallet/balances` or `/wallet/balancesWithUnconfirmed` endpoints.

- **Q** (@c8, 2024-06-10): Three-step security model for verifying wallet ownership in a web app?
  **A** (@c8 + @HQΣr): (1) Browser proves address ownership via signed message/transaction; (2) server verifies proof; (3) server checks valid "Login Key" asset at address. ErgoAuth provides a "connect" flow without requiring asset transfer — preferable to rolling your own.

- **Q** (@c8, 2024-06-10): Can bots circumvent wallet-signing captcha?
  **A** (@c8): Yes — bots can generate 1,000 wallets and sign transactions with no added effort. ErgoAuth or social proof (e.g. verified Twitter) is a stronger deterrent.

- **Q** (@mgpai, 2024-06-10): What is the relationship between sigmastate-interpreter and AppKit?
  **A** (@ergomorphic): AppKit depends on sigmastate-interpreter transitively via build.sbt. SNAPSHOT versions can be used directly. Standard Scala transitive dependency relationship.

- **Q** (@c8, 2024-06-12): How to test individual inputs in a signed transaction with `ergo-lib-wasm-nodejs` without an external API?
  **A** (@dusek_): Use `validate_tx` for complete transaction validation or `verify_tx_input_proof` for single-input proof validation.

- **Q** (@c8, 2024-06-14): What keywords help learn the signing math behind Ergo?
  **A** (@dusek_): Study Schnorr signatures (conduition.io post); note Ergo notation "a * G" equals "g^a" in standard literature. DLOG is the foundation.

- **Q** (@kii, 2024-06-17): How to extract SigmaProp from a testnet multisig address created in Minotaur for use in ErgoScript contracts?
  **A** (@kushti): Use `ErgoAddressEncoder(TestnetNetworkPrefix).fromString(addr).get.script.toProposition(true)` to get the proposition with constants substituted. Wrap in `SigmaPropConstant(script)` for register placement. For mainnet, prefer constructing directly from keys via `CTHRESHOLD(1, keys)` to avoid address-parsing complexity.

- **Q** (@c8, 2024-06-18): Definition of a full mempool and how to measure utilisation?
  **A** (@kushti): Mempool capacity is set via `ergo.node.mempoolCapacity` (default 1,000 transactions). Max tx size from P2P is ~100 KB; no limit applies from local submission. No direct utilisation percentage metric is exposed in the standard API.

- **Q** (@Aberg, 2024-06-19): How to convert the new `Ecp` type returned by `CryptoConstants.dlogGroup().generator()` to a BouncyCastle `SecP256K1Point`?
  **A** (@ergomorphic): On the JVM, use `.value` to extract the underlying point.

- **Q** (@Aberg, 2024-06-19): Alternative to the removed `sigmastate.basics.SecP256K1` class for decoding points?
  **A** (@ergomorphic): Use `Platform.createContext().decodePoint(pointBytes)`, which returns `Ecp`. Reference: `CryptoContextJvm#L29`.

- **Q** (@mgpai, 2024-06-24): What is the security argument against 15-second block times on Ergo?
  **A** (@kushti): The 8M execution cost limit per block requires ~8 seconds validation on average hardware. Multi-hop network propagation compounds this. Even 2-minute blocks are conservative; shorter blocks force smaller block sizes, eliminating throughput gains. Miners could also issue maximally heavy blocks to handicap competitors (the verification dilemma).

- **Q** (@c8, 2024-06-24): How would Ergo handle a DDoS spam attack similar to Cardano's (10,000 TXs/min)?
  **A** (@kushti + team): The fee market solves this. Reference client (v4.0.44+) has spam protection limits — nodes reject excessive transactions from a single peer. Mempool timeout flushes unconfirmed low-fee transactions after ~1–2 hours. Node operators can raise minimum propagation fee rate to block spammers from reaching their mempool entirely. A fee estimation API exists on the node but is not yet widely used by applications.

- **Q** (@c8, 2024-06-26): Does Ergo include a commitment to UTXO state in block headers?
  **A** (@kushti): Yes. Ergo block headers include an AVL+ tree digest over the UTXO set, enabling partially stateless clients. Reference: https://m.youtube.com/watch?v=PHY7JnLrK5o

- **Q** (@c8, 2024-06-28): Can chained transactions be bulk-submitted in one API request?
  **A** (@ergomorphic): Proposed as part of the ATD design — a single API request with a TX list is possible — but requires mempool admission and eviction policy design first. Currently requires sequential submission.

- **Q** (@kii, 2024-06-28): What are the practical limits on data storage on Ergo?
  **A** (@team): ~8 MB blocks yield ~5.7 GB/day maximum. Single transaction limit is 96 KB. Large submissions require very high miner fees. Block size is intentionally constrained to prevent full-node centralisation.

## Open Questions Entering Next Month

- **5.0.22 / 5.0.23 stability**: 5.0.22 proved unstable; the slicing-only approach was chosen but timing and confirmation of stability remain TBD before 5.0.23 (large changeset) can proceed.
- **Bulletproofs verification in ErgoScript**: kushti working toward completion; no confirmed timeline for the 500 SigmaUSD bounty deliverable.
- **ATD mempool spam vector mitigation**: ergomorphic confirmed the trade-off between ATD chain size and propagation time but eviction policy specifics remain unresolved.
- **C bindings / ergo-lib-go performance gap**: dusek_'s Rust optimisations (900ms) are not translating to Go (50s+). Root cause suspected in C binding cloning but needs deeper investigation.
- **Fleet SDK documentation**: Multiple developers flagged lack of practical tutorials for register serialisation, transaction building, and Nautilus connector integration. Fleet docs repo opened but no prioritised bounty or assignment made.
- **Stable RocksDB rollout**: kushti recommends RocksDB over LevelDB but is waiting for 5.1.0 stability confirmation before production guidance.
- **Mobile node architecture**: Debate between auto-provisioning light nodes in consumer apps vs. user-selectable server options unresolved; Satergo autostart/system service integration status unclear.
- **Chain reorganisation statistics**: Request for Ergo reorg frequency and depth data went unanswered across multiple weeks.
- **Satergo Android cold signer via Bluetooth**: Fundraiser concept raised; demand assessment and goal TBD.
- **ErgoTree decompilation in escript.online**: ErgoTree → ErgoScript restoration is technically possible but non-trivial; flagged as unfinished work with no assigned timeline.
- **Dexy Gold mainnet LP launch**: Under discussion as of W26; no confirmed date.
- **Michael's wallet restore address derivation issue**: Address differs across restores on Windows 5.0.21 nodes; root cause (config, mnemonic format, node version) still unclear despite kushti requesting config PM.
