# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer | **Month:** 2024-08

---

## Executive Summary

August 2024 was dominated by the progression of Sigma 6.0 toward testnet activation, with multiple PRs merged into the candidate branch and a 6.0.0-alpha0 release shipped by month's end. Alongside this, sub-blocks research advanced from concept to early implementation refactoring, while the broader ecosystem saw significant tooling releases (Nautilus v0.13.0, Fleet SDK v0.6.1, Sigma Rust 0.28) and a wave of dApp projects (ChainCash, Dexy, Paideia, The Field, Lithos) moving from development into active testnet and mainnet phases.

---

## Key Topics This Month

### Sigma 6.0 Development & Activation

The month began with multiple Sigma 6.0 PRs (#972, #968, #965, #1008, #962, #1013, #1015, #1016, #979, #1025) under review and polishing. kushti completed costing tests, merged several PRs into the 6.0.0 candidate branch, and created a devnet PR (#1025) specifically for testing Lithos protocol features including `checkPoW`, nBits methods, and header serialization/deserialization. By week 35, a 6.0.0-alpha0 release was shipped with successful 6.0 transaction submission tests confirmed. Activation details were being refined in PR #2152, with mainnet activation date not yet finalized. Community contributors were called to assist with Merkle batch proof tests and cross-version (5.x/6.0) transaction compatibility testing.

### Sub-blocks Architecture & Block Time Research

Sub-blocks moved from playground experimentation into production architecture work: pre-subblocks refactoring was merged (#2180), and the p2p message layer for sub-blocks entered development. kushti proposed a community discussion post-Ergoversary on block time improvements and sidechain constructions. The Tailstorm proposal was flagged as an alternative sub-blocks scheme for review. A recurring concern throughout the month was the impact of faster block times on dApps relying on block height rather than timestamps — kushti noted these dependencies require careful analysis before any parameter changes. A testnet channel was made available for interested contributors wanting to experiment with tweaked parameters; no-code participation (running testnets, mining, observing) was explicitly welcomed.

### Context Extension Serialization Bug

A significant technical issue was identified late in the month: transaction serialization order for context extension variables differs between sigma-rust and the Ergo reference implementation due to HashMap ordering. This caused transaction ID mismatches when transactions passed through Nautilus (which sorts inputs before signing), breaking contract execution for time-sensitive or context-dependent scripts. kushti confirmed the root cause: sigmastate-interpreter uses a standard HashMap whose order depends on insertion order and map size; a `SigmaMap` implementation (PR #951) was started but remains unfinished. Fixing this properly would require a hard fork, as soft-forks cannot change serialization format. A related practical manifestation was Luivatra's P2S contract with a time-based condition that expired between construction and submission — partially obscuring the underlying serialization issue.

### LevelDB → RocksDB Migration

The Ergo node's dependency on `leveldbjni-all` from an Ethereum fork became a blocking issue when the source repository (Spring/softmotions) became unavailable. Short-term: Pgr456 published the artifact to a GitLab public Maven registry as a temporary fix. Long-term: RocksDB migration was confirmed for v5.0.23 (released during the month), eliminating the problematic dependency. The broader indexer rewrite — pragmaxim's new Rust-based `chain-indexer` using RocksDB — was announced with a prototype API target of late October and production readiness expected by winter 2024, designed to run on raspberry pi-level hardware.

### Node Stability & API Improvements

Ergo node v5.0.22 was released at the start of the month; v5.0.23 followed with RocksDB and a fix for the indexer/API thread pool conflict (where the indexer gradually displaced API threads, causing UI lockups). PR #2164 by Pgr456 added block refreshing every 20 seconds to include new mempool transactions, reducing average confirmation latency without affecting consensus rules. kushti also proposed new mempool API endpoints (`GET /transactions/unconfirmed/ids` and `POST /transactions/unconfirmed/byTransactionIdList`) and a `lastMempoolUpdate` timestamp in `/info` for efficient differential tracking — both approved in principle with GitHub issues opened.

### Ecosystem dApp Progress

Multiple projects reached significant milestones concurrently:
- **ChainCash**: First mainnet transactions executed (reserve mint, note mint, note spending) via the HTTP server API; a contract bug preventing note double-spend per pubkey was found and fixed; RAMICS paper in progress.
- **Dexy**: LP mint deployed, LP redemption deployed, arbitrage mint contract ready for deployment.
- **Paideia**: Backend synchronization completed; generic notification service infrastructure finished with plugins for on-chain transaction confirmations; sigma-rust bug (#763) and contract bug (#232) found during testing.
- **Lithos Protocol**: Initial client finished; MEV integration and NISP (NIPoPoWs in smart contracts) development started for 6.0 devnet.
- **The Field** (peer-to-pool pledging): MVP contracts complete with redemption logic; semi-private testing underway.
- **Rosen Bridge**: EVM chain support (Ethereum) added across watcher, scanner, guard service, and health check components; first and second round EVM testing completed with ongoing bug fixes.

### Wallet & SDK Releases

Nautilus progressed from v0.13.0-beta.1 (40% memory reduction, 60% faster loading) through beta.3 (mempool support at dApp Connector level, Fleet SDK blockchain provider migration) to the stable v0.13.0 release. The `qrcode` library was replaced with `uqr`, removing 29 sub-dependencies. Fleet SDK shipped v0.6.1 (token limit enforcement at 100 per box, query pagination fixes for 20+ addresses, Header#votes encoding fix) and `@fleet-sdk/blockchain-providers@0.6.2` (take/skip streaming parameter support). Sigma Rust 0.28 was released with breaking changes. A practical developer note: Fleet SDK's `TransactionBuilder#configure((s) => s.setMaxTokensPerChangeBox(100))` should be used to prevent signing failures caused by box size limits being exceeded when change boxes hold many tokens.

## Technical Q&A Worth Preserving

- **Q** (@c8, 2024-07-31): Any plan to add socket subscription for new blocks, reorgs, sub-blocks, and mempool events to the node?
  **A** (@kushti): No concrete plan yet. Recommended approach: poll `/info` endpoint regularly, detect changes via `unconfirmedCount` and block info fields, then call API methods for details. Alternative is p2p connections but harder to maintain and p2p behavior may change.

- **Q** (@c8, 2024-07-31): Does mempool transaction ordering guarantee reliable offset pagination for tracking new transactions?
  **A** (@kushti): Transactions are sorted by inclusion priority (fee), not insertion order. Cannot rely on offset/limit for differential queries. New endpoints (`GET /transactions/unconfirmed/ids`, `POST /transactions/unconfirmed/byTransactionIdList`) and a `lastMempoolUpdate` timestamp in `/info` were proposed and approved in principle.

- **Q** (@arobsn, 2024-08-02): Vite config for Svelte + WASM (`ergo-lib-wasm-browser`)?
  **A** (@c8): Use `vite-plugin-wasm` with `optimizeDeps` including `ergo-lib-wasm-browser` and set `target: 'esnext'`. SvelteKit uses Vite instead of webpack, so no webpack config needed.

- **Q** (@Aco, 2024-08-07): Fleet SDK splits change boxes at 120 tokens but doesn't account for max box size (4096 bytes), causing signing errors.
  **A** (@arobsn): Use `TransactionBuilder#configure((s) => s.setMaxTokensPerChangeBox(100))` to set a safer limit; this will become the default in future releases.

- **Q** (@Aberg, 2024-08-12): What is the correct way to calculate minimum box value for large transaction sets (e.g., 1000 outputs)?
  **A** (@kushti): Take output candidate size, add 34 bytes for transaction ID + output index (both serialized with VLQ encoding, so variable-length for large indices). Tokens also consume space in the token ID map. The naive `bytes × 360 nanoErg` formula only accounts for output candidate bytes and misses transaction-level overhead at scale.

- **Q** (@Luivatra, 2024-08-14): "Malformed transaction: Scripts of all transaction inputs should pass verification" with code 403 — what does this mean?
  **A** (@dusek_): Code 403 here is not an HTTP error code but a script execution cost; it indicates the input script reduced to `false` on the Ergo node but passed sigma-rust. Root cause was a P2S input with a time-based condition that had expired by the time the transaction was submitted.

- **Q** (@Luivatra, 2024-08-14): What is included in transaction ID calculation?
  **A** (@kushti): Transaction ID is a hash of transaction bytes without signatures (Ergo implements witnesses segregation). Differing tx IDs between sigma-rust and node indicate serialization differences upstream — specifically, context extension variable ordering via HashMap.

- **Q** (@deathgripson, 2024-08-19): Could you set a different generator point in ErgoScript to use a different elliptic curve?
  **A** (@kushti): secp256k1 subgroup has cofactor = 1. GroupElements have deserialization checks enforcing curve validity. Not all curves are DDH-secure; any candidate would need careful vetting.

- **Q** (@dusek_, 2024-08-26): Why does context extension serialization order differ between sigma-rust and the Ergo reference implementation?
  **A** (@kushti): sigmastate-interpreter uses a standard HashMap for context variables, making order dependent on insertion order and map size. A `SigmaMap` implementation (PR #951) was started but remains unfinished. Fixing this properly requires a hard fork, as soft-forks cannot change serialization format.

- **Q** (@Michael, 2024-08-28): What causes the node UI to become unresponsive, requiring restart?
  **A** (@kushti): The indexer and API share the same thread pool; the indexer gradually displaces API threads over time. A fix isolating the thread pools was tested and included in v5.0.23.

- **Q** (@deathgripson, 2024-08-28): Node process randomly stops without error.
  **A** (@kushti): Likely receiving SIGHUP when the terminal session ends. Use `nohup ./start.sh &` to prevent signal delivery to the node process.

- **Q** (@Luivatra, 2024-08-26): Can context extension order be preserved through Nautilus signing?
  **A** (investigation): Nautilus sorts inputs before signing (see prover.ts#L104), likely causing reordering. Whether JSON serialization or sigma-rust prover is the reordering point requires further investigation. EIP-12 may need to define canonical ordering.

## Open Questions Entering Next Month

- **Context Extension Serialization Canonical Order**: Should EIP-12 be extended to define canonical ordering for context extension variables to prevent tx ID mismatches between clients? Is a hard fork the only proper fix, or can applications work around HashMap ordering in the interim?

- **Sigma 6.0 Mainnet Activation Timeline**: Devnet testing ongoing; exact activation date not set. Dependent on community testing of Merkle batch proofs and cross-version (5.x/6.0) transaction compatibility.

- **Faster Block Times & Timestamp Dependencies**: Which deployed dApps and contracts rely on block height rather than timestamps? This analysis is a prerequisite before any block time parameter changes can proceed safely. No volunteer has yet taken ownership of the testnet experiment group.

- **Sub-blocks Production Architecture**: p2p message layer under development but no integration test plan yet. Tailstorm as alternative scheme awaits formal evaluation. Community discussion post-Ergoversary not yet scheduled.

- **pragmaxim Chain-Indexer REST API Design**: Autogenerated vs. hand-written + WASM client remains undecided; WASM integration producing mysterious bugs. Prototype API compatibility target is late October.

- **AppKit Maintenance & CVEs**: Aberg identified vulnerable dependencies in AppKit but specifics were not documented in chat. Remediation status and whether a coordinated release plan exists remains unclear.

- **Node Wallet Derivation Path vs. Nautilus**: Why does the Node REST API derivation endpoint produce different keys than Nautilus for the same path `m/44'/429'/0'/0`? Whether this is an API parsing bug or intentional difference is unresolved.

- **Ergo-lib-iOS Deprecation**: No response recorded to the question of whether `ergo-lib-ios` bindings are deprecated or planned for update. Swift development pathway for Ergo unclear.

- **NISP Integration in Lithos**: Specific details on how NIPoPoWs integrate with Lithos smart contracts deferred to a forum post that had not been published by month's end.

- **Documentation Gaps**: kushti flagged five topics needing forum coverage — Merkle tree format, Extension block section format, soft-forkability rules, signature scheme internals, and ErgoScript compiler phases — but work had not started as of month's end.
