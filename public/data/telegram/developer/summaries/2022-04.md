# Ergo Developer Chat — Monthly Summary
## April 2022

---

## Executive Summary

April 2022 developer discussions centred on node infrastructure resilience, protocol internals, and tooling gaps in the Ergo ecosystem. A recurring theme was the centralisation risk posed by over-reliance on a small number of public nodes, alongside detailed technical exchanges on UTXO semantics, cost estimation, and sync protocol compatibility. The month also saw active documentation and release work, with node version 4.0.27 shipped and 4.0.28/4.0.29 planned.

---

## Key Topics This Month

### Node Centralisation Risk
A significant concern was raised regarding the majority of Ergo wallets relying on kushti's single public node (213.239.193.208), exposing the ecosystem to DDoS attacks, government action, or compromise. Aberg documented that Satergo is the primary exception, as it supports local node hosting. The architectural question of whether HTTP node APIs are even necessary was raised, with direct P2P connections proposed as a more decentralised alternative. Aberg acknowledged the problem as serious and noted work was underway to address it, though specifics were not disclosed.

### UTXO Semantics & Transaction Ordering
Developers explored whether transactions could be guaranteed inclusion in the same block. The conclusion was that no reliable mechanism exists for block-level co-inclusion guarantees in Ergo. Execution sequencing is governed by UTXO ordering rather than block-level scheduling — critically, a UTXO cannot be spent before it is created. This has implications for multi-transaction dApp flows that assume atomic cross-transaction dependencies.

### Cost Estimation & Proof Size
A notable technical discovery was that proof size — not computation time — is the dominant factor in Ergo's cost estimation model. Proof size scales at O(log n), per the cryptographic research on authenticated dynamic dictionaries (ePrint 2016/994). This has practical implications for contract designers and wallet developers optimising transaction costs.

### EIP-27 & AppKit Integration
Discussion covered EIP-27 flag requirements in wallet implementations. It was clarified that AppKit can leverage the ergo-wallet box selector from the Ergo node repository following the relevant updates, streamlining wallet integration with the new emission soft-fork flag handling.

### Node Releases & ErgoHack Preparation
kushti outlined a near-term release plan: 4.0.28 to ship with additional PRs, followed by 4.0.29 assembled from a backlog of ready PRs. There was expressed interest in having outstanding work finished and tested in time for ErgoHack. Documentation PRs covering P2P networking and sync info formats were also under active review.

## Technical Q&A Worth Preserving

- **Q** (@unknown, 2022-W16): What Scala versions are supported?
  **A** (@kushti): 2.12 and 2.13 are both supported.

- **Q** (@unknown, 2022-W16): Can transactions be guaranteed inclusion in the same block?
  **A** (@kushti): No good mechanism exists for this guarantee. UTXO ordering defines execution sequence — a UTXO cannot be spent before it is created, but block-level co-inclusion cannot be enforced.

- **Q** (@unknown, 2022-W16): What is the dominating factor in cost estimation?
  **A** (developers, 2022-W16): Proof size, not computation time, dominates cost estimation. Proof size is O(log n) per authenticated dynamic dictionary research (ePrint 2016/994).

- **Q** (@unknown, 2022-W16): When a script reduces to false, what happens?
  **A** (developers, 2022-W16): The box cannot be spent — the guardian condition of the script is not satisfied, so the spending transaction is invalid.

- **Q** (@unknown, 2022-W16): What format are `.conf` files in?
  **A** (developers, 2022-W16): HOCON (Human-Optimized Config Object Notation), per the Lightbend specification.

- **Q** (@unknown, 2022-W16): Are the old and new sync info formats backwards compatible?
  **A** (@kushti): Yes, backwards compatibility is maintained. Aberg subsequently noted his local git clone was outdated and he had missed the relevant changes.

## Open Questions Entering Next Month

- **Node Decentralisation**: Aberg indicated work was underway to reduce reliance on single public nodes, but no specific solution was detailed. This remains an open ecosystem risk.
- **Fee Estimation API**: The API exists in the node but lacks adoption and testing by applications. Whether this gets prioritised or documented remains open.
- **Transaction Co-inclusion Guarantees**: No mechanism exists for guaranteeing multiple transactions land in the same block. Whether a protocol-level solution is desirable or feasible was not resolved.
- **Documentation PR Reviews**: PRs #9 (P2P network docs) and #11 (sync info docs) were awaiting review at week's end.
