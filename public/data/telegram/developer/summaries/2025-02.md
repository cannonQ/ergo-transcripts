# Ergo Developer Chat — Monthly Summary: February 2025

---

## Executive Summary

February 2025 was dominated by two converging tracks: the progression of the Protocol 6.0 soft-fork toward mainnet readiness (including sub-blocks/input blocks implementation, public testnet activation, and sigmastate-interpreter audit work), and a wave of dApp and tooling maturation across Dexy Gold, Rosen Bridge, Nautilus wallet, and oracle infrastructure. A secondary theme emerged around decentralization of dApp architecture and novel applications of LLMs for contract auditing and test generation.

---

## Key Topics This Month

### Protocol 6.0 Soft-Fork & Sub-Blocks

Public testnet voting for the 6.0 soft-fork activated on 2025-02-14 with 6.0.0-RC2, following the introduction of block version parameter ID 120 (soft-fork signal requiring >90% hashrate consensus across 128-block epochs). By week's end, 780,000 votes had been recorded. The core team focused on finalising the sigmastate-interpreter PR #1054 audit and code cleanup throughout the month. Sub-blocks (input/ordering blocks) implementation progressed from initial DAG representation in RAM and the `bestInputBlock`/`bestInputChain` API methods toward addressing fork resolution, out-of-order block handling, and missing parent reconstruction — all of which remained works in progress entering March. Context extension API methods were identified as missing and slated for v5.0.26. A dedicated testnet channel was flagged as needing motivated community members to run parameter-tweaking experiments.

### Node Stability & Infrastructure Releases

A significant node stability improvement landed mid-month: Ergo 5.1.2 replaced Java-based LevelDB with RocksDB, cutting memory usage from 5GB+ to ~300MB and resolving persistent issues on macOS and Windows. The companion 5.0.25 release introduced extra indices support. Sigma SDK 5.0.15 added `executeFromVar` (macro-like script deserialization from context extension variables) and `bigInt` construction from scripts, with ContractTemplate JSON parsing and Box serialization/deserialization in JavaScript. Fleet SDK integration of Sigma 5.0.15 was confirmed planned but without a committed timeline.

### Oracle Infrastructure & SigmaUSD Bank Upgrade

The ERG/USD oracle pool underwent a successful ~20-minute update ceremony (2025-02-14), enabling migration to Oracle Pool v2 and unblocking downstream SigmaUSD and Dexy Gold workflows. By late February, a SigmaUSD bank contract upgrade was planned for the same week, with contract hash votes to be submitted and mainnet Dexy token issuance targeted for the following week. Separate discussion emerged around whether the upgrade should migrate away from fixed input/output indices — a change described as trivial to audit but requiring roadmap clarity on minting limits, key destruction, and future update governance.

### Dexy Gold

LP contracts were deployed to testnet in week 6. Throughout the month, the ErgFi (Savonarola Labs) team built out the swap widget, history integration, mempool_server connectivity, and ErgoPay support. The alpha UI was targeting end-of-week release by week 8, with ergfi.xyz confirmed ready for full release in week 9. A LP/bank widget and fake token swap testing pipeline were actively in progress as the month closed.

### Rosen Bridge

R&D on Bitcoin Runes integration completed and documentation finalised; development phase was set to begin imminently. Binance token support and integration were planned for release within days as of week 7. Watcher V2 expanded its observation status service with transaction pool, database, and scanner dependencies. Full QA test coverage was achieved across five scenarios (invalid to-address, cold storage, arbitrary order, event reprocess, event synchronization). Nami wallet support was discontinued. Manual wallet testing preceded public site opening.

### Nautilus Wallet Redesign

The wallet underwent a substantial architectural overhaul across the month: component refactoring (AddWalletIndexView, CreateWalletView, LedgerConnectView), Tailwind v4 and reka-ui v2 migration, chart.js integration, recovery phrase expansion to 12–24 words, and new token logos (rsBNB, rsETH). A first beta was released to Google Play's Abyss channel on 2025-02-28. Active debate continued on feature prioritisation: internationalization, a headless `nautilus.ts` library, password caching, multi-recipient transactions, and 0-conf UTXO streaming via `window.postMessage()` (constrained by Manifest v3 service worker limitations) were all flagged but not committed. Ledger hardware wallet integration completed a third-party audit and was confirmed production-ready, awaiting an official release date from Ledger HQ.

### dApp Architecture & Decentralisation

A substantive architectural debate ran through weeks 8 and 9. The observation that most dApps depend on centralised explorer backends prompted a proposal for a lightweight, stateless, open-source `mempool_server` socket abstraction (Elixir-based) allowing dApps to query nodes directly. The key framing: "open source + works with local node → dApp; closed source → website." Tension between socket abstraction introducing new centralisation risks versus the practical UX advantages of centrally hosted services was partially resolved by the self-hosting option but not fully settled.

### LLM-Based Contract Auditing & Test Generation

mgPai proposed using GRPO (Group Relative Policy Optimization) reinforcement learning fine-tuning on 8 × A100 80GB GPUs to automate contract test generation and verification. A dedicated Discord and Telegram group were spun up. The Sigmanauts mining pool fundraised 190 ERG for LLM-generated smart contract auditor fees, and a local-execution UI was built. Concurrently, kushti proposed using LLMs for automated vulnerability pattern detection (box spending under same contract, forgotten register checks, premature box satisfaction). Phoenix HODLcoin contracts and mockchain test suites were identified as high-quality training data sources. No proof-of-concept results were published during the month.

### Lithos Protocol

Fraud proof iteration and significant codebase cleanup were underway with a public release targeted before testnet launch. Stratum implementation was being reworked due to nbminer compatibility issues.

## Technical Q&A Worth Preserving

- **Q** (@TheStophe, 2025-02-04): What does `120 = 1` in the testnet mining node config represent?
  **A** (@cannon_q / @dusek_): Parameter ID 120 is a votable soft-fork signal. When >90% of hashrate votes for it across an epoch, the block version increments (soft-fork activation). Other votable parameters: block size (ID 3), cost per transaction output (ID 8), storage fee (ID 1). Reference: [voting.tex](https://github.com/ergoplatform/ergo/blob/8e1489314c06406b2979df27e08efa2c71b45adb/papers/yellow/voting.tex#L48)

- **Q** (@devjoe4ever, 2025-02-13): Can `executeFromVar` be called recursively within a script it deserialises? How is infinite recursion prevented?
  **A** (@dusek_): Not possible. `executeFromVar` instances are replaced with expressions from the ContextExtension *before* script evaluation — it operates as a macro. Recursion is detected at compile-time, not runtime.

- **Q** (@Aco Šmrkas, 2025-02-17): Node `/info` returns incorrect version; `scorex.restApi.nodeName` has no effect.
  **A** (@Glasgow): Correct parameter is `scorex.network.nodeName`, placed under the `[scorex.network]` bracket — not the `restApi` section. See [Scorex network configuration docs](https://docs.ergoplatform.com/node/conf/conf-scorex/#network-configuration).

- **Q** (@Aco Šmrkas, 2025-02-17): UTXO set database corruption ("Readers are not initialized yet") on Ubuntu VPS.
  **A** (@kushti): Identified as UTXO set corruption. Resolved by copying a clean `.ergo` directory from a stable system. Resulting RAM usage ~620 MB.

- **Q** (@Luivatra, 2025-02-19): Should dApps normalise the ability for users to choose their own node rather than relying on centralised explorers?
  **A** (@c8e4d2a): Direct node queries lead to poor dApp UX. Proposed solution: a lightweight, stateless, open-source `mempool_server` socket abstraction layer. dApps query the socket rather than the node directly. Can run on a local node for fully offline capability: "open source + works with local node → dApp; closed source → website."

- **Q** (@c8e4d2a, 2025-02-26): With 0-conf support, should Nautilus stream UTXO changes via `window.postMessage()`, or should dApps poll every 10 seconds?
  **A** (@arobsn): Streaming is ideal but problematic under Manifest v3 — service workers are designed to passively respond, and hacks to keep workers alive indefinitely risk Google Play review rejection. Requires further research.

- **Q** (@krasaviceblasen, 2025-02-26): Will the SigmaUSD upgrade move away from fixed input/output indices?
  **A** (@c8e4d2a / @Luivatra): No PR or audit exists yet for an index-agnostic update. The primary goal of the current upgrade is saving the update UTXO from storage rent. The change itself would be trivial to audit (effectively a one-line change) but depends on broader roadmap clarity around minting limits and key destruction.

- **Q** (@novamon_25735, 2025-02-26): How to debug ErgoTree directly (AST interpretation, not bytecode)?
  **A** (@kushti): ErgoTree is direct-style interpretation of typed ASTs — a compact on-chain representation. ErgoScript compiles to ErgoTree with optimisations applied. Direct debugging is non-trivial (unlike bytecode-level tools) but has been approached in systems like GraalVM; the methodology needs review.

- **Q** (@mgpai, 2025-02-28): Does Base blockchain implement sub-blocks per their recent announcement?
  **A** (@kushti): No. Base is a corporate EVM execution environment, not a PoW blockchain. Sub-blocks are unnecessary for such architectures; more performant solutions exist for that design space.

## Open Questions Entering March 2025

1. **Sub-blocks fork resolution**: Fork handling, out-of-order block processing, and missing parent reconstruction in the sub-blocks implementation remain incomplete. No mainnet activation timeline confirmed.

2. **SigmaUSD bank upgrade scope**: Will the contract update include migration away from fixed input/output indices? Roadmap for minting limits, key destruction, and future update governance not yet published.

3. **Dexy Gold mainnet deployment**: Exact deployment plan and timing for mainnet Dexy token issuance and contract activation not finalised as of month end.

4. **Fleet SDK / Sigma 5.0.15 integration**: Confirmed planned by @arobsn but no ETA committed; large number handling via built-in JSON parser flagged as a concern.

5. **Nautilus 0-conf UTXO streaming**: Whether passive streaming under Manifest v3 is feasible without jeopardising Google Play review remains unresolved.

6. **LLM contract auditing proof-of-concept**: GRPO fine-tuning framework identified (Unsloth), community and compute organised, but no results published from the month's work.

7. **DexLimitOrder template hash validation**: Whether published contract hashes are correct and whether limit order contracts are actively used or superseded remains unanswered.

8. **Compiler reproducibility**: Whether the ErgoScript compiler can accept determinism parameters or support source-level hashing to decouple contract identity from binary representation — raised but not addressed.

9. **Keystone hardware wallet integration**: Test phase revised to March; development resources confirmed tight.

10. **Testnet block production anomaly**: Only ~176 blocks/day observed in public testnet despite minimum difficulty — flagged for investigation, unresolved.
