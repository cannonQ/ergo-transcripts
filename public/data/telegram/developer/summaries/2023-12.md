# Ergo Developer Chat — Monthly Summary
**Month:** December 2023

---

## Executive Summary

December 2023 was defined by infrastructure troubleshooting (node indexing regressions, database performance, logging gaps), forward-looking scalability debates (block time reduction, sub-blocks, Rosen Bridge throughput), and API/tooling optimisation work. Community projects ErgoSanta and FleetSharp/.NET 8 migrations shipped, while unresolved questions around NIPoPoWs bootstrap stability and block-time contract dependencies carried into the new year.

---

## Key Topics This Month

### Node Stability: 5.0.15–5.0.17 Indexing Regression
A Sigma serializer "fix" introduced in 5.0.15 broke rollback logic, causing multiple nodes to become stuck at specific heights. kushti identified the culprit and released a dockerfile fix in 5.0.17. A separate intermittent crash involving `empty.last` exceptions in NipopowAlgos and LevelDB manifest corruption appeared when running with `nipopowBootstrap=true` and `utxoBootstrap=true` simultaneously; restarting resolved some cases but root cause remained under investigation. A related logging gap was flagged: ERROR-level exceptions appearing in console do not persist to `ergo.log` due to multiple uncoordinated logger instances.

### Scalability: Block Time Reduction and Sub-Blocks
An extended multi-week debate centred on whether reducing block time from 2 minutes is viable. The core obstacle is that emission contracts, time-locked boxes, and other dApps use the `blockHeight` context variable with an implicit 2-minute assumption — reducing block time without addressing this would cause early unlocks and accelerated emission. Pulsarz proposed an ErgoTree version flag as an opt-in workaround; consensus was that any backward-compatible path is "messy." Sub-blocks were discussed as an alternative throughput mechanism, with Luivatra expressing preference for sub-blocks over block-time reduction, but the developer migration path and dApp integration specifics remained unspecified.

### Rosen Bridge Transaction Volume
Pulsarz raised a practical concern: with 77 watchers online, each bridge event generates approximately 154 transactions across two chains. This prompted questions about whether fee market saturation is imminent and what throughput strategy — sub-blocks, block size increases, or cost-limit adjustments — is prioritised. Block size can be increased by miner governance vote (no hard fork required), and kushti noted the max execution cost per block could fairly safely double. No concrete prioritisation decision was reached.

### Database and Storage Architecture
Discussion compared LevelDB (Ergo's current choice), RocksDB (used by Solana and Kaspa, noted as superior with better JNI drivers), and LMDB (Monero's choice, with potential multi-process safety advantages). No migration decision was made, but RocksDB was discussed as a longer-term candidate. A related finding: the `/blockchain/box/unspent/byTokenId` endpoint supports batch limits up to 16,384 items, dramatically outperforming the default 100-item pagination that was causing rich list timeouts.

### JavaScript BigInt Precision and Token Amount Parsing
A critical bug was identified where i64 token amounts (e.g., `9223372036854776000`) exceed JavaScript's 58-bit integer precision, causing silent rounding during JSON deserialization. The solution requires fetching API responses as bytes, converting to UTF-8 string, and parsing with the `json-bigint` library rather than native `JSON.parse`. This affected ergo-lib-wasm consumers and was linked to the known GitHub issue #1900.

### UTXO Management and Airdrop Tooling
HQΣr reported accumulating 1,240–1,400 UTXO boxes from GreasyCex missions and dev fees. Flying Pig confirmed up to ~1,000 boxes can fit per transaction (content-dependent, 96 KB max), with the max inputs ceiling raised after node 5.0. For mass airdrops, qx() recommended YAAT (Flying Pig's tool, handles large output counts), ErgoPy (mgPai, ~1,000 outputs/tx limit), and a CSV-to-TX builder frontend.

### Mempool Behaviour and Fee Markets
Discussion clarified several mempool mechanics: multiple load-balanced explorer instances cause inconsistent mempool query results — direct node queries are more reliable. Conflicting (zombie) transactions are evicted after a timeout of several hours. The ErgoTree timestamp context variable represents the previous block's timestamp plus one, meaning a contract checking a time-based unlock cannot be submitted until after the first block mined past that timestamp. A fee estimation API exists in the node but is untested and unused by any application.

## Technical Q&A Worth Preserving

- **Q** (@Aco Šmrkas, 2023-12-08): How do you parse token amounts larger than i64 max (e.g., `9223372036854776000`) when `ergo-lib-wasm`'s `TokenAmount.from_i64()` only accepts i64, and JavaScript silently rounds the value?
  **A** (@Pulsarz, @Luivatra): Root cause is JavaScript's 58-bit integer precision. Fetch the API response as raw bytes → convert to UTF-8 string → parse with the `json-bigint` library instead of native `JSON.parse`. The signed 64-bit limit is sufficient in the reference client with overflow checks. See GitHub issue [#1900](https://github.com/ergoplatform/ergo/issues/1900).

- **Q** (@HQΣr, 2023-12-14): How can 1,240+ UTXO boxes be consolidated efficiently?
  **A** (@Flying Pig): Restore wallet in Nautilus and send to self across multiple transactions. Up to ~1,000 boxes can fit per transaction (content-dependent, 96 KB max); this ceiling was raised after node 5.0. Mobile wallets handle ~80, Nautilus desktop ~200 per tx.

- **Q** (@Aco Šmrkas, 2023-12-12): How do you restrict ErgoPay user address selection to a single address instead of allowing multiple?
  **A** (@Aco Šmrkas, self-resolved): Use a POST pre-check to the URL with `#P2PK_ADDRESS#` replaced by `"multiple_check"`. A 2xx return code allows multiple selection; non-2xx blocks it.

- **Q** (@CW, 2023-12-08): How can you check current indexing status on a node?
  **A** (@Flying Pig): Query `http://node_url/blockchain/indexedHeight` to retrieve the current indexed height.

- **Q** (@Pulsarz, 2023-12-19): After updating dockerised nodes from 5.0.16 to 5.0.17, one node fails with a LevelDB manifest error (`MANIFEST-489013: No such file or directory`). Probable cause?
  **A** (@kushti): Corrupt history index. Safe to delete the history folder and resync.

- **Q** (@Pulsarz, 2023-12-23): If block time is reduced from 2 minutes to ~15 seconds, how do contracts using the `blockHeight` context variable behave? Does the timestamp in context represent the current block or previous block?
  **A** (@Pulsarz, resolved through discussion): ErgoTree is evaluated at submission time using last block + 1 timestamp, and rechecked on each block if the transaction remains in the mempool. Edge case: if a contract's time-lock opens at 3:00 PM but the last block was at 2:59 PM, the transaction cannot be submitted until the first block mined after 3:00 PM is confirmed.

- **Q** (@Pulsarz, 2023-12-23): Can block size be increased without a hard fork?
  **A** (@Pulsarz, confirmed via docs): Miners can vote to increase block size via governance — no hard fork required. See [docs.ergoplatform.com/mining/governance/](https://docs.ergoplatform.com/mining/governance/). The max execution cost per block is often the binding constraint rather than raw size; kushti noted this could fairly safely double. Since node v5, transactions tend to hit the size limit before the complexity limit.

- **Q** (@Aco, 2023-12-26): Why does the mempool API return different transactions on repeated calls?
  **A** (@Pulsarz): Multiple load-balanced explorer instances each maintain their own mempool state. Direct node mempool queries are more consistent and recommended for production use.

- **Q** (@Flying Pig, 2023-12-28): What is the maximum safe batch limit for `/blockchain/box/unspent/byTokenId` queries?
  **A** (@qx()): 16,384 items per request.

## Open Questions Entering Next Month

- **NIPoPoWs + UTXO bootstrap crash**: The `empty.last` exception and LevelDB manifest corruption when combining `nipopowBootstrap=true` with `utxoBootstrap=true` is under investigation by kushti; root cause (possible race condition or state-machine edge case) not yet identified.
- **Block time reduction viability**: No consensus on how to handle contracts dependent on the implicit 2-minute blockHeight assumption. ErgoTree version flag proposed but not approved; design path remains open.
- **Sub-blocks developer adoption path**: How existing dApps (e.g., Spectrum DEX) would integrate sub-blocks, what protocol-level changes are required, and what the migration path looks like remain unspecified.
- **Rosen Bridge fee pressure**: With 77 watchers generating ~154 transactions per bridge event, no concrete throughput prioritisation decision (sub-blocks, cost limit increase, or block size vote) has been made before further watcher expansion.
- **Node logging coordination**: ERROR-level exceptions appearing in console do not persist to `ergo.log`; multiple uncoordinated logger instances (GitHub issue #2056) unresolved.
- **Node version bump automation**: Recurring omission of version number in releases causing false update prompts in Satergo and ergonodes.net; automation not yet implemented.
- **SPF rich list timeout**: 50,000-box query still exceeds Vercel's 10-second execution limit even with 16,384-item batching; workaround not finalised.
- **Lean transaction ID endpoint**: No endpoint exists to retrieve only transaction IDs (not full transaction data) for an address; feature request open.
- **Fee estimation API**: Exists in node but untested and unused by any application; flagged as worth testing and integrating into wallets and DEXes.
- **GraalVM vs. RoboVM for mobile Ergo apps**: No technical resolution on which compiler target is optimal for Satergo or other mobile projects.
