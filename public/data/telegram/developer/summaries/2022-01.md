# Ergo Developer Chat — Monthly Summary
**Month:** January 2022

---

## Executive Summary

January 2022 developer activity centred on wallet UX improvements and dApp connector capabilities, with Nautilus wallet receiving significant attention through its new Drydock testing programme. Infrastructure improvements (SSL certificates, Discord–Telegram bridging) were completed, while several technical questions around message signing, fee markets, and contract verification patterns remained open or partially resolved.

---

## Key Topics This Month

### BIP39 Mnemonic Passphrase Terminology
Discussion focused on the correct naming convention for the optional BIP39 passphrase feature in Nautilus wallet. Terms considered included "additional word," "custom word," and "mnemonic extension." The team settled on "mnemonic extension" with an accompanying tooltip explanation to balance technical accuracy with user clarity.

### dApp Connector & Message Signing
kushti confirmed that arbitrary message signing is currently supported only in the low-level interpreter API and has not yet been implemented in Sigma Rust or the dApp connector. A PR for a Node API signing method exists but is stalled, and the work was flagged as an open contribution opportunity. All signing operations in Ergo are transaction-based — modifying register data requires spending and recreating a box.

### ErgoTree Hash Verification in Contracts
Luivatra raised a technical question about using ErgoTree byte hashes to verify output box contracts within staking and profit-sharing scripts (referencing ErgoPad's staking contract). This pattern is used for efficient on-chain contract verification; Luivatra planned to implement equivalent logic in Python.

### Transaction Confirmation Delays & Fee Markets
An investigation into a 24–30 minute transaction confirmation on mainnet concluded the cause was mempool fee market dynamics rather than network issues — higher fee-per-byte transactions received priority. Community feedback also highlighted that existing Ergo wallets lack UI controls for configuring custom per-byte fees, leaving users unable to act on this information.

### Nautilus Wallet Drydock Programme
Armeanio launched a private testing and feedback group for Nautilus wallet stability and feature validation. Community members were invited to join and report issues ahead of broader release.

### Developer Infrastructure
Glasgow completed SSL certificate deployment for ergoplatform.com/docs and initiated bridging of the Telegram `#development` channel to Discord `#dev-support`, chosen over `#development` on Discord due to noise concerns. Both changes were well received by the community.

## Technical Q&A Worth Preserving

- **Q** (@Ilya, 2022-01-W02): What can be signed with the dApp connector — arbitrary data or only UTXO transactions? Can a signed transaction JSON reveal which wallet signed it?
  **A** (@Luivatra): All signing operations in Ergo are transactions; changing data in registers requires spending the box and recreating it with updated register values. Whether a signed transaction JSON can identify the signing wallet is unresolved.

- **Q** (@Luivatra, 2022-01-W02): Why does the ErgoPad profit-sharing script use a hash of ErgoTree bytes to verify the output box contract — is this a script size optimisation?
  **A** (implied, via contract context): This is a standard Ergo contract pattern for efficient on-chain contract verification. Luivatra intended to replicate the equivalent logic in Python.

- **Q** (@Ilya, 2022-01-W02): Why did a simple transaction take 24–30 minutes to confirm on mainnet?
  **A** (@Glasgow): Mempool congestion and fee market competition — transactions with higher fee-per-byte are prioritised. The transaction in question was outcompeted by higher-fee submissions.

- **Q** (@kushti, 2022-01-W02): What is the current status of arbitrary message signing in the dApp connector?
  **A** (@kushti): Arbitrary message signing is supported only in the low-level interpreter API. It is not yet available in Sigma Rust or the dApp connector. A PR for a Node API method exists but is stalled and open for contribution.
