# Ergo Developer Chat — Monthly Summary: December 2025

---

## Executive Summary

December 2025 was a highly productive month for Ergo's developer community, with parallel progress across protocol research (sub-blocks/Matrix Protocol, ChainCash/Basis), tooling (ErgoScript LSP, LLM-assisted contract development), and ecosystem infrastructure (testnet migration, Ergo Wallet App 2.5 RC, USE stablecoin integration). The month's standout theme was the maturation of developer tooling — from vibe-coded language servers to LLM context repositories — alongside a wave of novel IPFS-hosted decentralised dApps demonstrating Ergo's censorship-resistance properties.

---

## Key Topics This Month

### Sub-blocks / Matrix Protocol
kushti advanced sub-blocks research throughout December, moving from whitepaper drafting in W49 to devnet deployment and expanded pruning test coverage by W50, with the whitepaper sent for peer review by W52. The deployment strategy involves gradual miner adoption followed by lock-in voting once 90%+ hashrate support is reached. A public devnet node was made available at `213.239.193.208:9058` for community testing. The input blocks whitepaper was published to the `weak-blocks` branch on GitHub. Sub-blocks were consistently cited as a prerequisite for improved UX in dependent applications like CruxFinance.

### ChainCash / Basis
The ChainCash paper was presented by kushti at the 1st Stability Workshop on December 15. Development continued throughout the month: tracker box updates and live blockchain reserve testing completed, `/reserves` endpoint fixed and functional, and redemption testing underway by W52. Promotional materials were in progress at month end. The Stability Workshop also featured five peer-reviewed Ergo community research papers covering Djed Shu, Dexy Gold, Gluon W, the Orb oracle, the Fate Protocol prediction market, and RainDrop reward distribution.

### USE Stablecoin Integration
USE was deployed in W49 with token IDs and deployment transactions merged in PR #5 to the dexy-stable repository. The month's integration work spanned multiple fronts: logos added to Nautilus (merged), USE/DORT/CLB pool tracking added to CruxFinance with charts and order history (Luivatra, W49), token tracking added to Ergo Explorer (kushti, W51), and offchain bots deployed. A DORT buyback bot fix was required in W52 before buyback testing could proceed. LP implementation and full DexyGold-style USE implementation (c8) remained in progress at month end.

### ErgoScript Language Tooling (LSP & LLM)
Two complementary tooling tracks advanced in parallel. Luivatra developed a vibe-coded ErgoScript compiler with LSP support, type inference, and error reporting (W51), plus a Zed editor extension. Meanwhile, Arohbe built the Ergo-Context repository — a structured machine-readable collection of langspec.md, known issues, examples, and references for local LLM development — releasing v1.2 in W51 with local LLM file support and contract audit examples. kushti and Arohbe explored using Qwen Coder for contract development achieving ~90% accuracy, with edge-case handling (e.g., ensuring box token counts match sales) addressed through embedded context rules. The Unstoppable Hackathon at hackathon.stability.nexus generated proposals for a formal ErgoScript LSP, with anon_br and mgPai both expressing long-standing interest in a native sigmastate-interpreter checker without JVM dependency.

### Ergo Wallet App & Mobile
Superellipse's PR #188 (Android 15 and iOS support requiring MobiVM 2.3.24 and XCode 26.1.1) was approved and tested on iPhone 12 mini (iOS 18.6.2) in W49, incorporated into PR #208 (merged W52) as Ergo Wallet App 2.5 RC with bounty collection available. PRs #182 and #183 queued for later merge pending conflict checks.

### Explorer & Infrastructure
qx() migrated all testnet explorer services (testnet.ergoplatform.com, api-testnet, gql-testnet) from old Hetzner host (195.201.82.115) to locally-hosted instance (128.253.41.110) in W51, completing decommissioning by W52. A consolidation plan for mainnet explorer infrastructure was announced in W50: two high-performance machines (64-thread, 128GB RAM at cornell.edu for large queries; 48-thread AMD Threadripper, 128GB RAM community node) with P2P infrastructure as fallback. Dashboard updated at https://infra-p2p.ergoplatform.com/cgi-bin/dashboard.cgi. qx() also added "Other Explorers" and "Community" sections to the testnet explorer UI, with a "Commercial" category planned.

### Rosen Bridge
Active development throughout December: Base chain integration research underway; on-chain token map integrated with UI; architecture for unbridgeable tokens introduced; guard service updated for multiple bridge fee destinations; watcher service v6.0.2 and guard service v8.1.0 released with USE and NIGHT token support.

### Lithos Protocol
Two releases shipped: v2.0.2 (testnet, W49) fixing NISP tree synchronisation, fraud proof evaluation, and stratum sync issues; v2.2.0 (W51) adding multi-threaded sync, individual rollup/NISPTree syncing, and database error fixes. Lithos Panel integration with Xergon planned post–Lithos Client 3.0.

### Decentralised IPFS-Hosted dApps
A notable cluster of new censorship-resistant dApps appeared in W52, all hosted on IPFS and leveraging Ergo's eUTXO model:
- **SIGHT**: Anonymous reporting using blockchain-stored tokens, client-side PoW spam prevention, GPS truncation to neighbourhood level, ~$0.01/report.
- **SAY**: P2P social platform with SAY-{timestamp} tokens, on-chain follows via SAYFOLLOW tokens, 0.001 ERG/post, ErgoPay integration.
- **Ephemeral Messenger v1.1**: Self-destructing on-chain messages with configurable lifetimes, incentivised cleanup (earn ERG for burning expired tokens), nation-state-proof encryption passphrases.

### Tor Privacy / Anonymous Node Support
x10zyn initiated investigation in W51 into fully anonymous Ergo node operation via Tor, proposing a user-friendly implementation (ADD_ONION creates hidden service at runtime, auto-discovers SOCKS port, persists keys across restarts, handles Tor restarts). kushti confirmed PeerSpec appears flexible enough for variable-length addresses including Tor v3 .onion addresses without a hard fork. x10zyn published Sigma State interpreter learning documentation at zutxo.github.io/sigma-book/.

### Ergo Go P2P Library
mgPai implemented the Ergo P2P protocol in Go (ergonnection-go), using the Java implementation as direct reference, published in W51.

## Technical Q&A Worth Preserving

- **Q** (@Bruno Woltzenlogel Paleo, 2025-12-01): Is there a "good first issues" list on Ergo, similar to Stability Nexus? The current bounty list seems focused on harder issues rather than entry-level contributions.
  **A** (@kushti): Some repos have them; will check and update.

- **Q** (@Luivatra, 2025-12-01): Need to identify which pool actions were taken on specific pool contracts — how do pool contracts differ across stablecoins?
  **A** (@kushti): Pool contract is identical across USE, DORT, and CLB — only token IDs differ for the stablecoin and action NFTs. The compiler extracts all tokens from the same template.

- **Q** (@Arohbe, 2025-12-03): How should an LLM-based contract builder handle edge cases like ensuring the contract box token count matches tokens sold?
  **A** (@kushti): Include langspec.md, UTXO model description, known issues documentation, and examples (whitepaper, advanced ErgoScript whitepaper) in the LLM context. Basic exchange rules in the whitepaper cover most cases.

- **Q** (@kushti, 2025-12-10): Does anyone have experience with LSP? Is it feasible to write one for ErgoScript these days?
  **A** (@anon_br): On my long-term todo list. The key question is whether we can have a native ErgoScript checker from sigmastate-interpreter without a JVM dependency. Debug function instrumentation and code coverage tooling would also be invaluable.
  **A** (@mgPai): Was looking into this ~2 years ago, attempting to copy the approach from Aiken. Active research area.

- **Q** (@Josemi, 2025-12-17): ErgoScript failing with `winnerAuth || resolverAuth` but working with `winnerAuth` alone (where `winnerPK == resolverPK`). MockChain passes; mainnet fails.
  **A** (@kushti): Asked for clarification on P2PK prefix value used for detection.
  **A** (@Josemi): P2PK prefix is `0008cd`. `resolverPK` and `winnerPK` are `proposition_bytes`; the prefix indicates whether `proveDlog` can be performed or if authenticity must be proven by spending an input. *(Root cause unresolved — potential issue with OR logic in Sigma protocol composition or serialization.)*

- **Q** (@x10zyn, 2025-12-18): Does the Ergo P2P protocol support variable-length addresses (e.g., Tor v3 .onion) without a hard fork? Has BIP 155-style addrv2 or Tor Controller integration been discussed?
  **A** (@kushti): PeerSpec appears to support variable-sized addresses and looks flexible enough for onion addresses. Asked for details on the Tor Controller auto-configuration concept.
  **A** (@x10zyn): Proposed approach: ADD_ONION creates a hidden service at runtime, auto-discovers SOCKS port, persists keys across restarts, handles Tor restarts automatically. User sets `tor.enabled = true` and the node handles the rest. Referenced Bitcoin's `torcontrol.cpp` as implementation model.

- **Q** (@Aco Šmrkas, 2025-12-23): How can the Explorer height be greater than the Node height?
  **A** (@qx()): Likely a label swap error; investigating.

- **Q** (@kushti, 2025-12-27): Is the NFT viewer in the node panel using extra indices?
  **A** (@ArØhβΣ): Uses fallback to Explorer API if no extra index is present, pulling tokenId metadata via `/blockchain/token/byId/<tokenId>`. Always attempts local call first; falls back only if a user-provided explorer URL is entered — prevents forced dependency on a single explorer instance.

- **Q** (@cannon_q, 2025-12-31): Best practice for pulling NFT images into a viewer/site?
  **A** (@ArØhβΣ): Pull token metadata from Explorer or node by tokenId, convert metadata to IPFS CID, use `http://ipfs.io/ipfs` as prefix — most stable gateway compared to alternatives.
