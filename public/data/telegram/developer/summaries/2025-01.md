# Ergo Developer Chat — Monthly Summary: January 2025

## Executive Summary

January 2025 was a high-velocity month for Ergo core development, dominated by three converging themes: finalisation of the 6.0 protocol soft-fork (fixing serialisation issues, addressing spam vectors, and reaching testnet activation readiness), rapid advancement of the Dexy Gold integration with novel UX philosophy, and a broad push across infrastructure, wallets, and tooling to reduce friction for both developers and end-users. The month closed with 6.0 Sigma issues fully resolved and public testnet activation imminent, alongside a maturing ecosystem bounty programme (100 open bounties, 28,000+ ERG in pool) and multiple wallet releases.

---

## Key Topics This Month

### 6.0 Protocol Soft-Fork
The 6.0 soft-fork progressed from bug-fixing to testnet-ready status across the month. Early weeks addressed a BigInt overflow in `Global.decodeNBits` (issue #1044), prohibition of new 6.0 types in registers and context extension variables for backward compatibility with pre-6.0 nodes, and a `coll.distinct` spam attack vector (removed in PR #1052). dusek_ led versioning and type serialisation fixes throughout, with tree-versioned serialisation for new types landing in PR #1043. By week five, kushti confirmed all Sigma issues resolved and announced readiness for public testnet activation, with voting expected to take approximately four days under testnet settings. The 6.0 audit fundraiser raffle was launched on Ergo Raffle mid-month.

### Sub-blocks / Input-Ordering Blocks (EIP)
The input/ordering blocks EIP moved from draft to active prototyping. The initial draft was published in PR #2055, with the pointer to previous input blocks written into the extension section and transactions packed as they arrive. Incremental transaction accumulation, proof regeneration, and API endpoints (`/blocks/bestInputBlock`, `/blocks/bestInputChain`, `subblocks/byIndexRange`, `/transactions/unconfirmedNotInSubblock`) were added or proposed. Input blocks are stored in RAM and pruned after two ordering blocks. Mining pool stratum implementations will require updates to support sub-block submission, and explorer updates remain outstanding. kushti is exploring making contracts dependent on input blocks for sidechains and rollups, with security assumptions still being documented.

### Dexy Gold Integration
c8e4d2a began integrating Dexy Gold into ergfi.xyz, porting Scala unit tests to JavaScript and minting test tokens for mainnet simulation. A critical serialisation bug — `Int.MaxValue` (2147483647) producing different byte representations between Scala and JavaScript due to JavaScript's 53-bit Number precision — was discovered and resolved in Fleet SDK v0.8.2. By month end, c8e4d2a articulated a unified swap interface philosophy: in the eUTXO model, all interactions (bank mint/redeem, LP operations) can be presented as a single "tokens in → tokens out → swap" interface regardless of underlying mechanics. Key outstanding questions include whether Dexy Gold can function without bots setting R7 in tracking boxes, and the precise conditions under which the bank removes Dexy from LP pools to restore the peg.

### Wallet Ecosystem (Nautilus, Satergo, Minotaur, Keystone)
Significant wallet progress across four parallel efforts. Nautilus underwent a comprehensive UI overhaul (Transaction History, Receive view, address filtering, AssetInfoDialog, SettingsView, AppConnections, TransactionSign component migration to Drawer for mobile UX) with fiat denomination support via CoinGecko and Spectrum DEX. Keystone hardware wallet integration completed transaction parsing, user display, and signing on-device with QR-based workflow; Rust SDK and JavaScript Base SDK PRs submitted, pending Keystone team review. Satergo finalised wallet file format changes, awaiting sigma-state v5.0.15. Minotaur 2.4.0 shipped with token detail display on signing pages, multiple token burning support, and animated QR code scanning.

### Oracle Pool v2 Migration
kushti completed code to upgrade Oracle Pool to v2 by modifying only the update contract, enabling migration without full redeployment. Testnet validation was in progress with pool refresh and bank update planned after validation. Oracle Core is switching from scans to extra indices to simplify running multiple lightweight oracle instances on the same node. Vector oracles — proposed as dynamic key-value maps `Coll[(Coll[Byte], Int)]` for multi-ticker price feeds — were proposed for collateralised lending use cases; Palmyra was identified as a potential early adopter for commodity vector oracles.

### Lithos Protocol (Optimistic Rollup)
The Lithos optimistic rollup design came into focus. The design uses temporary NISP (Non-Interactive Schnorr Proof) storage with off-chain fraud checking; fraud proof contracts verify conditions such as low-difficulty shares and incorrect timestamps on-chain, with fraudulent proofs removed via contract execution. Beta testing with larger miner cohorts is planned for the public testnet after 6.0 activation. Mainnet is targeted for June 2025 or slightly later.

### Infrastructure & Tooling
mgPai published detailed hardware benchmarks: a Hetzner ARM instance (16 core, 32GB RAM, ~$30/month) achieved a 15-hour full sync versus Linode's 25-hour sync at $192/month, with both completing the indexer sync in approximately two hours. Minimum viable production configuration is 8 core + 32GB RAM. Go bindings for Sigma-Rust (`ergo-lib-go`) were tested in production for off-chain bots and blockchain scanners, with a minor memory management bug discovered and under fix. The Ergo Bounties registry launched at mid-month with 100 open bounties and grew to 28,074 ERG in the pool by month end, with coding language metadata added.

### ChainCash
Refund-related registers and actions were removed from ChainCash in favour of note redemption (PR #10). Note registration contracts are being added. Layer 2 note transfer options for peer-to-peer off-chain cash backed by on-chain reserves are under investigation; kushti identified register preservation gaps in reserve refund handling and implemented a fix via note token issuance tracking with timestamp validation.

### SigmaUSD Contract Improvements & ERGFI
Discussion on moving SigmaUSD away from hard-coded fixed-index box references (`OUTPUTS(1)`) toward filtering logic (by NFT token or output tokens) to enable flexible transaction composition by dApps and UIs. kushti suggested context variables as an alternative. c8e4d2a deployed channel-based socket subscriptions for real-time mempool state in ERGFI, shifting from REST to socket architecture; the SigmaUSD UI was released within the month with mempool subscription state updates integrated.

### Celaut & Bene
jossemii reported Celaut node fault tolerance improvements via peer tree search and reputation visualisation progress. The primary 2025 goal for Celaut is enabling payments for services without a network connection, leveraging PoW — a mechanism more complex than current infrastructure. Bene UI fixes were finalising for v1, with v2 improvements deferred.

## Technical Q&A Worth Preserving

- **Q** (@krasaviceblasen, 2025-01-02): Should the SigmaUSD contract move away from fixed-index box references to allow flexible transaction composition by dApps and UIs?
  **A** (@kushti): "Why not use provided via context vars indices?" — Context variables suggested as cleaner approach for dynamic box ordering without contract code changes.

- **Q** (@c8e4d2a, 2025-01-04): Can a fully indexed node run on minimal hardware (~1 core, 4GB RAM)?
  **A** (@mgpai): "It kinda works" but is unreasonable for production. Minimum viable configuration is 8 core + 32GB RAM. Hetzner ARM at $30/month outperforms Linode at $192/month: 15-hour vs. 25-hour full sync, with both completing indexer sync in ~2 hours.

- **Q** (@c8e4d2a, 2025-01-08): Web wallet signing timeout (>30 sec) rejects promise but provides no error UI hint; password input becomes unresponsive.
  **A** (@arobsn): Likely a client-side Promise timeout; issue to be opened to investigate error signalling.

- **Q** (@jossemii, 2025-01-08): Can AppKit sign a message with a wallet private key and verify with `verify(message, public_key)`?
  **A** (@kushti): Yes — use the basic method from `AppkitProvingInterpreter`: `verify(env: ScriptEnv, ergoTree: ErgoTree, context: CTX, proof: Array[Byte], message: Array[Byte])`.

- **Q** (@c8e4d2a, 2025-01-15): Will oracle-dependent transactions fail if the oracle input is spent before confirmation, given 4–6 minute mempool chaining delays?
  **A** (@kushti): Spent data inputs are readable in the same block — oracle-dependent transactions do not fail provided both the oracle spend and the dependent transaction occur in the same block. See [data-inputs-semantics](https://www.ergoforum.org/t/data-inputs-semantics/654).

- **Q** (@luivatra, 2025-01-15): Why switch from Rust to Go for the Crux aggregator backend when Rust has more Ergo libraries?
  **A** (@kushti): Foreign function interfaces are "the most cursed part of any codebase." (@Pgr456): Go bindings for Sigma-Rust (`ergo-lib-go`) exist and are production-tested in off-chain bots and blockchain scanners.

- **Q** (@c8e4d2a, 2025-01-22): How do different buffer values (2, 3, 5) in Dexy contracts work? Can they all be set equal?
  **A** (@kushti): Yes — set to 3 everywhere; means the TX must be included in one of the next 3 blocks to be valid.

- **Q** (@c8e4d2a, 2025-01-22): `Int.MaxValue` (2147483647) serialises to `"04feffffffffffffffff01"` in Scala but `"04feffffff0f"` in JavaScript, causing box parsing errors.
  **A** (@arobsn): Caused by JavaScript's 53-bit Number precision limitation. Fixed in Fleet SDK v0.8.2.

- **Q** (@Pgr456, 2025-01-22): Are input blocks held in RAM like the mempool? Disk I/O would be high with frequent block production and indexer polling.
  **A** (@kushti): Input blocks are stored in RAM and pruned after 2 ordering blocks. Ordering blocks serve as the compaction and pruning mechanism.

- **Q** (@c8e4d2a, 2025-01-22): How can socket services efficiently fetch unconfirmed transactions? Can input boxes be pre-included in unconfirmed transaction JSON?
  **A** (@kushti): Augmenting transaction JSON with input boxes is possible. Proposed additional APIs: `subblocks/byIndexRange`, `/transactions/unconfirmedNotInSubblock`, and an `/info` field `lastSubBlockIndex`.

- **Q** (@kii, 2025-01-22): Did we figure out random numbers in contracts?
  **A** (@c8e4d2a): Write a `futureHeight` to the UTXO; after the block at that height is minted, use its `headerID` (combining NFT `tokenID` + `headerID` for improved distribution) modulo element count. Requires a 2-block wait but avoids third-party trust, unlike multiparty schemes such as hashmon.

- **Q** (@c8e4d2a, 2025-01-22): Is there an open-source Ergo Rust wallet?
  **A** (@Glasgow, @Pgr456): `sigma-rust` includes wallet code at `ergo-lib/src/wallet`. For transaction building, reference `oracle-core`'s `publish_datapoint.rs` using `tx_builder` and `box_builder`.

- **Q** (@mohsen6070, 2025-02-01): When signing a TX that doesn't need the wallet key, getting "Unable to find the corresponding secret for input 2" when calling `prover.signTransaction(tx, [])`.
  **A** (@mgpai): Use `ErgoHDKey.fromMnemonic()` and pass the `rootKey` array to `prover.signTransaction()` — provided Fleet SDK example code showing correct transaction building pattern.

- **Q** (@c8e4d2a, 2025-01-29): If all LP tokens land in the pool during mint, `val supplyLpIn = $initialLp - lpReservesIn._2` prevents further minting — is this a bug?
  **A** (@Michael): No — the bank can remove Dexy from the LP pool via extraction transactions to restore the peg.

## Open Questions Entering Next Month

- **Sub-blocks pool and explorer support**: Mining pool stratum implementations need updating for sub-block submission; explorer updates outstanding. No timeline confirmed.
- **6.0 testnet voting coordinator**: Activation voting will take ~4 days but a coordinator to manage deployment to public testnet has not been formally identified.
- **Dexy bot dependency**: Can Dexy Gold operate without bots setting R7 in tracking boxes, or is headless automation or peer-to-peer user incentivisation required? Not resolved.
- **Dexy bank peg restoration mechanism**: Under what exact conditions does the bank remove Dexy from LP pools to restore the peg? Mechanism not fully documented.
- **L2 architecture selection for Sigmachains**: Which rollup or sidechain architectures are compatible with existing Ergo DeFi contracts? Sigmachain partner identification still pending.
- **Vector oracle consensus rules**: How consensus enforces identical oracle maps across the network for multi-ticker feeds not fully specified; static vs. dynamic deprecation model still being designed.
- **AES-GCM encryption length leakage (Satergo)**: No better solution found beyond random garbage padding; Aberg's full constraints not resolved.
- **Celaut offline payment proof**: Technical mechanism for proving a transaction without network connectivity using PoW still under design.
- **Old bounty validity**: Whether 2022–2023 GitHub issues with attached bounties remain active or should be closed is unresolved.
- **Socket service API shape for sub-blocks**: Final design of `lastSubBlockIndex` field and sub-block streaming endpoints not confirmed.
- **Keystone integration completion**: Remaining Web SDK, Web USB SDK, and Nautilus/Satergo integration pending Keystone team resource availability.
- **Historical balance API**: Efficient exposure of historical balances and token snapshots deferred as explorer/indexer responsibility; no concrete plan exists.
