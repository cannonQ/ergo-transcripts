# Ergo Developer Chat — Monthly Summary
**Month:** November 2022 | **Chat:** Ergo Developer

---

## Executive Summary

November 2022 was dominated by node infrastructure work (IPv6 support, peer discovery, TOR integration) and protocol-level discussions around scalability, block speed, and mempool/fee market behaviour. Alongside this, several active dApp development threads emerged — including Dexy Protocol testing, ErgoFlip game architecture, and SPF token bridge strategy — reflecting a maturing ecosystem beginning to stress-test both the network layer and smart contract tooling simultaneously.

---

## Key Topics This Month

### Node Infrastructure: IPv6, Peer Discovery, and TOR

A critical infrastructure gap was identified early in the month: zero reachable IPv6 nodes existed on the network, blocking IPv6-only users entirely. Community members self-organised to deploy IPv6-accessible nodes using the `declaredAddress` configuration field, without waiting for an official directive. Aberg deployed the first working node and documented the configuration approach; ergonodes.net was updated to reflect IPv6 reachability. A deeper protocol limitation was also surfaced: the peer information format does not support multiple public addresses per node (e.g., IPv4 + IPv6 simultaneously), requiring either a new protocol field or a peer feature extension. TOR integration was flagged by kushti as a related open problem — Ergo's P2P layer currently broadcasts only IP addresses, not onion domains, and no documentation or implementation exists for proper TOR support.

### Scalability Planning and Block Speed

kushti published a scalability plan update to the Ergo Forum, opening structured discussion on throughput improvements. The feasibility of faster block times was debated; a key concern raised was that many on-chain contracts depend on `HEIGHT` or block timestamps, meaning parameter changes are non-trivial and require auditing the full contract ecosystem. Sub-blocks were mentioned as one architectural option alongside layer 2 approaches. kushti noted that Ergo's existing contract set (Rosen Bridge, Oracle Pools, Spectrum DEX, AMM pools, lending contracts) now provides a concrete workload profile, making optimisation targets more well-defined than for chains with empty parallel infrastructure. A community testnet channel was flagged as available for those wishing to experiment with parameter tweaks — no code contribution required, just coordination and scientific observation.

### Mempool, Fee Market, and Spam Protection

Discussion across multiple weeks covered transaction replacement behaviour, fee estimation, and node-level spam protection. kushti confirmed that from v4.0.44, the reference client includes filters limiting transaction downloads from a single peer. The fee estimation API exists in the node but is reportedly untested and unused by dApps. Bandwidth limits were planned for the 5.0.4 branch (~10 MB per peer), balancing spam resistance against normal propagation needs. The broader pattern — fee markets as organic spam protection — was discussed with reference to the Bitcoin block size wars as historical precedent.

### ErgoScript Auditability and ErgoTree

A substantive thread clarified how deployed smart contracts can be reviewed. Consensus: only ErgoTree (compiled bytecode) is stored on-chain; ErgoScript source is not visible. Decompilation is theoretically possible but difficult due to parameterised contracts producing different ErgoTree hashes for different configurations. Developers must publish source code separately to enable public auditing. This mirrors Cardano's behaviour. The implication for ecosystem trust — that contract auditing depends entirely on developer disclosure — was noted as a friction point for dApp adoption.

### Dexy Protocol Testing

scalahub progressed LP and Intervention contract testing publicly on GitHub, with a dedicated test branch under the `ergo-jde` repository. This represented one of the more technically advanced contract development threads of the month, serving as a reference implementation for complex multi-contract interactions.

### dApp Architecture: ErgoFlip and Gaming Backends

Grand Gambit sought guidance on building a coinflip game with a 1 ERG → 2 ERG payout model. The discussion surfaced an important architectural tension: simple reserve-pool models are fragile (reserves deplete on losing streaks), and backend-based fund custody introduces trust assumptions. The Obolflip liquidity pool model — 24-hour bet cycles with pooled winner payouts — was recommended as a more robust alternative. The broader question of when to use smart contracts versus backend services for fund custody was discussed in the context of ErgoFlip, with community members noting that sending funds upfront to a backend server creates counterparty risk absent from on-chain implementations.

### SPF Token Bridge Strategy

A proposal was discussed to mint 2 billion SPF tokens (1 billion on each chain), with 1 billion locked into Rosen Bridge once ready. This would enable native token support on both chains rather than relying on wrapped representations, treating the bridge as a supply management mechanism from inception.

### BigInt in ErgoScript Registers

kushti confirmed that BigInt values can be provided via context extension for signature checks within contracts, with registers expected to behave analogously. Moein Zargarzadeh and mhs_sam were pointed to as sources for bridge-specific implementation details.

## Technical Q&A Worth Preserving

- **Q** (@Ilya Oskin, 2022-11-07): Can BigInt values be stored in ErgoScript registers?
  **A** (@kushti): BigInt can be provided via context extension for signature checks within contracts; registers should work the same way. Moein Zargarzadeh and mhs_sam can provide details on bridge-specific implementations.

- **Q** (@Grand Gambit, 2022-11-07): How do I build a backend for a coinflip game that receives 1 ERG and pays out 2 ERG on a win?
  **A** (@Unknown): A simple reserve-depletion model is fragile. Prefer a liquidity pool incentive model (as in Obolflip's 24-hour bet cycles with pooled winners) or scale bet/payout caps to available liquidity.

- **Q** (@Aberg, 2022-11-14): How can a node broadcast multiple public addresses?
  **A** (@Aberg): You can't currently. The peer information format in the protocol does not allow multiple public addresses. Either a new field or a new peer feature ("extra addresses") would need to be added to the protocol.

- **Q** (@Odiseus Me, 2022-11-15): What is the difference between reachable and unreachable nodes?
  **A** (@Aberg): Reachable nodes accept incoming connections. Unreachable nodes are those broadcasted by other peers but not open for incoming connections. This has nothing to do with web accessibility.

- **Q** (@zhang zee, 2022-11-28): When a smart contract is deployed on-chain, can the ErgoScript be reviewed without the developer publishing it?
  **A** (@anon_br / @kushti): No. Only ErgoTree (compiled bytecode) is stored on-chain. Decompilation is theoretically possible but difficult due to parameterised contracts producing different ErgoTree hashes. Developers must publish source code separately to enable auditing. Cardano behaves similarly.

- **Q** (@HQΣr, 2022-11-14): Is there a way to create a QR code for a product such that scanning and completing a transaction automatically sends the product to the buyer's wallet?
  **A** (@MrStahlfelge): Yes — see EIP-25. Or EIP-20 if the product is a token.

- **Q** (@HQΣr, 2022-11-07): Is it hard to create a simple betting pool for the World Cup using coin-flip mechanics?
  **A** (@Luivatra): Requires a trustworthy oracle source. (@Flying Pig): Someone is expected to launch a similar project in 2023.

## Open Questions Entering Next Month

- **TOR integration for P2P**: No design or documentation exists for broadcasting onion domains via Ergo's peer discovery layer. Bitcoin's approach identified as precedent but not yet examined in detail.
- **Multiple public addresses per node**: Protocol-level change required to support nodes advertising both IPv4 and IPv6 (or primary + secondary addresses); no EIP or PR initiated.
- **Block speed and contract timestamp dependencies**: Faster block times require auditing all contracts using `HEIGHT` or timestamp fields — scope and approach undefined.
- **Fee estimation API**: Node API for fee estimation exists but is untested and unused by dApps; no owner or timeline for integration.
- **Nautilus reduced transaction signing**: kii flagged the gap but no implementation approach or timeline was established.
- **Asset burning vs. vault mechanism**: PR closed; refinement of vault approach deferred to anon_br with no concrete timeline.
- **ErgoScript auditability tooling**: No standardised mechanism for developers to publish and associate ErgoScript source with deployed ErgoTree; friction for ecosystem trust noted but unaddressed.
