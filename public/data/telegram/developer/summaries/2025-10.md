# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer | **Month:** October 2025

---

## Executive Summary

October 2025 was dominated by three interlinked themes: the successful activation of the 6.0 protocol soft-fork on mainnet and the subsequent groundwork for the 6.0.2 release, the ongoing research and implementation of sub-blocks (input/ordering blocks) as a path toward higher L1 throughput, and a concerted effort to systematise Ergo's contract library through AI-assisted tooling, bounties, and design pattern documentation. Infrastructure incidents — an explorer indexer bug affecting wallet history queries, a storage rent integer overflow bug, and SigmaUSD oracle pool stalls — surfaced important technical debt that the community moved quickly to address.

---

## Key Topics This Month

### 6.0 Protocol Activation & 6.0.2 Development

The 6.0 soft-fork activated successfully on mainnet on 6 October, with 6.0 and 5.0 nodes remaining in consensus and mining operating normally. Attention shifted immediately to 6.0.2, with PRs #2244 and #2247 targeting fixes for issue #2223, and PR #2242 covering sub-block implementation. A tree version address generation discrepancy was identified in the 6.0 node API — `treeVersion` 0 and 1 produce the same address despite representing different protocol versions — and flagged for fix in the 6.0.2 release. Version 1 trees include tree size metadata, benefiting light clients; both versions execute correctly in the interim.

### Sub-blocks (Input/Ordering Blocks)

Sub-blocks remain the primary L1 scalability research vector. Core mempool and wallet support for input blocks was completed, with the exception of rollback processing. The associated whitepaper is in progress. Implementation requires accounting for block cost limits during input block processing, and PR #1069 in sigmastate-interpreter is the critical dependency under review. A devnet relaunch was targeted for early November once forking processing fixes landed, with a request for CPU mining participation using 2–3 nodes to generate forks at sufficient frequency. The distinction between sub-blocks and L2 was debated: sub-blocks are motivated by faster sidechain settlement and native fee markets, while L2 solutions remain preferable for most applications requiring decentralization.

### Contract Library & ErgoScript Design Patterns

kushti initiated a systematic effort to collect, test, and document ErgoScript design patterns from the Ergo Forum (dating to 2019) and ErgoHack events into an open-source library co-located near the ErgoScript compiler. New contracts added to sigmastate-interpreter this month include: Perpetual Token (PR #1082), Daily Withdrawal Limit (PR #1083), Stealth Address (PR #1084), and Token with Whitelisted Transfers (PR #1080). Work-in-progress contracts include a Trustless Hashrate Oracle (PR #1085) and Bulletproof range proof verification (PR #1079). AI agents were explicitly restricted to test generation and documentation within `src/test/` only — production code in security-critical modules must remain human-written.

### Storage Rent Integer Overflow Bug

A critical integer overflow was discovered in the sigma-rust box size calculation used during storage rent collection: box size × storage fee factor (e.g., 2,239 × 1,250,000 = 2,798,750,000) exceeds `Int.MaxValue` (2,147,483,647), causing at least 45 boxes to be unable to collect rent. The overflow location was pinpointed in `ErgoInterpreter.scala#L43`. A bounty was assigned to sigmastate-interpreter#602 for the fix.

### Explorer Infrastructure & SigmaUSD Oracle Stall

The explorer's GraphQL indexer developed a corruption issue causing HISTORY query failures for wallets with history predating 2023. qx() deployed a temporary Perl proxy on `gql.ergoplatform.com` while a full indexer redesign is investigated. GQL v0.5.5 was released with fallback routing using height + timestamp sorting, automatic indexer health detection, and a fix for the stale API circulation amount. Separately, multiple SigmaUSD oracle operators disconnected from the old pool, causing an oracle stall at block 1,644,679. Four operators were brought back online to resolve it, with a proposal to surface oracle operator status on a public dashboard.

### ChainCash / Basis

The first Basis reserve was deployed to mainnet (TX: `9342f78bd290c8057632588b6dc908c71b19fb9a33e859716a0c323d1eb5a346`). A notable storage rent design pattern was documented: "claim box before rent if you burn tokens" — applicable after 3 years of inactivity and potentially reusable by other dApps. Browser wallet implementation in JavaScript commenced. kushti proposed using GitCircles for ChainCash/Basis development coordination.

### Dexy / USE Stablecoin

The new USD Oracle Pool v2 (Dexy-powered) was launched by Pgr456. A paper on USE was submitted to the 1st Stability Workshop (hybrid, Jaipur/remote, 15 December; paper deadline 19 October). Bank intervention simulations are being finalised before token issuance; a liquidity bootstrapping event was announced as forthcoming.

### Ecosystem Projects

- **Nautilus**: v1.3.1 released with Ledger transport internationalisation; Javanese and Malaysian language support PRs pending merge; Bitcoin Runes integration research ongoing.
- **Lithos Protocol**: HTML UI completed; stratum fixes for Rigel and NBminer; HiveOS compatibility work started; AppKit 6.0 tests being added; testnet launch script added.
- **GitCircles**: Bro Etiket completed additional PRs (3 milestones remaining); domain acquired; specification work ongoing for issue #11; new contributor onboarded.
- **Hermes**: Improved error handling and serialisation; mempool client added; still in MVP development.
- **Cyberverse**: Combat mechanics implemented — enemy attack/defence animations, real-time CON stat upgrades, position escape logic.
- **Machina Finance**: No major updates; still in MVP development.

## Technical Q&A Worth Preserving

- **Q** (@Patato, 2025-09-29): Node stuck at block height 41,854 while devnet progresses to 42,663; desynchronisation persists even after DB deletion.
  **A** (@kushti): Root cause is rapid block arrival with few peers triggering incorrect penalty scoring. Temporary mitigation: set `scorex.network.penaltyScoreThreshold = 500000` in the top-level `scorex` section of config. Fix in progress.

- **Q** (@Josemi, 2025-10-01): Can a token be omitted from transaction outputs to remove it from network state?
  **A** (@Pulsarz): Yes — that is exactly how token burning works, unless the box script prevents it.

- **Q** (@Josemi, 2025-10-01): Is token burning different from ERG burning? Is burning ERGs only possible with an always-false script (ignoring storage rent)?
  **A** (@kushti): For ERG, the sum of inputs must equal the sum of outputs. For other tokens, input sum ≥ output sum is sufficient. Token burning is erasure via non-inclusion in outputs; ERG burning requires destruction via an always-false script or storage rent depletion.

- **Q** (@Dũng Nguyễn, 2025-10-06): Where are coinbase transactions in an Ergo block?
  **A** (@kushti): Ergo has a flat design — there are no coinbase transactions. Instead, a genesis-state UTXO is protected by an emission contract. An ordinary transaction charges from it (usually first in a block, but not required). Reference: https://explorer.ergoplatform.com/en/blocks/c6b0bf24c53cb1034d7e2490dc42441a4fb41a8255dd75e8bcaf1d51768e3891/transactions

- **Q** (@Josemi, 2025-10-11): What is the difference between box context variables (`getVar`) and box registers?
  **A** (@kushti): Register values are fixed at box creation time; context extension variables are provided during box spending. Reference: https://docs.ergoplatform.com/dev/scs/global-functions/#getvar

- **Q** (@Dũng Nguyễn, 2025-10-15): Is ErgoScript verified off-chain when signing a transaction?
  **A** (@kushti): Yes — when a transaction is not dependent on blockchain context (e.g., block height), it executes identically off-chain and on-chain. Off-chain computation generates a proof; on-chain verification confirms it.

- **Q** (@Dũng Nguyễn, 2025-10-16): Why implement sub-blocks instead of L2? Won't sub-blocks add unnecessary L1 complexity?
  **A** (@kushti, @Jayedii via Discord): Sub-blocks and L2 solve different problems. Sub-blocks optimise L1 for true P2P currency — faster throughput, native fee markets, decentralisation. L2 solutions are less secure but reduce computational load and are strictly better for most applications requiring decentralisation. Sub-blocks are specifically motivated by: (1) faster sidechain settlement, and (2) faster settlement-layer confirmations.

- **Q** (@Quibbix, 2025-10-23): Why do escript.online and the node API `/script/p2sAddress` generate different addresses for the same ErgoScript lock script?
  **A** (@kushti): Same ErgoTree, different versions (1 vs 0). The bug is that `treeVersion` is used in execution but not in address generation, so both addresses are valid and execute correctly. Version 1 includes tree size metadata, which is better for light clients. Fix will be included in an upcoming release.

- **Q** (@Quibbix, 2025-10-23): What do ErgoTree versions 0 and 1 represent?
  **A** (@kushti, @mgPai): Version is the protocol version. New 6.0 features are only available in v3 trees; v0 and v1 are equivalent in practice. v1 includes a tree size requirement benefiting light clients. Both v0 and v1 must be used consistently.

- **Q** (@Pgr456, 2025-10-31): Why are 45+ boxes unable to collect storage rent past their collection height?
  **A** (@kushti): Integer overflow in the sigma-rust box size function. Box size × storage fee factor exceeds `Int.MaxValue` (e.g., 2,239 × 1,250,000 = 2,798,750,000 > 2,147,483,647). Overflow location: `ErgoInterpreter.scala#L43`. Bounty assigned to sigmastate-interpreter#602.

- **Q** (@Luca_D'Angelo, 2025-10-31): If N inputs share the same script and reference an ErgoTree in a data-input register, is the transaction size smaller than if each input had the ErgoTree in its own register with no data-input?
  **A** (@kushti): Yes — the former approach is more compact.

## Open Questions Entering Next Month

- **Sub-blocks devnet stability**: Rollback processing for input blocks remains incomplete; devnet relaunch pending. Will forking processing fixes be sufficient for stable multi-node testing?

- **Storage rent integer overflow fix**: Bounty assigned to sigmastate-interpreter#602, but fix not yet merged. How many additional boxes are at risk, and is there a mitigation path for already-affected boxes?

- **Explorer indexer root cause**: The temporary Perl proxy on `gql.ergoplatform.com` addresses symptoms; full indexer redesign scope, owner, and timeline are undefined.

- **SigmaUSD oracle monitoring**: Proposal to display oracle operator status publicly was raised but not assigned or scoped. How should oracle health be surfaced to prevent future stalls?

- **Modular oracle architecture**: How to structure the oracle system so non-developers can launch oracles for new data types without codebase-level changes remains architecturally open.

- **Sigma-Rust JS/WASM documentation**: `sigma-rust/bindings/ergo-lib-wasm` lacks formal documentation. LLM-assisted generation was proposed but not assigned or started.

- **Bounties list maintenance**: The Ergo-Bounties GitHub list was flagged as outdated by a new contributor; no confirmed owner or update process established.

- **Privacy features roadmap**: The question of concrete next steps to make Ergo's zero-knowledge features more accessible to end users was raised but not answered with a concrete plan.

- **Reputation proof / SafeW wallet compatibility**: Josemi's reputation forum system throws `ergo.get_current_height is not a function` with SafeW but works with Nautilus; root cause and fix not yet identified.

- **AppKit 6.0 update**: 100 ERG bounty assigned (ergo-appkit#250) but technical blockers not yet characterised; scope unclear.
