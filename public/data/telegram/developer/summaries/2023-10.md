# Ergo Developer Chat — Monthly Summary
**Chat:** Ergo Developer | **Month:** 2023-10

---

## Executive Summary

October 2023 was dominated by infrastructure reliability concerns — particularly Explorer API instability and its downstream effects on wallets and DEXes — alongside significant developer tooling work across AppKit, Fleet SDK, and node configuration. A secondary theme was UTXO spam and mempool management, with productive discussion on prevention strategies, fee markets, and node-level mitigations, though no consensus on a definitive solution was reached.

---

## Key Topics This Month

### Explorer API Reliability & Infrastructure Dependency

Explorer downtime was a recurring pain point throughout the month, surfacing in W40, W41, and W43. The official Ergo platform API experienced multiple outages and stale-state issues, causing cascading failures in downstream tools — notably Spectrum DEX via its Nautilus → GraphQL dependency chain. Independent explorer instances operated by qx() and Sigmanauts provided partial relief. The deeper architectural question — why wallets depend on the explorer for transaction generation rather than querying the node directly for UTXO data — was raised but not resolved. A testnet GraphQL endpoint (`https://gql-testnet.ergoplatform.com/`) was deployed by end of month to support Nautilus and other dApps on testnet.

### UTXO Spam, Box Consolidation, and Mempool Management

Flying Pig demonstrated a critical usability vulnerability: for approximately 2.5 ERG, an attacker can spam a target address with 60,000+ unspent boxes, rendering the wallet nearly unusable. Current wallet consolidation tools are severely limited (Nautilus capped at 200 boxes; SafeW unable to generate consolidation transactions). Discussion covered node-level prevention (limiting boxes per transaction), wallet-side tooling (emergency consolidate buttons), and lite node approaches. Separately, mempool spam mitigation was clarified: the reference client v4.0.44+ implements per-peer cumulative cost limits, operators can adjust `mempool.cleanupDuration` (default 20 minutes), and minimum propagation fee rates can be raised to prevent spam from entering the mempool at all. No consensus on a permanent box-spam prevention approach was reached.

### Node Configuration, Indexing, and Operational Issues

Significant discussion across W41 and W43 around node operations: the `extraIndex = true` setting enables indexed queries but is incompatible with pruned node mode and requires a full resync. Mining pool configuration surfaced through Aberg's Satergo solo mining pool work — a key issue was that `miningPubKeyHex` requires the address in hex format (via `utils/addressToRaw`) rather than base58. Database corruption from improper node shutdowns was identified as a recurring operator problem requiring full resync to recover. AppKit version compatibility (2.12 vs 2.13 for Java 21 projects) and whether the ErgoScript parser can be excluded via modularization remained unresolved.

### Re-emission Contracts and Protocol Mechanics

The re-emission mechanism (EIP-27) was formally discussed: re-emission tokens are injected at the consensus layer, miners must include them in reward boxes, and pay-to-re-emission contracts support periodic merging and ERG withdrawal at 3 ERG per block. This served as a useful reference thread for developers building contracts that interact with emission mechanics.

### Developer Tooling: Fleet SDK, AppKit, and API Integrations

Fleet SDK saw a meaningful bug fix: `STuple` was not being exported from the serializer library, breaking UTXO manipulation in TypeScript/JavaScript contracts. AppKit discussions identified issues with change amount calculation when a single box contains multiple instances of the same token. Token price discovery methods for Satergo were discussed extensively — options include querying Spectrum pool data on-chain via `/boxes/unspent/byTokenId/{pool_id}`, the Spectrum API `/v1/amm/pools/stats` endpoint, and CoinGecko for ERG price conversion.

### Spectrum Network White Paper

kushti released Spectrum Network white paper v1 in W41, featuring a PoS-based global synchronization mechanism, improved VRF lottery selection, and detailed block validation and chain selection algorithms — a significant protocol research milestone for the month.

## Technical Q&A Worth Preserving

- **Q** (@Flying Pig, 2023-10-04): How can mass UTXO spam to a single address be prevented? Testing an airdrop tool accumulated 20,000+ unspent boxes on own address at minimal cost (~2.5 ERG for 60,000 boxes). Average users wouldn't know consolidation procedures.
  **A** (@Luivatra): Prevention can't be 100% guaranteed — tooling to fix it needs to be user-friendly. (@qx()): Wallets should integrate existing consolidation scripts (e.g., from Spire) as an "emergency consolidate" button.

- **Q** (@Flying Pig, 2023-10-04): Does a lite node (NIPoPoWs superblock-based) store all UTXOs? Can it handle mass consolidation without full node or indexed explorer?
  **A** (@qx()): NIPoPoWs superblocks allow ~80% blockchain reduction. Lite nodes can securely interact with the UTXO set without downloading the entire blockchain. Unspent boxes can be retrieved by address or ErgoTree from a non-indexed node — it's unclear why wallets use the explorer for transaction generation rather than the node directly.

- **Q** (@Aberg, W41): What is the best method for token price discovery in Satergo wallet display?
  **A** (@HQΣr): Multiple viable options: (1) on-chain Spectrum pool data via `/boxes/unspent/byTokenId/{pool_id}` calculating from token/ERG ratio — fastest; (2) Spectrum API `/v1/amm/pools/stats`; (3) CoinGecko for ERG price with conversion; (4) `ergcube.com/prices.php` for hourly cached prices.

- **Q** (@Flying Pig, W41): Can a single transaction spend from both a P2S contract address and a user-controlled address, with only the user input signed?
  **A** (@Flying Pig / @Luivatra): Yes — the contract doesn't require a cryptographic signature (it self-validates), only the user address input needs signing via `prover_index`. Use specific box IDs rather than addresses as inputs; multiple box types can coexist in a single transaction.

- **Q** (@Aberg, W43): Is the mnemonic phrase itself stored in the Satergo wallet?
  **A** (@kushti): The seed derived from the mnemonic is stored, not the mnemonic phrase itself. The term "mnemonic" is now avoided in documentation because it implies the phrase needs to be memorized.

- **Q** (@Flying Pig, W43): What settings enable node indexing? GitHub documentation doesn't mention it clearly.
  **A** (@Glasgow): See https://docs.ergoplatform.com/node/indexed-node/ — the setting is `ergo.node.extraIndex = true`. Important caveat: `extraIndex` is incompatible with pruned node mode; enabling it requires a fresh full sync from scratch.

- **Q** (@Flying Pig, W43): How can mempool cleanup of transactions spending already-spent boxes be accelerated?
  **A** (@Pulsarz): Adjust `mempool.cleanupDuration` in node config (default 20 minutes); setting to 2 minutes causes a recheck roughly every block. Reference: node `application.conf` line 83.

- **Q** (@Aberg, W43): Why does providing `miningPubKeyHex` in node config prevent the HTTP server from starting?
  **A** (@Pulsarz): The address must be in hex format obtained via the `utils/addressToRaw` API endpoint — not the base58 address format used elsewhere.

- **Q** (W41): How does mempool spam protection work at the node level?
  **A** (@kushti): The mempool tracks cumulative cost per peer across all contracts in submitted transactions. Once a peer exceeds the threshold, the node stops accepting further transactions from that peer. Reference client v4.0.44+ also adds limits on transactions per peer.

## Open Questions Entering Next Month

- **UTXO box spam prevention**: No consensus on whether mitigation belongs at the transaction layer, node config level, or wallet-side tooling. The fundamental vulnerability (60,000 boxes for ~2.5 ERG) remains unaddressed at the protocol level.

- **Explorer architectural dependency**: Why do wallets use the explorer for transaction generation rather than querying the node directly for UTXO data? No resolution or migration plan established.

- **Stealth address enumeration resistance**: Whether stealth addresses are vulnerable to attacker-generated address enumeration for deanonymization; question raised by Aberg but not resolved.

- **AppKit modularization for Java 21**: Which AppKit version (2.12 vs 2.13) is correct for Java 21 projects? Can the ErgoScript source parser be excluded via modularization to reduce build size?

- **Light testnet node sync**: Aberg reported the light testnet node failed to sync after completing headers download — unclear if this is an architectural issue or configuration problem.

- **Block time variance reduction**: Whether Ergo should pursue variance-reduction mechanisms (e.g., Bobtail proposal); kushti expressed skepticism but no decision was made.

- **LP token verification / DEX ticker disambiguation**: Proposal for verification checkmarks on DEX token listings to prevent confusion with fake LP tokens; no implementation approach decided.

- **Explorer API stale state root cause**: Intermittent stale state served by some Explorer instances remains undiagnosed; affects Spectrum DEX and Nautilus wallet integration.
