# Ergo Developer Chat — Monthly Summary
**Month:** 2022-05
**Chat:** Ergo Developer

---

## Executive Summary

May 2022 was defined by the successful lock-in of EIP-27 re-emission, with ~92.68% hashrate support, marking a critical milestone for Ergo's long-term emission schedule. Alongside this, developers worked through breaking API changes in AppKit and continued progress on Satergo wallet infrastructure. Several unresolved ErgoScript specification questions surfaced, pointing to documentation gaps that carried into the following month.

---

## Key Topics This Month

### EIP-27 Re-emission Lock-In and Activation
EIP-27 achieved lock-in with 949 votes representing ~92.68% hashrate support. Miners were required to update their nodes before the activation block. The re-emission mechanism works by counting re-emission tokens in transaction inputs, sending corresponding nanoERGs to the re-emission proxy contract, and burning the re-emission tokens — with 1 re-emission token corresponding to approximately 1 nanoERG. The next node release after lock-in was slated to remove the `eip27Supported` flag entirely.

### AppKit API Migration
AppKit introduced breaking changes deprecating `ctx.getUnspentBoxesFor()` and removing `BoxOperations.selectTop()`. The replacement pattern centres on `BoxOperations.createForSender()` combined with `ExplorerAndPoolUnspentBoxesLoader()` and `.loadTop()`. Developers were actively migrating and sharing working code patterns to ease the transition.

### Extended Public Key Derivation
A practical solution was found for deriving child addresses from an extended public key without requiring access to the secret key. The approach uses `ExtendedPublicKey.child(index)` on a key derived via `DerivationPath.fromEncoded("m/44'/429'/0'/0")`, enabling read-only wallet integrations — relevant both for Satergo and potential third-party integrations such as Koinly.

### Satergo Wallet and ErgoSock
The ErgoSock library — providing P2P connection to the Ergo network without requiring an HTTP API — was completed and ready for release. Satergo stable release 3 was in final refinement. A release-tagging issue on GitHub was identified where testnet releases were being picked up by Satergo's automatic node downloader; kushti offered to mark such releases as pre-release to prevent this.

### Dexy Protocol
Early-stage work on the Dexy stablecoin protocol was underway, with kushti writing a paper and discussing contracts with @scalahub. No further technical detail was shared publicly in this period.

## Technical Q&A Worth Preserving

- **Q** (@Aberg, 2022-05-20): How do you derive child addresses from an extended public key without access to the secret key?
  **A** (@Aberg): Use `ExtendedPublicKey.child(index)`:
  ```java
  ExtendedPublicKey parentKey = ((ExtendedSecretKey) rootSecret
      .derive(DerivationPath.fromEncoded("m/44'/429'/0'/0").get()))
      .publicKey();
  P2PKAddress.apply(parentKey.child(0).key(), new ErgoAddressEncoder(...))
  ```

- **Q** (@Aberg, 2022-06-13): What replaces `BoxOperations.selectTop` in the updated AppKit?
  **A** (@MrStahlfelge): Use a `BoxOperations` instance with `.loadTop()`. The deprecated method has been moved to `BlockchainDataSource`.

- **Q** (@MrStahlfelge, 2022-06-13): What does the replacement usage of `getUnspentBoxesFor` look like?
  **A** (@MrStahlfelge):
  ```scala
  val boxesToSpend = BoxOperations.createForSender(buyerAddress, ctx)
    .withInputBoxesLoader(
      ExplorerAndPoolUnspentBoxesLoader().withAllowChainedTx(true)
    )
    .withFeeAmount(minerFee)
    .withAmountToSpend(amountToSend)
    .loadTop()
  ```

- **Q** (@kushti, 2022-06-17): How does the EIP-27 re-emission logic work at the transaction level?
  **A** (@kushti): Count re-emission tokens in transaction inputs; send corresponding nanoERGs to the re-emission proxy contract; burn the re-emission tokens. 1 re-emission token ≈ 1 nanoERG.

## Open Questions Entering Next Month

- **ErgoScript paper ambiguities** (@zhang zee): Atomic exchange example — do boxes return to original owners or swap? What is the precise semantics of `propositionBytes` for single-owner boxes? Does the emission contract genesis box require signing at every block, and is this on-chain or off-chain? None of these received definitive answers.
- **Accidental token burn prevention** (@Luivatra): Whether showing burned amounts and using signing screens is sufficient UI-level protection, or whether contract-level constraints are needed — complicated by ErgoPad staking's dependency on current burn behaviour.
- **ErgoScript human-readable formatting** (@Luivatra): Current pretty-print output noted as insufficient; no solution proposed.
- **Testnet ERG sourcing** (@Aberg): Ergofaucet now requires Discord login, complicating testnet workflows; resolved ad hoc in this period but no permanent fix established.
