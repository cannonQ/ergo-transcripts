# Ergo Developer Chat — Monthly Summary
**Month:** 2023-11 (Weeks 45, 47)

---

## Executive Summary

November 2023 developer discussions centred on eUTXO design constraints — particularly around token enforcement, UTXO-based minting rules, and the fundamental limits of on-chain blacklisting — alongside active work on tooling ergonomics (AppKit API design, Satergo solo mining pool, ChainCash Rust implementation). A secondary thread ran through the month on protocol-level architecture questions: sub-block naming conventions, fast-confirmation feasibility for DEXes, and the long-term viability of the sigmastate-interpreter as a separate repository.

---

## Key Topics This Month

### eUTXO Token Constraints and Blacklisting

A recurring theme was the fundamental limits of on-chain enforcement in Ergo's eUTXO model. Discussion clarified that peer-to-peer token transfers cannot be prevented at the protocol level; enforcement can only be achieved through dApp participation and singleton NFTs acting as control mechanisms. Token minting rules were also clarified: only one unique token per transaction is permitted, the box ID of the first input becomes the token ID, and input boxes must be unspent — meaning the same box cannot be reused across multiple minting operations. EIP-4 was identified as the canonical metadata standard governing these constraints.

### Sub-blocks, Fast Confirmations, and P2P Architecture

HQΣr proposed distinguishing "blocks" (weak/sub-blocks) from "Blocks" (full blocks) to reduce protocol terminology ambiguity. Related discussion explored whether a fast-confirmation layer for dApps could be cleanly separated from the peer-to-peer user layer — with no firm conclusion reached. The feasibility of fast confirmations for DEX operations under such an architecture was flagged as an open and complex question requiring further design work.

### AppKit API Design and Tooling Ergonomics

Aberg raised several practical issues with AppKit internals: accessing private fields such as `InputBoxImpl.getExtension()`, converting between `OutBox` and `ErgoBox` instances, and choosing between Scala 2.11, 2.12, and 2.13 builds. Casting to implementation classes was confirmed as the workaround for low-level access, though officially unsupported and fragile across updates. MrStahlfelge noted Scala 2.12 has Android/Proguard bytecode incompatibility issues. No definitive AppKit version guidance was provided. BIP 32 derivation path components were confirmed to use `Seq[Int]` in Ergo's Scala implementation, with hardened components appearing as negative values without reported issues.

### sigmastate-interpreter Repository Separation

A debate arose over whether sigmastate-interpreter should be merged into the main Ergo repository. The consensus was to keep it separate: it is independently versioned, contains JavaScript module dependencies, and merging would introduce cyclic dependencies. The development processes for the two repositories are sufficiently different to warrant continued separation.

### Satergo Solo Mining Pool and ChainCash Rust Bounties

Aberg solicited community testers for the Satergo solo mining pool feature (available on the `solo-mining-pool` branch), requiring a full node wallet setup. Separately, kushti announced bounties on issues in the ChainCash server Rust repository (`chaincash-rs`), inviting Rust developers to contribute.

### ErgOne Protocol Updates

The ErgOne content protocol moved its voting period to the last day of the month, introduced 1 free submission point per registered user (not vote points), and scheduled October rewards distribution. A rewards halving from 5,000 to 2,500 $ErgOne per cycle was announced to take effect the following month.

### Auction House Fee Adjustment

The Auction House increased fees from 2% to 3% to cover development costs and increase profit-sharing rewards for AHT stakers. Improvements to price-per-token display and a verified token process were also added.

### Node API and Synchronisation Issues

Flying Pig reported that certain tokens (B Paideia, remission tokens) could not be found via the Node API `/blockchain/token/byId` endpoint even on indexed nodes — suspected to be an indexing or API bug. HQΣr reported slow node sync despite bootstrap configuration with UTXO and NIPoPoWs enabled. Luivatra observed repeated "Failed to parse modifier" warnings and misbehaviour penalties from a specific peer IP, prompting a node upgrade from 5.0.14 to 5.0.15.

## Technical Q&A Worth Preserving

- **Q** (@Luivatra, 2023-11-20): How do you force the inclusion of a dataInput in every transaction a certain token is in?
  **A** (@Pulsarz): You cannot force inclusion at the protocol level. The workaround is a singleton NFT as a control mechanism, but peer-to-peer transactions still cannot be prevented on-chain — this is a fundamental eUTXO design trade-off.

- **Q** (@Pulsarz, 2023-11-10): Can you mint multiple tokens in one transaction?
  **A** (@Pulsarz / @Luivatra): Only one unique token per transaction with an arbitrary amount. The box ID of the first input becomes the token ID. Input boxes must be unspent, so the same box cannot be reused across minting operations. EIP-4 is the canonical metadata standard describing these constraints.

- **Q** (@Aberg, 2023-11-25): Should BIP 32 path components use `int` (32-bit) or `long` (64-bit) in Java?
  **A** (@kushti): Use `Seq[Int]` as in Ergo's Scala `DerivationPath.scala`. No issues reported despite hardened components appearing as negative values in 32-bit signed representation.

- **Q** (@Aberg, 2023-11-25): Why does `InputBox` not expose `getExtension()` when `InputBoxImpl` does?
  **A** (@Aberg): It is not public API. Casting to implementation classes is the workaround, but this risks breakage on future AppKit updates and is officially unsupported.

- **Q** (@Pulsarz, 2023-11-27): What parts of sigma-rust should be examined to port ErgoTree evaluation to another language?
  **A** (@Pulsarz): Start with `ergotree-ir` and `ergotree-interpreter` from the sigma-rust repository.

- **Q** (@HQΣr, 2023-11-25): Mnemonic restoration from node wallet produces a different address; derived addresses not visible.
  **A** (@kushti): Likely the mnemonic passphrase is not configured correctly. If no passphrase was used during wallet creation, it should be left blank during restoration.

- **Q** (implied, 2023-11-10): Should sigmastate-interpreter be merged into the main Ergo repository?
  **A** (@Aberg): No. sigmastate-interpreter is independently versioned, contains JavaScript module dependencies, and merging would create cyclic dependencies. Development processes differ sufficiently to keep them separate.

- **Q** (@HQΣr, 2023-11-07): Can someone summarise sub-blocks (weak blocks) and how they work?
  **A** (@HQΣr): Proposed naming convention: weak blocks as "blocks" and full blocks as "Blocks". Fast confirmation for DEXes via a P2P layer would require separating user-facing and dApp protocol layers — the architecture is still exploratory and feasibility for DEX operations is unclear.

- **Q** (@Flying Pig, 2023-11-22): Node API cannot find certain tokens (B Paideia, remission tokens) via `/blockchain/token/byId` despite confirmed blocks. Why?
  **A** (Unresolved): Issue confirmed on both local indexed and remote nodes; suspected indexing or API bug, not investigated further in this period.
