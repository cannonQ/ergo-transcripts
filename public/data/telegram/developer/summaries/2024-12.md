# Ergo Developer Chat — Monthly Summary: December 2024

---

## Executive Summary

December 2024 was a month of significant protocol infrastructure work, centred on advancing the Ergo 6.0 soft-fork (tree-based versioning, AVL+ tree semantics, sub-blocks implementation) towards testnet activation. Alongside this, several ecosystem security events — a critical Paideia staking contract exploit, Oracle Pool v1 operator downtime, and mainnet node DDoS/rate-limiting issues — prompted meaningful discussion about auditing culture, incentivisation design, and infrastructure resilience. Developer tooling matured substantially, with Fleet SDK gaining network-aware compilation, Sigma Rust PRs merged for Keystone hardware wallet integration, and Nautilus UI undergoing a significant redesign.

---

## Key Topics This Month

### Ergo 6.0 Soft-Fork Progress

The 6.0 protocol update dominated technical discussion throughout the month. Key milestones included: finalising AVL+ tree semantics — fixing `insert`/`update` operation error handling and adding the new `insertOrUpdate` operation (PR #1038); implementing tree-based versioning (PR #1043), where ErgoTree version rather than script version gates 6.0 semantics changes such as lazy evaluation in `opt.getOrElse(default)`; and completing nuanced serialization versioning (PR #1041). kushti clarified that new operations and the unsigned `BigInt` type will be v3-only ErgoTree features. By end of month, versioning was passing tests, with devnet functional testing planned before public testnet activation requiring 90%+ hashrate voting threshold. Sub-blocks implementation also advanced: linked chain structure was finalised, with complex transaction packing into input blocks and UTXO set commitment updates identified as the most involved remaining work. The double-spend prevention rule was confirmed — a double-spend cannot appear in a follow-up sub-block after protocol lock-in.

### AVL+ Trees in JavaScript / Fleet SDK

A sustained thread across weeks 49–50 explored AVL+ tree support in JavaScript tooling. The core problem: `ergo-lib-wasm` lacks AVL utilities, creating a gap for dApp developers building Fleet-based offchain logic. The established pattern — storing the AVL tree digest (32-byte hash) in a register, passing lookup proofs as context variables — was confirmed as sufficient; the full tree need not be passed. kushti confirmed that AVL+ classes have been translated to `sigmastate-js`, making Fleet support technically feasible, but noted JavaScript expertise is limited in the core team. arobsn indicated a dedicated Fleet AVL+ wrapper package (analogous to Plasma lib) is planned. Existing references — GetBlok Plasma documentation, the offchain Bank layer-2 example, and Paideia's plasma staking contract — were provided as interim guidance.

### SigmaUSD Bank V2 / V3 and Oracle Pool Migration

The SigmaUSD bank contract update followed a complex path this month. Bank v2 was deployed to testnet (week 49), with 0.5% per-oracle-update mint/redeem limits (~16k SigmaUSD per epoch at current collateralisation). The contract was subsequently renamed v3, with TokenJay targeting December for UI support. Oracle Pool v1 operator downtime (week 51) exposed the core incentivisation gap: operators are not compensated by the bank/protocol, unlike Gluon and Dexy Gold reward schemes. This prompted kushti to propose a full Oracle Pool V2 migration — new update NFT, fixed contract removing token check, DORT token injection support — with the EIP PR #29 update contract fix as a prerequisite. By week 52, a pragmatic interim decision emerged: updating the bank contract to itself to allow more time for V2 detail refinement, with Luivatra in agreement.

### Paideia Staking Contract Exploit & Auditing Culture

Luivatra discovered a critical vulnerability in Paideia staking contracts (week 51): missing validation allowed staking zero or negative token amounts, exploitable via AVL tree manipulation. Luivatra preemptively removed 55 million staked Paideia and offered a 25k Paideia bounty for responsible disclosure. The fix was deployed and verified; exploited assets were returned via a Paideia DAO proposal by week 52 — notably without any backdoor, purely through on-chain governance. The incident sparked wider discussion about the auditing culture gap: most projects operate on shoestring budgets without access to professional audits. CannonQ and others flagged availability for protocol review work. A related Paideia off-chain bug was also discovered — mempool filtering corrupted state when chained transactions spent UTXOs before confirmation — and a general fix was merged.

### Node Infrastructure: DDoS, Memory, and RocksDB

Several infrastructure issues converged in weeks 50–51. Node 213 experienced sustained connection flooding (possible DDoS from an Azerbaijan subnet), prompting kushti to implement rate limiting and subnet banning. A Hetzner server was generating tens-of-megabyte responses per request and was flagged but not yet banned. Separately, multiple operators reported RocksDB `OutOfMemoryError` and block checksum mismatches; one operator experienced disk-level corruption. Node 5.1.1 was released in week 52 with versioning fixes, and sync performance data was gathered: 20 hours on a 20-thread/8GB/mirrored NVMe machine versus 60 hours on a 4-core/8GB VPS. A silent configuration failure mode was identified — unreadable `ergo.conf` files cause indexing to silently not activate, with no error logged. Best practice established: sync fully first, then enable extra indexing.

### SigmaUSD UI, ErgoPay Architecture, and Contract Metadata

c8e4d2a developed a new SigmaUSD UI that communicates directly with a node rather than an explorer, improving transparency. This prompted a substantive debate on proxy contracts and ErgoPay's role in user safety: c8e4d2a argued that proxy contracts enable insecure dApp design without visible address verification, while MrStahlfelge countered that ErgoPay is transport-agnostic and that dApp design is the real issue. A proposed solution emerged: a curated verified contract repository (EIP-5 style JSON with templates) in the `ergo-address-labels` repo, allowing wallets to display human-readable metadata for known contract addresses. The Nautilus ErgoPay gap remained unresolved — Captain Nemo's philosophical objection to running backend services was cited but not confirmed.

### Fleet SDK, Sigma Rust, and Keystone Integration

arobsn added network-aware compilation support to Fleet (PR #156), enabling testnet/mainnet address handling in ErgoScript via a `{ network: "testnet" }` compile option. A `Blake2b` function was added to the Fleet crypto package. On the Sigma Rust side, PRs #781 and #782 (deterministic nonce handling) were merged after kushti's review, and PR #796 (deterministic reduced transaction signing) was added for Keystone support. Keystone firmware (PR #1427) and SDK (PR #105) integrations reached draft PR stage, with Bluetooth added to Satergo enabling larger UR-standard QR payloads. Nautilus UI redesign progressed with dark theme support, NFT/Assets view consolidation, and removal of Emurgo's `cip4-js` dependency.

### Cornell University Ergo Payment System

qx() deployed a tipper-bot-based blockchain payment system for large-format plotting at Cornell University (week 51). The workflow: users upload files, receive a Discord QR with a preset tipper command, the server scans the wallet for ERG or Cornell BRICK (a LETS token) payment. The system is launching next semester in production and was noted as a replicable model for other institutions. The Cornell BRICK token represents a LETS-style local exchange implementation on Ergo.

### Lithos Protocol and Celaut

Lithos Protocol advanced its optimistic rollup design for NIPoPoWs share proofs and miner payments, with the last NISP contracts being finalised and fraud proof contract interaction model completed (week 52). A 3-month activation timeline for voting was considered acceptable — voting to start immediately, activation in March. Celaut's reputation system reached a significant UX milestone: node reputation proof sharing now works with zero ERG balance and zero configuration, enabling "launch with one command."

### Documentation Expansion

Mark Glasgow significantly expanded `docs.ergoplatform.com` in week 51, adding: Rosen Bridge token integration guide, Babel Fees documentation, Merkle Trees, soft-fork mechanics, Sigma protocols internals, signature scheme details, ErgoScript compiler phases, and a data model overview. Community requests for a comprehensive node installation guide for non-technical users were addressed with references to `ergo-installer.sh`, Christophe Garant's Docker Compose setup, and a Raspberry Pi full node guide.

## Technical Q&A Worth Preserving

- **Q** (@c8e4d2a, 2024-12-03): Is it viable to store the AVL tree hash in a contract register, provide the tree as a context variable, and provide a lookup proof?
  **A** (@cheeseenthusiast): Yes. A register of type `AVLTree` holding the 32-byte digest plus lookup proofs in context variables is sufficient — the full tree does not need to be passed as a context variable.

- **Q** (@c8e4d2a, 2024-12-04): What happens to NFTs/tokens in a box collected for storage rent if insufficient ERG remains?
  **A** (@Pgr456): Assets go to the party claiming the box. The claimer can choose to burn or keep them — this is not automatic during the collection step.

- **Q** (@dusek_, 2024-12-04): For `insertOrUpdate`, must both `isInsertAllowed` and `isUpdateAllowed` flags be set on the AVL tree?
  **A** (@kushti): Yes. The operation checks both flags and fails if either is not set.

- **Q** (@c8e4d2a, 2024-12-05): Does the Fleet compiler have a mainnet/testnet compilation option?
  **A** (@arobsn): Yes, added in PR #156. Usage: pass `{ network: "testnet" }` as an option to the compile function. The `PK()` function uses network-specific address encoders, so testnet addresses in a mainnet context would cause encoder failure.

- **Q** (@c8e4d2a, 2024-12-11): How can I verify in ErgoScript that an NFT was minted by a specific contract, without relying on offchain historical data?
  **A** (@ldgaetano): Pass the issuer box as a context variable and check that its box ID matches the NFT token ID. Due to Ergo's token semantics, if the NFT ID equals the spent box ID, that box must have existed and been spent to mint the token — this is cryptographically enforced.
  **Additional** (@kushti): For evolution contracts, create an unspendable receipt UTXO at minting time containing the NFT ID and token reference. Evolution logic can use this receipt as a data input for on-chain validation.

- **Q** (@c8e4d2a, 2024-12-11): Will the contract fail if a tampered fake box is passed as a context variable?
  **A** (@ldgaetano): Yes. Box ID is a hash of the box contents. Creating a fake box with the same ID as an existing one is computationally infeasible.

- **Q** (@jossemii, 2024-12-17): Can it be demonstrated that an ErgoScript is equivalent to a given ErgoTree?
  **A** (@Luivatra): Compile the ErgoScript to ErgoTree and compare. However, future compiler versions may produce different tree shapes with identical semantics, and the original source cannot be fully recovered from ErgoTree due to compiler optimisations and lost variable names.
  **Additional** (@kushti): Compilation is deterministic for a given compiler version — the test suite relies on this extensively.

- **Q** (@DiscordBridge/c8e4d2a, 2024-12-17): How exactly will RBF work with sub-blocks?
  **A** (@kushti): A double-spend cannot be included in a follow-up sub-block after the full lock-in protocol completes.

- **Q** (@Aco Šmrkas, 2024-12-17): Is iterating over inputs to check if any contains a specific token a valid ErgoScript pattern?
  **A** (@Luivatra): Yes, that pattern is fine.

- **Q** (@Luivatra, 2024-12-23): Node not indexing despite `isExplorer=true` — what's happening?
  **A** (@Luivatra, self-resolved): File permission issue on `ergo.conf` — the node silently ignores unparseable or unreadable config files without logging an error.
  **Additional** (@kushti): The node merges three config files (`application.conf`, `mainnet.conf` from the JAR, and user-provided conf). Inaccessible files are silently ignored. Best practice: sync fully first, then enable extra indexing.

- **Q** (@mgpai, 2024-12-04): Are transactions returned in order when querying a full block?
  **A** (@Luivatra): Yes in practice, though not explicitly guaranteed by the API contract.
  **Confirmed** (@kushti): Yes.

- **Q** (@mgpai, 2024-12-06): A transaction ID sometimes differs between AppKit's computed hash and what appears in the node mempool — why?
  **A** (@kushti): The transaction bytes must have changed somehow. Requires investigation of the specific case — transaction ID is a deterministic hash of bytes.

- **Q** (@c8e4d2a, 2024-12-26): How to get input boxes of mempool transactions?
  **A** (@mgpai): Requires additional queries or pre-indexing all relevant UTXOs beforehand.

## Open Questions Entering Next Month

1. **6.0 Testnet Activation**: Who is actively mining the testnet? The 90%+ hashrate voting threshold needs assessment before public testnet activation can proceed.

2. **SigmaUSD Bank V2/V3 Finalisation**: Should the bank contract be updated to itself to allow more V2 refinement time? Agreed in principle between kushti and Luivatra but not formally decided. UI timeline (mid-January target from c8e4d2a) unconfirmed.

3. **Oracle Pool V2 Migration Path**: DORT token injection requires fixing the update contract (EIP PR #29). No active proposals yet; V1 operator incentivisation gap remains a live risk for SigmaUSD stability.

4. **Sub-blocks Impact on Existing DeFi Contracts**: Faster blocks and sub-blocks will affect timestamp-based logic, block reward emission contracts, and potentially AMM/lending contracts. Systematic analysis not yet conducted; flagged for post-Ergoversary technical roundtable.

5. **Node Memory Requirements for 5.1**: 1GB verified acceptable in 5.0; 512MB crashes after days in 5.1 with extra indexing. Formal documentation and minimum spec guidance needed.

6. **RocksDB vs. LevelDB Stability**: Multiple disk-level corruption reports on indexed nodes. Outcome of operators switching to RocksDB pending.

7. **ErgoPay Architecture and Proxy Contract Verification**: No consensus on whether curated contract metadata repository (EIP-5 style), transport-layer improvements, or dApp design guidelines are the right path. Flagged as potential ErgoHack topic.

8. **Nautilus ErgoPay Support**: Whether the philosophical objection to backend services is the actual blocker remains unconfirmed.

9. **Fleet AVL+ Wrapper Package**: arobsn indicated intent to build this; no timeline or design decisions made yet.

10. **Gluon SDK Architecture Refactor**: Mixed state throughout the SDK makes pure mathematical operations infeasible. No implementation plan or assignee established.

11. **Speed Improvement Path (Sub-blocks vs. Microblocks vs. Layer 2)**: Community interest confirmed; dedicated testnet coordination channel offered by kushti; no concrete path chosen. Post-Ergoversary discussion scheduled.
