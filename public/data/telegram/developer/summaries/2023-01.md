# Ergo Developer Chat — Monthly Summary: January 2023

## Executive Summary

January 2023 was a technically active month for Ergo development, centred on protocol stability and the road ahead to node v6.0. Key activity included the release of reference client v5.0.6, active Dexy Protocol testing, and a community discussion on the roadmap. Oracle price feed infrastructure and block time optimisation emerged as significant open research areas with no final resolution.

---

## Key Topics This Month

### Reference Client v5.0.6 Release
kushti shipped v5.0.6 of the Ergo reference client, following shortly after v5.0.5. The release was accompanied by documentation work on the Ergo block header format, including sorting headers during proof generation and cross-referencing with the Rust implementation. A community discussion was scheduled for 20 January 2023 specifically to map the roadmap toward node v6.0.

### Dexy Protocol Testing
scalahub was actively testing the Dexy Protocol, with work focused on implementing test cases for "Extract to future" operations. kushti indicated a merge and release was on track for the following week after a final testing and polishing phase. This represented meaningful progress toward a production-ready Dexy deployment on Ergo.

### Oracle Price Feed Architecture
A substantive discussion emerged around sourcing price data for oracle integration. Aberg evaluated several APIs — metals-api.com, CoinGecko, and direct exchange APIs (KuCoin, CoinEx) — weighing trade-offs across API rate limits (50–100 requests/month on most services), market pair coverage (USD/USDT vs BTC pairs), source reliability (concerns raised about Tether and Bitfinex), and resilience against single points of failure. No consensus architecture was reached.

### Block Time Optimisation & Contract Constraints
Discussion touched on the complexity of reducing block times on Ergo, noting that many dApps are tied to block height rather than timestamps. This creates a significant constraint: faster blocks would affect emission contracts and other height-dependent logic. kushti flagged that optimisation requires careful testnet validation, and invited interested community members to engage with the existing testnet parameter-tweaking infrastructure.

### Language Choice: Scala for Ergo
A discussion thread examined why Scala was chosen for core Ergo components rather than more widely adopted languages. kushti explained the primary driver was expressiveness. Notably, some components — such as AVL+ trees with batched proofs — were contributed by developers (including Prof. Reyzin, whose background was C++) who learned Scala specifically for the project.

## Technical Q&A Worth Preserving

- **Q** (@Aberg, 2023-01-16): Why were most Ergo components written in Scala rather than more popular languages like Java — is it performance or expressiveness?
  **A** (@kushti): Better expressiveness. Some parts were written by developers not previously familiar with Scala — Prof. Reyzin implemented AVL+ trees with batched proofs and primarily used C++ beforehand.

- **Q** (@Aberg, 2023-01-16): Did you fix the get peers filter that was broken a few months ago?
  **A** (@kushti): Taika is working on it.

- **Q** (@kushti, 2023-01-19): How should box ID be handled in contract design?
  **A** (@kushti): The ID is a hash of the box contents.

- **Q** (@kushti, 2023-01-19): How should byte array iteration be handled in ErgoScript contracts?
  **A** (@kushti): The simplest approach is to put the bytes array in a register and iterate over it.

- **Q** (@HQΣr, 2023-01-25): What is the correct API call to fetch an ErgoLaunchpad project by ID?
  **A** (@Luivatra): Use `https://api.ergopad.io/projects/16` — a simple REST endpoint rather than a filters parameter.

- **Q** (@Luigi, 2023-01-25): Getting error "Unable to define external address. Specify it manually in `scorex.network.declaredAddress`" when running a node — how to resolve?
  **A** (@Aberg): Not resolved in chat; Aberg suggested using Satergo as an easier alternative setup path.

## Open Questions Entering Next Month

- **Oracle price feed architecture**: No consensus on the optimal data source strategy. Trade-offs between API rate limits, market pair coverage, Tether/Bitfinex reliability concerns, and single-point-of-failure risk remain unresolved.
- **Block time optimisation**: The constraint that many dApps rely on block height rather than timestamps complicates any parameter changes. Community testnet validation work needed before any proposal can advance.
- **Node external address configuration**: The `scorex.network.declaredAddress` error encountered by Luigi was not resolved; potential documentation gap for operators running nodes for off-chain bots.
- **Peer filter bug**: Get peers filter breakage acknowledged; Taika assigned but no resolution confirmed within the month.
