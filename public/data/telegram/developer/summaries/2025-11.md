# Ergo Developer Chat — Monthly Summary: November 2025

---

## Executive Summary

November 2025 was dominated by three converging themes: the active development and testnet launch of Ergo 6.0 (including sub-blocks, Autolykos v2, and new protocol features), significant infrastructure work on Explorer performance and query optimization, and a broadening ecosystem with new projects (Degens.World, Lithos Protocol, Palmyra/USE stablecoin) building on Ergo's UTXO model. Cross-cutting these was an emerging conversation about AI-assisted development governance, with the community establishing early norms around LLM-generated code contributions.

---

## Key Topics This Month

### Ergo 6.0, Sub-blocks & Testnet Launch

The month's most significant technical thread was the progression of Ergo 6.0 toward testnet. The existing testnet emission contract ran out of ERG in W46, prompting a clean restart: the new LITHOS_NET testnet launched with 6.0 activated from genesis block, Autolykos v2 enabled immediately, and a target block time of 1–2 minutes. Node operators were instructed to wipe their databases, update peer endpoints (`128.253.41.110:9020`, `213.239.193.208:9023`), and build from PR #2252.

Sub-blocks implementation continued throughout the month, with kushti reworking fork processing logic, adding tests, and running a local devnet with two mining peers. A dedicated devnet endpoint (`213.239.193.208:9058`) was made available by W47. The whitepaper is in progress but not yet published. PR #2249 and #2250 were reviewed and merged; issue #2251 tracks remaining mitigation strategies. A new mining reward API route (PR #2255) was submitted for review, with the caveat that the internal wallet doesn't natively track miner rewards and documentation is required.

A key 6.0 feature highlighted by kushti is the ability to provide a script via context extension of one input and read it across all inputs — potentially reducing transaction formation complexity for developers.

### Explorer Backend: Performance & Query Optimization

A sustained effort to reduce reliance on the Hetzner-hosted Cornell instance (128GB RAM, high-spec SSDs) was central to W45–W47. Heavy queries — particularly `byErgoTree` lookups with 500+ offsets and address transaction scans — timed out on p2p nodes (32GB RAM) but succeeded on Hetzner. Root causes were identified as missing database indices and RAM disparity rather than fundamental algorithmic issues. By W47, a 16GB RAM node was achieving 14-second responses on previously failing queries via targeted index tuning, and a 32GB node was being tested for the full query suite. A hash-comparison index approach (`hashtextextended(address, 0)` with composite index on hash, timestamp, spent, tx_id) was proposed and tested. GitHub issue #265 tracks DB optimization progress. Explorer backend PRs #260, #261, and #256 were all active during the month; PR #260 was approved and merged with direct compensation from kushti to avoid DevDAO overhead.

The Hetzner outage on 2025-11-03 (W45) — which temporarily redirected API, GraphQL, and Explorer to Cloudflare workers and p2p nodes — served as a live stress test, exposing these performance gaps and motivating the optimization push.

### Rosen Bridge: Bitcoin Runes, Handshake Support & Watcher Updates

Rosen Bridge had a productive month across multiple fronts. The Bitcoin Runes release entered final QA, replacing the P2P system with Rosenet, integrating a new Ergo Multisig package, and upgrading services to Node v22.18. A TokenMap system is being implemented in two stages to decentralize token configuration management. Separately, @AcoSmrkas submitted Handshake (HNS) support PRs across the full Rosen stack (Utils, Scanner, Chains, Health Check, UI, Watcher, Guard Service). DefiLlama token mappings were updated to include Doge.

In W48, users were instructed to update their watchers, with kushti noting that Rosen Bridge requires 60%+1 consensus across each supported network before updates take effect. The Rosen Bridge SDK (`github.com/rosen-bridge/rosen-sdk`) was confirmed available for third-party integration, though EVM support gaps were noted.

### Lithos Protocol Mining Integration

Lithos Protocol moved from concept to testnet across the month. v1.0-test was tagged in W47 (by @Armeanio), followed quickly by v1.1-test in the same week with bug fixes, improved stratum error handling, and simplified configuration. Cheese Enthusiast is leading client development; the mining integration path clarified significantly, with the Lithos client expected to track miner reward UTXOs and Nautilus expected to scan the mining reward contract. PR #2255 (mining reward API route) is the upstream Ergo node change needed to fully support this. A recurring research question — how NIPoPoWs' "goodness" concept might translate to LLM inference work in a Lithos context — was raised but remains unanswered.

### USE Stablecoin & Dexy Configuration

The USE stablecoin (essentially DexyUSD with DexyGold-like parameters) moved toward deployment throughout November. By W48, kushti published updated Dexy configuration changes (PR #5 on kushti/dexy-stable) specifying new oracle data normalization, 1200% overcollateralization, and a 0.1% weekly payment maximum. @Pgr456 provided the DORT token data needed for the deployment (pool NFT, oracle token ID, LP NFT). An unresolved question at month end was whether the USE pool contract ErgoTree template differs from the DexyGold template — flagged by @Luivatra for confirmation before final launch.

### Palmyra & Chainlink Partnership

Dan (Palmyra/Winter Protocol) explained the Chainlink integration during W47 discussions: the partnership uses Chainlink's proof-of-reserve and oracle functions (via proxy function in Winter Protocol) to issue on-chain warehouse receipts for decentralized trade financing. Commodity producers deposit into bonded warehouses and receive on-chain receipts backed by IoT sensors (weight, motion, camera — depending on commodity). Chainlink was chosen for network size and support breadth; however, the plan explicitly includes building out native Ergo Oracle Pool infrastructure post-token-sale. Palmyra currently runs exclusively on Ergo. A community note from W47 cautioned against tribalistic reactions to multi-chain tooling choices: companies may use one, two, or all aspects of the Ergo ecosystem, and that constitutes meaningful adoption.

### AI-Assisted Development Governance

A cross-week debate emerged about AI-generated code contributions to Ergo repositories. Concerns raised by @Pgr456 in W47 included: unrelated file changes mixed into PRs, poor PR descriptions, and increased review burden without clear quality guarantees. kushti proposed `AGENTS.md` files in every repo establishing boundaries: AI assistance acceptable for tests, documentation, and refactoring; consensus-critical code should be explicitly locked out. A parallel thread in W48 focused on safe LLM context management for ErgoScript generation — simple markdown files are insufficient; a dedicated agent directory with security guidelines and explicit instructions (e.g., "only use details within agent files; notify user if information not found") is needed to prevent web searches from injecting malicious code. "AI-friendly" bounty labels (e.g., for documentation and non-consensus refactoring tasks) were proposed as a channel for this work. `AGENTS.md` was merged into the 6.0.2 protocol candidate (PR #2242) in W45.

### NIPoPoWs for Offline & Trustless Verification

A noteworthy W48 discussion explored using NIPoPoWs for trustless transaction verification without internet access — the motivating example being a coconut shop accepting ERG payments offline. kushti confirmed that NIPoPoW proofs can work with a minimum work threshold (derived from mainnet parameters) plus sufficient confirmations, and that proofs can even be printed on paper for physical presentation. The recommended libraries are sigma-rust or Ergo core (JVM side). This connects to broader Basis/ChainCash development goals around remote-area trading using mesh networks.

### DuckPools v2, Fleet SDK & Ecosystem Tooling

DuckPools v2 deployed and audited new proxy contracts with an updated fee structure in W47 (cannon_q); a database resync (~24 hours) was underway with dashboards and portfolio statistics to follow, targeting a December 8 public feedback release. Fleet SDK saw significant dependency updates in W47: tsup replaced with tsdown for faster builds, and @noble/* and @scure/* crypto libraries bumped to v2 with substantial performance gains. v1 stable release is approaching; v2 API planning is underway led by greenhat/arobsn. Nautilus wallet integration of Keystone hardware wallet support was flagged as a supply chain concern (160+ transient dependencies), with @anon_br investigating mitigations before merge.

## Technical Q&A Worth Preserving

- **Q** (@TheStophe, 2025-11-17): LITHOS_NET testnet node not syncing despite adding known peers.
  **A** (@qx()): Key config requirements: `declaredAddress` (external IP), `apiKeyHash`, `nodeName`; firewall rule needed on port 9020. Root cause turned out to be missing `--testnet` execution flag.

- **Q** (@Alien, 2025-11-17): Is USE stablecoin similar to SigmaUSD?
  **A** (@kushti, 2025-11-19): USE is essentially DexyUSD with the same parameters as Dexy Gold — not SigmaUSD. See ChainCash/Basis channels for mechanics.

- **Q** (@ArOhBeK, 2025-11-17): Is it possible to send messages between nodes without knowing their IP?
  **A** (@kushti): No messaging layer or DHT storage currently exists on the Ergo p2p network. Discussed in 2019–20; worth reviving. Temporary workaround: `/allpeers` + node name search + handshake confirmation.

- **Q** (@Josemi, 2025-11-20): Template ID differs when using 32 zeros vs. other 32-char hex in the Bene contract. Why?
  **A** (@kushti): Compiler optimization likely strips the unused `isErgBased` variable from the ErgoTree when value is all zeros; the length difference changes the template ID. Resolution: work with both template IDs; zeros treated as empty `Coll[Byte]`.

- **Q** (@anon_br, 2025-11-20): What happens to transactions in the mempool during a block rollback?
  **A** (@kushti): Transactions from rolled-back blocks are resubmitted to the mempool, subject to current mempool size limits.

- **Q** (@Pgr456, 2025-11-20): AI-generated PRs mixing unrelated changes and poor descriptions — how to enforce standards?
  **A** (@kushti): Propose `AGENTS.md` in every repo with explicit rules: AI assistance permitted only for tests, documentation, and refactoring; consensus-critical code must be isolated from AI contributions.

- **Q** (@Josemi, 2025-11-24): Can NIPoPoWs verify transactions without a trusted node or internet connection?
  **A** (@kushti): Yes. Use a NIPoPoW with a minimum work amount sourced from mainnet parameters plus sufficient confirmations. Proofs can even be printed on paper for offline physical verification.

- **Q** (@Josemi, 2025-11-26): Is there a library to validate NIPoPoWs?
  **A** (@kushti): Use sigma-rust or Ergo core (JVM side).

- **Q** (@ArOhBeK, 2025-11-24): How to safely manage LLM context for ErgoScript generation?
  **A** (@kushti, @Arohbe): Create a dedicated repository with a master `agent.md` directory structure. Add explicit instructions such as "only use details within agent files; notify the user if information is not found" to prevent web searches from injecting potentially malicious code.

- **Q** (@Sheetoshi, 2025-11-25): Why does a locally run explorer show 93B ERG circulating supply vs. 82B on the mainnet API?
  **A** (@Michael, @kushti): Difference due to reemission and emission proxy accounting. Use explorer version 9.17.4 or later; supply figure should exclude reemission and emission proxy balances.

- **Q** (@Cheese, 2025-11-22): Can the mining reward API route from PR #2255 be added to mainnet/testnet builds?
  **A** (@kushti): PR looks good but review is pending. Complexity: the internal wallet does not track miner rewards natively; external tracking and documentation are required. The Lithos client will handle this; Nautilus should scan the mining reward contract for new UTXOs.

- **Q** (@Alien, W45): Does Rosen Bridge expose APIs for third-party platform integration?
  **A** (@Michael): Yes — see `github.com/rosen-bridge/rosen-sdk`. EVM support is incomplete on some chains.

- **Q** (@kushti, W45): Starting from 6.0, can context extensions reduce transaction formation friction?
  **A** (@kushti): Yes. From 6.0, a script can be provided via the context extension of one input and read across all inputs — useful for reducing per-input script duplication in complex transactions.
