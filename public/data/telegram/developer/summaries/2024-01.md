# Ergo Developer Chat — Monthly Summary
## January 2024

---

## Executive Summary

January 2024 was dominated by in-depth technical discussion around sub-blocks as a scaling mechanism, with significant community effort clarifying the distinction between dApp-layer throughput improvements and mainchain confirmation speed. Secondary themes included stealth address cryptography implementation, register serialization tooling, and developer ecosystem metrics placing Ergo in the top 100 active blockchain ecosystems globally.

---

## Key Topics This Month

### Sub-blocks, Scaling, and Confirmation Times

The most sustained discussion of the month centred on sub-block implementation and what it can and cannot achieve. The core clarification that emerged: sub-blocks are a dApp-layer scaling solution that improves network throughput and UX, but do not reduce mainchain p2p confirmation time. Exchanges and bridges such as Rosen Bridge impose artificial confirmation thresholds (e.g., 20 confirmations ≈ 40 minutes) for security reasons; sub-blocks alone cannot lower these without policy changes from those operators. Rosen specifically requires approximately 80 minutes of security guarantees regardless of block structure, and faster blocks would increase reorg depth rather than reduce settlement time. Changing the 2-minute block interval would require a hard fork; the rationale for this interval is documented on ergoplatform.org. Sub-blocks remain the preferred path for use cases where weak confirmations are acceptable, and incentives for miners to generate and propagate sub-blocks are expected to come via application-specific fee agreements or merge-mined sidechains rather than core protocol changes.

### Mempool Configuration and Spam Resilience

Related to scaling, the default mempool size of 1,000 transactions was noted as configurable up to 10,000 with `rebroadcastCount` changes, allowing nodes to propagate stuck transactions to peers that have not yet accepted them. Node operators can raise the minimum propagation fee rate to prevent spam transactions from entering the mempool entirely. A fee estimation API exists in the node but is not yet used by any known applications and has not been thoroughly tested — this was flagged as a gap worth addressing.

### Stealth Address Implementation

Aberg (Satergo dev) began reimplementing stealth address logic from JavaScript into Java, working through BouncyCastle ECPoint multiplication and serialization. The question of how to convert an ECPoint (result of scalar multiplication) back to a BigInteger representation in BouncyCastle was raised but not resolved in chat. A broader conceptual question — whether stealth addresses provide meaningful privacy if on-chain transaction linking remains visible post-withdrawal — was raised by Flying Pig but received no definitive answer.

### Register Serialization and Python Tooling

Flying Pig worked through deserializing Ergo register values (specifically `r4` and `r5`) from SigmaUSD bank boxes in Python, ultimately forking an existing serialization library to add deserialization support. The process was largely iterative and self-resolved. This highlighted a gap in accessible Python tooling for register introspection.

### Address Derivation Discrepancy

Flying Pig reported that node and Nautilus wallet derived matching addresses only for the first address from the same seed, diverging on subsequent derivations. kushti suggested checking for a mnemonic passphrase or an old derivation flag as the likely cause, but the root cause was not confirmed within the reporting period.

### Developer Ecosystem Metrics

Ergo was ranked in the top 100 blockchain ecosystems in the Electric Capital Developer Report, with 58 monthly active developers and 17 full-time contributors — placing above Tron. A policy change in Electric Capital's methodology excluded some recent repositories due to missing repo checks, slightly understating activity. The Ergo ecosystem entry is maintained at the Electric Capital crypto-ecosystems GitHub.

### Ergo Website and Infrastructure

The Ergo website was noted as undergoing a refresh, with the active content repository moved to `ergoweb/content/locales/`. Contributors were advised not to submit PRs to the old repository. Aberg also updated the ergonodes.net server from an obsolete web library to a modern one, enabling future HTTPS support without user-facing changes.

## Technical Q&A Worth Preserving

- **Q** (@Hasnain, 2024-01-03): Will sub-blocks reduce p2p transaction confirmation time? If exchange confirmations require 10–15 minutes, is sub-block impact limited to dApps only?
  **A** (@cafebedouin.org): Sub-blocks are a scaling solution for dApp users, not the mainchain itself. Exchanges impose artificial confirmation thresholds for security; sub-blocks improve UX and network utilisation but will not lower those thresholds.

- **Q** (@Hasnain, 2024-01-03): Will 20 confirmations take less time after sub-block implementation?
  **A** (@Glasgow): 20 confirmations will take the same time. Sub-blocks are smaller blocks within the confirmation window with lower difficulty — analogous to super-blocks used in NIPoPoWs. Exchanges and Rosen set manual confirmation limits; if block time decreased, they would increase required confirmation counts to compensate.

- **Q** (@Hasnain, 2024-01-03): Can Ergo achieve ~10–15 second confirmation times without a hard fork?
  **A** (@Glasgow): No — changing the 2-minute block interval requires a hard fork. The rationale is documented at ergoplatform.org. Sub-blocks are the preferred solution for use cases where weak confirmations are acceptable.

- **Q** (@Hasnain, 2024-01-03): Could Rosen Bridge use a mixed approach — 50% weak blocks, 50% strong blocks — to reduce settlement time?
  **A** (@Pulsarz): Rosen is not a good candidate for this; it cares about elapsed time, not block count. Faster blocks increase reorg depth and risk. Rosen requires approximately 80 minutes of security guarantees regardless of block structure.

- **Q** (@Luivatra, 2024-01-03): What benefit do sub-blocks provide to dApps already using mempool transaction chaining?
  **A** (@Luivatra, self-answered): Sub-blocks improve transaction propagation and network load distribution. Wallets should offer unconfirmed balance display to resolve UX issues; sub-blocks may help reduce ghost transactions in the mempool.

- **Q** (@Flying Pig, 2024-01-18): Node and Nautilus derive different addresses from the same seed beyond the first address. Why?
  **A** (@kushti): Check for a mnemonic passphrase or an old derivation flag. Root cause was not confirmed in chat.

- **Q** (@Flying Pig, 2024-01-25): How do you deserialize Ergo registers to numbers in Python? What do `r4` and `r5` represent in a SigmaUSD bank box?
  **A** (@Flying Pig, self-resolved): Forked an existing serialization library to add register deserialization; `r4` and `r5` values were confirmed through iterative testing.

- **Q** (@Aberg, 2024-01-27): In Java (BouncyCastle), after multiplying an ECPoint by a BigInteger, how do you convert the result back to a BigInteger?
  **A**: Not resolved in chat; deferred to anon_br Σ for cryptography guidance.
