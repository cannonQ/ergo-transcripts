# Ergo General Chat — Monthly Summary: August 2020

---

## Executive Summary

August 2020 was a landmark month for Ergo, defined by three major milestones: the release of Oracle Pools on mainnet (developed in partnership with EMURGO), the first live privacy implementation via ErgoMixer with token support, and the announcement of the first Ergo network hard fork. These achievements collectively demonstrated that Ergo's extended UTXO model could support production-grade DeFi infrastructure, attracting significant attention from the Cardano community and broader crypto ecosystem.

---

## Key Topics This Month

### Oracle Pools Launch & Architecture

EMURGO and Ergo shipped Oracle Pools to mainnet by the end of August, with ADA/USD and ERG/USD pools live and a frontend integrated directly into the Ergo Explorer. Robert Kornacki released a comprehensive video breaking down the UTXO-based architecture, and a full Q&A transcript was published. The design offers several structural advantages over Chainlink: no custom token required (native ERG staking), epoch-based incentive frameworks, flexible governance (internal oracle votes or external DAO), Sybil resistance via coordinated pool formation, eUTXO hierarchical datapoint structures, and near-zero access costs making oracle data effectively a public good. Governance and stake slashing are designed into the protocol but were not included in the initial release, with future upgrades planned requiring no structural changes. Community members raised questions about whether Oracle Pool adoption benefits ERG directly; the answer was that dApps consuming oracle data are built on Ergo, driving network activity and fees, and large dApps can enforce tiny automatic payments to oracle pools from user interactions.

### ErgoMixer & ZeroJoin Privacy

ErgoMixer 2.0.0 shipped on August 8 with full token mixing support, improved performance, and platform-specific binaries. The implementation is based on the ZeroJoin protocol, which was accepted to the CBT 2020 workshop. A subsequent update introduced covert addresses for automatic mixing once a threshold is reached. kushti also demonstrated and tested a Zero-Knowledge Treasury using distributed threshold signatures — described as potentially the first public blockchain with ZK multi-sig capability — with a community member offering 2,000 ERG to test the system. Future mixer roadmap includes Tor integration and stealth address support.

### First Ergo Hard Fork Announcement

kushti announced the first Ergo network hard fork, planned for late 2020, containing three elements: a PoW memory-hardness fix to the Autolykos algorithm, non-critical security fixes, and support for updating the contractual layer via soft-fork in future. The fork was described as minimal, ideally the last hard fork required, and designed to be invisible to users not running full nodes — no snapshot, swap, or new tokens needed. The PoW discussion throughout the month included extensive debate with Reuben from the Zcoin team on ASIC/FPGA resistance: Equihash was ruled out (easily ASIC-optimised), MTP failed despite memory-hardness recommendations, and ProgPoW was identified as the most promising direction. RandomX was cautioned against unless a project is as committed to it as Monero.

### Node Release 3.3.0 / 3.3.1 & Technical Progress

Node 3.3.0 shipped on August 7 (+4,720 / -1,929 lines) with a massive wallet rework and external application support. Version 3.3.1 followed with bug fixes. kushti reported a 2.5x increase in transaction throughput over 14 months. The P2P layer rework and NIPoPoWs-based bootstrapping were confirmed in progress, with the goal of nodes syncing in minutes. A fee estimation API exists in the node but was noted as untested and not yet used by applications.

### Wallet Ecosystem & Exchange Situation

Yoroi wallet ERG integration was confirmed with a September–October target; by week 35, ERG appeared in Yoroi Nightly but was not yet syncing. Ledger hardware wallet support was in active development by Jason Davies, expected in autumn 2020. Community concerns centred on low liquidity across Hotbit, Tidex, ProBit, Waves, and CoinEx, with CoinEx experiencing withdrawal bugs and CitEx confirmed to have fake volume and a disabled ERG wallet. Tier-1 exchange listing fees were estimated at $300k–$500k. kushti confirmed "talks are going slooow" with Binance. Ergo Foundation acknowledged discussions with multiple exchanges but gave no commitments or timelines.

### Scalability, Block Speed & Sidechain Design

Extended discussion covered on-chain throughput improvements, sub-blocks/microblocks, and layer 2 options. kushti argued that Ergo is now in a strong position to consider these upgrades because a concrete set of production contracts exists (GARD protocols, Rosen, Oracle Pools, Spectrum, AMM pool contracts, Trade House, lending contracts) — unlike chains that pursued sharding or parallel chains before having real transaction profiles to optimise against. Merge-mined sidechains were described in detail: miners deliver sidechain data and perform trustless pegging via smart contracts (analogous to but improving on BIP 301), with the mainchain emission contract rewarding miners for sidechain data delivery. A testnet for parameter tweaking (including block time and difficulty) was noted as available, with kushti inviting community members to lead the effort.

### DeFi Philosophy & Ecosystem Positioning

Several blog posts and forum discussions articulated Ergo's DeFi vision: supporting only economically viable, non-custodial financial applications long-term; avoiding account-model pitfalls; and providing first-class privacy through native tools rather than bolt-ons. The EMURGO partnership and Robert Kornacki's appointment to the Ergo Foundation board were framed as a bridge to the Cardano/Goguen ecosystem. A Cointelegraph article (incorrectly calling Ergo a "Cardano-based project") and an IOG tweet acknowledgment drove significant new user influx, primarily from the Cardano community.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2020-08-03): Can Ergo contracts be formally verified? What makes them more secure?
  **A** (@Unknown): Non-Turing complete per transaction but Turing complete across multiple transactions, making formal verification tractable (crowdfunding contract already verified). The eUTXO model allows protocols to be parallelised into sub-protocols and to reference state from any contract trustlessly via data-inputs at near-zero cost.

- **Q** (@pepeblue, 2020-08-09): If Oracle Pools remove the need for a custom token like LINK, doesn't that limit ERG value accrual?
  **A** (@Unknown): Oracle Pools use native ERG (no custom token overhead). dApps consuming oracle data are built on Ergo, driving network fees. Large dApps can enforce tiny automatic ERG payments to oracle pools from user interactions, making key datapoints a public good while still generating on-chain activity. ERG is likely to be used as collateral in DeFi contracts as first-class citizen.

- **Q** (@Unknown, 2020-08-25): How will merge-mined sidechains work for miners?
  **A** (@kushti): Miners deliver sidechain data and perform trustless pegging via smart contracts, similar to but better than BIP 301. The emission contract rewards mainchain miners for delivering sidechain data. The sidechain should recognise mainchain blocks natively, with a trustless relay to update sidechain data. This is an Ergo-specific construction with clearer security than Bitcoin's sidechaining approach.

- **Q** (@Unknown, 2020-08-28): What are the advantages of Ergo's Oracle Pools over Chainlink's threshold signatures approach?
  **A** (@bobertk): On-chain oracle pools provide game-theoretic benefits like stake slashing that are not possible off-chain with threshold signatures. Ergo has no tx congestion and plans sidechains for scaling, so the on-chain approach captures more benefits while remaining cheap.

- **Q** (@Unknown, 2020-08-28): How are Oracle Pools funded and incentivised long-term?
  **A** (@bobertk): In the eUTXO model, large dApps can enforce that users pay a tiny percentage fee output automatically to Oracle Pools. Well-used dApps fund core oracle pools through usage with negligible per-user cost. More specialised data may require direct funding, but key datapoints like ERG/USD function as infrastructure available to the entire ecosystem.

- **Q** (@Unknown, 2020-08-19): Could ERG be listed on Uniswap?
  **A** (@Jason Davies): Uniswap compatibility would require wrapped ERG on Ethereum. Alternatively, Ergo's own DEX can wrap ETH on Ergo and compete with Uniswap directly.

- **Q** (@Jonesy, 2020-08-25): What will the first hard fork mean for holders?
  **A** (@kushti): Totally painless and invisible for those not running full nodes. No snapshot or swap needed — designed for soft-forkability so old nodes remain operational while allowing significant protocol changes.

- **Q** (@Unknown, 2020-08-17): Is token rebalancing (like Ampleforth) possible on Ergo?
  **A** (@Jason Davies): Yes, feasible via a pricing oracle in a rebase operation. Cannot directly change token balances in wallets, so would use an internal multiplier similar to AMPL's approach. Exchanges may have difficulty with open order semantics during rebasing.

- **Q** (@Unknown, 2020-08-19): What applications would complement Ergo for developer practice?
  **A** (@kushti): Mixer improvements, Zero-Knowledge Treasury, wallets, and privacy-preserving dApps. Stack flexibility includes Java/Scala for core contributions and Rust projects (sigma-rust, crowdfunding-cli, oracle pools, desktop wallet). Internship opportunities available for students; weekly dev chats on Discord Wednesdays 1PM UTC.

- **Q** (@Unknown, 2020-08-30): What spam/DDoS protections exist in Ergo?
  **A** (@kushti): Node v4.0.44 includes mempool limits — nodes will not download too many transactions from a single peer. Fee market dynamics mean spammers are eventually priced out as legitimate users raise fees. Node operators can raise minimum propagation fee rates to prevent spam from entering their mempool. A fee estimation API exists in the node but is untested and not yet used by applications.
