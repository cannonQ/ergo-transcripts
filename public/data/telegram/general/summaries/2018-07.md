# Ergo General — Monthly Summary: 2018-07

## Executive Summary

July 2018 was dominated by testnet infrastructure work, early community education about Ergo's design philosophy, and unresolved questions around the EFYT-to-ERG conversion mechanism. Discussions ranged from low-level node configuration and JVM tuning to high-level protocol design questions covering NIPoPoWs, Sigma protocols, PoW algorithm selection, and governance via miner voting — establishing the foundational technical narrative of the project ahead of the Q4 2018 mainnet target.

---

## Key Topics This Month

### Testnet Infrastructure & Node Setup
Early testnet participants encountered practical deployment challenges, particularly around running nodes in Docker environments with insufficient JVM heap memory. Developers confirmed that nodes could be run standalone (downloading the JAR and config directly) rather than exclusively via Docker. The testnet explorer at testnet.ergoplatform.com was made available for community testing, with bug reports welcomed. Memory requirements remained unclear for lower-specification machines.

### Mainnet Timeline & Roadmap
The team consistently pointed to Q4 2018 as the mainnet target throughout July, contingent on completing two further testnet cycles that would bring the environment close to production specifications. The Scorex 2.0 framework underpinning Ergo was expected to reach its final version by Autumn 2018, with Scorex 3.0 planned later with more incremental changes. A mobile wallet was announced as being in development. Four to five new team members were noted as forthcoming announcements.

### EFYT-to-ERG Conversion Mechanics
The EFYT swap mechanism was one of the most actively debated topics of the month. The confirmed structure was 1 EFYT → 1 ERG (Σ-Cash) + 1 ERGOdata (Σ-Data), with conversion funded by a 10% foundation mining tax as coins are mined post-launch. However, the precise timing, priority ordering, and fairness mechanism for redemption remained undefined throughout the month. Multiple distribution models were proposed by the community but none were adopted. A founders' meeting was scheduled to finalize conversion rate details, with an announcement expected "next Friday hopefully" as of early August.

### PoW Algorithm & Mining Decentralization
Ergo's initial PoW used Equihash-like parameters, but the team was actively evaluating alternatives — including proof-of-data-possession schemes and non-outsourcable mining designs — before the mainnet prototype. GPU mining was confirmed as supported. A mechanism to incentivise solo mining alongside pool operation was noted as under development to improve network decentralisation. The research thread on pool resistance was tracked via a GitHub issue.

### Smart Contract Architecture & Sigma Protocols
Ergo's smart contract approach via the Sigma-State language (based on zero-knowledge Sigma protocols) was explained to new community members throughout the month. The novel "tail recursion via transaction sequencing" pattern — where loops are induced through chained transactions rather than in-contract loop constructs — was identified as foundational. Ergo's positioning as "real Bitcoin 2.0" emphasised safe, practically deployable contracts, light client verification, and a soft-forking-only governance philosophy.

### NIPoPoWs, Sidechains & Scalability
NIPoPoWs were confirmed as built into Ergo to support light client verification and cross-chain value transfer between the Σ-Cash and Σ-Data chains. Full nodes store superblock data required for NIPoPoWs; light-full nodes most likely do not. The AVL+ tree stateless client solution was targeted for early 2019 with no hard fork required, enabling most clients and miners to operate without storing full state. Scalability was framed as extending beyond TPS to encompass user count and state size management.

### Governance via Miner Voting
A multi-layer on-chain governance model was described, where miners vote per 256-block epoch to adjust protocol parameters including block size, computational cost limits, and individual instruction costs. Storage rent and computation fees for contract execution were discussed as additional miner income streams. The emission curve was noted as final, while the economic model and voting system details remained in development.

## Technical Q&A Worth Preserving

- **Q** (@Komninos, 2018-07-02): Does a full node store which blocks are superblocks and references to older blocks for NIPoPoWs?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): Full nodes yes; light-full nodes most likely do not.

- **Q** (@Komninos, 2018-07-02): How do you configure JVM memory limits for Ergo nodes running in Docker?
  **A** (@Alexey Kiselev): Docker containers see all host memory by default. Modify the Dockerfile to add `-Xmx` to line 13 to constrain heap allocation.

- **Q** (@Komninos, 2018-07-02): What is the relationship between Ergo and Scorex?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): Ergo created Scorex; it remains actively developed and serves as the core framework for Ergo Platform.

- **Q** (@Crypto Bunny 🐜, 2018-07-09): Will Ergo be able to execute smart contracts?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): Certainly. Alexander (kushti) has a long history with smart contract protocol design.

- **Q** (@Unknown, 2018-07-11): Given Equihash ASICs and recent 51% attacks on ZenCash, what are Ergo's countermeasures?
  **A** (@Unknown): Equihash parameters will be custom per the original paper. Alternative PoW schemes based on proof-of-data-possession are being researched, and non-outsourcable mining designs are under study.

- **Q** (@Unknown, 2018-07-11): How will the EFYT-to-on-chain coin distribution work?
  **A** (@Unknown): 1 Σ-Cash + 1 Σ-Data per 1 EFYT, funded as coins are mined gradually via the 10% foundation tax on mainnet.

- **Q** (@Rejinderi, 2018-07-18): How does the Avalanche protocol compare to Ergo?
  **A** (@Dmitry): Avalanche is far from production and contains unproven claims; 1,300 TPS is unimpressive. Avalanche is only a network protocol, whereas Ergo is a complete blockchain system with smart contracts, light client verification, and an economic model. Implementations likely contain vulnerabilities.

- **Q** (@Mihai, 2018-08-01): Is the only way to configure a node via Docker?
  **A** (@hawky): No — download the config file, configure it, download the JAR, and run it directly as a standalone deployment.

- **Q** (@Unknown, 2018-08-03): Will stateless clients be supported?
  **A** (@Unknown): An AVL+ tree solution is planned for early 2019 with no hard fork required, enabling most clients and miners to operate without storing full state.

- **Q** (@Unknown, 2018-08-04): How does Ergo implement NIPoPoWs for sidechains?
  **A** (@Unknown): Ergo uses the Sigma-State language (zero-knowledge based) for smart contracts, and NIPoPoWs enable sidechains allowing value transfer between chains (e.g., Σ-Cash ↔ Σ-Data) via built-in sidechain support.

## Open Questions Entering Next Month

- **EFYT swap fairness and priority**: No mechanism finalised for determining who can redeem EFYT first, how to prevent launch chaos, or how to handle partial supply availability. Multiple community proposals discussed but none adopted.
- **PoW algorithm selection**: Custom Equihash parameters versus proof-of-data-possession or non-outsourcable designs still under evaluation; no decision made before the end of July.
- **Economic model and wallet design**: Both flagged as still in development; parameter voting system, foundation governance, and how the mining tax evolves post-swap all remain TBD.
- **Yellow paper release**: Draft available on GitHub but final version tied to the Q4 2018 mainnet launch window.
- **Solo mining incentive mechanism**: Pools confirmed for mainnet but the specific decentralisation incentive design is still being worked on.
- **Smart contract inter-call semantics**: Whether contract A can synchronously await a return value from contract B, and the atomicity implications, was raised but never formally answered during the month.
- **Node memory requirements**: A user with 3 GB RAM experienced crashes even after JVM tuning; root cause not fully diagnosed.
