# Ergo General — Monthly Summary: August 2018

---

## Executive Summary

August 2018 was a foundational month for the Ergo community, dominated by early-stage clarification of tokenomics (particularly the EFYT-to-ERG swap mechanics), technical architecture discussions around the UTXO model and ErgoScript's safety guarantees, and repeated Q&A about the Q4 2018 mainnet launch timeline. The period reflects a pre-mainnet project consolidating its design philosophy and onboarding its first wave of community members.

---

## Key Topics This Month

### EFYT-to-ERG Token Swap Mechanics

The mechanics of the EFYT (Ergo First Year Token) swap were among the most frequently revisited topics. The conversion rate was confirmed as 1 EFYT = 1 ERGOcash + 1 ERGOdata, though the "cash and data" subdivision was later acknowledged as nomenclature still under development pending the yellow paper. The swap can only occur after mainnet launch and new coin minting. Token holders on the Waves wallet sought clarification on whether tokens needed to be moved before the swap; this remained partially unresolved. The official EFYT asset on Waves was linked for verification.

### UTXO Model and Smart Contract Architecture

Dmitry provided substantive explanations of Ergo's preference for the UTXO model over account-based systems, emphasising support for multi-user transactions, mixing protocols, and transaction chaining without forward dependencies. It was noted that emulating an account-oriented system in UTXO is possible but carries privacy costs equivalent to the account model. ErgoScript's compiler-enforced safety guarantees were detailed: static type safety, zero-knowledge operation validation, termination guarantees via non-recursive design, execution cost bounds, and memory consumption limits — effectively providing "gas-free" contracts with predictable costs.

### Mainnet Timeline and Planned Releases

The Q4 2018 mainnet target was repeated across multiple weeks, with the team framing progress as conditional on testnet stability. Intermediate releases were planned covering wallet prototypes, fee model, system reconfiguration, and proof-of-work finalisation. A testnet explorer was deployed at `https://testnet.ergoplatform.com/` and a privacy-first wallet prototype was demonstrated. No precise date was locked, with the team noting that "each upgrade made to testnet brings us closer to the target."

### ASIC Resistance Strategy

Ergo's intent to pursue ASIC resistance was confirmed, with an explicit differentiation from Monero's approach: Ergo would not use hard forks to achieve this. The exact mechanism was not detailed in August discussions and remained an open item.

### Formal Verification Interest

A community member raised the prospect of formal verification tooling for ErgoScript, noting that the language's non-recursive design already narrows verification complexity. The team acknowledged the long-term value and referenced IOHK's work on refinement types for Plutus as a comparable effort, but committed to no near-term roadmap. ErgoScript's existing compile-time guarantees were positioned as a pragmatic substitute for the time being.

## Technical Q&A Worth Preserving

- **Q** (@LLB, 2018-08-15): Why does Ergo prefer the UTXO model?
  **A** (@Dmitry): It is more flexible — it supports transactions from many users to many users, which is required for various mixing protocols, and allows chaining transactions such that tx2 is only valid if tx1 was processed first.

- **Q** (@LLB, 2018-08-15): Can you emulate an account-oriented system using UTXO with a single address reused many times, and will the transaction model impose restrictions?
  **A** (@Dmitry): You can do it; however, from a privacy perspective it is as bad as the account model.

- **Q** (@Komninos, 2018-08-15): Is there any documentation about the Sigma language?
  **A** (@Dmitry): Not finalised yet, but current version (LaTeX sources) is available at https://github.com/ScorexFoundation/sigmastate-interpreter/tree/master/docs/wpaper

- **Q** (@Unknown, 2018-08-21): Will Ergo block ASIC miners as Monero is attempting to do?
  **A** (@Dmitry): Yes, but not using Monero's hard-fork approach; the methodology differs.

- **Q** (@Unknown, 2018-09-18): Can you explain what "1 EFYT = 1 ERGOcash + 1 ERGOdata" means?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): The Waves token will be swapped for ERG after mainnet launch. The cash/data subdivision is still being defined in the yellow paper.

- **Q** (@Unknown, 2018-09-18): How many tokens will be generated in total?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): Emission curve details are available at https://www.ergoplatform.org/news_emission_curve.html

- **Q** (@Unknown, 2018-09-18): Are there plans to incorporate formal verification tooling for smart contracts?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): ErgoScript already enforces (1) static type safety, (2) zero-knowledge operation validation, (3) termination guarantees via non-recursive design, (4) execution cost bounds, and (5) memory consumption bounds. Formal verification is a long-term roadmap interest; IOHK's refinement types work for Plutus was noted as a reference point.

- **Q** (@Kaichao | Keyring, 2018-08-19): Running Scorex with `sbt run` produces a null pointer exception — what did I miss?
  **A** (@𝖒𝖊𝖓𝖙𝖆𝖑𝖎𝖘𝖙կշօ): Is your config file correct?
