# Ergo General Chat — Monthly Summary: July 2022

---

## Executive Summary

July 2022 was a technically dense month dominated by three interlocking themes: wallet infrastructure maturity (Ledger stalls, UTXO token limits, multi-wallet seed compatibility), scaling and DeFi protocol development (Babel fees, Dexy stablecoin, Oracle Pool v2, Trade House order books), and node network health (v4.0.35 peer-drop regression, ergonodes.net launch, public REST API coverage). Bridge development — particularly Rosen Bridge — remained the most-watched work-in-progress, while storage rent, post-emission security, and block time trade-offs generated sustained philosophical debate throughout the month.

---

## Key Topics This Month

### Babel Fees (EIP-54)
Babel fees moved from proposal to working implementation during July, with Nautilus shipping a full testnet-ready version by week four. The mechanism allows users to pay transaction fees in arbitrary tokens via order book–style "Babel boxes" that atomically swap tokens for ERG; liquidity providers set rates and profit from convenience premiums. Technical discussions clarified that each box still requires minimum ERG (the swap happens under the hood), that Babel boxes have their own liquidity mechanism independent of Spectrum DEX, and that two-sided order book extensions are architecturally possible. The implementation was identified alongside NIPoPoWs as a major UX unlock for mass adoption. An open question remains around whether Babel box liquidity will be sufficient at launch without bootstrapping incentives.

### Dexy Stablecoin & Oracle Pool v2
kushti introduced and elaborated on Dexy (v2), a market-driven stablecoin using emission contracts and burn-as-last-resort stabilization. The first live instance will be pegged to gold price using Oracle Pool v2, with GORT rewards replacing UTXO-dust payouts from v1. Oracle Pool v2 economics were discussed in detail: rewards are paid via dedicated tokens, eliminating the dust accumulation problem. A community DeFi/Dexy discussion was held on 2022-07-26. Separately, kushti published a forum post on decentralized grid trading contracts — sell orders that can only be unlocked by corresponding buy orders at lower prices, enabling on-chain arbitrage strategies executable by off-chain bots.

### Rosen Bridge & Cross-Chain Infrastructure
Rosen Bridge testnet work progressed but encountered delays: 5–10 second block times on the test environment were unsuitable for debugging, requiring a switch to a local mainnet simulation. A Sigma Rust multisig issue was identified and partially fixed. The overall testnet ETA remained "a few weeks" throughout the month without firm confirmation. The SUSY Bridge was effectively declared dead or severely stalled — Armeanio confirmed hiring a new developer to complete the remaining work. No mainnet bridge launch date was announced for any cross-chain solution.

### Node Infrastructure & Network Health
The ergonodes.net dashboard launched, revealing 1,070+ total nodes with ~600 publicly reachable, adding ~11 nodes per day. This prompted detailed discussion of port-forwarding best practices (9030 for P2P, 9053 for REST API), the distinction between outbound-only and inbound-accepting nodes, and the importance of inbound connections for true decentralization. Node v4.0.35 introduced a regression causing peer counts to drop significantly (from 30+ to 1–20 range); reverting to v4.0.34 resolved the issue for most operators. Separately, only ~32 nodes were found to be publicly reachable via REST API — a concern for light wallet clients aiming to deprecate centralized explorers.

### UTXO Token Limit Bug
A critical bug was discovered early in the month: wallets holding more than 255 different tokens in a single UTXO box would cause transaction-building failures. The issue was traced to transaction-building tools (notably Nautilus), not the protocol itself. Nautilus added a safeguard limiting outputs to 100 tokens per box; Ergo Mobile Wallet was confirmed safe (also capped at 100 tokens per box). Sigma Rust required patching for related accidental-burn scenarios. The fix timeline across all affected tools remained partially unresolved at month end, with particular urgency flagged for projects like EXLE that issue large numbers of NFTs.

### Hardware Wallet Support (Ledger & Trezor)
Ledger integration stalled with the integration team unresponsive; an unofficial sideload version (with a 10-token transaction limit due to Nano S memory constraints) remained the only option, accessible via GitHub. Trezor was confirmed as not accepting new coin integrations and requiring top-100 market cap status — plus a complex reimplementation of Ergo's full signing model on the device. No ETA was provided for either hardware wallet's official support.

### Trade House, Order Books & DEX Landscape
qx() gave an extended explanation of Trade House's significance as an order book DEX contrasted with AMM-based Spectrum DEX: order books provide full depth/liquidity transparency, precise slippage control, and peer-to-peer matching without impermanent loss risk. Partial order fills are supported through box duplication (leftover change returned with identical parameters). The discussion touched on historical precedents (MuesliSwap on Cardano, Serum on Solana and its FTX-related collapse), with the key architectural advantage of Ergo's model being that orders live on-chain and no single upgrade key controls the DEX.

### Storage Rent, Post-Emission Security & Token Burning
Extended debates across multiple weeks addressed whether storage rent will adequately incentivize miners after ERG emission ends. The community acknowledged the problem is real but unknowable without future price/cost/hashrate data. A related proposal suggested dApps voluntarily burn a fraction of tokens as ecosystem norm to support long-term storage rent reserves — debated but without foundation guidance or formal proposal. The philosophical distinction between storage rent (recirculates parked coins back into the economy) and Bitcoin's permanently locked coins (e.g., Satoshi's holdings) was repeatedly highlighted as an Ergo advantage.

### Palmyra / Chainlink Partnership
Armeanio and Dan (ZenGate/Palmyra) detailed plans for on-chain warehouse receipts enabling decentralized trade financing: producers deposit commodities into bonded warehouses, receive on-chain receipts issued via Chainlink oracles coupled with IoT devices (cameras, weight/motion sensors), and use those receipts as collateral. Chainlink was chosen for network size and support breadth, not exclusivity — the plan includes building out native Ergo oracles using capital raised through a token sale, with Oracle Pool v1 and v2 as the native targets. The discussion also prompted a broader community note about tribalism: companies may adopt partial Ergo infrastructure, and that is still a net positive for the ecosystem.

### Block Time, Scaling & Sub-Blocks
kushti raised the possibility of revisiting micro-blocks, sub-blocks, sidechains, and faster confirmation protocols as concrete scaling directions now that Ergo has a well-defined set of deployed contracts to benchmark against. Key concern flagged: timestamp-based dApp logic would be affected by faster blocks differently than block-height-based logic. A community testnet group was proposed for parameter experimentation (no code required — tweaking difficulty and block parameters). No decision was made; the initiative needs a community champion to lead it. BIP 301–style merge-mined sidechain constructions were confirmed as under active development, with trustless pegging via smart contracts proposed as the mechanism.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2022-07-06): What is the easiest way to run an Ergo node?
  **A** (@Gazza): Satergo wallet allows a full node to be run in a few clicks, with wallet access included.

- **Q** (@Unknown, 2022-07-06): What is the benefit of running a public HTTPS node vs. a private one?
  **A** (@MrStahlfelge): Public nodes can serve light wallets that need access to blockchain state. The long-term goal is to deprecate centralised explorers in favour of decentralised node infrastructure.

- **Q** (@Unknown, 2022-07-06): How do light wallets discover public nodes?
  **A** (@MrStahlfelge): Currently they cannot. A PR is open to allow nodes to advertise their URLs in settings so light wallets can build a discovery list.

- **Q** (@Unknown, 2022-07-10): Is the >255 token issue a chain bug or a token design problem?
  **A** (@Luivatra): It is a bug in transaction-building tools (e.g., Nautilus), not in the chain or token design itself.

- **Q** (@K Chan, 2022-07-14): Why does a simple p2p transaction from Ergo Mobile Wallet show 57 inputs and 3 outputs?
  **A** (@Luivatra): Each block payout or receipt creates a new UTXO. To send a larger amount the wallet consolidates multiple UTXOs as inputs. 57 mining payouts = 57 UTXOs; consolidating them produces one recipient UTXO, one change UTXO, and one fee output.

- **Q** (@Unknown, 2022-07-14): Do dApps automatically benefit when they become popular, or must they monetise explicitly?
  **A** (multiple): No built-in protocol reward exists. dApps must implement their own monetisation — fees, tokens, grants, or sponsorship. Transaction fees route only to miners.

- **Q** (@Ilya, 2022-07-19): What are the pros and cons of sending all boxes as transaction inputs by default?
  **A** (@Aberg, @Klaus): **Cons**: larger transactions increase blockchain size; privacy leak revealing total wallet balance; complicates FIFO tax accounting per address. **Pro (future)**: reduces unspent box bloat when NIPoPoWs are deployed. Compromise: consolidate all boxes from a single address only.

- **Q** (@Unknown, 2022-07-26): How do Babel fees determine exchange rates?
  **A** (@Capt. Nemo): Liquidity providers set the price at the moment of Babel box creation, analogous to a token buy order book. The market establishes equilibrium. Babel fees have their own liquidity mechanism, separate from Spectrum DEX.

- **Q** (@kii, 2022-07-26): Is there still an ERG cost for Babel fees despite paying in tokens?
  **A** (@Capt. Nemo): Yes — the token-to-ERG swap happens under the hood. Each box still requires minimum ERG; the Babel box mechanism handles the exchange atomically.

- **Q** (@Ilya, 2022-07-26): What incentives exist for Babel box liquidity providers?
  **A** (@Capt. Nemo, @kushti): Babel boxes sell convenience. LPs can set prices above market rate and profit from the premium. When the market is thin, a meaningful premium is needed to justify providing liquidity.

- **Q** (@Chris Ray, 2022-07-26): How difficult is it to build collateralised lending (Compound/Aave-style) in eUTXO?
  **A** (@kushti): Probably possible; other designs may be more optimal or interesting for the UTXO model. No detailed implementation exists yet.

- **Q** (@Unknown, 2022-07-26): Can Ergo smart contracts execute simultaneously?
  **A** (@Gazza): Yes, but only one transaction spending a given box can execute per block; others queue or fail and must resubmit.

- **Q** (@Unknown, 2022-07-26): What is the security risk of receiving unexpected tokens in an Ergo wallet?
  **A** (@qx(), @Ilya): Ergo native tokens carry no executable code — there is no smart contract risk from receipt. Preventing receipt would require the receiver to spend fees to return or burn them. From a technical standpoint, the wallet is simply a UI onto blockchain state.

- **Q** (@DrTwentyOne, 2022-07-30): Can Satergo be trusted for offline seed phrase generation?
  **A** (@Aberg): Generation uses system randomness and is impossible to predict. A 12-word seed has 2^128 (340 undecillion) possible combinations. Satergo is open source, portable, and runnable from source with a single command.

- **Q** (@Unknown, 2022-07-26): How do bulletproofs relate to zero-knowledge proofs on Ergo?
  **A** (@kushti): Bulletproofs are a technique for zero-knowledge proofs — particularly range proofs — that require no trusted setup (unlike SNARKs). They can be used for generic statements and are a specific form of Sigma protocol. If bulletproof verification can be done inside ErgoScript, it opens the door to a wide variety of ZK protocols (verifiable credentials, Groth one-out-of-many proofs) being verified on-chain without revealing sensitive data.

- **Q** (@Unknown, 2022-07-29): What causes node v4.0.35 peer connections to fluctuate or drop?
  **A** (community): Root cause unclear; multiple operators report dropping from 30+ peers to 1–20. Reverting to v4.0.34 resolves the issue. WiFi connections separately cause nodes to appear synced while stopped — wired connections recommended.

## Open Questions Entering Next Month

1. **Rosen Bridge testnet ETA**: Still "a few weeks away" at month end; Sigma Rust multisig issue partially resolved but no confirmed mainnet date. SUSY Bridge effectively dead pending new developer.
2. **Node v4.0.35 regression**: Root cause of peer-count drops not publicly identified; fix not confirmed shipped.
3. **UTXO token limit fix completeness**: Nautilus and Mobile Wallet patched; Sigma Rust patch status unclear; impact on high-NFT-count projects like EXLE unresolved.
4. **Ledger official integration**: Team unresponsive; no ETA beyond "in progress." Sideload-only option persists.
5. **Block time / sub-block scaling**: Community interest established but no champion identified to run testnet experiments. Timestamp-vs-block-height logic impact on existing dApps not fully mapped.
6. **Storage rent post-emission adequacy**: Philosophical consensus that the problem is real and solvable but no protocol-level proposal formalised; token-burning-as-norm debate unresolved.
7. **Public node discovery for light wallets**: PR open to allow URL advertising in node settings, but no decentralised discovery mechanism designed yet; only ~32 nodes publicly reachable via REST API.
8. **Babel fee liquidity bootstrapping**: Who provides initial Babel box liquidity and at what rates? No bootstrapping plan announced.
9. **dApp monetisation protocol support**: Whether Ergo should add protocol-level mechanisms to reward dApp creators remains an open philosophical question.
10. **Faster block time impact on emission contracts**: If block times decrease, emission contracts (tail end of curve) and timestamp-dependent dApps would be affected; analysis not completed.
