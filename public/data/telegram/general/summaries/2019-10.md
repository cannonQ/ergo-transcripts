# Ergo General Chat — Monthly Summary: 2019-10

---

## Executive Summary

October 2019 was dominated by Ergo's early exchange listing activity, with Hotbit and ProBit integrations accompanied by significant third-party infrastructure friction (withdrawal issues, incorrect CMC data, and low liquidity). Alongside these ecosystem-building challenges, the community made meaningful progress on governance — establishing a formal Development Fund Board with multisig control — and continued active discussion of oracle architecture, wallet roadmaps, and mining infrastructure.

---

## Key Topics This Month

### Exchange Listings & Market Infrastructure

Hotbit launched ERG/BTC trading on 2019-10-09, followed by ProBit announcing a listing for 2019-10-29. However, both integrations were accompanied by immediate friction: Hotbit users reported inactive withdrawal buttons, account locks, and incorrect ERG amounts sent during withdrawals. P2PB2B users experienced widespread withdrawal delays. CoinMarketCap displayed an incorrect ticker (ERGO instead of ERG), broken links, and wrong supply figures, with corrections submitted but circulating supply discrepancies persisting into late October. CoinGecko was recommended as a superior alternative. These issues were attributed to third-party platform problems rather than protocol-level concerns.

### Oracle Design & Architecture

Early discussions of oracle architecture centred on an 8-trusted-party design intended to reduce collusion risk and eliminate single-point coding failures. A key unresolved question was whether oracle results should be aggregated by a single contract or whether distributed per-contract logic was preferable. The oracle pool announcement was made via Twitter in W42. The question of update frequency (how often oracle prices refresh) was raised but not answered in the available record.

### Wallet Ecosystem & Ledger Support

Community members sought guidance on storing ERG securely. Magnum Wallet was recommended as the near-term option, with the caveat that it does not support script-backed outputs or mined coin balances. Waves DEX linked to Ledger was identified as the safest current option. The team confirmed that a simplified wallet UI, a light wallet using NIPoPoWs, and Ledger hardware wallet support were all in development, though no ETAs were provided.

### Mining Infrastructure

Mining setup questions were frequent across the month, with users directed to the `ergo_mining` Telegram channel and AMD/Windows blog posts. The mining pool contract was estimated at 6–7 weeks out (referenced from BitcoinTalk). A 720-block maturity delay for mined coins was clarified. NVIDIA hashrate display issues were reported but unresolved. The Autolykos mining algorithm was noted as requiring no programming knowledge to participate in.

### Development Fund Governance

A formal Development Fund Board was established with 6 members (kushti, morphic, mobybit, Martin-mx, Justin, Mentalist420), operating on a 2-of-3 multisig control structure with a 4-of-6 voting threshold for spending approvals. The fund holds approximately 740K EFYTs (equivalent to ~2.9M ERG post-swap), with monthly spend under 20K EFYT. An outstanding Bitcoin debt from pre-mainnet conference expenses was acknowledged and justified. Decred's treasury model (percentage of block rewards with community voting) was cited as a governance precedent, with the note that Ergo miners retain ultimate consensus power over protocol changes.

### Node Infrastructure & Sync

Users running nodes on limited hardware (Raspberry Pi 3, Core2Duo) sought guidance on sync performance. Pre-synced blockchain data transfer was recommended. A reported issue of two nodes stalling at block height 87,080 was suspected to relate to Chinese network partitioning but was not conclusively resolved. The EFYT-to-ERG swap on Waves DEX was confirmed as a direct 1:1 process.

### Community Channels & Communication

Debate arose around whether to expand beyond Telegram to Discord, Reddit, or other platforms. Consensus settled on Telegram + ergoforum.org + GitHub as sufficient for the current community size, with a suggestion to strengthen ergoforum.org as the primary venue for development discussion. WeChat and Weibo accounts were shared for Chinese community engagement.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2019-10-W40): How does the 8-oracle architecture work — does a single aggregating contract read all 8 oracles, or does each contract implement that logic itself?
  **A** (implicit from context): The 8-trusted-party model reduces both collusion likelihood and single-point coding error failure modes. The aggregating vs. distributed architecture pattern was raised but remained unresolved in this period.

- **Q** (@Unknown, 2019-10-W40): What is the derivation path when importing a wallet seed into Magnum Wallet?
  **A** (@Ilya Oskin): The derivation path is returned from the node API when deriving a new key. It is only needed if you require keys other than the root key. Reference: [Ergo OpenAPI spec, line 1841](https://github.com/ergoplatform/ergo/blob/898aa239c3da080760f6a9a3c4547711179bd862/src/main/resources/api/openapi.yaml#L1841).

- **Q** (@Al, 2019-10-W44): How do I re-sync a wallet?
  **A** (@Dmitry Usov): Delete the `.ergo/wallet` directory in the node startup directory.

- **Q** (@god, 2019-10-W44): Can Magnum Wallet import mnemonics from the Ergo Core wallet?
  **A** (@Ilya Oskin): Yes — but with an important caveat: Magnum does not support scripts, so it is incompatible with mined coins or script-backed outputs.

- **Q** (@god, 2019-10-W44): Mined ERG is visible in the explorer but not appearing in the local wallet query, even with the node always running. Why?
  **A** (@Dmitry): A 720-block maturity delay applies before mined coins are reflected in the wallet balance.

- **Q** (@god, 2019-10-W44): Receiving HTTP 403 "authentication not authorized" when querying wallet balance via API.
  **A** (@Unknown): Refer to the Ergo REST API documentation at the GitHub wiki.

- **Q** (@Unknown, 2019-10-W40): How does Ergo's mempool handle spam transactions? Can node operators filter them?
  **A** (@kushti, context): Node operators can raise the minimum propagation fee rate to prevent spammer transactions from entering the mempool at all. From node version 4.0.44, limits exist on how many transactions a node will download from a single peer. Fee estimation is available via a node API endpoint, though it was noted as untested by applications at the time. Spammer transactions that are not picked up will eventually time out and be flushed from the mempool.

## Open Questions Entering Next Month

- **Oracle aggregation architecture**: Whether to use a single aggregating contract or per-contract distributed logic for the 8-oracle system remains unresolved.
- **Difficulty adjustment threshold**: The exact mechanism by which a 50%+ block majority within a 1024-block epoch triggers a 1% block size limit increase was raised but not definitively answered.
- **Hotbit withdrawal reliability**: Root cause of inactive withdrawal buttons and account lock issues was not explained; no resolution timeline provided by the exchange.
- **Ledger hardware wallet support ETA**: Confirmed as in development but with no release date.
- **Mining pool contract**: Estimated 6–7 weeks out as of late October/early November — delivery and functionality to be tracked.
- **CMC circulating supply**: Treasury contract unlock schedule noted but CMC correction timeline unconfirmed.
- **Node sync stall at block 87,080**: Suspected Chinese network partition; not conclusively diagnosed or resolved.
- **Paper wallet / offline key generation**: No native browser-based Ergo tool confirmed; only a YouTube tutorial provided.
