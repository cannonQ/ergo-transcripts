# Ergo General Chat — Monthly Summary
**Month:** March 2019
**Chat:** Ergo General

---

## Executive Summary

March 2019 was a pivotal pre-launch month for Ergo, dominated by mainnet launch preparation, GPU mining readiness, and technical onboarding for early node operators and developers. The community received the landmark announcement of a May 26, 2019 mainnet target date (contingent on security audit), alongside the release of Ergo v2.0.0, TestNet 2.0, and the ErgoScript whitepaper draft.

---

## Key Topics This Month

### Mainnet Launch Timeline & Roadmap
TestNet 2.0 launched on March 14th with all planned mainnet features. The mainnet launch was officially targeted for May 26, 2019, contingent on a successful security audit, wallet integration, and wide community testing. The two-month window between TestNet 2.0 and mainnet was explicitly reserved for hardening, documentation, and marketing preparation.

### GPU & CPU Mining
CPU mining was available immediately via full node configuration (`mining = true`). A GPU miner was confirmed in development throughout the month — initially described as "probably this or next week" in W12, then entering testing in W13, where testnet instability caused by high block frequency and a discovered bug delayed the tutorial release. Contributors familiar with C and CUDA were actively recruited. ASIC resistance was confirmed as a core design priority; mining pool resistance was acknowledged as a more complex, ongoing challenge addressed through algorithm configuration and chain evolution adjustments.

### EFYT Token Swap & Emission
The EFYT token swap was confirmed as a 1:1 conversion to ERG at mainnet launch, funded from the foundation tax allocation rather than mining rewards. The maximum circulating supply was approximately 19 million ERG in the first year. Community members were advised that holding EFYT on Waves created counterparty risk and that direct ERG holding post-mainnet was preferable.

### ErgoScript & Smart Contract Capabilities
The ErgoScript whitepaper draft (v2.0) was released for community review. ErgoScript was clarified as both non-Turing-complete in single-transaction context and effectively Turing-complete when multiple transactions are considered — a formal academic paper was linked. Working contract examples available covered oracles, crowdfunding, atomic swaps, and mixing, with more complex examples (cold wallets, LETS system, ICO scenarios) in progress. Sponsored transactions (fees paid in custom tokens) were confirmed implementable via an exchange contract pattern.

### Node Configuration & Wallet Setup
Early node operators encountered common configuration issues: API key hashing (Blake2b, case-sensitive), Swagger endpoint access (`ip:9052/swagger`), and seed phrase placement in plain-text config. The initial wallet implementation used in-memory storage only; a more complete wallet with secure seed storage was promised before mainnet. External multicurrency wallet integrations were also planned.

### ErgoData & Sidechains
ErgoData, originally conceived as a separate blockchain, was reconceived as a future sidechain rather than an independent chain, postponed approximately 2–3 years post-mainnet pending sidechain infrastructure maturity and demand. The protocol was described as designed for flexible community-driven sidechain implementations, with community members already working on early-stage efforts. Lightning-style off-chain solutions were also noted as supportable.

### Light Nodes & UTXO Architecture
Light node modes using ADProofs and NIPoPoWs were discussed, with the full UTXO set required for mining. Stateless client research was noted as a future path to light mining nodes. Storage bloat is addressed via persistent updateable storage where only a digest plus ~40 bytes is stored in the UTXO set, with storage rent eliminating dust.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2019-03-07): Could we build dApps on Ergo?
  **A** (@Dmitry): "For sure. I hope we'll have working examples quite soon."

- **Q** (@Amir Boziev, W12): Is there an option to generate ERC-721-style tokens that cannot be split?
  **A** (@Dmitry): One token per transaction can be issued for free with a max supply of 1, achieving a non-fungible effect. No standard class system exists yet, but one is expected to emerge.

- **Q** (@Amir Boziev, W12): Is there an option to broadcast a batch of transactions and have them included in the same block?
  **A** (@Dmitry): No dedicated batch broadcast API exists, but sequential broadcasting with no delays achieves the same practical effect. Single-block inclusion for a batch is possible via contract logic but requires custom miner mempool collection rules.

- **Q** (@Amir Boziev, W12): Can a transaction output contain 0 ERG?
  **A** (@Dmitry): No. Every transaction output must contain some minimum amount of ERG, though the transaction fee itself may be 0.

- **Q** (@Amir Boziev, W12): Can sponsored transactions be implemented — where fees are paid in a custom token rather than ERG?
  **A** (@Dmitry): Yes, implementable using an exchange contract pattern: create a contract that buys the custom token for ERG; the sender sells the required token amount and transfers the remainder in a single transaction.

- **Q** (@Amir Boziev, W12): Will there be support for Lightning Network-style off-chain solutions and community-driven sidechains?
  **A** (@Dmitry): The goal is to provide a flexible protocol for sidechain implementation. Community members are already working on early-stage implementations, and all such activity is supported.

- **Q** (@Unknown, W12): What are the minimum hardware requirements to run a node?
  **A** (@Dmitry): Recommended configuration is 4 GB RAM, 2 CPU cores, 80 GB SSD. Exact minimum not precisely measured and depends on node regime.

- **Q** (@Unknown, W13): What happens to EFYT at mainnet launch?
  **A** (@Unknown, W13): EFYT converts 1:1 to ERG, funded from the foundation tax allocation — not from mining rewards.

- **Q** (@Unknown, W13): Is ErgoScript Turing-complete or restricted for safety?
  **A** (@Dmitry): Both. ErgoScript was designed to be non-Turing-complete, but formal analysis proved it is effectively Turing-complete when multiple transactions are considered. A formal paper and the ErgoScript manual are available.

- **Q** (@Unknown, W13): API authorization returns a 403 error despite correct Blake2b hash configuration — why?
  **A** (@Dmitry): `apiKeyHash` is case-sensitive. Also confirm the "Authorize" button is clicked after entering the secret in the Swagger authentication window.

- **Q** (@Unknown, W12): How do I get the `apiKeyHash` value?
  **A** (@Unknown): Use the Swagger endpoint `/swagger#/utils/hashBlake2b` to generate the Blake2b hash of your chosen API key.

## Open Questions Entering Next Month

- **GPU miner release and tutorial**: Miner was in testing at month's end with a bug identified; tutorial and stable release still pending entering April.
- **AMD GPU support**: A community member raised concerns that the mining protocol appeared to favour NVIDIA over AMD hardware; no technical response was documented.
- **Token class standard (ERC-721 equivalent)**: No standard for grouping non-fungible tokens exists yet; flagged as future community work.
- **Mining pool resistance**: Acknowledged as an unsolved and complex problem; ongoing algorithm configuration and chain evolution adjustments planned but no concrete protocol defined.
- **Wallet security and seed storage**: Secure wallet setup (replacing plain-text seed in config) confirmed as in-progress but not yet delivered.
- **Testnet instability root cause**: One bug identified during GPU miner testing; full resolution not confirmed by month's end.
- **ErgoData sidechain architecture**: Reconceived as a future sidechain but specific technical architecture, consensus mechanism, and merge-mining details remain TBD.
- **Website and documentation currency**: Outdated material flagged on the official website; updates deprioritised below mainnet launch work.
