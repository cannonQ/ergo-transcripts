# Ergo General — Monthly Developer Knowledge Base Summary
**Chat:** Ergo General
**Month:** 2019-06

---

## Executive Summary

June 2019 was dominated by pre-mainnet preparation, with the Ergo mainnet launch date shifting from June 20 → June 28 → July 1, 2019 (10:00 AM UTC) as security audits, AMD miner development, and Waves gateway integration remained blocking items. Community activity centred almost entirely on Autolykos GPU mining setup, node/wallet configuration troubleshooting, and clarifying the EFYT-to-ERG token swap mechanics, with substantive protocol discussions covering storage rent, emission curves, ErgoScript capabilities, and the first designs for atomic exchange and privacy features.

---

## Key Topics This Month

### Mainnet Launch Timeline
The mainnet launch date slipped three times across the month. The initial June 20 target was pushed to June 28 pending security audit completion and Waves DEX gateway delivery, then pushed again to **July 1, 2019 at 10:00 AM UTC** for additional core testing. The AMD miner, Waves gateway integration, and a clean testnet environment were consistently cited as the critical-path blockers. A minor breaking change (stricter box/script size limits and increased instruction costs) was introduced in the v3.0.0 release on June 28 for security hardening, though testing could continue with the updated JAR on existing configs.

### Autolykos GPU Mining Setup
Autolykos was the dominant technical topic throughout the month. The algorithm requires ≥3 GB VRAM (8 GB recommended for `keepPrehash=true` mode, which needs ~7.85 GB). NVIDIA CUDA miners were live on testnet; AMD support was promised "shortly after mainnet" from an external developer team. GTX 1060/1070/1080/1080 Ti and RTX 2080 Ti were confirmed suitable; typical hashrates ranged 12–20 MH/s. Recurring issues included CUDA architecture code selection (`sm_60` for GTX 1080 Ti), the `keepPrehash` flag for sub-8 GB cards, OpenSSL build dependencies on Windows, and the mnemonic/public-key-hex workflow required before mining could start.

### Node Configuration & Wallet Initialization
Consistent friction around node setup was present all month. Key pain points: generating the Blake2b-256 API key hash, the relationship between `mnemonicPass` (empty string if no password set) and the miner's pubkeyhex, 403 errors from missing API key headers, and the 720-block maturation delay before mined rewards become spendable (~2 days at 2-minute block times). The address format discrepancy between 52-character wallet addresses (`3W...`) and longer explorer-format addresses caused repeated confusion. Java 64-bit was required; 32-bit JRE caused heap allocation failures on Windows.

### EFYT → ERG Token Swap & Emission Model
EFYT holders on the Waves blockchain swap 1:1 to native ERG post-mainnet via a Waves DEX gateway; the treasury purchases and burns all EFYT received. The treasury holds 4,330,791.5 ERG, releasing 7.5 ERG/block for 2 years then declining amounts over 3-month periods, funding ~2.5 years of development. The ErgoData token was discussed as a planned 2:1 airdrop to EFYT holders, potentially as a sidechain 2–3 years post-mainnet. Exact swap queue mechanics (FIFO vs. market-based) were not fully confirmed by month's end.

### Storage Rent & Economic Design
Storage rent was explained as a mechanism that deducts small fees from dormant UTXO boxes to prevent state bloat, return lost coins to circulation, and provide post-emission miner rewards. Miners can adjust storage fee parameters post-launch; the development team retains no direct governance authority over these parameters after mainnet. The dormancy threshold before fees apply was referenced as approximately 4 years, but exact initial parameters were not confirmed in chat.

### ErgoScript, Atomic Exchange & Privacy
A foundational protocol discussion early in the month covered ErgoScript capabilities: the core protocol natively supports atomic exchange, and off-chain protocols allow bundling exchange with on-chain posts in a single transaction — described as a "DEX of DEXes" design. Non-interactive tumblers and mixing using protocol-level mechanisms (no third-party custodians) were highlighted as planned privacy features. Complex ErgoScript contract deployment via user-friendly tooling was explicitly deferred to post-mainnet.

## Technical Q&A Worth Preserving

- **Q** (@ponte, 2019-06-03): Getting `ErgoTreeSerializer.deserialize` error when submitting a transaction with R4 register data.
  **A** (@Ilya Oskin / @Dmitry): Register data must be serialized using Sigma serialization format; no user-friendly tooling exists yet. Workaround: remove the `registers` field from the JSON payload entirely, and the transaction will process.

- **Q** (@Overcooked Panda, 2019-06-10): Wallet initialization returns a 403 error when calling `/wallet/init`.
  **A** (@Ilya Oskin): Ensure `apiKeyHash` is set in `ergo.conf` (Blake2b-256 hash of your secret) and the corresponding `api_key` header is present in each request via Swagger.

- **Q** (@Rejinderi, 2019-06-12): GTX 1070 throws CUDA error code 2 (out of memory).
  **A** (Community): Set `keepPrehash = false` in miner config for GPUs with less than 8 GB VRAM; simultaneous system use on 6 GB cards is impractical as `keepPrehash = true` requires ~7.85 GB.

- **Q** (@Unknown, 2019-06-12): "Generated and received public keys don't match" error in Autolykos GPU miner.
  **A** (Community): Ensure the `testMnemonic` field in node config is commented in with the exact mnemonic used during wallet initialization. If a `mnemonicPass` was set during wallet init, add `"mnemonicpass": "yourpass"` to the miner config; use an empty string `""` if no password was set.

- **Q** (@Rumato Estorsky, 2019-06-07): `test.out` GPU benchmark reports 19 MH/s across 3 GPUs — same as single GPU.
  **A** (@Dmitry): `test.out` prints per-GPU hashrate separately; total throughput is the sum. 19 MH/s is a correct single-GPU figure for a GTX 1070 8 GB without overclocking at `keepPrehash = false`.

- **Q** (@Rimov R, 2019-06-20): Build error: `openssl/bn.h: No such file or directory`.
  **A** (Community): Use OpenSSL v1.0.2s full installer (23 MB) rather than source-only packages; verify that `C:\OpenSSL\include\openssl\bn.h` exists after installation.

- **Q** (@Ilya Oskin, 2019-06-19): Mined blocks but wallet balance remains zero.
  **A** (Community): Miner rewards are locked for 720 blocks (~2 days); coins are held in a special P2S address derived from the mining script, not the user's deterministic wallet address.

- **Q** (@UpwindStrategy, 2019-06-20): Does storage rent charge annual fees from all wallets?
  **A** (Community): Storage rent deducts small fees from dormant UTXO boxes (not active wallets) to prevent state bloat and reward miners post-emission. Miners adjust parameters post-launch; developers only propose changes.

- **Q** (@Unknown, 2019-06-29): What happens when all coins are mined in ~8 years?
  **A** (Community): Storage rent continues providing miner incentives after emission ends. Fee parameters are changeable by miners; Ergo launches with minimum parameters and the team has no direct influence post-mainnet.

- **Q** (@CryptoTeller, 2019-06-29): Java heap error on Windows: "could not reserve enough space for 3145728 KB object heap."
  **A** (Community): 32-bit JRE installed. Must use 64-bit Java; the node JVM requires addressable memory beyond 4 GB.

- **Q** (@Sykoboy, 2019-06-29): How to reduce excessive node log output?
  **A** (@Alex Slesarenko): Launch with `java -jar -Xmx4G -Dlogback.stdout.level=WARN ergo.jar --testnet -c ergo.conf` (valid levels: TRACE | DEBUG | INFO | WARN | ERROR).

- **Q** (@Dmitry V, 2019-06-25): Where is the rigorous VM and block format specification?
  **A** (Community): Only examples and computational model documentation are currently available; a full formal specification for independent reimplementation does not yet exist.

## Open Questions Entering Next Month

- **AMD Miner Delivery**: External developer team promised AMD miner by ~June 21, then "by mainnet." No confirmed delivery before July 1 mainnet launch; status unknown.
- **EFYT Swap Mechanics**: Exact queue/ordering mechanism (FIFO vs. market-based) for EFYT-to-ERG conversion during Year 1 not confirmed; "@Unknown: exact details are unknown at this moment."
- **Address Format Discrepancy**: 52-character wallet addresses (`3W...`) vs. longer explorer-format addresses raised repeatedly without clear resolution or documentation.
- **Complex ErgoScript Tooling**: User-friendly tooling for deploying complex ErgoScript contracts explicitly deferred to post-mainnet; no timeline given.
- **Exchange Listing Pipeline**: Only Tidex confirmed at launch; negotiations with other exchanges (including Binance) ongoing but no commitments or timelines disclosed.
- **Faster Block Times & Height-Dependent Contracts**: Community interest in reduced block times noted, with open concern about impact on dApps using block height rather than timestamps, and on emission contracts. A testnet parameter-tweaking group discussed but not formally organised.
- **Discord / Permanent Forum**: Multiple requests for a Discord server and a permanent community archive; acknowledged but not committed to.
- **Storage Rent Dormancy Threshold**: Exact initial dormancy period before fees apply (~4 years referenced) and initial fee amounts not confirmed in public chat.
- **Governance Process**: Mechanism for miners to formally propose and vote on protocol parameter changes (block size, storage rent, difficulty) described loosely but not specified.
