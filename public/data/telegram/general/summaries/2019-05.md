# Ergo General Chat — Monthly Summary: 2019-05

---

## Executive Summary

May 2019 was dominated by pre-launch preparation activity as the Ergo mainnet approached its June 20, 2019 target date. Core discussions centred on testnet mining configuration and troubleshooting, the EFYT-to-ERG swap mechanism, and early ErgoScript developer onboarding. Community growth brought a surge of new users with procedural questions about wallet setup, GPU mining, and exchange listings.

---

## Key Topics This Month

### Mainnet Launch Timeline & Postponement
The mainnet was originally targeting May 26, 2019 but was pushed back to **June 20, 2019** due to a security audit progressing more slowly than expected and testing delays. The postponement was announced by @Dmitry (msg#6989, 2019-05-08). Throughout the month, community members were advised that the current tradeable token (EFYT) lived on the Waves platform, entirely separate from the upcoming independent Ergo blockchain — a distinction that required repeated clarification for new arrivals. Charles Hoskinson's reference to Ergo's "Proof of NO Premine" in a YouTube video was noted as a positive external endorsement during this period.

### EFYT Token, Treasury Distribution & Swap Mechanism
The EFYT (Ergo First Year Token) distribution and post-mainnet swap process was a recurring topic. 100K EFYT were distributed proportionally to Waves Community Token (WCT) holders, with treasury funds (~815K tokens) to be distributed over approximately seven months post-mainnet. Treasury tokens not distributed would not be swapped to mainnet ERG. At mainnet launch, the process would be: swap EFYT → ERG on Waves, then withdraw to a personal Ergo wallet. A one-day (~720 block) delay between block minting and spendable rewards was confirmed. A community voting mechanism for treasury fund allocation was referenced as under development but with no design details available.

### Testnet Mining: Configuration, Bugs & Hardware
A significant portion of community support traffic concerned testnet mining setup. Key issues included:
- The **"Generated and Received Public Keys do not match"** error in Autolykos miner, traced to incorrect seed handling — from node version 2.0.4 onwards, the seed must not appear in the node config but instead be initialised via the `/wallet/init` Swagger route, with the returned mnemonic placed in the miner config.
- **Wallet persistence bug**: early builds did not persist wallet data to disk, causing balances to disappear after restart. Restoring from seed only recovered keys, requiring a full blockchain rescan.
- **OpenSSL compatibility**: miners failed to compile on Ubuntu 18.04 due to OpenSSL 1.0.x vs 1.1.x conflicts; a workaround using `libssl1.0-dev` packages was documented. OpenSSL 1.1 support was flagged as a priority.
- **Multi-GPU mining**: the miner auto-detects all GPUs; the `keepPrehash` flag optimises VRAM usage but auto-disables when fewer than 8GB VRAM are available per device.
- **720-block maturity delay**: multiple users interpreted zero balance as a bug; the delay between block minting and reward availability was the consistent answer.
- **AMD GPU support**: confirmed as not yet available in this period.

### Pool-Resistance, Mining Fairness & Autolykos Design
The Autolykos PoW algorithm's pool-resistance property was explained multiple times. The design treats pool operator centralisation as a greater risk than individual large miners, and the algorithm deliberately prevents pool formation. Dmitry offered a rough model: with ~40K GPUs mining, an individual miner could expect approximately one block per week, comparable to what a pool payout would yield — removing the incentive for pools without penalising smaller miners. Soft-forking and miner voting allow protocol parameter changes (block size, difficulty) without requiring developer intervention.

### ErgoScript & Early Developer Interest
Several developers expressed interest in writing smart contracts on testnet. The `/wallet/p2s_address` and `/wallet/p2sh_address` API endpoints were identified as the routes for deploying custom ErgoScript. Learning resources consolidated and shared included `ErgoScript.pdf`, `AdvancedErgoScriptTutorial.pdf`, the testnet playground, Scala examples in the sigmastate-interpreter repository, and kushti's blog post on smart contracts for the people. A community member with React/Redux/CSS expertise offered to contribute to UI development and was directed to contact @DMeshkov.

### Wallet API & Authentication Setup
Repeated troubleshooting occurred around Swagger UI access. The resolution: set `scorex.restApi.apiKeyHash` in the node config (computed via the `/utils/hashBlake2b` endpoint or any external Blake2b calculator), then pass the raw key as the `api_key` **header** — not a body parameter. This was a common friction point for new node operators.

## Technical Q&A Worth Preserving

- **Q** (@Unknown, 2019-05-02): How to resolve the "Generated and Received Public Keys do not match" error in Autolykos miner?
  **A** (@Dmitry, msg#6947): From node version 2.0.4, do not put the seed in the node config. Instead, initialise the wallet via the `/wallet/init` Swagger route and place the returned mnemonic in the miner config. Leave `mnemonicPass` empty — the miner does not yet support it.

- **Q** (@shn, msg#7384, 2019-05-27): Cannot access wallet API via Swagger UI; receiving "supplied authentication is not authorized."
  **A** (@Ilya Oskin, msg#7385–7387): Set `scorex.restApi.apiKeyHash = yourKeyHash` in the node config file. Pass the raw key as the `api_key` **header** in requests — not as a body parameter.

- **Q** (@shn, msg#7388, 2019-05-27): How to compute the Blake2b hash for API authentication?
  **A** (@Dmitry, msg#7391): Use any external Blake2b hash calculator or the node's built-in `/utils/hashBlake2b` API route. Public nodes exist but are not recommended for mainnet use.

- **Q** (@Rejinderi, msg#7099, 2019-05-14): Miner compilation failing on Ubuntu 18.04.
  **A** (@Andrey Andreev, msg#7105): Run `sudo apt install libcurl3 libcurl-openssl1.0-dev libssl1.0-dev && make` to resolve OpenSSL version conflicts.

- **Q** (@Rejinderi, msg#7430, 2019-05-28): Can `keepPrehash=true` be set with 2 GPUs at 6GB VRAM each?
  **A** (@Dmitry, msg#7431–7432): You can always set it to true; the miner will auto-disable it if insufficient VRAM is detected. Current implementation requires more than 8GB, so it will be disabled on your setup.

- **Q** (@Александр Кравцов, msg#7152, 2019-05-15): Mined ERG visible in wallet, but balance disappears after restart. Restoring from seed gives identical addresses — is that correct?
  **A** (@Ilya Oskin, msg#7154–7156): The wallet is not persistent in the current build. Restoring from seed only recovers keys; a full blockchain rescan is required to recover the displayed balance.

- **Q** (@Unknown, msg#7050, 2019-05-12): GPU miner found solutions but wallet shows zero balance.
  **A** (@Dmitry, msg#7052): There is a 720-block (~1 day) maturity delay between block minting and reward availability. Wait for the reward to mature before expecting a balance update.

- **Q** (@Unknown, msg#7074, 2019-05-13): Can transactions with custom ErgoScript be posted on testnet?
  **A** (@Kirill, msg#7076): Yes. Use the `/wallet/p2s_address` API method to create an address from a script, then submit a transaction to that address.

- **Q** (@shn, msg#7395, 2019-05-27): How do small miners survive without pools? Won't the network centralise to large miners?
  **A** (@Ilya Oskin, msg#7396): Autolykos is pool-resistant, preventing pool formation entirely. (@Unknown, msg#7404): The algorithm reduces hardware-specific advantages; miner voting and soft-forking allow parameter changes without trusted parties, further limiting centralisation pressure.

- **Q** (@toni, msg#7212, 2019-05-17): How many transactions per second can Ergo handle?
  **A** (@Unknown, msg#7213): The core design raises TPS without compromising blockchain security assumptions. Block size and other parameters are adjustable by miners; payment network solutions handle additional throughput beyond on-chain capacity.
